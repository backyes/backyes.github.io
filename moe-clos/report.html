1. 执行摘要

Mixture-of-Experts (MoE) 架构已成为大语言模型的主流选择（DeepSeek-V3、Mixtral 8×22B、Qwen3-MoE 等）。MoE 的核心特征是稀疏激活——每个 token 仅激活 top-k 个 expert（如 DeepSeek-V3 的 top-8/256）。这一稀疏性在分布式推理中引发了一个根本矛盾：

"通信是稀疏的，但 CLOS 网络是全互联的 —— 过度配置导致浪费"

本调研的系统性发现：传统 CLOS 网络的全二分带宽（full bisection bandwidth）对 MoE 推理是过度配置。通过拓扑简化、可重构光网络、软件通信优化三个层面，可在保持性能的前提下节省 20-77% 网络成本和 23-75% 网络功耗。

20-77%网络成本节省
0-5.6%性能损失
1.2-2.3×成本效率提升

这一定量结论来自多个独立研究团队（MIT CSAIL、UC Berkeley、HKUST、Microsoft、华为）在 2024-2026 年间发表的 8+ 篇顶级论文，已形成显著的学术共识。

https://arxiv.org/abs/2605.00254 | https://arxiv.org/abs/2604.14690 https://arxiv.org/abs/2501.03905 | https://arxiv.org/abs/2602.12521

2. 问题：MoE 为什么要全互联？

2.1 MoE 的推理通信模式

在 Expert Parallelism (EP) 下，MoE 推理层涉及两次 All-to-All 通信：

┌─ Input Tokens ──────────────────────────────────────┐ │ [t0, t1, t2, ..., tn] 分配到 GPU0..N │ ├────────────────────────────────────────────────────┤ │ Step 1: Token Dispatch (All-to-All Scatter) │ │ Router 给每个 token 分配 top-k experts │ │ → tokens 被发往 hosting experts 的 GPU │ │ │ │ Step 2: Expert Computation (Sparse FFN) │ │ 每个 GPU 只计算被分配到的 tokens │ │ │ │ Step 3: Token Gather (All-to-All Gather) │ │ → 输出 tokens 返回原始 GPU │ └────────────────────────────────────────────────────┘

即每层 MoE 需要 两次 A2A——一次 dispatch、一次 gather。这是 MoE 推理的核心通信瓶颈。

https://arxiv.org/abs/2605.00254 (§2.1)
"communication consumes up to 59% of runtime in a 96-GPU MoE serving setup"

2.2 传统 CLOS 网络的假设

CLOS（fat-tree）网络设计的基本假设是：任意节点可能与任意其他节点以全带宽通信。因此 CLOS 提供：

  • 全二分带宽：spine 层提供足够的 uplink 使任意叶间通信无阻塞
  • 对称拓扑：所有路径等长、等带宽
  • 无状态转发：通过 ECMP/flow-level 负载均衡，不感知流量模式

但 MoE 的 A2A 流量特征恰好与这些假设相悖。

3. CLOS 网络为什么对 MoE 低效？

3.1 稀疏性（Sparsity）

每 token 只激活 top-k experts（如 top-8/256 = 3.1%）。在 Expert Parallelism 下，每个 GPU 只需与少数持有对应 experts 的 GPU 通信，而非全部。CLOS 却为全互联准备好所有带宽。

https://arxiv.org/abs/2604.14690
"demonstrate how the symmetric, distributed switching of 3D-Torus and Rail-Optimized architecture align with sparse or imbalanced LLM training traffic, and show that All-to-All traffic from Mixture-of-Experts models severely degrades their port utilization and routing efficiency"

3.2 不均衡性（Imbalance）

Expert load imbalance 是 MoE 的固有问题——不同 experts 接收不同数量的 token 分配。这导致 A2A 通信中部分 GPU 发送/接收远多于其他 GPU，造成 CLOS 网络的 incast 拥塞和局部热点。

https://ieeexplore.ieee.org/abstract/document/11450478/ (RailS, IEEE ToN 2026)
"sparse MoE traffic is difficult to balance effectively using [traditional] rigid mapping"
"the sparse, dynamic, and imbalanced communication of MoE"

3.3 短流特性（Short Flows）

MoE 的 A2A 消息通常较小（per-token 数据量受限），形成大量短流（short flows）。CLOS 的 flow-level 负载均衡（如 ECMP）对短流的哈希不均匀导致路径冲突和乱序，加剧拥塞。

3.4 距离无关性（Distance Agnostic）

传统 CLOS 负载均衡不考虑通信的拓扑亲和性。MoE 通信实际上具有强 locality——token 倾向于激活相似的 experts（如特定 topic 的 tokens 常去同一组 experts）。MixNet 的工作发现，利用这一 locality 可以大幅降低全局 reconfiguration 的需求。

https://arxiv.org/abs/2501.03905 (MixNet)
"MoE dynamic communication pattern has strong locality, alleviating the requirement of global reconfiguration"

4. 解决方案全景：四层 "Sparse CLOS"

业界和学术界从四个层面应对 CLOS 对 MoE 的低效问题，每层对应不同的抽象级别和收益来源：

🔷 Layer 1: 拓扑简化（Topology Simplification）

核心思想：MoE 不需要 spine 层——直接去除或简化 CLOS 的 spine 交换机。

Rail-only / Rail-optimized

MIT CSAIL 的 Manya Ghobadi 团队提出了 Rail-only 网络，直接去掉 spine 层，仅在"rail"（按 NIC 编号连接的 GPU 行）内保持全互联。这个设计在 LLM 训练上取得了与 full CLOS 相同的训练性能。

Rail-only 收益（来源：IEEE HotI 2024）
• 网络成本降低：38-77%
• 网络功耗降低：37-75%
• MoE A2A 性能损失：仅 4.1-5.6%（通过 forwarding 补偿）
• 训练性能：与 full CLOS 相同
https://ieeexplore.ieee.org/abstract/document/10664412/
"LLM training generates sparse communication patterns in the network and, therefore, does not require any-to-any full-bisection network to complete efficiently"

Switchless Topologies（无交换机拓扑）

UC Berkeley（Scott Shenker、Sylvia Ratnasamy、Yakun Sophia Shao 等）的论文系统性对比了 scale-up（CLOS）、scale-out、3D torus、3D full-mesh 四种拓扑在 MoE 推理场景下的成本效率。

Switchless 拓扑收益（来源：arXiv 2605.00254, Apr 2026）
• 3D full-mesh 是 Pareto-optimal 拓扑
• 成本效率比 scale-up 提升 20.6-56.2%
• 降低 scale-up 链路带宽（从 450→150 GB/s）可使每成本吞吐提升 27%
• 预测在 Blackwell 和 Rubin 时代优势仍将持续
https://arxiv.org/abs/2605.00254
"lower-cost switchless topologies are more cost-effective than the scale-up topology across all serving scenarios explored"

RailX（华为）

华为提出 Rail-Ring 架构，用 ring 拓扑连接 rail 之间，替代 CLOS spine。

https://arxiv.org/abs/2507.18889

🔶 Layer 2: 可重构光网络（Reconfigurable Optical Fabrics）

核心思想：用 OCS（Optical Circuit Switching）动态重构网络拓扑，在 MoE 通信阶段切换到最优配置。

MixNet（HKUST）

首个在训练中动态重构网络拓扑的系统。发现 MoE 通信的 locality 后，用 OCS 在电互联之上构建可重构的光域。

MixNet 收益（来源：SIGCOMM 2025）
• 100 Gbps 链路：成本效率 1.2-1.5×
• 400 Gbps 链路：成本效率 1.9-2.3×
• 32 A100 GPU 实际原型验证
• 性能与非阻塞 fat-tree 相当
https://arxiv.org/abs/2501.03905

Opus（Microsoft）

用 OCS 替代电交换机实现 rail fabric。"Parallelism-driven rail reconfiguration"——利用并行维度的非重叠通信阶段时间复用物理端口。

Opus 收益（来源：arXiv 2602.12521, 2026）
• 网络功耗降低：23×
• 网络成本降低：4×
• 测试于 Perlmutter 超算和 2048 GPU 模拟
• 训练开销轻微
https://arxiv.org/abs/2602.12521

🟣 Layer 3: 软件通信优化（Software Communication Optimization）

核心思想：在不改拓扑的前提下，通过智能调度降低 A2A 通信量，等效于使网络变得"更稀疏"。

方案方法收益来源
UBEP 重写 A2A 原语：消除 BSP 序列化+距离感知调度+同步优化 A2A 延迟降低 52.4%，TPOT 降低 11.1% arXiv 2607.06202
Speculative MoE 预测 token 路由路径，预调度 token/expert 以减少跨 GPU 通信 在 fast/slow interconnect 上均显著加速 arXiv 2503.04398v1
Semantic Parallelism 模型-数据协同调度，collocate 高激活概率的 expert-token 对 大幅降低 A2A 通信量 arXiv 2503.04398
MegaScale-Infer 解耦 attention/FFN，ping-pong pipeline 隐藏通信开销 per-GPU 吞吐提升 1.9× arXiv 2504.02263
MegaScale-MoE EP+TP 自定义通信 + 生产级大规模 MoE 训练 大幅降低通信瓶颈 SIGCOMM 2026
PROBE 实时 predictive prefetching + continuous lookahead pipelining 计算-通信 co-balancing arXiv 2602.00509

UBEP 的深层洞察

UBEP 论文揭示了一个关键问题：即使在高带宽 superpods（NVL72/576, CloudMatrix384）上，MoE 通信仍然受限于三个根本瓶颈，而非原始带宽。

"three fundamental bottlenecks: (1) Strict execution serialization imposed by coarse-grained BSP; (2) Prohibitive synchronization overhead; (3) Severe load imbalance resulting from distance-agnostic scheduling"

这意味着，即使 CLOS 提供了无限带宽，MoE 的 A2A 效率仍因软件/调度问题而受限。这也是为什么拓扑简化方案几乎不损失性能——真正的瓶颈不在带宽，而在同步和负载均衡。

🟢 Layer 4: 分析框架（Analytical Frameworks）

Switching Efficiency Framework

提出 Switching Efficiency (η) 指标，量化计算有效数据吞吐量 per 单位交换容量，分解为 Data Efficiency、Routing Efficiency、Port Utilization 三个因子。

"MoE's All-to-All traffic severely degrades their port utilization and routing efficiency"

该框架为设计 future-generation AI 数据中心网络提供了分析工具。

https://arxiv.org/abs/2604.14690

6. 收益原理分析

为什么 Sparse CLOS 方案（拓扑简化/可重构/软件优化）能在几乎不损失性能的前提下大幅降本？

6.1 通信-计算重叠（Computation-Communication Overlap）

UC Berkeley 论文发现，Dual Batch Overlap (DBO) 可以几乎消除暴露的通信时间。当 batch size 足够大时，MoE 推理的计算时间（attention + FFN）可以完全覆盖 A2A 通信时间。这表明感知到的通信瓶颈更多是调度问题而非原始带宽问题。

"DBO nearly equalizes high- and low-bandwidth performance ... as long as communication time stays below compute time"

6.2 通信的 locality 禀赋

MixNet 的实际测量发现，MoE 的 token-expert 分配具有强 locality。特定输入的 tokens 倾向于激活相同子集的 experts。这意味着：

  • 不需要全局全互联——局部区域内的通信占主导
  • 拓扑可以针对 locality 优化（如 rail 内全互联、rail 间稀疏连接）
  • OCS 局部 reconfiguration 就足够覆盖大部分流量

6.3 "Memory Wall" 转移瓶颈

MegaScale-Infer 指出，MoE 推理时 FFN 层从计算密集型变为内存密集型（受限于 GPU HBM 带宽而非 compute）。这意味着 GPU 实际计算时间并不随网络带宽提升而缩短——网络带宽的边际收益递减。

"sparsely activated architecture shifts FFNs from being compute-intensive to memory-intensive during inference"

6.4 BSP 同步开销

UBEP 揭示了 BSP 同步开销在高带宽网络中反而更突出——随着互联带宽增加，同步开销增速超过带宽收益，导致"带宽浪费在等待同步"。

6.5 综合：为什么成本曲线那么陡？

综合上述因素，Sparse CLOS 方案的收益原理可以拆解为：

┌─────────────────────────┐ GPU Compute Cost │ 固定（不受网络影响） │ └──────┬──────────────────┘ │ ┌──────────────────────▼───────────────────────┐ │ 传统 CLOS 成本结构 │ │ GPU: $$$$$ | CLOS 网络: $$$$ (~40% TCO) │ └──────────────────────┬───────────────────────┘ │ ┌──────────────────────▼───────────────────────┐ │ Sparse CLOS 成本结构 │ │ GPU: $$$$$ | 简化网络: $ (~10% TCO) │ │ 节省的 30% TCO → 再买一批 GPU → 聚合吞吐↑ │ └──────────────────────────────────────────────┘

这里的核心洞察是：网络成本节省不是线性收益——节省下来的资金可以购买更多 GPU，从而实现聚合吞吐量的超线性增长。这在大规模部署中特别显著。

https://arxiv.org/abs/2605.00254 (Figure 1)

7. 专家视角洞察

7.1 站在模型设计者的角度

MoE 模型设计正向更高的 sparse ratio（更多 experts，更少 active per token）演进。DeepSeek-V3 的 top-8/256 意味着仅 3.1% 的 experts 被激活，未来趋势是 1/1024 甚至更低。这带来：

  • 更高的 expert 稀疏度 → A2A 通信量减少（更少的 token 需要路由）
  • 更多的 experts → 每 GPU 需要 hosting 更少的 experts，EP degree 更高
  • 更细粒度的 routing → 拓扑感知调度空间更大

推演：模型架构的稀疏度提升会持续降低对网络全互联的需求，使得 Sparse CLOS 方案的收益在未来模型上更大。

7.2 站在网络架构师的角度

AI 数据中心网络设计的范式正在从"提供最大可能带宽"转向"提供恰好足够的带宽并匹配流量模式"。具体趋势：

  • Rail-optimized 架构成为主流：NVIDIA NVL72/576 实际上已经是 rail 内的 full-mesh（NVSwitch）+ rail 间的稀疏连接
  • OCS 进入生产：Google 的 Jupiter OCS、Microsoft 的 Opus、HKUST 的 MixNet 都指向光交换成为 MoE 工作负载的关键使能技术
  • Multi-plane 设计：不同通信模式（TP all-reduce、EP A2A、DP all-reduce）跑在不同 plane 上，各自优化

7.3 成本效率的临界点

当前行业在 MoE 推理上的网络投资已接近边际收益为零的区域。UC Berkeley 的研究显示 450 GB/s 的 link BW（H100 NVLink）已经不是最优选择，150-300 GB/s 更经济。而 Opus 的 23× 功耗降低表明光互连在能效上有数量级优势。

8. 未来趋势

~2026Rail-only/rail-optimized 广泛部署
~2027OCS 可重构 fabric 进入生产
~2028全光互联 / photonic rail

8.1 短期（2026-2027）

  • Rail-only 拓扑：从 MIT 论文走向实际部署。NVIDIA NVL576 本质上已经是 rail-only 的变体
  • UBEP 类软件栈：在现有 CLOS 网络上即可部署，无硬件改造成本，可立即获得 11-52% 性能提升
  • DBO + SD 普及：计算-通信重叠技术成为标准推理优化

8.2 中期（2027-2028）

  • OCS reconfigurable fabric：类似 MixNet/Opus 的方案进入生产。OCS 的 reconfiguration latency 降低到 μs 级后将成为 MoE 网络的标配
  • 拓扑-模型 co-design：模型架构设计时即考虑通信拓扑约束

8.3 长期（2028+）

  • Photonic switching：Opus 的 23× 功耗降低预示了全光互联的终极形态
  • Switchless 大规模部署：3D torus/full-mesh 在更大规模（10K+ XPU）上的扩展
