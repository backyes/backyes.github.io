一、核心结论：一句话回答

不是。Google Rapid Storage（含 Rapid Bucket 和 Rapid Cache）本质上是为 训练数据加载、checkpoint 写入/恢复、推理模型权重冷启动 设计的高性能对象存储层，而非为推理时 KV Cache 的存取而生。

Google 为 KV Cache 外部化设计的存储方案是 Managed Lustre（并行文件系统，支持 RDMA）和未公开细节的 "Dedicated KV Cache scalable storage subsystem"。

但这个"不是"背后，隐藏着一个更有趣的故事：Google 的整个 AI Hypercomputer 存储栈确实是为 KV Cache 经济学而重构的——只是不同层级承担不同的角色。Rapid Storage 解决的是"数据如何快速到达加速器"，而 KV Cache 的外部化需要的是"推理状态如何在节点间共享和快速恢复"。

二、Google Rapid Storage 是什么？

Google Cloud Storage Rapid 是 Google 在 2026 年 4 月 Next '26 大会上发布的高性能对象存储产品线，基于 Google 内部使用的 Colossus 分布式存储系统（与 Gemini 和 YouTube 共享同一套基础设施）[1]。

它包含两个核心产品：

关键设计特征：

4.3 TPU 8i 的片上 KV Cache 设计 [4]

Google 为 KV Cache 优化的真正"主力"不是外部存储，而是 TPU 8i 芯片本身：

Google TPU 8i 技术深潜引用："With 3x more on-chip SRAM over the previous generation, TPU 8i can host a larger KV Cache entirely on silicon, significantly reducing the idle time of the cores during long-context decoding." [4]

TPU 8i 的 Pod 级 SRAM 总量：==384 MB × 1,152 chips = 432 GB==。这是 Google 为 KV Cache 设计的"第一道防线"——让 KV Cache 尽可能驻留在最快的存储层级上。

4.4 "Dedicated KV Cache scalable storage subsystem" [5]

在 Next '26 的 AI infrastructure 博客中，Google 还宣布了一个未公开细节的组件：

"Dedicated KV Cache scalable storage subsystem" [5]

这个组件在公告中被单独列出，与 TPU 8t/8i、Virgo Network、Managed Lustre 并列。截至调研时（2026-07-22），Google 尚未发布该子系统的技术细节。它很可能是专门用于 KV Cache 外部化的独立存储层，但具体架构未知。

五、为什么会产生"为 KV Cache 而生"的误解？

这个误解有其合理性——以下是几个导致混淆的因素：

5.1 同时发布，语境混同

Rapid Storage 和 TPU 8i 的 SRAM 扩容在同一场 Next '26 大会上发布，且都被包装在"AI Hypercomputer for the agentic era"叙事中。TPU 8i 的 3x SRAM 确实是为 KV Cache 设计的，但这不意味着 Rapid Storage 也是。

5.2 "Agentic Era"叙事中的存储重定义

Google 官方博文明确指出："During AI inference, storage is the access layer that makes it responsive, acting as the source for the context that AI agents need to be effective." [1]

这里的"context"被一些读者理解为 KV Cache，但实际上指的是 Agent 运行时的完整上下文数据（包括历史对话、工具调用结果、系统提示等），其存储方式远不止 KV Cache 一种。

5.3 命名暗示

"Rapid"（快速）这个名字暗示低延迟，容易让人联想到 KV Cache 外部化时对延迟的苛刻要求。但实际上，Rapid 指的是对象存储层级的吞吐量和 IOPS 提升，而非针对 KV Cache 的细粒度随机访问优化。

5.4 行业趋势的投射

2026 年，KV Cache 外部化是整个行业的热点话题（LMCache、Mooncake、NVIDIA 的 KV Cache offload 等）。在这个背景下，任何新发布的"AI 存储"产品都容易被默认解读为"为 KV Cache 而生"。

六、Rapid Storage 的真实定位

6.1 训练阶段：数据供给线

Rapid Bucket 的核心价值是确保昂贵的 TPU/GPU 不会因为等待数据而闲置：

  • ==50%== 减少 GPU blocked time（对比传统区域存储）
  • ==2.5x== 加速数据加载
  • Checkpoint 写入 ==3.2x== 加速，恢复 ==5x== 加速

这些指标指向的是训练吞吐量和加速器利用率，而非推理延迟。

6.2 推理阶段：模型权重分发

Rapid Cache 的核心价值是加速推理服务的冷启动：

  • 将模型权重预热到推理节点同区域的 SSD 缓存中
  • ==2.5 TB/s== 聚合读取吞吐，消除推理冷启动延迟
  • 与新发布的 Run:AI Model Streamer for vLLM 集成

这解决的是"模型如何快速加载到 GPU 上"的问题，而非"KV Cache 如何在推理过程中存取"的问题。

6.3 为什么 Rapid Storage 不适合 KV Cache？

架构层面的不匹配：

  • 对象存储语义：KV Cache 需要细粒度的随机读写（按 token 索引存取 KV block），而对象存储是大文件顺序读写优化
  • 一致性模型：KV Cache 需要强一致性（推理状态不能丢失或过期），对象存储的最终一致性模型不适合
  • 访问粒度：KV Cache 外部化需要 ~KB 级别的随机读取，对象存储的 ~MB 级别块访问粒度过粗
  • 延迟要求：推理时 KV Cache 的读取延迟直接影响 TTFT，需要亚毫秒级响应，对象存储难以满足

相比之下，Managed Lustre 作为 POSIX 并行文件系统，天然支持：

  • 细粒度文件级随机访问
  • 跨节点的共享访问（多 worker 可同时读取同一 KV Cache）
  • RDMA 直达（绕过 CPU，TPU Direct 直连）
  • 亚毫秒级延迟

八、分析师与市场观点

8.1 IO Fund 分析 [6]

Beth Kindig（IO Fund 首席技术分析师）的分析聚焦于 TPU 8i 的推理优势，特别是：

  • 384 MB SRAM 对 KV Cache 的加速作用
  • Boardfly 拓扑的 50% 延迟降低
  • 与 NVIDIA NVL72 的内存一致性对比（331.8 TB vs 20.7 TB）

注意：她的分析中并未将 Rapid Storage 与 KV Cache 直接关联。

8.2 Blocks & Files 分析 [2]

Chris Mellor 的报道明确区分了 Rapid Storage 和 Managed Lustre：

"Managed Lustre can be used as a shared KV-cache for AI inference, improving performance and economics. It reduced the mean time to first token by more than 40 percent compared to using KV Cache in host memory alone."

Managed Lustre 被定位为共享 KV Cache 方案，而非 Rapid Storage。

8.3 Stratechery 观点 [7]

Ben Thompson 在 Google Cloud CEO Thomas Kurian 的访谈分析中提到 Rapid Storage 可提供 15 TB/s 吞吐，但同样未将其直接与 KV Cache 关联，而是作为 AI 数据管道的一部分。

8.4 存储行业分析 [8]

StorageNews 的分析：

"Managed Lustre remains preferable for inference workloads requiring shared KV-cache states, yet Smart Storage dominates dataset selection phases."

明确将 Managed Lustre 定位为共享 KV Cache 的首选，而 Rapid Storage/Smart Storage 用于数据准备阶段。

设计逻辑推演

从 Google 的 TPU 设计哲学可以推导出其存储层级的设计逻辑：

Google 的 KV Cache 策略是"尽可能不上外部存储"：

  1. L0 优先：384 MB 片上 SRAM → 让 KV Cache 驻留在最快层级
  2. L1 扩展：288 GB HBM → 单芯片内容纳完整 KV Cache
  3. L2 共享：Managed Lustre → 跨节点共享/恢复 KV Cache
  4. Rapid Storage 的角色：确保训练数据/模型权重/Checkpoint 不成为加速器瓶颈

Rapid Storage 解决的是"如何让 TPU/GPU 持续满负荷运转"的问题——这是一个供给端的问题。KV Cache 外部化解决的是"如何让推理状态在节点间高效流转"的问题——这是一个状态管理的问题。两者在 AI Infra 中都是关键组件，但解决的是不同层面的问题。

一句话总结

Google Rapid Storage 是 AI 数据管道的"高速公路"，让数据快速到达加速器；而 KV Cache 外部化需要的是"保险箱"——Managed Lustre 提供的共享、持久、细粒度状态存储。两者互补，但不可互相替代。

对模型设计者的启示

从模型架构设计的角度看 Google 的存储层级设计：

  • 如果你设计的是 长上下文模型，TPU 8i 的 384 MB SRAM 意味着你可以将模型的 KV Cache 完全驻留在片上，这是比 GPU 方案（HBM 有限，需要 offload）更具优势的架构
  • 如果你设计的是 Agent 系统，Managed Lustre 的 KV Cache 共享能力意味着多 Agent 可以共享上下文状态，减少重复计算
  • Rapid Storage 对你模型设计的直接影响最小——它更多影响的是训练效率和推理部署的冷启动时间
