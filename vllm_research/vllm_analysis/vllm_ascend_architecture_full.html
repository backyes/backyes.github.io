vLLM-Ascend 全模块源码架构分析

一份覆盖 442 文件 / 127k 行 的全模块深度分析。围绕 5 个核心问题：为什么要 vllm_ascend、设计要解决什么问题、工程关键挑战、软件架构设计、与 upstream vLLM 的关系。由 6 个 agent 并行切片深读后综合。

0. Executive Summary — 5 条 Headline Facts

Headline 1 — vllm_ascend 不是"后端适配"，而是 6 层 overlay：Platform 子类 → Config injection → Monkey-patch → Worker 子类 → Device Op Adaptor → Custom Ops/Compile Backend。核心哲学：extension not fork。

Headline 2 — NPU 与 GPU 在 6 个维度不可调和：全局 init-once 约束、无 device capability 概念、4 种芯片变体（A2/A3/310P/A5）算子语义不同、ACL Graph vs CUDA Graph、HCCL vs NCCL、Sleep 模式 + CPU 绑定。

Headline 3 — EPLB 是 Ascend 对 vLLM 唯一净新增子系统：五层流水线（updator → worker 子进程 → policy → D2D loader → adaptor），FlashLB 用 μ+z·σ 风险调整 + depth=4 树搜索 + LPT + Hungarian 最小化迁移。零 upstream 修改。

Headline 4 — KV 传输已从 cache 进化为 memory tier：L1 HBM P2P（Mooncake RDMA）→ L2 跨节点 Pool（Ascend Store, 三后端）→ L3 NVMe SSD offload。对应 OS cache→DRAM→swapfile 同构。

Headline 5 — 5 种 Attention 变体 = 模型结构 × NPU 硬件特性 的笛卡尔积：FIA（标准 paged）、MLA（latent absorption）、DSA（indexer+compressor+SWA 三级缓存）、SFA（block-sparse + 独立 indexer cache）、FA3（第三方旁路）。update_graph_params() 是 NPU full-graph replay 的硬需求。

1. 为什么要 vllm_ascend — NPU 与 GPU 的 6 维不可调和差异

§1硬件差异根因source: agent1 §1 + agent2 §2 + agent5 §1

1.1 全局 init-once 约束 vs 可重入初始化

GPU 的 cudaMemGetInfo 是可重入的；NPU 的 torch_npu 全局状态机只允许一次初始化。platform.py:275 的 get_device_total_memory 被迫 raise NotImplementedError——不是"值不同"，而是"调用时机有有毒副作用"。

1.2 Device capability 模型不存在

GPU 用 (major, minor) 标识计算能力（sm80 vs sm90），vLLM 据此分派 kernel。NPU 没有这层概念——platform.py:244 返回 None。芯片变体通过 AscendDeviceType 枚举区分（A2/A3/_310P/A5），capability 对上层透明。

1.3 四种芯片变体 = 四种算子语义空间

芯片MoE RoutingAttention内存模型权重布局
A2 (910B)npu_moe_init_routingFIA kernel标准 HBMND
A3 (910C)npu_moe_init_routingFIA (不同 CANN)标准 HBMND
310P (300I DUO)AllGather-basedKV-quantized sparseRC 共享内存NZ (fractal_z)
A5 (950, 2026)routing_v2 (dtype 透传)A5 FIA variant标准 HBMND

1.4 图执行：ACL Graph vs CUDA Graph

核心差异：CUDA Graph 是"录制-回放"模型（replay 零 host 介入）；ACL Graph 是"录制-参数更新-回放"模型。NPU attention 的 block_table/seq_len 无法 capture 时固化，必须通过 update_graph_params() 在 replay 前逐 layer 显式更新。

1.5 通信：HCCL vs NCCL

三个本质差异：进程组绑定（HCCL 需显式 torch.npu.device 上下文）、AllReduce stream 转换（pyhccl.py:186 封装）、UniqueId 分发（HCCL 需显式 broadcast）。

1.6 NPU 独有特性：Sleep 模式 + CPU 绑定

GPU 显存独占不需要腾挪。NPU 在多租户/RL 场景需要 sleep/wakeup 让出显存（sleep_mem_optimized.py：ACL graph + HCCL 双层释放/恢复）。HCCL 依赖 host-side PCIe 拓扑亲和，CPU 绑定对性能敏感（cpu_binding.py：3 种模式 + A5 UVB poll window）。

结论：vllm_ascend 不是"vLLM 的 NPU 后端"，而是一个完整 overlay——6 层适配覆盖 Platform 到 Custom Ops 的全栈。

2. vllm_ascend 要解决的 5 个核心设计问题

§2设计问题source: agent1-6 §2 综合

2.1 设计问题 1：Platform 接口的"承诺 vs 能力" mismatch

vLLM 的 VllmConfig 是 GPU 中心的配置空间。NPU 必须在 init 早期做一次配置翻译——_fix_incompatible_config（platform.py:1025-1243）系统性清除 10 个配置域、30+ GPU 专用参数（cascade_attn / cpu_kvcache / nvtx_tracing / nsight / partial_prefills / flash_attn_version 等）。check_and_update_config（360 行 14 步）是枢纽。

2.2 设计问题 2：为什么需要 5 种 Attention 变体

"模型结构 × NPU 硬件特性"的笛卡尔积。MLA 的 KV cache 存 latent [num_blocks, block_size, kv_lora_rank]；DSA 叠加 compressor + indexer + SWA 三级缓存；SFA 的 indexer 是独立虚拟 AttentionLayer。每种变体的 get_kv_cache_shape() 返回不同 shape，物理分配无法统一。

2.3 设计问题 3：NPU Roofline 决定 Scheduler 策略

NPU 算力/带宽比（~200-267 FLOPS/byte）显著高于 GPU（~156-295），使得 cost(recompute) < cost(swap)。RecomputeScheduler 把 preemption 从"swap to CPU"改为"drop-and-recompute"。ProfilingChunkScheduler 用二次公式 a·x²+(2aL+b)·x=T 解出等时 chunk 大小，消除 PP 气泡。

2.4 设计问题 4：KV Cache 从 cache 进化为 Memory Tier

PD 分离场景下多对多 KV 共享需求迫使引入 Pool 路径（Ascend Store）。三级 tier：L1 HBM P2P → L2 跨节点 Pool（Mooncake/Memcache/Yuanrong 三后端）→ L3 NVMe SSD offload。与 OS cache→DRAM→swapfile 同构。

2.5 设计问题 5：MoE Expert 负载不均衡

DeepSeek-V3/R1 的 routed expert 引入负载不均衡——hot expert 所在 NPU 成为瓶颈。EPLB 在推理阶段动态调整 expert 到 NPU 的物理映射：把 hot expert 复制到多个 NPU，让 gate 路由到空闲副本。这是 Ascend 对 vLLM 生态的唯一净新增子系统。

3. 工程实现的关键挑战

§3工程挑战source: agent1 §3 + agent3 §3 + agent4 §3 + agent6 §3 综合

3.1 310P 变体的 6659 行独立代码

5 个不可参数化的差异迫使独立子类：

  • 无 Triton（Atlas 300I DUO 不支持）→ CPU numpy slot_mapping + copy_to_gpu
  • RC 共享内存（host/device 共享物理内存）→ get_device_total_memory 语义不存在
  • 5D KV cache（head_dim 按 16 对齐切分）→ 独立 attention 实现
  • NZ 权重布局（fractal_z 格式 + transpose）→ 独立 fused_moe
  • CPU RNG sampler（无 Triton）→ 独立 sample 路径

is_310p() 在全库被调用 30+ 次，形成隐式分派网——可维护性瓶颈。增加第 5 种芯片变体必须在所有分派点添加新分支。

3.2 Patch 系统的工程约束

vLLM 在 engine-core 子进程加载 plugin，E2E test conftest hook 不运行。全局 patch 必须在 plugin 入口点内部自举——_GLOBAL_PATCH_APPLIED 一次性语义 + _ensure_global_patch() 在 4 个入口调用。28 + 25+ 子模块覆盖 vLLM 内部 GPU 硬编码路径。

33 个 patch 文件遵循"Less is more / 写移除计划 / 随时清理"三原则，通过 vllm_version_is() / is_310p() 条件导入实现版本隔离。

3.3 量化方案选择的工程逻辑

场景方案原因

W4A4∧EPLB 不兼容：W4A4 权重被 pack 成 int4pack 格式，无法运行时按 expert 粒度拆分（w4a4_flatquant.py:161）。

3.4 EPLB 的跨进程设计

FlashLB 的 numba JIT 编译 + 树搜索耗时可达数十 ms，不能阻塞 forward pipeline。EplbProcess 跑在独立子进程中，通过 Manager().dict() 做跨进程数据交换。三段式状态机：Heat Collection (600 步) → Algorithm Execution (50 步) → Weight Update (逐层 D2D)。

3.5 ACL Graph 的精细工程

  • GC 禁用：piecewise 模式下每层 capture 一个 graph，反复 gc 极慢
  • ExternalEvent 排序：graph stream 等待 host update_stream 上的 metadata 写入完成
  • weak_ref 内存优化：capture 后立即释放 output 和 workspaces 强引用
  • 双后端编译：npugraph_ex（首选）vs torch._inductor + fusion_pass_compile（fallback）

6.2 DSA index → paged attention 数据流

hidden_states
  ├─ _mla_prolog_multistream() → q, qr (dual-stream overlap)

9.4 双后端编译

AscendCompiler 通过 enable_npugraph_ex 开关分派：

  • npugraph_ex 路径（首选）：自带 pattern matcher，不经过 GraphFusionPassManager
  • fusion_pass_compile 路径（fallback）：torch._inductor + GraphFusionPassManager

BasePattern.register() 同时向两者注册，保证 pass 在两个后端都能匹配。

11.2 最脆弱的架构决策

310P 分派网：is_310p() 在 30+ 处被调用形成隐式分派。增加第 5 种芯片变体时必须在所有分派点添加新分支。建议：把分派逻辑集中到一个 DeviceCapability 对象中，用策略模式替代 if/else 网。

Patch 技术债：33 个 patch 文件中至少 10 个标注了 upstream PR 作为移除条件。vLLM upstream 迭代快，patch 容易 stale。建议：定期审计 patch 的 upstream 状态。

11.3 最有价值的架构决策

"Extension not fork"：6 层 overlay 在 vLLM 的 4 个扩展点（Platform 子类 / worker_cls / custom ops / additional_config）上做文章。EPLB 零 upstream 修改。这使得 vllm-ascend 能跟随 vLLM 主仓迭代而不产生合并冲突。

Scheduler 的 mixin 注入：SRF 只替换了 waiting queue 数据结构，schedule() 主循环零修改。这是"关注点分离"的最佳实践。

KV memory tier 三级架构：L1 P2P → L2 Pool → L3 SSD 的演化路径与 OS 内存层级同构，为未来 CXL/持久内存预留了扩展空间。

11.4 与 vLLM 统一报告的关联

  • 本报告回答"vLLM-Ascend 为什么需要 overlay"（硬件差异 + 工程约束）
