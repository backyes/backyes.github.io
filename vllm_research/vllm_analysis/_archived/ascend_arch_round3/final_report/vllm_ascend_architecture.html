1. Executive Summary

vllm-ascend is a community-maintained, out-of-tree hardware plugin that lets vLLM run on Huawei Ascend NPUs (Atlas A2/A3/A5 training, 800I A3/A5 inference, 310P edge). It implements vLLM's "Hardware Pluggable" RFC, but because that RFC is incomplete, the plugin goes far beyond a clean platform layer — it ships ~60 monkey-patches against vanilla vLLM, 4 novel schedulers, an RDMA-based PD disaggregation connector, a 4-strategy EPLB, and two innovative weight loaders.

Headline architectural facts:
  • 3-tier model support — only 3 models have custom ForCausalLM classes; most support lives in the Xlite graph runtime + attention backends + per-model patches.
  • Device fragmentation — 4 silicon generations (A2/A3/A5/310P) drive materially different operator trees; 310P is a full fork with no MLA/FIA support.
  • MLA absorption trick — instead of vanilla vLLM's two-stage MLA, Ascend absorbs W_UK into Q (bmm(q_nope, W_UK_T)) and scores in latent space via a single fused FIA-v2 call.
  • Pull-model KV transfer — Decode-side actively RDMA-reads Prefill KV via Mooncake, the opposite of vanilla vLLM's Nixl push model.
  • ProfilingChunkScheduler — the most novel contribution: offline profile → quadratic fit → closed-form per-request, per-position chunk sizing.

3. Core Scheduler Innovations high novelty

vanilla vLLM's scheduler is FCFS + fixed max_num_scheduled_tokens. vllm-ascend ships 4 scheduler-side innovations in vllm_ascend/core/, addressing hardware-specific bottlenecks.

3.2 Deep-dive: ProfilingChunkScheduler (the most novel) — click to expand

Subclasses Scheduler, overrides only schedule(). The upstream schedule() is forked nearly verbatim, deltas marked # >>> PROFILING CHUNK >>> / # <<< PROFILING CHUNK <<<.

Concept 1 — startup profile the hardware curve:

for i in range(num_samples+1):
    result = model_executor.collective_rpc("profile_prefill_latency", args=(chunk_size,))
predictor.fit(seq_lens, latencies)    # least-squares quadratic

Concept 2 — closed-form chunk from quadratic: for already-computed L, target T, solve f(L+x)−f(L)=ax²+(2aL+b)x = T:

A = a ;  B = 2*a*L + b ;  C = -T
x = (-B + sqrt(B²-4AC)) / (2A)
smoothed = base + smooth_factor*(x-base)
chunk = align(max(smoothed, min_chunk), max(page_size,64))

Why quadratic, not linear? because prefill attention cost grows super-linearly with L (just like GPU), so marginal cost grows — solving yields a smaller chunk as L grows: early prefill steps are large, later steps shrink. O(1) per request at schedule time.

Concept 3 — dual constraint loop: RUNNING/WAITING loops gain a time_budget counter initialized from predictor.target_latency; loop exits when time-budget hits 0 even if token-budget remains. Decode requests skip time accounting so they never starve.

Concept 4 — online refinement: record_batch_execution_time accumulates (Σ(C+H)·C, Σ(C+H), N, elapsed) points, refits a history-aware f(C,H)=a·C·(C+H)+b·C+c·H once ≥5 points, adapting to batch-interference the offline profile can't see.

Source: core/scheduler_profiling_chunk.py, core/profiling_chunk_predictor.py — report 01 §4.2

Composition: recompute_scheduler_enable and profiling_chunk_config.enabled are mutually exclusive top-level schedulers; ShortRequestFirst is a waiting-queue mixin recompute can carry as a passenger; over all of them, CompressAttentionManager patches KV accounting for compressed-MLA groups.

4.2 MLA on Ascend — weight absorption + fused FIA-v2 — expand

Key divergence from vanilla vLLM: instead of the explicit two-stage mla_decode (full K/V materialized in-kernel), Ascend (a) absorbs W_UK into the query, (b) caches only the joint latent (c_kv, k_pe), (c) scores in latent space.

# decode preproc  (mla_v1.py:_q_proj_and_k_up_proj)
q_nope, q_pe = self.q_proj(q_c).view(...).split([qk_nope_head_dim, qk_rope_head_dim])
ql_nope = torch.bmm(q_nope, self.W_UK_T)          # ← weight absorption
k_pe, k_nope = exec_kv_decode(kv_no_split, cos, sin, kv_cache, slots)
    # → torch_npu.npu_kv_rmsnorm_rope_cache(..., cache_mode="PA_NZ"|"PA")
# decode attention call
attn_output,_ = torch_npu.npu_fused_infer_attention_score_v2(
    q_nope, k_nope, k_nope,
    query_rope=q_pe, key_rope=k_pe,               # RoPE fused in-kernel
    input_layout="BSND_NBSD"|"BNSD_NBSD"|"TND_NTD",
    block_table=..., actual_seq_qlen=..., actual_seq_kvlen=...)
# v-up
x = torch_npu.npu_transpose_batchmatmul(x, self.W_UV, perm_y=(1,0,2))

MLAPO deep fusion (A2/A3): _C_ascend.mla_preprocess fuses Q/KV projections + LayerNorm + RoPE + cache-write into one CANN custom op; on KV-consumer (PD-decode) instances the original weights are freed after absorption. A5: npu_mla_prolog_{v2,v3} + npu_dynamic_{mx_}quant for FP8/MXFP decode.

Source: attention/mla_v1.py — report 03 §2.2-2.5

4.3 KV-compression = Hamming-hash sparse (NOT GLA/Mamba)

attention/kvcomp_attn/attention_utils.py implements Hamming-distance hash-top-k sparse attention, mutually exclusive with C8 quant. _C_ascend.npu_hamming_dist_top_k(hashq, hashk_cache, topk, ...) picks top-K blocks → context collapses from L to top_k * chunk_size, retaining sink+recent. This is H2O/StreamingLLM-style, implemented purely in custom ops.

4.5 310P — a forked operator tree

310P (Atlas 300I Duo, soc 200-205)predates and sits below the FIA-v2+MLAPO stack: it lacks npu_fused_infer_attention_score entirely, uses older _npu_flash_attention/_npu_paged_attention, requires 16-aligned NZ KV shapes, supports only W8A8 quant, and has no MLA/SFA. Hence a separate _310p/ tree for attention/ops/quant/sample/spec_decode/MoE. Only MHA/GQA + Flash-attn + linear-recurrent SSMs run on 310P.

6. KV-cache PD Disaggregation + EPLB Mooncake

Pull model ≠ Nixl push model. vllm-ascend's Mooncake path has the Decode node actively RDMA-read Prefill's KV via batch_transfer_sync_read, the opposite direction from vanilla vLLM's NixlConnector. Addresses computed in user-space from data_ptr + block_stride*block_id (2MB-aligned), so transfers are zero-copy and HCCL-bypassing.

6.2 MooncakeConnector lifecycle

  1. Handshake: read PD tp/dp/pp_size (P.tp ≥ D.tp), get global_te.get_transfer_engine(host, dev) in "P2PHANDSHAKE"/"ascend" mode; per-NPU unique port.
  2. register_kv_caches: emit {layer_names, spec_type, num_kv_heads} per group; per layer compute base_addr + block_stride + block_len; require 2MB-aligned base; register_buffer(ptrs, lengths); build MooncakeAgentMetadata (msgspec struct); start recv/send threads.
  3. Scheduler: mark do_remote_prefill requests, build MooncakeConnectorMetadata, delay-free finished blocks.
  4. Transfer: _transfer_kv_cache_all_groups loops group × layer × cache-tensor → engine.batch_transfer_sync_read(session_id, src_list, dst_list, length_list). Supports CP block mapping, Mamba state, GQA multi-pull reformat, KV NZ format.
  5. Completion: KVCacheTaskTracker → release + DONE_RECVING_MSG (zmq).

7. Patch System — ~60 Patches, 6 Categories adaptation layer

Why so many patches? vLLM's "Hardware Pluggable" RFC exposes clean hooks for only a subset of the stack. Everything else — scheduler loop, config validators, torch.distributed, torch.accelerator, the FusedMoE factory, rejection sampler, tool-call parsers — is still reached by direct imports of concrete symbols. vllm-ascend chose monkey-patch over fork; each patch carries a "Future Plan: Remove when upstream merges #XXXX".

7.1 Patch mechanics

# utils.py:537  — a deliberately import-for-side-effects dispatcher
def adapt_patch(is_global_patch: bool = False):
    if is_global_patch:
        from vllm_ascend.patch import platform   # ordered import list → each module rebinds
    else:
        from vllm_ascend.patch import worker     # ordered import list

Two phases: global (process-wide, at pre_register_and_update or via _ensure_global_patch()) and worker (per-NPU, top of NPUWorker.__init__). Spawn-mode re-patch problem: fresh interpreter in child → patches lost → fixed by wrapping EngineCoreProc.run_engine_core to re-run _apply_profiling_patches().

7.3 Top-3 most impactful patches — expand

patch_mla_prefill_backend (cleanest): registers a no-op AscendMLAPrefillBackend stub so upstream get_mla_prefill_backend() (which crashes on Ascend via a FlashAttn assert) completes — Ascend MLA impl owns the whole forward anyway.

patch_profiling_chunk (structurally hardest): patches EngineCore.__init__ + Scheduler.update_from_output + Scheduler.schedule + subprocess entry point; cross-process feedback loop for dynamic chunk sizing.

patch_fused_moe (highest leverage): rebinds FusedMoE in both fused_moe.layer and the package, injecting AscendMoERunner (AscendMoERunner310 on 310P), translating Ascend-only kwargs (tid2eid) and dropping CUDA-only ones (hash).

8.3 Xlite — C++ whole-model fusion runtime

xlite ≠ edge runtime. It is a graph-based, whole-model fusion execution mode backed by a C++ library xlite._C exposing Model / Runtime / AttnMeta / ScoringFunc{Softmax,Sigmoid}. Architecture-strategy adapter registry: LlamaXliteModel (Qwen2/3), QwenMoeXliteModel, Glm4MoeXliteModel, MiniMaxM2XliteModel. Opt-in via ascend_config.xlite_graph_config.enabled.

9. Device Layer, Allocator & CPU Binding

BaseDeviceAdaptor (device_op.py)

1000+ line static-method op-dispatch shim — Ascend's analog of a CUDA kernel library. Covers KV scatter/gather, FIA, MLA prolog (MLAPO), sparse DSA attention, GDN linear-recurrent, MXFP quant. CUDA routes cuDNN/FlashInfer/Triton directly; Ascend routes through torch_npu.* + _C_ascend.* with extra cache_mode/quant_mode/layout_* enums (FRACTAL_NZ, PA_BSND, TND) — a format+kernel translation shim.

Sleep allocators

CaMemAllocator — singleton NPUPluggableAllocator, C++ extension vllm_ascend_C, ACL-memcpy-based mmap, tagged pools → pinned CPU backup. SleepWakeupManager — tears down HCCL groups + ACL graphs on sleep, recaptures on wake. Differs from CUDA caching allocator: offload-not-reuse, HCCL-aware, graph-aware.

Source: device/device_op.py, device_allocator/{camem,sleep_mem_optimized}.py — report 06 §4

11. Cross-cutting Architectural Assessment

For an AI-systems / engine researcher, the signal vllm-ascend sends about hardware-plugin design:

  1. "Plugin" is a misnomer for now. A clean platform layer would be ~5 files; vllm-ascend is ~60 patches + custom schedulers + custom loaders + custom KV transport. The "Hardware Pluggable" RFC is necessary but not sufficient — until vLLM exposes hooks for KV transfer, scheduler policy, config validation, fused-MoE factory, and torch.accelerator semantics, any serious NPU plugin must patch.
  2. Scheduler is the real competitive advantage. ProfilingChunkScheduler (quadratic-fit closed-form chunk) and RecomputeScheduler (stateless-D via free-and-recompute) are contributions that would read well as vRLM-community PRs — they generalize beyond Ascend.
  3. Transport matters as much as compute. The Mooncake pull-model PD disaggregation and the RFork seeded-RDMA loader are Ascend-specific answers to real deployment problems (elastic preemption recovery, disaggregated serving) that vanilla vLLM's Nixl + default loader don't solve cleanly.
  4. Device fragmentation is the main complexity tax. 4 silicon gens (A2/A3/A5/310P) with materially different operator sets — 310P alone is a near-full fork — dominate the LOC and the patch surface.
  5. MLA absorption + FIA-v2 is the right kernel bet. Absorbing W_UK into Q + a single fused FIA-v2 call avoids the head-materialization overhead that plagues two-stage Triton/CUDA MLA; the MLAPO extension (Q/KV proj + LN + RoPE + cache-write in one CANN op) is Ascend-specific but points at a general "fuse the prolog" direction.

12.5 第一性原理修订指引

社区调研对第一性原理文档 first_principles/vllm_first_principles.html 产生以下修正（详见该文档对应章节）：

  • P2 修正：ProfilingChunkScheduler 主要目的是 PP 流水线气泡消除（O(n²) attention 导致同 size chunk 越往后越慢），借 P2 调度回路实现 -- 本质是 P6 工具而非 P1 工具。（来源：官方 design_dcpp.md）
  • P5 强化：patch.md 官方三原则（Less is more / 写移除计划 / 随时清理）把 P5 从分析洞察升级为项目政策。（来源：docs/Design_Documents/patch.md）
  • P4 补 lifecycle：#12119 删除 MatmulAllReduce pass 证明融合技术有生命周期，不是永恒设计。
  • P2 新增：Balance scheduling = 全局准入门，非主动重平衡（design_balance_schedule_refactor）。
