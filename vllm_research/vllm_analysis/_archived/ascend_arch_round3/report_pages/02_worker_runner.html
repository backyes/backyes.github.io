Worker, Model Runner, Block Table & Compilation Passes — Ascend Deep-Dive

Scope: /vllm_ascend/worker/ and /vllm_ascend/compilation/. This slice covers the NPU worker lifecycle, the v1 model runner hot path (execute_model → forward), the v2 model runner rewrite, NPU-block table / input-batch extensions for MLA + PCP, and the full torch.compile pass pipeline (fusion + rewrites) that adapts vLLM’s CUDA-centric graph stack to Ascend’s ACL/NPUGraph execution model.

1. Worker lifecycle

NPUWorker(WorkerBase) lives at worker/worker.py. It is the per-NPU process owner: it owns the distributed environment, the device, the KV cache, and the model runner.

Init order (key sequence in __init__ + init_device)

  1. Patch vLLM & register NPU ops (worker.py:108-119): adapt_patch() is applied, dummy fusion ops registered, ATB extensions loaded for non-A5 silicon, custom ops registered, AscendConfig and soc version initialized.
  2. super().__init__ — the vLLM WorkerBase scaffold (cache dtype, profiler stub, sleep-mode manager SleepWakeupManager).
  3. _init_device() → self.device (worker.py:399-501): - DP-aware local-rank shift: mirrors upstream gpu_worker — local_rank += dp_local_rank * tp_pp_world_size so each DP group binds a distinct NPU slice, with an explicit bypass when --device-ids (assigned_physical_gpu_ids) pre-shards devices (lines 404-416). This is an Ascend-specific guard absent in vanilla vLLM. - Device binding via torch.npu.set_device. - torch_npu._inductor imported lazily after set_device to avoid re-initialization in worker subprocesses (lines 454-457) — required for the Triton/Ascend graph mode path. - A5-only local comm resource init; memory snapshot (MemorySnapshot) taken; gpu_memory_utilization sanity-checked against free_memory (lines 466-478).
  4. Distributed env (_init_worker_distributed_environment, worker.py:981-994): batch-invariance init, init_distributed_environment(..., "hccl") (note "hccl" backend, not NCCL), ensure_model_parallel_initialized with both prefill and decode CP sizes, init_ascend_model_parallel, and EC-transfer init.
  5. Model runner construction (worker.py:512-518): 
    if self.use_v2_model_runner:
    from vllm_ascend.worker.v2.model_runner import NPUModelRunner as NPUModelRunnerV2
    self.model_runner = NPUModelRunnerV2(self.vllm_config, self.device)
else:
    self.model_runner = NPUModelRunner(self.vllm_config, self.device)
    The v2 path is gated behind use_v2_model_runner and is force-disabled on vLLM 0.24.0.

Memory profiling & graph capture (compile_or_warm_up_model, worker.py:698-788)

  • Iterates warmup_sizes in descending order calling _dummy_run.
  • Removes cudagraph_capture_sizes from warmup (those get a dedicated capture call), and appends range-end sizes to ensure every compile range compiles at least once.
  • Graph capture: self.model_runner.capture_model() (NPU-side warmup + NPU graph capture).
  • Emits a --kv-cache-memory= suggestion derived from weights + peak activation + non-torch + npugraph memory, with a 150 MiB safety margin for ACL/HCCL/driver overhead.
  • Special ATB matmul warmup (_warm_up_atb) for non-A5, then CPU binding.

2. Model Runner v1 deep-dive

NPUModelRunner(GPUModelRunner) at worker/model_runner_v1.py:265 (5188 lines) is the execution hot path. It subclasses GPUModelRunner and overrides most of the input-prepare / forward / sample path. Class-level enabler: torch.npu.config.allow_internal_format = True — permits NZ (fractal) weight layouts.

2.1 __init__ (lines 266-594) — notable Ascend additions

  • PCP pad budget: max_pcp_pad_tokens = pcp_size * 2 * max_num_seqs is temporarily added to max_num_batched_tokens before super().__init__, then removed (lines 271-332) — a workaround until upstream PR #28988 merges.
  • Ascend buffers beyond the CUDA set: query_start_loc (+2 for FIA padding), group_len, group_key_idx, group_key_cache_idx (MLA/DSA bookkeeping), a gdn_query_start_loc unpadded clone for GDN, plus CPU pinned _positions_np_buf / _dsa_positions_np_buf for position computation (lines 306-456).
  • Attention backend selection, use_compress (DSA compressed-KV via compress_ratios), use_sparse (DSA top-k), use_sparse_c8, PCP manager (PCPManager), EPLB (D2DExpertWeightLoader + EPLbProcess + multiprocessing.Manager shared dict for expert-load heat), and NPUInputBatch construction (lines 488-558).
  • NPU-graph decision: self.use_aclgraph = cudagraph_mode != NONE && mode == VLLM_COMPILE && !enforce_eager (lines 655-660).

2.2 Execution hot-path: execute_model (lines 1951-2403)

The method is structured in five phases. The central orchestration (greatly condensed):

execute_model(scheduler_output, intermediate_tensors)
 ├─ [pre-work] ngram deepcopy / PP new_token_ids / KV preemption handle
 ├─ [prepare input]  (synchronize_input_prep)
 │     _update_states → _prepare_inputs(scheduler_output, num_scheduled_tokens)
 │       ├─ block_table.commit_block_table          (overlap copy w/ CPU work)
 │       ├─ positions, req_indices, cu_num_tokens
 │       ├─ PCP/DCP: slot_mapping pre-split, pcp_manager.init_batch_info
 │       ├─ query_start_loc / gdn clone / M-RoPE / X-DRoPE
 │       ├─ discard_request mask (optimistic seq_lens), num_accepted_tokens sync
 │       ├─ num_computed_tokens GPU update (async-spec-decode correction)
 │       └─ slot_mapping (non-PCP, GPU kernel)
 │     → _determine_batch_execution_and_padding → (cudagraph_mode, batch_desc, ...)
 │     → _build_attention_metadata → AscendCommonAttentionMetadata + per-group builder
 ├─ [preprocess] _preprocess → input_ids/positions/intermediate_tensors/model_kwargs
 ├─ [forward] set_ascend_forward_context(...) wrapping:
 │     _model_forward → self.model(...) ; optional flash_comm all_gather
 └─ [post process] compute_logits → append ExecuteModelState; return None (continues in sample_tokens)

The CUDA-graph mode is resolved per forward by _determine_batch_execution_and_padding (lines 2955-3052): it decides FULL vs PIECEWISE vs NONE, pads to TP size for sequence parallelism, and critically coordinates DP ranks via _sync_metadata_across_dp (packed num_tokens + cudagraph_mode.value tensor, all-reduced; optional NPU-side all-reduce dp_allreduce_on_npu to avoid CPU-side dirty data, lines 662-710).

2.3 Forward-context population & MoECommType

set_ascend_forward_context (ascend_forward_context.py:57-195) extends vLLM’s ForwardContext with Ascend-specific fields. The pivotal dispatch is select_moe_comm_method (lines 303-367):

class MoECommType(Enum):
    ALLGATHER = 0; MC2 = 1; ALLTOALL = 2; FUSED_MC2 = 3

Selection is device-generation × token-count × EP-size × quant conditioned: - A2: MC2 when num_experts_per_device <= 24 && ep >= 16 && tokens <= mc2_capacity. - A3: FUSED_MC2 for w8a8_dynamic + small EP + dispatch_ffn_combine; else MC2/ALLTOALL. - A5: MC2 within capacity; ALLGATHER when world <= topK; else ALLTOALL. - 310P: always ALLGATHER.

The context also sets mmrs_fusion (TP≤8), flash_comm_v1_enabled (enable_sp && tokens > 1000, the 1000-token empirical threshold), flashcomm_v2_enabled, pad_size/padded_length (TP-divisible padding for the FlashComm collective), mc2_mask, sinks, layer_idx, and eplb_heat_collection_status. V1 vs V2 context access is unified by _ExtraForwardContextProxy (_EXTRA_CTX, lines 370-430): in V2 it reads/writes ctx.additional_kwargs instead of attributes, so the same Ascend extra fields work transparently across both runners.

_model_forward (lines 2860-2896) captures the context, builds model_inputs, and—when ENPU_ENABLE (soft-segmentation) is on—calls _update_full_graph_params_if_needed before the forward so the full-graph attention params are current; otherwise after.

2.4 CUDAGraph → NPUGraph mapping & capture differences

Ascend does not use PyTorch’s CUDA graph directly. Two related-but-distinct mechanisms:

a) ACLGraphWrapper (compilation/acl_graph.py:60-267) — piecewise + full capture of self.model. On __call__:

with torch.npu.graph(aclgraph, pool=self.graph_pool):
    output = self.runnable(*args, **kwargs)
    get_offloader().join_after_forward()
    output = weak_ref_tensors(output)   # last-graph mem saving
entry.aclgraph = aclgraph

Capture-time nuances: forward_context.capturing = True; offloader sync (sync_prev_onload / join_after_forward); GC disabled for all-but-first piecewise graph (patch("gc.collect", ...), lines 167-176); workspace weak-refing (weak_ref_workspaces); and Ascend-specific error translation (Alloc sq cq fail → upgrade HDK; stream-resource exhaustion → reduce capture sizes). Replay enforces input-address stability (debug-mode assert, lines 243-250) and a synchronize() barrier for async-scheduling ordering (lines 262-265).

b) graph_capture context manager (model_runner_v1.py:212-240) — swaps the current NPU stream so captured kernels are isolated from background default-stream work.

c) capture_model (v1) (lines 5035-5045) — wraps the upstream GPUModelRunner.capture_model inside _torch_cuda_wrapper() + _replace_gpu_model_runner_function_wrapper(): _torch_cuda_wrapper (lines 5112-5154) monkey-patches torch.cuda.* → torch.npu.* (Event/Stream/stream/synchronize/mem_get_info); the function-replace wrapper injects the Ascend graph_capture + EncoderAclGraphManager into the GPU module namespace just for the capture call, keeping the upstream GPUModelRunner.capture_model body reusable.

3. Worker v2 vs v1

worker/v2/ is the experimental Model Runner V2, gated by VLLM_USE_V2_MODEL_RUNNER. Per its README, it tracks vLLM’s v2 scheduler/worker split and is tracked in issue #5208.

Structural differences

Aspect v1 (worker/) v2 (worker/v2/)
Class NPUModelRunner(GPUModelRunner) NPUModelRunner(GPUModelRunner) (different GPUModelRunner — vllm.v1.worker.gpu.model_runner)
Request state InputBatch (persistent batch + block table) AscendRequestState(RequestState) — adds num_computed_tokens_cpu (Ascend attn backends require CPU seq_lens that upstream v2 deprecated)
Input buffer NPUInputBatch AscendInputBuffers (+ seq_lens_cpu / positions)
Speculative decoder drafters (AscendEagleProposer, …) AscendEagleSpeculator + init_speculator
Graph manager ACLGraphWrapper + capture_model ModelAclGraphManager(ModelCudaGraphManager)
CP / PCP / EPLB full NotImplemented (PCP, dcp, dynamic_eplb all raise)
Forward context via set_ascend_forward_context inherited GPU path + override_mrv2_in_profile_run ContextVar for the MC2 profile dummy run

The v2 runner’s __init__ (v2/model_runner.py:61-140) deletes the upstream req_states / input_buffers / speculator and re-creates Ascend-specific versions, re-adding CPU-side num_computed_tokens_cpu + a dedicated num_computed_tokens_stream/Event to copy computed-token counts back to CPU for seq_lens reconstruction. For full-graph mode it creates an update_stream used by run_fullgraph to push attention params.

What problem v2 solves

The v1 runner couples input preparation, DP sync, PCP splitting, and EPLB bookkeeping in one monolithic execute_model/_prepare_inputs. v2 adopts upstream’s separation of prepare_inputs (pure, padded, DP-coordinated) from postprocess and reuses upstream execute_model by only overriding prepare_inputs, postprocess, postprocess_sampled, _update_seq_lens_cpu, and the graph manager. This is visible in run_fullgraph (aclgraph_utils.py:71-101): v2 injects model_runner into ModelAclGraphManager precisely so it can inherit upstream execute_model and only hook update_full_graph_params — whereas v1 must carry the whole forward orchestration.

v1→v2 shim mechanics

  • torch_cuda_wrapper (v2/utils.py:10-27): wraps torch.cuda.*→torch.npu.* including graph_pool_handle, CUDAGraph = NPUGraph, and graph = torch_npu_graph_wrapper.
  • torch_npu_graph_wrapper (v2/utils.py:48-56): exits the graph context by weak-reffing the graph workspaces (weak_ref_workspaces) — an MRV2-specific cleanup that avoids another upstream monkey patch.
  • communicator_switch (v2/utils.py:31-44): swaps CudaCommunicator→NPUCommunicator for capture.
  • ModelWithContext (aclgraph_utils.py:135-163): wraps self.model so that inside capture capturing=True, is_draft_model, is_draft_model_prefill are set on _EXTRA_CTX — letting Ascend inherit upstream _capture_full_graph unchanged.

Known v2 gaps (README): set_cos_and_sin (MLA RoPE), custom KV-init (2M-alignment, PD disagg), the graph-manager↔model_runner coupling, and the speculator graph-manager coupling all still need to be upstreamed before the v1-only workarounds can be deleted.

4. Block Table & Input Batch

BlockTable (worker/block_table.py)

BlockTable + MultiGroupBlockTable extend vLLM’s block management with three Ascend concerns:

  1. Compressed-KV (MLA/DSA) scaling — compress_ratio is read off UniformTypeKVCacheSpecs and max_num_blocks_per_req is divided by it (lines 33-49). DSA’s “compress-head” layout stores fewer logical blocks per request.
  2. Hybrid (kernel) blocks — kernel_sizes selects a kernel block size that evenly divides physical_block_size; blocks_per_phys_block splits a physical block into multiple logical blocks (use_hybrid_blocks), with _convert_physical_to_logical_blocks (lines 61-309). Required so attention kernels that assume a small block size can address a coarser physical allocation.
  3. PCP/DCP context-parallel slot mapping — compute_slot_mapping dispatches to the GPU Triton _compute_slot_mapping_kernel for the plain case (with CP_KV_CACHE_INTERLEAVE_SIZE, PAD_ID, KV_CACHE_BLOCK_SIZE, BLOCKS_PER_KV_BLOCK constexprs), and to _compute_pcp_dcp_slot_mapping (lines 240-285) for CP. The CP path uses a virtual block = physical_block_size * cp_world_size, computes a per-rank mask virtual_block_offsets // cp_kv_cache_interleave_size % total_cp == current_rank, and interleaves slots across ranks — DCP stores KV with a rank-interleaved layout so each token i lives on the NPU with dcp_rank == i % pcp_world_size (per the code comment).

MultiGroupBlockTable (lines 329-467) maintains one BlockTable per KV-cache group (MLA vs standard attention vs Mamba vs encoder-only), skipping Mamba groups in compute_slot_mapping. It is constructed in NPUInputBatch.__init__ with per-group block_sizes, kernel_block_sizes, and kv_cache_groups.

NPUInputBatch (worker/npu_input_batch.py)

NPUInputBatch(InputBatch) (:33) carries the persistent batch. Ascend-specifics over upstream:

  • block_table = MultiGroupBlockTable(...) holding multi-group block tables.
  • M-RoPE / X-DRoPE handled in the runner (_calc_mrope_positions, _calc_xdrope_positions).
  • prev_sampled_token_ids, prev_req_id_to_index for async spec-decode token re-insertion; req_prompt_embeds dict for multimodal prompt embeddings.
  • Sampling metadata (temperature/top_p/top_k, penalties, logits processors) duplicated as {gpu, cpu_tensor, numpy} triplets for non-blocking CPU→GPU copies via copy_snapshot_to_gpu (worker/utils.py:15-18), the canonical “pinned snapshot copy” helper. This helper is used pervasively in _prepare_inputs to overlap CPU compute with GPU upload.

The batch flow is: _update_states (writes new token ids into token_ids_cpu, allocates blocks), then _prepare_inputs (positions → slot_mapping → commit_block_table → GPU copy). The num_accepted_tokens / num_computed_tensors sync between CPU and GPU is the mechanism that makes optimistic async spec-decode safe.

5. Compilation passes

Ascend cannot rely on vLLM’s Triton-centric Inductor path, so it defines a separate pass manager + compiler interface and a library of IR-level rewrites. Two execution backends are supported, selected by ascend_compilation_config.enable_npugraph_ex:

  • fusion_pass_compile (compiler_interface.py:47-68) → GraphFusionPassManager → torch.compile Inductor pattern-matcher passes (Triton generation, the path actually used for custom Triton kernels like muls_add_triton).
  • npugraph_ex_compile (compiler_interface.py:149-223) → npugraph_ex / torchair NPU-backend compilers that lower FX → Ascend IR → ACL operators; also runs the same GraphFusionPassManager inside its inner compile via compile_inner.

GraphFusionPassManager (graph_fusion_pass_manager.py) is the Ascend analog of vLLM’s PostGradPassManager (“Since torch_npu does not support triton for now, we define our own pass manager”). configure (:49-79) toggles each fusion via ascend_compilation_config:

Pass File Toggle
AddRMSNormQuantFusionPass norm_quant_fusion_pass.py fuse_norm_quant (default on, off on 310P)
QKNormRopeFusionPass qknorm_rope_fusion_pass.py fuse_qknorm_rope (default on)
MatmulAllReduceAddRMSNormPass allreduce_rmsnorm_fusion_pass.py fuse_allreduce_rms (default on)
MulsAddFusionPass muls_add_pass.py fuse_muls_add (default on, off on 310P)
SequenceParallelismPass sequence_parallelism.py pass_config.enable_sp
SequenceParallelismMoePass sequence_parallelism_moe.py pass_config.enable_sp

BasePattern (base_pattern.py) is the uniform abstraction: get_pattern / get_replacement / get_inputs feed both the Inductor PatternMatcherPass (pm.register_replacement) and the NPU graph nge.register_replacement with an extra_stream_scope_check (npugraph_ex_utils_check). Duplicate registration is avoided by a global _registered_patterns set.

5.1 Pass-by-pass transforms

QKNormRopeFusionPass (qknorm_rope_fusion_pass.py:187) — fuses QKV-split + per-head RMSNorm + RoPE into one DeviceOperator.split_qkv_rmsnorm_rope:

# pattern (lines 53-76)
q, k, v = qkv.split([q_size, kv_size, kv_size], dim=-1)
q_by_head = q.view(...)
q_norm_out, _ = torch.ops.npu.npu_rms_norm(q_by_head, q_weight, self.eps)
k_by_head = k.view(...)
k_norm_out, _ = torch.ops.npu.npu_rms_norm(k_by_head, k_weight, self.eps)
q_rope, k_rope = torch.ops.vllm.npu_rotary_embedding(
    positions, q_flat, k_flat, cos_sin_cache, self.head_dim, self.rope_dim, True)
# → replacement
DeviceOperator.split_qkv_rmsnorm_rope(input=qkv, q_weight=, k_weight=, ...)
Why: collapses 2× split + 2× reshape + 2× rms_norm + 1× rope into a single ACL op launch; only active for head_dim=128, eps ∈ {1e-5,1e-6}, with/+bias variants.

MatmulAllReduceAddRMSNormPass (allreduce_rmsnorm_fusion_pass.py:134) — matmul + TP all-reduce + residual-add + RMSNorm → torch.ops._C_ascend.matmul_allreduce_add_rmsnorm:

# pattern (lines 57-64)                       # replacement (lines 68-84)
mm = unquantized_gemm(x, weight, None)        out0, out1 = npu_add_rms_norm_bias x weight residual
ar = tensor_model_parallel_all_reduce(mm)      rmsnorm_weight, tp_group_name, tp_size,
chunk = maybe_chunk_residual(ar, residual)     local_rank, eps, True, False
out  = npu_add_rms_norm_bias(ar, chunk, w, None)
The fused op internally performs ReduceScatter+AllGather instead of AllReduce, halving traffic. Gate: compile_range.start > 512 (ALLREDUCE_NORM_FUSE_THRESHOLD) to avoid hurting small-batch latency (lines 154-159).

AddRMSNormQuantFusionPass (norm_quant_fusion_pass.py:679) — the largest pass family; fuses AddRMSNorm + quantization into single ACL ops, e.g.:

# pattern (lines 50-67)                         # replacement (lines 72-89)
out = npu_add_rms_norm_bias(inp, res, w, None)  out = npu.npu_add_rms_norm_quant(
q = vllm.quantize(out[0], scale, s_recip, off)      inp, res, w, scale, offset, epsilon=eps)
Variants: ±bias, +SP (maybe_all_gather_and_maybe_unpad kept after the fused op), dynamic (npu_dynamic_quant→npu_add_rms_norm_dynamic_quant), and MXFP (npu_dynamic_mx_quant→npu_add_rms_norm_dynamic_mx_quant/npu_rms_norm_dynamic_mx_quant, fp8_e4m3). W4A4-int4 models explicitly disable this pass (lines 658-676).

MulsAddFusionPass (muls_add_pass.py:76) — element-wise x * routed_scaling_factor + y → torch.ops.vllm.muls_add (a Triton kernel, the rare Ascend path that uses Triton):

tmp = x * self.scale                 # → vllm.muls_add(x, y, self.scale)
out  = tmp + y
The scale is the model’s routed_scaling_factor (MoE silu-scale).

SequenceParallelismPass (sequence_parallelism.py:187) rewrites middle/layer SP epilogues. The SP minimum-token threshold is 1 for MoE, 1000 for dense (get_sp_min_token_num, lines 16-20), because MoE benefits from SP even at tiny batches. The core rewrite for MiddleAllReduceRMSNormPattern (lines 76-99):

# pattern                               # replacement
x = all_reduce(input)                   rs = reduce_scatter(input)
r,_,res = npu_add_rms_norm_bias(        res = maybe_chunk_residual(rs, res)
        x, residual, weight, eps)       r,_,res = npu_add_rms_norm_bias(rs,res,weight,eps)
                                        return all_gather(r), res
Plus a Qwen3VLM variant (+ deepstack_input_embeds add). Runs only when compile_range.start >= min_tokens.

SequenceParallelismMoePass (sequence_parallelism_moe.py:165) is the AllGather epilogue pass for MoE. MiddleLayerAllgatherAddRMSNormPattern (lines 32-52) moves the all_gather after rmsnorm to avoid middle-layer shape explosion:

# pattern                                   # replacement
ag = all_gather(input)                       res = maybe_chunk_residual(input, res)
sl = ag[:num_tokens]                         r,_,res = npu_add_rms_norm_bias(input,res,weight,eps)
r,_,res = npu_add_rms_norm_bias(sl,res,...)  return all_gather(r), res
Plus AllGatherChunkNoOpPattern folding all_gather + sequence_parallel_chunk_impl → identity.

NoOpEliminationPass (noop_elimination.py:11) and AllGatherChunkNoOpCleanupPass (allgather_chunk_noop_pass.py:10) clean up identity view/reshape and all_gather+chunk pairs that would otherwise become graph breaks or wasted kernels. NoOpElimination walks FX nodes, compares input/output shapes via statically_known_true, and erases no-op view/reshape. Both are preconditioning for capture: the fewer dead ops, the smaller/replayable the captured graph.

5.2 Why Ascend needs its own pass pipeline

  1. CUDA→Ascend operator lowering — /vllm.npu_rms_norm, /vllm.npu_add_rms_norm_quant, /vllm.matmul_allreduce_add_rmsnorm are ACL operators with no Triton equivalent; Inductor cannot emit them, so they must be pattern-matched and rewritten.
  2. Communication–compute fusion — ReduceScatter+AllGather, AllReduce→AddRMSNorm, and the matmul+allreduce atomics hide latency and halve interconnect traffic on HCCL.
  3. Graph-break removal — Triton kernel captures, all_gather+chunk noops, and reshape/view folding produce piecewise-capturable subgraphs.
  4. Structural shuffle — moving all_gather before/after rmsnorm (SP passes) and the TP token-padding (_pad_for_sequence_parallelism) reshape the FX graph into a form the NPU piecewise graph replay expects.

5.3 Ascend-specific compiler backend (AscendCompiler, compiler_interface.py:226)

  • compute_hash factors in torch_npu_version, enable_npugraph_ex, enable_static_kernel (lines 236-249) so graph caches invalidate on CANN/driver upgrades.
  • compile branches on enable_npugraph_ex → npugraph_ex_compile (preferred) else fusion_pass_compile (Inductor/Triton). The npugraph_ex path selects nge or torchair, and configures _configure_backend (lines 82-146):
  • npugraph_ex: force_eager=True, inplace_pass=False (avoids Gelu→CPU fallback), clone_input/output=False, optional static_kernel_compile for decode sizes.
  • torchair: mode="reduce-overhead", debug.run_eagerly=True, disable_reinplace_inplaceable_ops_pass=True, optional _aclnn_static_shape_kernel.
  • LOCAL_WORLD_SIZE is synthesized from local_world_size * data_parallel_size_local for static-kernel Gloo groups (lines 97-108) because vLLM doesn’t set it like torchrun does.
  • Cache/load serialize compiled py_code, but skip graphs containing Triton kernels (their kernel_side_table indices are process-local and non-serializable, lines 186-198).

6. ACL graph, torch.compile, and Triton — when each is used

Ascend has three execution modes, selected per-build/not-per-op:

Mechanism Where invoked When chosen
ACL Graph (piecewise or full) ACLGraphWrapper.__call__ / ModelAclGraphManager.run_fullgraph cudagraph_mode != NONE && mode==VLLM_COMPILE && !enforce_eager (_use_aclgraph). FULL = whole model per BatchDescriptor; PIECEWISE = per-layer segments (re-captured with GC disabled). Replayed via entry.aclgraph.replay(). Uses torch.npu.NPUGraph from a current_platform.get_global_graph_pool().
torch.compile (Inductor/Triton) fusion_pass_compile → GraphFusionPassManager Default compilation; needed to emit Triton kernels (e.g. muls_add_triton, _zero_kv_blocks_kernel in worker/utils.py:21, _compute_slot_mapping_kernel). The Ascend Triton path runs via torch_npu._inductor.
npugraph_ex / torchair npugraph_ex_compile When enable_npugraph_ex=True. FX→Ascend IR→ACL; replaces Inductor as the lowering backend. Still runs the same fusion passes.

ACL graph is layered on top of torch.compile in Ascend: compile_or_warm_up_model runs Inductor/compile (which applies the fusion passes and picks a backend), and the resulting callable is what ACLGraphWrapper captures into an NPUGraph. So the typical production path is: dynamo → FX → fusion passes → npugraph_ex/torchair lowering → ACL graph capture (FULL or PIECEWISE) → replay.

Encoder / ViT graphs

Vision encoders use a separate ACL-graph path for the FIA (fusion-in-attention) tasks: EncoderCudaGraphManager from vLLM is monkey-patched to EncoderAclGraphManager (worker/encoder_acl_graph.py, injected via _replace_gpu_model_runner_function_wrapper), with EncoderGraphParams keyed by token budget rather than batch size (set_encoder_graph_params(token_budgets), lines 37-62). The V2 manager uses the same budget bookkeeping in ModelAclGraphManager.

kvcomp (KV compression / sparse attention metadata)

worker/kvcomp_utils.py binds hash-k caches for sparse attention (DSA top-k / SFA): bind_hashk_cache / bind_hashk_cache_nope allocate per-layer hash-key caches, insert them into the forward_context for each Attention layer, and maintain a runner-side list ordered by extract_layer_index. This is the runtime metadata plumbing that the sparse-attention kernels (build_kvcomp_metadata, AscendKVBlockZeroer) consume alongside the block table.
