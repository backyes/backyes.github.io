1. Model support matrix

Architecture support in vllm-ascend is not confined to models/. Only three modelsregister custom Ascend ForCausalLM classes (models/__init__.py); everything else lives in the Xlite graph runtime (xlite/xlite.py — a whole-model graph capture for dense/MoE architectures), DSA/MLA/SFA attention backends (attention/), and model-specific patches (patch/platform/). This three-tier design is the key architectural fact.

2. Deep-dive: DeepSeek V4 on Ascend (the most-adapted model)

AscendDeepseekV4ForCausalLM (deepseek_v4.py:1206) composes a @support_torch_compile DeepseekV4Model and a DeepseekV2MixtureOfExperts. The headline Ascend adaptations:

2.1 MLA → DSA (Deepseek Sparse Attention) path

DeepseekV4Attention (deepseek_v4.py:705) does not run a conventional attention kernel. It builds a DSAModules bundle (wq_a, q_norm, wq_b, wkv, kv_norm, wo_a, wo_b, attn_sink, indexer, compressor, swa_cache) and instantiates AscendDeepseekSparseAttention (ops/dsa.py), which routes to an attention backend selected by AscendDSABackend (attention/dsa_v1.py):

self.dsa_attn = AscendDeepseekSparseAttention(dim, n_heads, scale, ..., dsa_modules=dsa_modules, ...)
# forward():
def forward(self, positions, hidden_states, llama_4_scaling):
    return self.dsa_attn(positions, hidden_states, llama_4_scaling)   # deepseek_v4.py:898

Sparse DSA caches (the MLA state compression) — three custom cache classes override vLLM’s GPU-side MLA caches to emit Ascend-padded specs: - AscendCompressorStateCache → AscendSlidingWindowMLASpec(page_size_padded=...) (:104) - AscendDeepseekV4IndexerCache → AscendMLAAttentionSpec(model_version="deepseek_v4", compress_ratio=..., scale_dim=...) (:137). On A5 devices the index cache dtype is forced to float8_e4m3fn (:149). - AscendDeepseekV4SWACache → AscendSlidingWindowMLASpec(model_version="deepseek_v4") (:173). A5 again uses FP8 and grows head_size += 128 (:191).

A block-size lookup table (DSV4_BLOCK_SIZES, from models/layer/attention/layer.py) pads all three cache levels per concrete block_size (:118,156,185,647,655).

Indexer / Compressor — for compress_ratio == 4 (the V3.2 / V4 sparse-attention regime) the layer adds an Indexer (low-rank q projection wq_b, weights_proj, top-k index selection) and a Compressor (learned ape positional embedding, wkv/wgate projections, RMSNorm, inter-block overlap via overlap_transform) (:525–703). The IndexCache optimization (:837) reuses top-k indices across layers via a configurable index_topk_pattern (e.g. "FSS" → full/share/share) so only ~1/L layers run the indexer.

2.2 HC (Hybrid Computation) pre/post kernels — Ascend-specific

The decoder layer is wrapped by custom C++ operators defined in ops/: 

def hc_pre(self, x, hc_fn, hc_scale, hc_base):
    return torch.ops._C_ascend.npu_hc_pre_v2(x, hc_fn, hc_scale, hc_base, self.hc_mult,
                                             self.hc_sinkhorn_iters, self.norm_eps, self.hc_eps)  # :967
def hc_post(self, x, residual, post, comb):
    return torch.ops._C_ascend.npu_hc_post(x.unsqueeze(0), residual.unsqueeze(0), post.unsqueeze(0), comb.unsqueeze(0))
(:966–976) plus a per-model hc_head gated-head that mixes hc_mult streams with a learned sigmoid router + rmsnorm (:1088). These implement the Multi-head Latent / Hybrid-Computation variant where hc_mult parallel residual streams are combined with sinkhorn-routed gates — standard vLLM has no equivalent; this is fully custom.

2.3 MoE path

DeepseekV4MoE (:348) is a standalone Ascend MoE: - Router: ReplicatedLinear gate with precast_fp32_weight=True; optional hash routing for early layers (gate.tid2eid, :412) vs. e_score_correction_bias. - EP-aware physical experts: n_physical_experts = n_logical + n_redundant, partitioned by EP rank, with EPLB support (:385–391, enable_eplb). - Shared-expert fusion (is_fusion_moe_shared_experts_enabled driven by ascend_config.mix_placement): when on, shared expert computation is folded into FusedMoE with n_shared_experts experts appended to the expert pool (:396,451) — this is the AITER/MC2 fused path. - FusedMoE from vLLM is used, but with DeepSeek-specific grouped-topk + routed scaling pulled outside the router so ordering is: normalize top-k → scale routed output (:442–452). - Sequence-parallel MoE: sequence_parallel_chunk before, tensor_model_parallel_all_gather after (:464,498).

2.4 MLP path

Standard DeepseekV2MLP (:299): MergedColumnParallelLinear → SiluAndMul(SiwgClamp) → RowParallelLinear, with disable_tp option when running under sequence-parallelism.

2.5 RoPE & load_weights

ComplexExpRotaryEmbedding (ops/rope_dsv4.py) supports two rope groups (default + c{compress_ratio}) with compress_rope_theta (:787). load_weights (:1295) is a 220-line Ascend-specific weight map handling: w1/w2/w3→gate/down/upProj renaming, sink weights distributed across TP ranks (enable_dsn_cp() → full copy vs. narrow, :1363), and the shared-expert fusion slice-and-load into appended expert slots (:1423–1428).

2.6 M Model-level spec-decode support

DeepseekV4Model.forward stashes a _mtp_hidden_buffer (pre-hc_head residual) with special all-gather handling for FlashComm v1 / sequence-parallel so the MTP draft sees the full token set (:1133–1151). Exposed via get_mtp_target_hidden_states (:1289).

3. Speculative decoding on Ascend

vllm-ascend spec decode lives in spec_decode/ and is much broader than vanilla vLLM’s MTP. It supports MTP (DeepSeek), Eagle3, DFlash (Qwen3), DSpark (Qwen3), Medusa, Ngram, suffix, plus a draft_proposer / extract_hidden_states_proposer.

3.1 MTP on Ascend (deepseek_v4_mtp.py)

DeepSeekV4MTP (:201) is a full custom draft model. DeepSeekMultiTokenPredictorLayer (:56) fuses the target hidden state and embedding: 

inputs_embeds = torch.where(positions.unsqueeze(-1) == 0, 0, inputs_embeds)  # mask pos 0
hidden_states = self.e_proj(inputs_embeds).unsqueeze(-2) + self.h_proj(previous_hidden_states)  # :121
hidden_states, residual = self.mtp_block(positions, hidden_states, residual)  # shared DeepseekV2DecoderLayer
It shares topk_indices_buffer, embed_tokens (always), and lm_head with the target — handled in AscendSpecDecodeBaseProposer._maybe_share_embeddings/topk_indices/lm_head (llm_base_proposer.py:408–513). The load_weights rewrites mtp.0.-prefixed checkpoint names into the in-model layout with mtp_block insertion (:300,480).

3.2 AscendSpecDecodeBaseProposer (spec_decode/llm_base_proposer.py)

This is the central Ascend spec-decode orchestrator (~2150 lines), subclassing vLLM’s SpecDecodeBaseProposer. Notable Ascend behavior: - TP-patch for draft: swaps the global _TP group so a draft model can run at draft_tensor_parallel_size == 1 different from target (patch_tensor_parallel_group, :65–78). - ACL Graph (cudagraph analog): wraps the merged draft in ACLGraphWrapper(..., runtime_mode=CUDAGraphMode.FULL) (:523). - PCP/DP integration: pcp_size/dcp_size from the runner; block-table cloning for context-parallel where step>0 needs an independent block table (:596,670); block_table_tensor_clone allocated after profile. - GLM fallback: _is_glm_model() forces the draft to eager mode with a warning (:137–145, 223). - Attention metadata: builds AscendCommonAttentionMetadata with dsv32-specific seq_lens/query_start_loc persistent pinned buffers for full-graph mode (:249–256,613). - Top-k / hidden sharing: _maybe_share_topk_indices rebinds topk_indices_buffer across all draft modules (:531). - multimodal: _get_model_name switches for Qwen2.5/3-VL, Step3.7, Pixtral, KimiK2.5 (:371–387). - Eagle3 / DFlash lm_head sharing with special handling for DFlash’s draft_id_to_target_id vocab remapping (keeps draft’s own lm_head, :479–508). - Merged drafting (_run_merged_draft) runs all K spec tokens through the draft model in one batched call (the default runnable, :262). - greedy_sample does TP-aware global argmax via all_gather (:118).

3.3 Other proposers

  • eagle_proposer.py, dflash_proposer.py (DFlashQwen3ForCausalLM — imported from vLLM model lib), dspark_proposer.py (Qwen3DSparkForCausalLM, version-gated on vllm_version_is("0.24.0"), :58), medusa_proposer.py, ngram_proposer.py + ngram_proposer_npu.py (NPU variant), suffix_proposer.py, step3p5.py, draft_proposer.py, extract_hidden_states_proposer.py. spec_decode/utils.py provides SlidingWindowAdapter for draft sliding-window attention with future_offset = 0 (DFlash) / num_spec_tokens (Eagle).
  • Eagle3-VWN for Llama: models/llama_eagle3_vwn.py registers Eagle3VwnLlamaForCausalLM — a small custom head (Vwn variant).

Integration with PCP/DP: PCP (prefill context parallel) needs block-table isolation per step; the proposer clones block_table_tensor_clone and uses prefill_context_parallel_metadata (:642). DP is handled via _sync_metadata_across_dp at the start of dummy_run (:582).

4.3 EP × quant interaction

  • get_moe_num_logical_experts (from base.py) excludes redundant EPLB experts — router_logits.shape[1] == num_logical_experts is asserted at apply time (w8a8_dynamic.py:250, w4a16.py:283).
  • AscendFusedMoEMethod.create_weights (method_adapters.py:222) partitions by intermediate_size_per_partition and tags scales as FusedMoeWeightScaleSupported.CHANNEL or GROUP; dynamic-EPLB clones per-expert weight/scale lists at process_weights_after_loading (w8a8_dynamic.py:360–384).
  • MC2коммуникация: MoE uses an MC2 all-to-all group name captured from the HCCL comm (get_mc2_group().device_group, :172); enable_fused_mc2 replicates scales in int64 for the fused path (:356).
  • AscendFusedMoEMethod.maybe_make_prepare_finalize returns None to prevent upstream modular-kernel initialization from overriding Ascend’s own MoE forward_impl (:217–220).

W4A16 weight flow (most carefully documented): checkpoint weight_packed[int32]/weight_scale[bf16]/weight_shape[int32] → load as w13/w2 → process_weights_after_loading unpacks → transposes → re-packs to the Ascend fused-MoE layout (:332–364). All-zero offset tensors are synthesized since the operator requires them.

5. KV-offload (two-tier)

Ascend implements two distinct CPU-offload systems — they coexist because they serve different predictability/latency profiles.

5.1 Full: kv_offload/ — the vLLM-v1 OffloadingSpec

NPUOffloadingSpec (kv_offload/npu.py) subclasses vllm.v1.kv_offload.spec.OffloadingSpec and is driven by events (kv_events_config.enable_kv_cache_events): - Manager: CPUOffloadingManager(block_size = gpu_block_size * block_size_factor, num_cpu_blocks) (:38). - Handler: CpuNpuOffloadingHandler(attn_backends, gpu_block_size, cpu_block_size, num_cpu_blocks, gpu_caches) (:53), yielding both GPULoadStoreSpec↔CPULoadStoreSpec directions. - Block size is scaled by block_size_factor; num_cpu_blocks comes from extra_config.

5.2 Simple: simple_kv_offload/ — the graph-capturable connector

SimpleCPUOffloadNPUWorker (simple_kv_offload/worker.py) subclasses SimpleCPUOffloadWorker and replaces only the copy backend with NPUDmaCopyBackend (copy_backend.py). Key Ascend realities: - K/V are separate allocations (not stacked) → must iterate every sub-tensor; dedup by untyped_storage().data_ptr() (:101–112). - The NPU runner over-allocates each KV for 2 MiB alignment, so block-views are sized from tensor stride/shape, not storage.nbytes() (_build_block_views, :160–224). - No cudaHostRegister → CPU mirrors are torch.zeros(pin_memory=True); transfer streams are plain torch.npu.Stream() (no priority hint since torch.npu lacks Stream.priority_range(), :124–151). - NPUDmaCopyBackend issues batched aclrtMemcpyBatchAsync across all sub-tensors.

5.3 Why both exist

kv_offload/ is the event-driven, vLLM-official mechanism (used with block-size factor / KV-events / disaggregated-PP KV connectors); simple_kv_offload/ is the lightweight, ACL-Graph-capturable path with deterministic pinned-buffer transfers and no event dependency — preferred when full-graph capture and P2P DMA overlap matter more than scheduler-driven eviction. The “simple” one stores the leading num_blocks blocks that the connector knows about, regardless of the runner’s alignment padding.

6. Model loader innovations (netloader, rfork)

vllm-ascend registers two custom loaders (register_model_loader → register_netloader/register_rforkloader, model_loader/{netloader,rfork}/__init__.py), both falling back to DefaultModelLoader on failure.

6.1 Elastic NetLoader (model_loader/netloader/)

ModelNetLoaderElastic (netloader.py::ModelNetLoaderElastic) — a network/distributed weight loader. Pipeline: 1. Read config from model_loader_extra_config keys SOURCE / MODEL / LISTEN_PORT / INT8_CACHE (hbm|dram|no) / INT8_CACHE_NAME / OUTPUT_PREFIX (:94–125). 2. initialize_model builds an empty model on the NPU in its target dtype. 3. elastic_load(model, device_id, model_path, sources, tp, pp, group_name) (load.py): - Filters sources for this device_id; opens an ElasticClient socket to the source addr:port. - Constructs P2PLoad(world_name, source_ip, source_port, group_name) (executor/elastic_load.py), which statelessly init’s an HCCL process group and recv’s every named parameter over HCCL on a dedicated torch_npu.npu.Stream, then barrier + synchronize (elastic_load.py:50–90). Weight arrives already distributed via HCCL so no CPU staging. 4. Optionally starts an ElasticServer (interaction/elastic.py) listening for downstream consumers (model cascading), with INT8_CACHE options (hbm|draft, :360). 5. Draft-model loading uses DRAFT_PORT_OFFSET=10000 so target/draft servers don’t collide (:258–273); a torch.distributed.barrier() synchronizes all target ranks before loading the draft (_sync_target_netloader_before_draft, :146).

Why faster: weights stream rank-to-rank over HCCL while compute setup happens; the source node broadcasts once and many receivers load in parallel. Avoids each rank independently reading shards from NFS (the default-path bottleneck), and the post-load process_weights_after_loading (quant reorder) runs on-device.

6.2 RFork loader (model_loader/rfork/) — the headline innovation

RForkModelLoader (rfork_loader.py::RForkModelLoader, registered "rfork", 2026) implements a seeded weight transfer: instead of re-loading weights from disk, a warmed “seed” instance RDMA-pushes its already-loaded weights to a fresh worker. Pipeline: 1. model_loader_extra_config keys model_url, model_deploy_strategy_name, rfork_scheduler_url, rfork_seed_timeout_sec, rfork_seed_key_separator — or env vars (:134–174). RFork is skipped entirely under dynamic EPLB (_is_dynamic_eplb_enabled, :231). 2. _ensure_rfork_worker creates an RForkWorker (rfork_worker.py) holding: - RForkSeedProtocol (seed_protocol.py) — builds a per-instance seed key "{model_url}${strategy_name}${node_rank}$pp{rank}$tp{rank}$ep{rank}$draft" (get_local_seed_key, seed_protocol.py:50), then GET http://scheduler/get_seed → returns {seed_ip, seed_port, ...} / release_seed / report_seed (heartbeat thread). A central scheduler hands out seed instances. - RForkTransferBackend (transfer_backend.py) — wraps yr.datasystem.TransferEngine (Yuanrong RDMA transfer engine). init_transfer_engine() opens it on ascend with npu:{device} (:263–288). 3. load_model (rfork_loader.py:217): - initialize_model (empty model, on device). For quantized models, process_weights_after_loading is run before transfer so weight dtypes/shapes match the seed’s (:268–270). - rfork_worker.pre_transfer(model): registers the model’s weight memory with the RDMA engine. It walks all parameters + buffers + Ascend post-load runtime tensors (MLA W_UV, W_UK_T etc., _iter_transferable_tensors/_collect_transferable_tensors, :167–203), coalesces torch.npu.memory.memory_snapshot() blocks that contain weight pointers, and batch_register_memory them (register_memory_region, :298–363). - rfork_worker.transfer(model): for each weight, pulls seed pointer/session from GET /get_rfork_transfer_engine_info, reshapes local tensor to seed shape if needed, then batch_transfer_sync_read(seed_session_id, client_ptrs, seed_ptrs, lengths) in chunks capped at 1 GiB / 512 weights (_iter_transfer_chunks, :215–249, MAX_TRANSFER_CHUNK_BYTES=1024**3). - post_transfer releases the seed; start_seed_service spins up an HTTP server (seed_server.py) advertising this instance as a future seed (:110–144). 4. Fallback on any failure (seed unavailable, transfer error, shape mismatch) → get_model(load_format="auto"), default vLLM loader. NPU cache is emptied aggressively (gc.collect() ×3 + torch.npu.empty_cache(), :299).

Why faster than default: the default loader has every rank deserialize shards from storage, paying CPU/GIL + PCIe × each new process. RFork replaces that with one RDMA read from a live, already-warmed seed (same model, same quantization, same layout). The seed-reuse pattern matters for elastic serving / KV-decode disaggregation / preemption recovery where a new worker must become ready in seconds, not minutes. The GIL concern is sidestepped because TransferEngine is a native RDMA library that pushes into GPU-mapped memory without Python-level tensor construction.

7. LoRA & Sampling

7.1 LoRA (lora/)

PunicaWrapperNPU (lora/punica_npu.py) wraps the punica bgmv/sgmv kernels. Ascend selection logic (:22–50): - On 310P or when max_lora_rank ≥ 128: uses vLLM’s PyTorch-native bgmv_*/sgmv_* ops (Triton-less path). - Else: imports Ascend-native vllm_ascend.lora.lora_ops bgmv/sgmv (custom AscendC kernels, csrc/kernels/bgmv_*.cpp). All ops are graph-capturable (static shapes, value-independent).

AscendFusedMoEWithLoRA / AscendFusedMoE3DWithLoRA (lora/fused_moe.py) implement MoE LoRA v2 — the key Ascend innovation: instead of a torch.unique/nonzero gather (data-dependent shapes not capturable by ACL Graph), they fold the (lora_id, expert_id) pair into a single gather index into a [max_loras * num_experts, ...] view of the weight stacks, with -1 sentinel rows skipped internally by the bgmv kernels (punica_npu.py:344–410). This keeps shapes value-independent → fully graph-capturable. Uses mul_routed_weight, adapter_enabled masks, and in-place bgmv_expand_slice.

lora/__init__.py, utils.py (refresh_all_lora_classes), and supported ops cover linear, embedding, logits, and fused-MoE. Scope: full LoRA for dense + MoE (shared + routed) experts where Ascend bgmv kernels apply; PunicaWrapperBase wiring handles the multi-LoRA metadata.

7.2 Sampling (sample/)

AscendSampler (sample/sampler.py:45) replaces the CPU/CUDA penalty loop with Triton-Ascend via apply_all_penalties → ops.triton.penalty.apply_penalties_triton when the Ascend Triton backend is available, else vLLM default (:47–68). It also overrides greedy_sample to do a TP-aware global argmax via tp_group.all_gather (:105–124). AscendTopKTopPSampler(:127) similarly gates its kernel path. sample/rejection_sampler.py is a ~1400-line custom rejection sampler with both NPU-optimized and pure-PyTorch (*_pytorch) paths — AscendRejectionSampler.apply_penalties reuses the Triton-Ascend penalties, prepare_sampling, apply_logits_processors, and forward, plus block-verify (rejection_random_sample_block_verify_pytorch) and recovered-token sampling, all with spec-length-1 fast paths. This breadth is required because speculative decoding (MTP/Eagle3/Medusa/DFlash) needs a different sampling semantic than vanilla single-token decoding.
