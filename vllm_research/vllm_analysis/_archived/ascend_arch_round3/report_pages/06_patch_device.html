06 — Patch system · Profiler · Device · Xlite · CPU binding

Slice focus. How vllm-ascend bends vanilla vLLM to Ascend without forking it. The single most revealing fact: vllm-ascend maintains ~60 active monkey-patch modules (~27 platform + ~33 worker) that rewrite vanilla vLLM symbols at import time. This is the “Hardware pluggable RFC is not yet complete” reality made code. The patch/ tree is the engineering diary of every place vanilla vLLM assumed CUDA-NCCL-FlashAttn-UDA and had to be re-wired for Ascend-CANN-HCCL-FIA.

1. Patch system mechanics

1.1 The two-phase hook: adapt_patch

The entire patch system is a thin import-for-side-effects dispatcher in vllm_ascend/utils.py:537:

def adapt_patch(is_global_patch: bool = False):
    if is_global_patch:
        from vllm_ascend.patch import platform  # noqa: F401
    else:
        from vllm_ascend.patch import worker  # noqa: F401

patch/platform/__init__.py and patch/worker/__init__.py are ordered import lists — each line import vllm_ascend.patch.<area>.patch_X runs that module’s top-level rebinding code. There is no decorator, no registry, no apply_patch(fn) helper. The mechanism is deliberately primitive:

  1. Capture the real original at module top: _original = SomeClass.method / _original_FusedMoE = layer.FusedMoE.
  2. Define a replacement (function, lambda, or subclass).
  3. Rebind the symbol: SomeClass.method = _replacement / pkg.FusedMoE = _ascend_FusedMoE.

This is the classic “import-time monkey-patch”. The only sophistication is ordering: the __init__.py imports are sequenced so a patch that depends on another’s rebinding runs after it (e.g. patch_fused_moe runs before model loading; patch_profiling_chunk wraps EngineCore.__init__ and then wraps the multiprocess entry point).

1.2 Global vs targeted (worker) patches

Phase When Where called Scope
Global / platform Once, before any worker/engine exists NPUPlatform.pre_register_and_update() (platform.py:184-188); also re-armed through plugin entry points register_connector/register_model_loader/register_service_profiling via _ensure_global_patch() (__init__.py:21-35) with a _GLOBAL_PATCH_APPLIED guard Process-wide, affects scheduler, engine-core, config validation, platform APIs
Worker / targeted Once per worker process, at the very top of NPUWorker.__init__ (worker.py:108-111) adapt_patch() (no arg = worker) Per-worker; affects model forward, attention, MoE, sampling, distributed comms, weight loading

Why two phases? Vanilla vLLM’s VllmConfig validation, scheduler construction, and engine-core run loops happen before a worker exists, and some validations (quant, CUDA graph, speculative whitelist, KV cache block sizes) run at config-parse time. Those must be patched globally or the process crashes before the worker even starts. Worker-phase patches are the ones that need local_rank, device, or per-process state, or that must run after ops.register_dummy_fusion_op() / _register_atb_extensions().

1.3 The multiprocess-spawn re-patch problem

A recurring motif (documented in patch/__init__.py and implemented in patch_profiling_chunk.py): in spawn mode the child process starts a fresh interpreter, so class-level monkey-patches applied in the parent are lost. The fix pattern is to wrap the subprocess entry point so unpickling re-imports the patch module:

# patch_profiling_chunk.py:221-229
_original_run_engine_core = EngineCoreProc.run_engine_core
def _patched_run_engine_core(*args, **kwargs):
    _apply_profiling_patches()          # re-apply in the child
    return _original_run_engine_core(*args, **kwargs)
EngineCoreProc.run_engine_core = _patched_run_engine_core

This is the same reason _ensure_global_patch() exists in __init__.py — vLLM loads general plugins inside engine-core subprocesses, so global patches must be re-applied there.

1.4 Why a hardware plugin NEEDS this many patches

Vanilla vLLM’s platform abstraction (vllm.platforms) was historically CUDA-first. The “Hardware pluggable” RFC exposes clean hooks for only a subset of the stack. Everything else — the scheduler loop, the config validators, the torch.distributed collectives, the torch.accelerator memory API, the FusedMoE factory, the rejection sampler, the tool-call parsers — is still reached by direct imports of concrete symbols. Until every one of those becomes a backend hook, an Ascend plugin has exactly two choices: fork, or monkey-patch. vllm-ascend chose monkey-patch and keeps each patch small, documented, and tagged with a “Future Plan: Remove this patch when upstream merges PR #XXXX” — see the exhaustive patch/__init__.py registry (lines 29-1200), which is itself the canonical index of “what is not yet pluggable in vLLM”.

2.1 The most important category: “Replace CUDA semantics” + “Enable Ascend features”

These two are the load-bearing core. Together they explain why vllm-ascend is a plugin, not a fork: rather than rewriting vLLM, it surgically replaces the CUDA-assuming leaves.

  • patch_torch_accelerator.py is the cleanest example of the whole thesis. Vanilla vLLM (commit 747b068) replaced current_platform.memory_stats() with torch.accelerator.memory_stats(), which does not delegate to NPU. Four lines rebind the entire memory-observation surface: 

    torch.accelerator.empty_cache = patch_empty_cache            # → torch.npu.empty_cache()
torch.accelerator.memory_stats = torch.npu.memory_stats
torch.accelerator.memory_reserved = torch.npu.memory_reserved
torch.accelerator.reset_peak_memory_stats = torch.npu.reset_peak_memory_stats
torch.accelerator.get_memory_info = torch.npu.mem_get_info    # NPU caching allocator is not a DeviceAllocator
    The comment is the giveaway: “NPU’s caching allocator is not a DeviceAllocator, so it crashes with ‘Allocator for npu is not a DeviceAllocator’.” — a CUDA-specific assumption baked into torch.accelerator.

  • patch_fused_moe.py (platform) is the single highest-leverage patch: it replaces the FusedMoE factory in both fused_moe.layer and fused_moe package with a wrapper that injects AscendMoERunner / AscendMoERunner310 as the default runner_cls, and forwards Ascend-only kwargs (tid2eid, drops DeepSeek hash). Every model that does from vllm.model_executor.layers.fused_moe import FusedMoE (deepseek_v2, etc.) silently gets the Ascend MoE path. The worker-side patch_fused_moe.py just re-imports the platform one to guarantee the binding holds during worker init.

  • _hccl_pg_registry.py is the deepest “replace CUDA semantics” patch: a full HCCL process-group reuse registry keyed by (backend, ranks, normalized hccl_config, reuse_domain), with refcounting and acquire/release/clear. This exists because HCCL groups are expensive and vanilla vLLM creates a fresh NCCL group per GroupCoordinator; Ascend reuses them. It even fail-closed on unknown non-default pg_options fields.

3. Top-3 most impactful patches — deep dive

3.1 patch_mla_prefill_backend.py — the cleanest architectural workaround

Problem. vLLM PR #32623 introduced MLAPrefillBackend. MLAAttention.__init__ calls get_mla_prefill_backend(), whose selector sees Ascend returns None for get_device_capability() and falls back to FlashAttnPrefillBackend, which asserts flash_attn_varlen_func is available — crashing on Ascend.

Why a patch is the right fix. Ascend’s AscendSFAImpl/AscendMLAImpl handle the full forward (including prefill) via impl.forward(); prefill_backend.run_prefill_* is never called. So the correct backend is a no-op that exists only to satisfy the constructor.

The patch (patch/platform/patch_mla_prefill_backend.py): 

class AscendMLAPrefillBackend(MLAPrefillBackend):
    @staticmethod
    def get_name() -> str: return "ASCEND"
    @classmethod
    def is_available(cls) -> bool: return True
    def run_prefill_new_tokens(self, q, k, v, return_softmax_lse):
        raise NotImplementedError("Ascend MLA prefill is handled by AscendSFAImpl/AscendMLAImpl")
    def run_prefill_context_chunk(self, chunk_idx, q, k, v):
        raise NotImplementedError("Ascend MLA prefill is handled by AscendSFAImpl/AscendMLAImpl")

vllm.model_executor.layers.attention.mla_attention.get_mla_prefill_backend = \
    lambda vllm_config: AscendMLAPrefillBackend

BEFORE (vanilla selector): get_device_capability() is None → FlashAttnPrefillBackend → assert flash_attn_varlen_func → crash.

AFTER (patched): selector returns AscendMLAPrefillBackend, constructor completes, and the no-op methods are never invoked because the Ascend MLA impl owns the whole forward. This is the canonical “vllm-ascend registers a backend stub so the upstream abstraction can run” pattern.

3.2 patch_profiling_chunk.py — the most structurally complex patch

This is the one that touches three symbols and solves the spawn-mode re-patch problem. It implements profiling-based dynamic chunk sizing: a one-shot profiling pass after model_executor is ready, then per-step latency feedback to a ProfilingChunkManager.

BEFORE (vanilla EngineCore): __init__ builds the scheduler and exits; update_from_output applies scheduler output; run_engine_core is the subprocess entry. No timing feedback exists.

AFTER (patched, patch/platform/patch_profiling_chunk.py): 

# 1) __init__ now boots the profiling pass and wraps two scheduler methods
def _patched_engine_core_init(self, *args, **kwargs):
    original_init(self, *args, **kwargs)
    if hasattr(self.scheduler, "run_profiling_chunk_init"):
        self.scheduler.run_profiling_chunk_init(self.model_executor)
    _ensure_update_from_output_wrapped(self.scheduler)   # record execution_time_ms
    _ensure_schedule_wrapped(self.scheduler)             # propagate timing-done flag
EngineCore.__init__ = _patched_engine_core_init

# 2) Subprocess entry re-applies the above inside the child process
_original_run_engine_core = EngineCoreProc.run_engine_core
def _patched_run_engine_core(*args, **kwargs):
    _apply_profiling_patches()
    return _original_run_engine_core(*args, **kwargs)
EngineCoreProc.run_engine_core = _patched_run_engine_core

The wrapper _record_execution_timing reads model_output.execution_time_ms (set by the NPU model runner), decomposes scheduler_output into per-request (chunk_size, hist_seq_len) pairs, bootstraps a target latency from the first 3 samples, then feeds ProfilingChunkManager.record_batch_execution_time for online fitting of a history-aware latency model. Once calibrated it flips need_timing=False end-to-end via a disable_profiling_timing flag on SchedulerOutput. This is a cross-process, cross-class feedback loop that vanilla vLLM has no hook for — hence the triple patch.

3.3 patch_fused_moe.py (platform) — the highest-leverage single patch

BEFORE (vanilla): deepseek_v2 does from vllm.model_executor.layers.fused_moe import FusedMoE and calls the upstream factory directly → CUDA MoE path.

AFTER (patch/platform/patch_fused_moe.py): 

_original_FusedMoE = _fused_moe_layer.FusedMoE
_DefaultAscendMoERunner = AscendMoERunner310 if is_310p() else AscendMoERunner

def _ascend_FusedMoE(*args, runner_cls=None, runner_args=None, **kwargs):
    if runner_cls is None:
        runner_cls = _DefaultAscendMoERunner
    kwargs.pop("hash", None)                          # DeepSeek V4 flag, consumed earlier
    tid2eid = kwargs.pop("tid2eid", None)             # Ascend-only → runner_args
    if tid2eid is not None:
        runner_args = dict(runner_args or {}); runner_args["tid2eid"] = tid2eid
    return _original_FusedMoE(*args, runner_cls=runner_cls, runner_args=runner_args, **kwargs)

_fused_moe_layer.FusedMoE = _ascend_FusedMoE
_fused_moe_pkg.FusedMoE = _ascend_FusedMoE

Two subtle points make this load-bearing: (1) it patches both the layer module and the package binding, because different call sites import FusedMoE from different paths; (2) it translates Ascend-only kwargs (tid2eid) and drops CUDA-only ones (hash) so the upstream factory signature stays satisfied. The worker-side patch_fused_moe.py is just import vllm_ascend.patch.platform.patch_fused_moe to guarantee the rebinding survives into the worker process. This is the patch that makes DeepSeek V2/V3 MoE actually run on NPU.

4. Device & allocator

4.1 Device layer (device/)

device/__init__.py is empty; the real surface is device/device_op.py — a 1000+ line BaseDeviceAdaptor (static method per op) that is the Ascend analogue of a CUDA kernel library. It is the op-dispatch backbone for the whole attention/MoE/KV-cache stack:

  • KV-cache scatter/gather: reshape_and_cache → torch_npu.npu_scatter_pa_kv_cache; kv_cache_load → torch_npu.npu_gather_pa_kv_cache.
  • Attention: npu_fused_infer_attention_score (FIA TND path), npu_flash_attention (_npu_flash_attention_unpad), npu_moe_init_routing, moe_gating_top_k, npu_moe_token_unpermute.
  • MLA preprocess: mla_preprocess_only_decode and sfa_preprocess_with_mlapo dispatch between torch_npu.npu_mla_prolog_v2 (quant) and torch.ops._C_ascend.mla_preprocess (non-quant), with cache_mode="PA_NZ" / "nzcache" / "krope_ctkv".
  • Sparse attention (DSA): execute_sparse_flash_attention_process, execute_kv_quant_sparse_flash_attention, indexer_select_post_process (three indexer branches: sparse_c8 / torch_npu_lightning_indexer / custom op), plus the full quant+scatter pipeline (indexer_quant_scatter, dsa_kv_compress_scatter, dsa_indexer_scatter_scale_part3, warmup_indexer_quant_scatter).
  • GDN / linear-attn / normalization: fused_gdn_gating, split_qkv_rmsnorm_rope, npu_gemma_rms_norm, apply_dsa_q_rms, and the triton chunk_scaled_dot_kkt_fwd / solve_tril_16x16 kernels.
  • MXFP quant: npu_dynamic_quant, npu_grouped_matmul_swiglu_quant, npu_grouped_matmul_gmm2 (all gated: MXFP MoE is A5-only, else RuntimeError).

device/utils.py holds the large-head prefill fallback (npu_large_head_prefill_attention) for the A2/A3 FIA-TND 512-dim head gap. device/mxfp_compat.py defines the MXFP quant dtypes.

How this differs from CUDA: CUDA vLLM mostly calls cuDNN/FlashInfer/Triton directly from attention backends. Ascend routes through torch_npu.* and torch.ops._C_ascend.* with explicit cache_mode/quant_mode/layout_* enums (FRACTAL_NZ, PA_BSND, TND) that have no CUDA equivalent — the device layer is a format+kernel translation shim for CANN.

4.2 Allocator (device_allocator/)

Two allocators, both for sleep mode (the Ascend answer to CUDA’s CuMem sleep mode):

camem.py — CaMemAllocator (a singleton NPUPluggableAllocator): - Backed by a C++ extension vllm_ascend_C exposing init_module, python_create_and_map, python_unmap_and_release (ACL runtime memcpy-based mmap). - use_memory_pool(tag) context routes every tensor through a tagged pool; sleep(offload_tags) ACL-memcpy’s tagged tensors to pinned CPU backup and unmap_and_releases the device handles; wake_up(tags) remaps and copies back. - Explicitly rejects expandable_segments:True (PyTorch issue #147851) and works around PyTorch issues #146431 / #145168. This is the NPU-native analogue of vLLM’s CUDA CuMem allocator, but built on ACL memcpy + NPUPluggableAllocator instead of CUDA virtual memory.

sleep_mem_optimized.py — SleepWakeupManager (the HCCL-aware layer): - AclGraphSleepWakeupManager: on sleep, clears attention workspaces and resets all acl_graph._graph_params + concrete graph entries; on wakeup, re-runs model_runner.capture_model(). - HcclSleepWakeupManager: on sleep, destroys all live HCCL process groups (iterates parallel_state._groups, calls group.destroy_hccl()); on wakeup, restores them and refreshes MoE HCCL groups via moe_comm_method.*.token_dispatcher.refresh_hccl_group().

Differences from CUDA caching allocator: 1. CUDA’s caching allocator reuses freed blocks in-place; Ascend sleep mode unmaps the device handle entirely and backs up to host — it is offload, not cache reuse. 2. It is HCCL-aware: CUDA sleep mode doesn’t tear down NCCL comms; Ascend must, because HCCL groups hold device memory that would otherwise leak across sleep/wake. 3. It is graph-aware: ACL graph workspaces and captured graphs must be torn down before sleep and recaptured after — a concern unique to the Ascend graph-capture path.

5. Profiler (profiler/)

profiler/torch_npu_profiler.py — TorchNPUProfilerWrapper(WorkerProfiler) wires vanilla vLLM’s WorkerProfiler interface to torch_npu.profiler:

  • Scope: CPU + NPU activities; ProfilerLevel.Level1; AiCMetrics.PipeUtilization (pipeline-bubble visibility); ExportType.Text; memory profiling optional (torch_profiler_with_memory); stack/modules off by default (the comment notes with_stack adds “significant time overhead” on NPU).
  • Guard: refuses to run alongside MSMONITOR_USE_DAEMON (the Ascend msmonitor daemon profiler) — they are mutually exclusive.
  • Output: tensorboard_trace_handler(torch_profiler_dir, worker_name=trace_name) — TensorBoard-compatible traces per worker.
  • Lifecycle: _start/_stop delegate to the torch_npu.profiler.profile context; _profiler_step returns True (step-based stepping is a no-op at the wrapper level).

Ascend profiling scope vs CUDA: CUDA vLLM uses torch.profiler (CUPTI/Nsight). Ascend uses torch_npu.profiler with CANN-level PipeUtilization and explicit data_simplification=True / op_attr=False to keep overhead manageable. HCCL timings and op-level traces come through the CANN integration rather than CUPTI. Flame-graph support is via the TensorBoard trace handler (the torch_npu.profiler equivalent of Nsight/Perfetto).

6. Xlite (xlite/)

6.1 What xlite is

xlite is NOT a new edge runtime. It is a graph-based, whole-model fusion execution mode layered on top of the standard NPUModelRunner, backed by an external C++ library xlite._C exposing Model, ModelConfig, Runtime, AttnMeta, AttnMHA, and scoring functions (ScoringFuncSoftmax, ScoringFuncSigmoid). Think of it as the Ascend analogue of a fixed-fusion “metal path”: the model’s weights and config are extracted once into an XModel, a per-rank Runtime owns a pre-allocated tensor pool, and each forward step runs xlite_model.forward(rt, input_ids, attn_meta, kv_caches, freq_cis, hidden_states, stream).

6.2 Architecture

  • XliteModel (ABC) + _architecture_strategy_map: each architecture subclasses it, implements _build_model_config / _build_model / _precompute_freqs_cis, and is auto-registered by _supported_architectures. Implementations:
  • LlamaXliteModel — Llama/Qwen2/Qwen3/Qwen3VL (dense, ScoringFuncSoftmax).
  • QwenMoeXliteModel — Qwen3Moe / Qwen3VLMoe.
  • Glm4MoeXliteModel — Glm4Moe (ScoringFuncSigmoid, partial rotary, shared experts).
  • MiniMaxM2XliteModel — MiniMaxM2.
  • XliteWrapper: wraps the runnable model; register_kv_caches; in __call__ it dispatches:
  • in_profile_run + full_mode → return preallocated hidden states (no double alloc).
  • non-AscendMetadata → fall back to the eager runnable.
  • DecodeOnly/SpecDecoding (or prefill in full_mode) → run the xlite graph.
  • prefill in decode-only non-DP mode → fall back to runnable.
  • DP + non-full-mode → use num_tokens <= num_reqs as the decode heuristic.
  • XliteWorker(NPUWorker) overrides init_device to build an XliteModelRunner.
  • XliteModelRunner(NPUModelRunner) wraps self.model = XliteWrapper(self.model, ...) and builds attention metadata during DP dummy runs when xlite is enabled.

6.3 Role in the system

xlite is an opt-in execution mode (ascend_config.xlite_graph_config.enabled, full_mode). It is the path that trades the eager/PyTorch-forward model for a C++-runtime, graph-captured forward with a preallocated tensor pool — the Ascend answer to CUDA’s “CUDA graph + FlashInfer fixed-fusion” decode path, but implemented as a separate xlite._C library rather than via vLLM’s own CUDA graphs. It coexists with the standard NPUModelRunner path and the ACL graph path (compilation/acl_graph.py).

7. CPU binding (cpu_binding.py)

7.1 Why per-device binding matters for multi-NPU

Ascend servers (A2/A3/310P/A5) pack many NPU chips sharing a NUMA-socket with dozens of ARM cores. Each NPU has dedicated IRQ threads (sq_send_trigger_irq, cq_update_irq), an acl_thread, and a release_thread. Without binding, a vLLM worker’s main thread, its ACL threads, and the NPU IRQs contend across cores and cross NUMA nodes — costing latency and jitter on the critical prefill/decode path. bind_cpus(rank_id) pins each worker to a per-NPU CPU pool and migrates its memory to that NPU’s NUMA node.

7.2 Topology modes

DEVICE_BINDING_MODE selects the strategy per silicon:

Device Mode Rationale
A2 topo_affinity Use npu-smi info -t topo NPU→CPU affinity
A3 global_slice Slice allowed_cpus by global logical NPU id (multiple DP processes may share one cpuset; global slicing avoids overlapping partitions)
310P topo_affinity Same as A2
A5 (Ascend 950) topo_affinity + 950-specific cluster pooling 8-physical-core clusters, no IRQ reservation, uvb_poll_window_thread binding
  • topo_affinity (_build_topo_affinity_cpu_pool): reads npu-smi info -t topo NPU→CPU map, extends each pool into the adjacent NUMA node (extend_numa), groups NPUs by identical CPU sets, and average_distributes when multiple NPUs map to the same pool. Non-visible NPUs are still considered to avoid cross-process overlap.
  • global_slice (build_global_slice_cpu_pool): ignores topo; partitions allowed_cpus into total_npus global slices by logical id, enforcing min_cpus_per_npu (5 with IRQ, 3 without). Deterministic and cpuset-safe.

7.3 Role split and IRQ binding

allocate() splits each NPU pool into roles: - default (A2/A3/310P): main = pool[2:-2], acl = [pool[-2]], rel = [pool[-1]] (first two cores reserved for IRQ). - A5 (Ascend 950): whole pool is main; ACL/release threads are not reserved.

bind_threads() pins the main PID (and its acl_thread/release_thread TIDs via taskset -acp) and runs migratepages to bind memory to the NPU’s NUMA node. bind_npu_irq() stops irqbalance, finds the NPU’s PCI MSI-IRQs, and writes /proc/irq/<irq>/smp_affinity to pin sq_send_trigger_irq/cq_update_irq to the first two reserved cores. run_all() = build pools → allocate → print plan → bind threads → bind IRQ.

The entry point is bind_cpus(rank_id) (cpu_binding.py:801), gated on is_arm_cpu() (x86_64 skips binding entirely), called from NPUWorker when ascend_config.enable_cpu_binding.
