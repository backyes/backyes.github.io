vllm-ascend 架构分析 — 分布式通信、KV 传输、权重传输与 EPLB

Slice: distributed/（device_communicators / kv_transfer / weight_transfer）、eplb/、memcache_comm_fence.py、ascend_forward_context.py（MoECommType）。 深度重点: KV-cache PD 分离传输、EPLB 专家负载均衡。

1. HCCL 通信器层（device_communicators）

Ascend 的集合通信原语由 HCCL（Huawei Collective Communication Library）提供，语义上对标 NVIDIA 的 NCCL。vllm-ascend 在 distributed/device_communicators/ 中封装了两层抽象。

1.1 底层 ctypes 直绑：pyhccl_wrapper.py

HCCLLibrary 通过 ctypes.CDLL 直接加载 HCCL 共享库（路径由 find_hccl_library() 或 HCCL_SO_PATH 环境变量决定），不依赖 torch.distributed 的 HCCL backend。它显式导出了 5 个 HCCL C 函数：

导出函数 HCCL C 签名对应 用途
HcclGetErrorString const char* HcclGetErrorString(HcclResult) 错误码转字符串
HcclGetRootInfo HcclResult HcclGetRootInfo(HcclRootInfo*) rank 0 生成唯一 ID
HcclCommInitRootInfo HcclResult HcclCommInitRootInfo(nRanks, rootInfo, rank, comm) 基于 rootInfo 初始化 communicator
HcclAllReduce HcclResult HcclAllReduce(sendBuf, recvBuf, count, dataType, op, comm, stream) AllReduce
HcclBroadcast HcclResult HcclBroadcast(buf, count, dataType, root, comm, stream) Broadcast
HcclCommDestroy HcclResult HcclCommDestroy(comm) 销毁 communicator

类型系统通过 hcclDataTypeEnum（支持 int8/16/32/64、uint8/16/32、fp16/32/64、bf16、int128）和 hcclRedOpTypeEnum（Sum/Prod/Max/Min）做 torch → HCCL 的映射。hcclUniqueId 是 4108 字节的 opaque 结构体（对标 NCCL 的 ncclUniqueId）。

关键设计点： - 库缓存：path_to_library_cache / path_to_dict_mapping 两个 class attribute 避免同一 .so 被重复加载。 - 仅导出 AllReduce + Broadcast：没有 AllGather、ReduceScatter、AllToAll 的直接导出。这意味着更复杂的集合通信要么通过 torch.distributed 的 HCCL backend 走（在 NPUCommunicator.all_to_all 中可见），要么通过更高层封装（如 MC2、FlashComm）实现。

1.2 PyHcclCommunicator：pyhccl.py

PyHcclCommunicator 是 HCCLLibrary 的面向对象封装，绑定到单一 NPU device（构造时强制 in_tensor.device == self.device，否则报 “illegal memory access”）。

初始化流程： 1. 接受 ProcessGroup | StatelessProcessGroup。若是普通 ProcessGroup，断言其 backend 不是 HCCL（"should be attached to a non-HCCL group"）—— 这是为了与 torch 的全局 HCCL 进程组隔离，典型场景是 EPLB 权重传输、权重热更新等需要独立通信域的场景。 2. rank 0 调用 hcclGetUniqueId() 生成 ID，通过 dist.broadcast（或 StatelessProcessGroup.broadcast_obj）分发给组内所有 rank。 3. 各 rank 在 torch.npu.device(device) 上下文中 hcclCommInitRank 创建 communicator。 4. Warmup：做一次小 tensor AllReduce + stream.synchronize()，避免首次通信冷启动延迟。

暴露的接口：all_reduce(tensor, op, stream)、broadcast(tensor, src, stream)。两者都接受可选的 stream 参数（默认 current_stream()），支持异步执行。

1.3 NPUCommunicator：npu_communicator.py

NPUCommunicator(DeviceCommunicatorBase) 是 vLLM 主框架要求的设备通信器。其特点： - ca_comm = None：没有自定义 AllReduce（custom AR）实现，注释明确说 “Refer to CudaCommunicator’s implementation to integrate PyHcclCommunicator”。这意味着 Ascend 上未走 custom AR 的 kernel 融合路径。 - all_to_all 通过 dist.all_to_all（torch.distributed，走 HCCL backend）实现，支持 scatter/gather sizes。 - _NpuAll2AllManager 是 no-op 的 fault-tolerance manager（support_fault_tolerance = False），因为 Ascend 用 MC2 / AllGather 做 MoE 通信，不注册真正的 all2all_manager。

1.4 HCCL vs NCCL 语义差距与桥接

维度 NCCL（CUDA） HCCL（Ascend） vllm-ascend 桥接方式
自定义 AR custom AR kernel（如 DeepEP） 无对应 ca_comm = None，未桥接
P2P 通信 ncclSend/ncclRecv HCCL 不直接暴露 P2P 通过 torch.distributed 的 isend/irecv（EPLB D2D loader）或 Mooncake TransferEngine（RDMA，KV 传输）
RDMA/高速互联 NVLink + NVSwitch + RDMA HCCS（Huawei Cache Coherence System，片间/模块内高速一致性互联）+ RDMA HCCL 内部封装了 HCCS/RDMA，未在 Python 层显式暴露；KV 传输层通过 Mooncake 引擎直接利用 RDMA
进程组隔离 多 ncclComm_t 多 hcclComm_t StatelessProcessGroup + PyHcclCommunicator 创建独立通信域

结论：HCCL 层在 Python 侧仅显式暴露了 AllReduce 和 Broadcast 两个集合原语。HCCS/RDMA 等底层传输对 Python 层是不透明的（由 HCCL 内部选择），只有 KV 传输层通过 Mooncake 引擎绕开 HCCL、直接操作 RDMA。

2. KV-cache 传输（PD 分离）— 深度重点

vllm-ascend 实现了 Prefill-Decode（PD）分离架构：Prefill 节点生成 KV-cache，Decode 节点拉取 KV-cache 做自回归生成。这是 vLLM 社区 disaggregation RFC 的 Ascend 实现，但传输层走 Mooncake TransferEngine（基于 RDMA），而非 vanilla vLLM 的 NixlConnector（基于 NIXL/GDR）。

2.1 连接器全景（kv_transfer/__init__.py 注册）

通过 KVConnectorFactory.register_connector 注册了多类连接器：

连接器 类型 说明
MooncakeConnectorV1 P2P 直传 主路径，基于 Mooncake RDMA
MooncakeHybridConnector P2P 直传 混合 KV cache 布局变体
MooncakeLayerwiseConnector P2P 逐层 逐层传输，与 attention 计算流水线重叠
AscendStoreConnector Pool（KV Pool） 基于共享内存池的 KV 缓存，支持 mooncake/memcache/yuanrong 后端
UCMConnector Pool 统一缓存管理
LMCacheAscendConnector Pool LMCache 适配
AscendSimpleCPUOffloadConnector CPU offload 覆盖上游实现，用 aclrtMemcpyBatchAsync
RecomputeCPUOffloadConnector CPU offload + 重计算 抢占恢复场景
AscendMultiConnector 多连接器组合 继承上游 MultiConnector，Ascend 覆盖版

AscendMultiConnector 的关键增强： - 要求所有子连接器支持 HMA（Hybrid Memory Attention，SupportsHMA），否则断言失败。 - get_num_new_matched_tokens 优先处理 has_preempted_request（重计算 offload 的未哈希部分块）。 - request_finished_all_groups 聚合多个子连接器的 async_save 与 kv_txfer_params。

2.2 MooncakeConnector 生命周期（核心 P2P 路径）

MooncakeConnector(KVConnectorBase_V1, SupportsHMA) 严格遵循 vLLM v1 连接器设计，分 Scheduler 侧 和 Worker 侧 两个角色。

2.2.1 初始化与握手（Worker 侧）

MooncakeConnectorWorker.__init__ 中： 1. 从 kv_transfer_config 的 extra_config 读取 prefill/decode 的 tp_size、dp_size、pp_size（PD 两侧可不对称，但要求 prefill_tp_size >= decode_tp_size，decode_pp_size == 1）。 2. 通过 global_te.get_transfer_engine(hostname, device_name) 获取 Mooncake TransferEngine 单例。Mooncake 引擎初始化走 "P2PHANDSHAKE" 模式 + "ascend" 设备标识。 3. 计算 handshake_port（基于 kv_port + dp_rank * tp * pp * pcp + device_index），确保每个 NPU 有独立端口。 4. 计算 tp_num_need_pulls：DeepSeek MLA 模型为 1；否则 num_d_block_heads // num_p_block_heads（decode 侧每块需从 prefill 侧拉的 TP 分片数）。

2.2.2 register_kv_caches：元数据注册

这是 KV 传输的核心准备步骤：

  1. 构建 kv_group2layeridx：将 KV cache group 按 spec 类型（FullAttention / MLA / Mamba / SlidingWindow / UniformType）拆分为 transfer group，每个 group 序列化为 {layer_names, kv_cache_spec_type, kv_cache_spec, num_kv_heads} 字典 + 物理 layer 索引列表。支持 MTP（speculative decoding）层的 ID 重分配。
  2. 逐层注册元数据：为每个 layer 的每个 cache tensor（K 或 V）计算： - kv_caches_base_addr[layer][cache]：data_ptr() 基址 - block_size_scale：tensor 块数 / 逻辑块数 - block_len_per_addr：单块字节数 - block_stride_per_addr：块间字节步长
  3. 注册内存到 Mooncake：根据布局类型选择注册策略： - Mamba 组 → _get_registered_kv_tensor_buffers（按 tensor 整体注册） - Hybrid（HMA）→ _get_registered_kv_tensor_buffers_hybrid - 普通 attention → collect_storage_merged_register_regions（合并共享底层存储的注册区域，避免超过 HCCL 每进程 region 上限） - 要求基址 2MB 对齐（base_addr % (2*1024*1024) == 0） - 最终调用 global_te.register_buffer(ptrs, lengths) 注册到 Mooncake 引擎。
  4. 构建握手元数据 MooncakeAgentMetadata（msgspec Struct）：包含 engine_id、te_rpc_port、kv_group2layeridx、block_size、kv_caches_base_addr、block_size_scale、num_blocks、block_lens、block_strides、local_ip、handshake_port。
  5. 启动收发线程：kv_producer 启动 KVCacheSendingThread，kv_consumer 启动 KVCacheRecvingThread，等待 ready_event（超时 5 分钟）。

2.2.3 传输生命周期（Scheduler → Worker）

Scheduler 侧（MooncakeConnectorScheduler）： 1. get_num_new_matched_tokens：若 kv_transfer_params["do_remote_prefill"]，返回 (actual - computed, True) —— 标记异步加载。 2. update_state_after_alloc：将需接收的请求加入 _reqs_need_recv。 3. build_connector_meta：将 _reqs_need_recv 转换为 MooncakeConnectorMetadata（含 ReqMeta 列表），清空待处理队列。 4. request_finished：对 do_remote_decode 的请求，延迟释放块，生成 kv_transfer_params（含 remote_block_ids、remote_engine_id、remote_host/port、remote_pcp/dcp_size 等），供下次 prefill 节点拉取。

Worker 侧传输核心（KVCacheRecvingThread._transfer_kv_cache_all_groups）：

# 关键传输逻辑（简化）
session_id = f"{remote_host}:{remote_transfer_port}"
src_list, dst_list, length_list = [], [], []

for group_pull in group_pulls:
    for layer_idx in layer_indices:
        for cache_idx in range(num_cache_tensors):
            # 计算 src/dst 物理地址（考虑 block_stride、inner_offset、tp_num_need_pulls 分片）
            src = src_layer_base_addr + local_block_id[0] * block_stride + inner_offset * inner_block_len
            dst = dst_layer_base_addr + remote_block_id[0] * remote_block_stride
            length = inner_block_len * len(local_block_id)
            src_list.append(src); dst_list.append(dst); length_list.append(length)

# 批量同步 RDMA 读（consumer 主动读 producer 侧 KV）
ret = self.engine.batch_transfer_sync_read(session_id, src_list, dst_list, length_list)

关键语义： - 传输方向是 batch_transfer_sync_read —— Decode 节点主动 RDMA 读取 Prefill 节点的 KV-cache（pull 模型），而非 prefill push。 - 地址完全由 kv_caches_base_addr + block_stride + block_len 在用户态计算，零拷贝（直接操作 NPU device memory 的 data_ptr）。 - 支持 CP（Context Parallelism） 的 block 映射：_local_kernel_ids_for_shard 处理 P/D 侧 block_size 不对称、CP rank 交错等复杂情况。 - 支持 Mamba 状态组（非 block 对齐，整体传输 conv/ssm state）。 - 支持 GQA 多 pull 重格式化：reformat_kv_cache_hybrid_linear_torch 将 [block, split, token, head_per_split, dim] 转回 [block, token, split, head_per_split, dim]。 - 支持 KV NZ 格式（Ascend 特定 fused attention 布局，enable_kv_nz）。

2.2.4 完成与资源回收

  • KVCacheTaskTracker 跟踪每个请求的传输状态，支持 延迟释放（delayed_free_requests）和 超时强制释放（VLLM_MOONCAKE_ABORT_REQUEST_TIMEOUT）。
  • 传输完成后发送 DONE_RECVING_MSG（zmq）通知 producer 释放资源。
  • get_finished 返回 (done_sending, done_recving) 集合，供 vLLM 调度器判断请求是否可调度。

2.3 AscendStoreConnector（Pool 路径）

AscendStoreConnector(KVConnectorBase_V1, SupportsHMA) 是另一条路径，基于 KV Pool（共享内存池）： - Scheduler 侧：KVPoolScheduler + LookupKeyServer（zmq REP 服务，处理 token hash 查找）。 - Worker 侧：KVPoolWorker + 多组 KVTransferThread（KVCacheStoreSendingThread、KVCacheStoreRecvingThread、KVCacheStoreLayerSendingThread 等）。 - 后端可插拔：MooncakeBackend、MemcacheBackend、YuanrongBackend（backend.py 抽象接口）。 - 支持 layerwise 模式（use_layerwise），逐层 save/load 与计算重叠。 - 使用 KVEventAggregator 聚合多 worker 的 KV cache 事件，仅保留 common events。

2.4 与 vanilla vLLM NixlConnector 的对比

维度 NixlConnector（vanilla vLLM） MooncakeConnector（Ascend）
传输引擎 NIXL（NVIDIA Inter-node eXchange Library，GDR/GPUDirect） Mooncake TransferEngine（RDMA，"P2PHANDSHAKE" 模式）
传输语义 push / pull 可配 pull 为主（batch_transfer_sync_read，consumer 主动读）
地址计算 NIXL 管理注册内存 用户态手动计算 data_ptr + block_stride * block_id，2MB 对齐
元数据握手 NIXL agent 协调 自研 MooncakeAgentMetadata（msgspec）+ zmq 侧信道
混合 KV 布局 标准 支持 HMA（Hybrid Memory Attention）、Mamba state、MLA、SlidingWindow、compress_ratio
CP 支持 有限 完整 P/D 不对称 block_size + CP rank 交错映射
重格式化 无 GQA multi-pull transpose、KV NZ fused 格式
Pool 模式 无 AscendStore（mooncake/memcache/yuanrong 后端）

核心差异：Ascend 的 Mooncake 路径是 用户态 RDMA（绕过 HCCL），直接操作 NPU 显存物理地址，因此需要做复杂的 block 地址计算和布局重格式化；而 Nixl 依赖 NVIDIA 的 GDR 生态，对上层更透明。

3. 权重传输引擎（weight_transfer）

权重传输用于 动态权重热更新：外部训练器（trainer）将更新后的模型权重广播到 vLLM 推理 worker。典型场景包括 RL（强化学习）的在线权重同步、MoE 专家热插拔（与 EPLB 配合）。

3.1 HCCL 引擎：hccl_engine.py

HCCLWeightTransferEngine 基于 HCCL Broadcast（trainer rank 0 → 所有 worker）：

  • 通信域隔离：通过 StatelessProcessGroup.create + PyHcclCommunicator 创建独立于推理通信域的 HCCL 组（trainer 是 rank 0，worker 从 rank_offset 开始）。
  • 两种传输模式：
  • 非 packed：逐参数 broadcast(weight, src=0)，简单但通信次数多。
  • Packed（默认推荐）：packed_broadcast_producer/consumer 将多个参数 cat 到 1GB 缓冲区（DEFAULT_PACKED_BUFFER_SIZE_BYTES = 1GB），用 2 缓冲（double buffering）+ 独立 NPU stream 实现广播与 pack 的流水线重叠，显著减少 HCCL 通信开销。
  • Layerwise reload：start_weight_update / finish_weight_update 调用 vLLM 的 initialize_layerwise_reload / finalize_layerwise_reload，支持逐层加载避免 OOM。
  • Trainer 侧：trainer_send_weights 静态方法，支持 dict 或 HCCLTrainerSendWeightsArgs 参数。

3.2 NPU IPC 引擎：npu_ipc_engine.py

NPUIPCWeightTransferEngine 基于 Ascend NPU IPC（进程间共享内存），适用于 trainer 与 worker 共置同一物理 NPU 芯片 的场景：

  • UUID 机制：npu_generate_uuid() 生成 {host_ip}-{physical_chip_id}，通过 ASCEND_RT_VISIBLE_DEVICES 映射逻辑设备到物理芯片。Trainer 和 worker 必须生成相同 UUID 才能匹配 IPC handle。
  • IPC handle 传递：通过 reduce_tensor 获取 IPC rebuild args，支持 ray 和 http 两种传输模式（http 模式下 IPC handle 经 base64 + pickle 序列化，需 VLLM_ALLOW_INSECURE_SERIALIZATION=1）。
  • 多 NPU 聚合：_all_gather_and_merge_handles 跨 rank all_gather IPC handle dict，使每个 worker 能找到自己 UUID 对应的 handle。
  • Packed 模式：packed_npu_ipc_producer/consumer 分块传输，bounded memory。
  • 无 layerwise reload：start_weight_update / finish_weight_update 是 no-op。

3.3 对比与用例

维度 HCCL 引擎 NPU IPC 引擎
传输介质 HCCL（HCCS/RDMA，跨节点） NPU IPC（共享内存，节点内）
适用场景 跨节点权重同步、RL 在线更新 共置 trainer+worker、低延迟热更新
通信模式 Broadcast（1 → N） IPC handle 共享（零拷贝）
内存效率 packed 1GB 缓冲 + double buffer packed 分块
Layerwise 支持 不支持

4. EPLB（Expert Parallelism Load Balancing）— 深度重点

EPLB 解决 EP（Expert Parallelism）MoE 中的专家负载不均衡 问题：路由器的 skew 导致部分 NPU 上的专家过热（hot expert），成为延迟瓶颈。Ascend 的 EPLB 是独立于 vanilla vLLM 的完整实现（注释多次提到 “Once vllm-project/vllm#24069 is merged, remove this”），包含 监控 → 决策 → 迁移 的闭环。

4.1 架构分层：core/ vs adaptor/

层 目录 职责
core/ eplb/core/ 核心算法与进程管理：策略（policy）、worker 子进程、D2D 加载器、工具函数
adaptor/ eplb/adaptor/ vLLM 模型适配：VllmEplbAdaptor 桥接 MoE 层参数、expert_map、moe_load

4.2 核心算法：四种策略（policy/）

PolicyFactory.generate_policy(policy_type) 根据 eplb_policy_type 选择策略：

type 策略 算法类别 说明
0 RandomLoadBalance 随机交换 每层随机（固定种子 42）交换 card 1 和 card 3 的最后一个专家。基线对照，非生产用
1 DefaultEplb 贪心装箱 + 冗余分裂 原始动态 EPLB 策略
2 SwiftBalanceEplb 冗余重分配 + 交换 V2：约束 shuffle 次数的局部交换
3 FlashLB 树搜索 + LPT 部署 联合优化、多 shot 增强、增量调整

4.2.1 DefaultEplb（type 1）—— 贪心装箱

核心循环（rebalance_experts）：

# 1. 聚合冗余专家的 workload 到原始专家
layer_workloads = self.add_redundant(placement_table, workload_table, num_original_expert)

# 2. 逐层计算全局平衡部署
for layer in range(layer_num):
    weights = [(expert_id, workload) for expert_id, workload in enumerate(layer_workloads[layer])]
    # 贪心分裂热专家为冗余专家，然后装箱到 num_npus 个 box
    result, layer_deployment = self.original_compute_balanced_pack_redundancy(
        weights, num_npus, num_redundancy_expert
    )
    global_deployment[layer] = layer_deployment

# 3. 约束局部交换（保持与旧部署的最大重叠）
new_global_deployment = self.constraint_expert_local_exchange(current_expert_table, global_deployment)

# 4. 按层优先级排序（改善比例最大的优先）
layer_changed_ratio = [max_heat_after / max_heat_before]
per_layer_priority = np.argsort(layer_changed_ratio)

# 5. 仅当整体热度下降 5% 以上才触发更新
change = 1 if npu_heat_all_after < 0.95 * npu_heat_all_origin else 0

original_compute_balanced_pack_redundancy 的贪心逻辑： - 每次将当前 workload 最高的专家分裂出一个冗余副本（route_expert_redundancy[hot_expert].append(new_id)），并重新计算该专家的 avg_weight = raw_weight / (redundancy_count + 1)。 - 然后将所有专家（含冗余副本）按 Best-Fit-Decreasing 装箱到 card_num 个 box：每次将专家放入当前总重量最轻且未满的 box。

4.2.2 SwiftBalanceEplb（type 2）—— 冗余重分配 + 交换

更精细的两阶段算法：

  1. 冗余重分配（redundancy_again）： - statistics_expert_distribution：统计每个 rank 上的冗余专家位置。 - compute_redundant_assignments：按 workload 贪心将冗余副本分配给热专家（new_avg_weight = original_weight * (r+1)/(r+2)）。 - distribute_redundant_experts：将冗余专家分配到负载最轻的 rank（约束 num_max_com 限制跨 rank 通信量）。 - fill_in_undeployed_ranks：填充未分配完的 rank。

  2. 专家交换（expert_exchange_between_ranks）： 

    while max_swap_times > 0:
    max_load_rank = argmax(rank_loads)
    for swap_rank_id in sorted_rank_idx[:-1]:
        if num_com_between_rank[swap_rank][max_rank] < num_max_com:
            # 找最优交换对（max_rank 的重专家 ↔ swap_rank 的轻专家）
            max_expert, swap_expert, max_weight = self.swap_experts_between_ranks(...)
            if max_rank_load - max_weight < swap_threshold: continue
            # 执行交换，更新 rank_loads 和 num_com_between_rank
    这是一个 局部搜索：每次找负载最重的 rank，尝试与轻 rank 交换专家以降低最大负载，直到收敛或达到 max_swap_times。

4.2.3 FlashLB（type 3）—— 树搜索 + LPT（最复杂）

FlashLB 是生产级策略，核心创新：

  1. 滑动窗口统计（register_hotness）：维护每层专家的 load 时间序列（hotness_window），支持增量更新（incremental_update_stats）和滑动更新（sliding_update_stats），计算 mu、var、cov（专家间协方差）。

  2. FlashTree 树搜索（optimize_balanceness）： - 将专家按 weight = mu + z_score * sqrt(var) 降序排列。 - 分 depth 层（默认 4），每层将专家分成 interval_size = ceil(num_experts / depth) 组。 - 对每组用 make_replica 分配冗余副本数（三种方法：min_max_replica / max_delta_replica / percentage_replica，均用 numba @njit 加速）。 - neighbor_search 在 [initial-max_range, initial+max_range] 范围内搜索最优副本数（width 默认 8）。 - 最终用 lpt_deployment（Largest Processing Time）贪心部署：每次将副本放到 风险最小（risk = mu + z_score * sqrt(var)，含协方差项）的设备。

  3. 最小化重部署（minimize_redeploy_with_inner_permutation）：用 匈牙利算法（linear_sum_assignment）将新部署与旧部署对齐，最小化专家移动。

  4. 触发条件（need_update）：当 average_to_peak_ratio < past_ratio * 0.9 或 < 0.85 时触发。

  5. 评分函数（compute_score）：(max_device_load * num_devices) / total_load，越低越好。

4.3 核心进程：EplbWorker 与 EplbProcess

EplbWorker 运行在独立子进程中（EplbProcess 通过 multiprocessing.Process 启动），通过 shared_dict（Manager().dict()）与主进程通信：

# EplbWorker.do_update 核心循环
def do_update(self):
    # 1. 读取当前 expert_map 和 moe_load
    old_expert_maps = self.get_init_expert_maps()          # shared_dict["expert_maps"]
    load_info = self.fetch_and_sum_load_info()            # shared_dict["moe_load"]

    # 2. 全局 → 局部 placement 转换
    old_placement = self.global2local(old_expert_maps, num_local_experts)

    # 3. 调用策略计算新部署
    changed, priority, new_placement = self.policy.rebalance_experts(old_placement, load_info)

    # 4. 校验（无重复专家、无 NPU 内移动）
    self.check_expert_placement(old_placement, new_placement)

    # 5. 局部 → 全局转换，计算每层的 send/recv 计划
    new_expert_maps = self.local2global(new_placement)
    update_info = self.compose_expert_update_info_greedy(new_expert_maps, old_expert_maps)

    # 6. 打包返回（send/recv/log2phy_map/layer_id）
    return self.pack_update_info(update_info)

compose_expert_update_info_greedy 生成每层的 P2P send/recv 计划： - dst_rank_indices, experts_to_recv = where((current == -1) & (updated != -1)) —— 需要接收的专家。 - src_rank_indices, experts_to_send = where((current != -1) & (updated == -1)) —— 需要发送的专家。 - 贪心选择源 rank（candidate_src_rank_indices[0]），TODO 提到未来考虑节点内/节点间拓扑。

4.4 D2D 权重迁移：D2DExpertWeightLoader

D2DExpertWeightLoader 执行实际的专家权重搬运，状态机：

WAITING → generate_expert_d2d_transfer_task() → READY
       → asyn_expert_weight_transfer()         → TRANSFERRING
       → update_expert_map_and_weight()        → WAITING
  • 传输方式：dist.batch_isend_irecv（torch.distributed P2P，走 HCCL backend），在 get_dynamic_eplb_group() 通信域内。
  • 异步执行：asyn_expert_weight_transfer 在异步流上发起 P2P，不阻塞主计算流。
  • 三要素更新：每层传输完成后，依次更新 expert_map（逻辑→物理映射）、log2phy_map（路由表）、expert_weight（实际权重 tensor，copy_ 到模型参数）。

4.5 编排器：EplbUpdator

EplbUpdator 是主进程侧的编排器，按 三阶段循环 驱动 EPLB：

[heat_collection_interval] → [algorithm_execution_interval] → [num_moe_layers (weight update)]
         ↑ 收集 moe_load      ↑ 唤醒子进程计算新部署      ↑ 逐层 D2D 迁移权重

关键方法： - forward_end：每个 forward 步调用，按 cur_iterations 判断当前阶段。 - wakeup_eplb_worker_flag：cur_iterations == heat_collection_interval - 1 时，all_gather 收集全局 moe_load 到 shared_dict，唤醒子进程。 - get_update_info_flag：子进程计算完成后取 update_info。 - update_expert_weight_flag：逐层执行 D2D 迁移。 - warm_up_eplb：预热 P2P 通信域（batch_isend_irecv 空 tensor）。

4.6 适配层：VllmEplbAdaptor

VllmEplbAdaptor 桥接 vLLM 模型与 EPLB 核心： - 注册机制：register_layer 静态方法在 MoE 层初始化时收集层引用（PPMissingLayer 不注册，自然过滤掉非本 PP rank 的层）。 - 参数索引：init_expert_param_per_layer 按量化类型（NONE/W8A8/W4A8/MXFP）和 fused MC2 开关选择权重名（w13_weight、w2_weight、scale、bias 等），建立 expert_param_per_layer[layer][expert] 索引。 - Buffer 预分配：init_buffer_tensor 预分配接收 buffer（torch.empty_like），避免传输时分配。 - 负载采集：get_rank_expert_workload 从每层的 layer.moe_load 采集负载。 - 更新执行：do_update_expert_map、do_update_log2phy_map、do_update_expert_weight 将新部署和权重写回模型。

4.7 通信域

EPLB 使用独立的 dynamic_eplb 进程组（get_dynamic_eplb_group()），在 parallel_state.py 中基于与 MC2 相同的 group_ranks 拓扑初始化，但独立于推理通信域，避免 P2P 权重传输干扰正常推理通信。

5. 通信/计算重叠：memcache_comm_fence

memcache_comm_fence.py 实现 AttentionComputeStartGate —— 一个连接 compute stream 与 MemCache 传输线程的门控机制。

5.1 问题背景

Ascend 上存在 异步通信与计算的重叠（如 KV 传输、权重 D2D）。MemCache worker 线程需要在 attention 计算开始前启动 H2D/L2D 传输，但如果仅依赖 Python 调用点触发（”Python call site reached”），此时 compute stream 可能尚未真正到达 attention 边界 —— 导致传输过早启动，数据未就绪。

5.2 机制

class AttentionComputeStartGate:
    def record(self, stream=None):
        # attention 路径在提交 attention op 前，在当前 compute stream 上 record 一个 NPU event
        event = torch.npu.Event()
        event.record(stream or torch.npu.current_stream())
        with self._condition:
            self._event = event
            self._condition.notify_all()

    def wait(self, timeout=10.0):
        # MemCache worker 线程等待该 event 完成（即 compute stream 真正到达 attention 边界）
        with self._condition:
            while self._event is None:
                self._condition.wait(timeout=timeout)
        event.synchronize()
  • record_attention_compute_start()：attention 计算路径调用，在提交 attention op 之前 record event。
  • wait()：MemCache worker 线程调用，阻塞直到 event 完成。
  • 逐层重置：reset_attention_compute_start_gate() 为每层创建新 gate，layerwise prefetch 任务持有当前 gate 的引用。

5.3 为什么 Ascend 需要显式门控

Ascend 的 NPU stream 模型与 CUDA 类似但 event 语义有差异。通过显式 event 门控，确保传输启动时机与 compute stream 的实际进度严格同步，避免： 1. 传输过早 → 数据在 buffer 中等待，浪费内存带宽。 2. 传输过晚 → attention 计算等待数据，增加延迟。

这是 Ascend 特有的 async comm/compute overlap 正确性保障机制。

6. MoECommType 与 EP 通信模式

ascend_forward_context.py 定义了 MoECommType 枚举，决定 MoE 层的专家并行通信策略：

class MoECommType(Enum):
    ALLGATHER = 0   # AllGather 方式（无 EP 或 EP=1 时的 fallback）
    MC2 = 1         # Matrix Compute Combine（华为定制的融合 AllGather+MatMul）
    ALLTOALL = 2    # AllToAll（标准 EP 通信）
    FUSED_MC2 = 3   # 融合 MC2（decode 阶段 w8a8_dynamic 量化优化）

6.1 选择逻辑（select_moe_comm_method）

按 芯片代际（A2/A3/A310P/A5）和 token 数 动态选择：

芯片 条件 选择 理由
任意 非 MoE 或 EP=1 ALLGATHER 无 EP 通信开销
A2 experts_per_device <= 24、EP >= 16、tokens <= mc2_capacity MC2 小专家+大 EP 时 MC2 效率高
A2 其他 ALLGATHER
A3 tokens <= capacity + w8a8 + 小 EP + 非 speculative FUSED_MC2 decode 阶段融合优化
A3 tokens <= capacity MC2
A3 大 token ALLTOALL
A5 tokens <= capacity + world_size > 1 MC2
A5 world_size <= top_k ALLGATHER
A5 其他 ALLTOALL
310P 任意 ALLGATHER 无 MC2 支持

6.2 MC2 对 EP 带宽/延迟的影响

MC2（Matrix Compute Combine） 是华为 Ascend 的定制原语，核心思想： - 将 AllGather（token 分发）+ MatMul（专家计算） 融合为一个算子，减少显存读写和通信启动次数。 - set_mc2_tokens_capacity 预计算 MC2 的 token 容量上限（min(ceil(max_tokens / tp_size), 512) * tp_size），并预分配 mc2_mask（bool tensor，标记实际 token 位置）。 - mmrs_fusion（npu_mm_reduce_scatter_base 融合）在 TP ≤ 8 时启用，但 MoE 模型强制关闭（mmrs_fusion = False）。

FUSED_MC2 进一步在 decode 阶段将 dispatch + FFN + combine 融合（dispatch_ffn_combine），要求 EP ≤ 32、w8a8_dynamic 量化。

带宽/延迟影响： - MC2 vs AllGather：MC2 减少了一次显存读写（AllGather 的输出直接进 MatMul），在 小 token、大 EP 场景下延迟显著降低（避免了 AllGather 的 all-to-all 通信瓶颈）。 - MC2 vs AllToAll：AllToAll 在大 token 时更优（通信量与 token 线性，但计算也线性）；MC2 在小 token 时更优（融合开销摊薄）。mc2_tokens_capacity 是切换阈值。 - FUSED_MC2：在 MC2 基础上再融合 combine 阶段，decode 阶段延迟进一步降低，但约束更严（量化类型、EP 大小、非 speculative）。

6.3 与 EPLB 的关联

  • forward_context.eplb_heat_collection_status 标志位控制 EPLB 的负载采集。
  • VllmEplbAdaptor 的 expert_weight_key 选择依赖 enable_fused_mc2（expert_weight_key = (quant_type, enable_fused_mc2 == 1)），影响 EPLB 权重迁移时传输哪些参数（fused MC2 时有额外的 fused_w1_scale_list / fused_w2_scale_list）。
  • EPLB 的 dynamic_eplb 通信组与 MC2 组共享相同的 group_ranks 拓扑（parallel_state.py），确保权重迁移的 P2P 路径与 EP 通信拓扑一致。
