vllm-ascend: Platform Entry / Core Scheduler & Plugin Infrastructure

Architecture analysis — plugin registration, NPUPlatform, forward-context extensions, and Ascend’s in-house scheduler innovations. Target audience: AI-systems / engine researchers.

1. Plugin registration sequence

vLLM exposes a set of named plugin hooks (register, register_connector, register_model_loader, …). vllm-ascend implements these in vllm_ascend/__init__.py to inject NPU behavior at four well-defined seams.

             ┌──────────────────────────────────────────────────────────┐
             │            vllm_ascend/__init__.py  (entry points)       │
             └──────────────────────────────────────────────────────────┘
                                      │
          ┌───────────────────────────┼───────────────────────────────┐
          ▼                           ▼                               ▼
   register() ★ first        register_connector()            register_model_loader()
   "vllm_ascend.platform     │  _ensure_global_patch()  ──►   │  _ensure_global_patch()
    .NPUPlatform"             │  kv_transfer.register_          │  netloader.register
    (platform string          │  connector()                    │  rfork.register_rforkloader()
     resolved by vLLM)        │  weight_transfer.register_      │
                              │  engine()                       │
                              ▼                                 ▼
                    register_service_profiling()      register_model()
                     _ensure_global_patch()            hunyuan_vl compat +
                    profiling_config.generate_         models.register_model()
                     service_profiling_config()

The canonical sequence and the role of each hook:

Hook Returns / side-effect Purpose
register() "vllm_ascend.platform.NPUPlatform" (string) The only hook called automatically on the main process, before engine-core spawns. It returns the platform class name; vLLM resolves it lazily and puts NPUPlatform in place as current_platform. This is the bootstrap domino — without it, none of the NPU overrides (device, KV cache, worker, attention backend, compile backend, etc.) ever wire up. Crucially it does not call _ensure_global_patch(), because it runs in the main process where e2e conftest patches might run instead.
register_connector() calls kv_transfer.register_connector() + weight_transfer.register_engine() Registers the Ascend KV/WD connectors (KV-offload / P-D disaggregation / weight broadcast). Requires global patch first because connector code references Ascend-patched kernels.
register_model_loader() register_netloader() + register_rforkloader() Hooks in Ascend’s net/rfork model-load paths. Also under _ensure_global_patch().
register_service_profiling() generate_service_profiling_config() Emits service_profiling_symbols.*.yaml for ms_service_profiler tracing hooks.
register_model() (import-time) installs HunyuanVL compat + registers model classes register_model() is invoked at import time of the package (last lines of __init__.py), not via a plugin hook.

Why _ensure_global_patch() guards the connector/loader hooks (__init__.py:20-35):

_GLOBAL_PATCH_APPLIED = False

def _ensure_global_patch():
    global _GLOBAL_PATCH_APPLIED
    if _GLOBAL_PATCH_APPLIED:
        return
    from vllm_ascend.utils import adapt_patch
    adapt_patch(is_global_patch=True)
    _GLOBAL_PATCH_APPLIED = True

The docstring is explicit: “vLLM loads general plugins in engine-core subprocesses. E2E test conftest hooks do not run there, so global patches that affect scheduler and engine code must also be applied through these plugin entry points.” In other words, the conftest that normally monkey-patches vLLM for Ascend is test-only code; in production the worker subprocess is spawned without it, so adapt_patch(is_global_patch=True) must run itself through these hooks. The module-level boolean reuses the main-process patch if already applied, so it is idempotent across hooks. The patch rewrites scheduler/utils/worker code ahead of any import vllm.… in the forks — without it the scheduler subclasses and Ascend kernels referenced in recompute / profiling-chunk / SRF paths would resolve to their vanilla CUDA implementations.

import vllm_ascend.logger # noqa at the bottom of __init__.py is a load-time side effect that installs Ascend’s logger configuration (handlers, formatter) before any other module logs.

2. NPUPlatform

NPUPlatform(Platform) (platform.py:128) implements vLLM’s PlatformBase out-of-tree via _enum = PlatformEnum.OOT. It overrides the interface almost exhaustively because Ascend’s device lifecycle differs from CUDA. Map of overrides and their lifecycle significance:

Identity device_name="npu", device_type="npu", dispatch_key="PrivateUse1", ray_device_key="NPU", device_control_env_var="ASCEND_RT_VISIBLE_DEVICES" Swaps the CUDA runtime, device enumeration, and Ray device key for NPU. VLLM_DISABLE_SHARED_EXPERTS_STREAM=1 is force-set at module import (platform.py:32) because Ascend’s shared-expert concurrent stream is unsupported.
simple_compile_backend="eager" Disables torch.compile() globally. Ascend uses its own ACL-graph / compile pipeline, not inductor.

check_and_update_config — the central config surgery

check_and_update_config (platform.py:410) is where the platform rewrites the VllmConfig in place. It is long but its structure is a sequence of surgical Ascend-specific passes:

  1. Bail-early guards: skip if device_type isn’t npu or model config is missing.
  2. Auto-detect quantization; validate DCP draft head-splitting; validate DP≠PCP mutual exclusion.
  3. _fix_incompatible_config (§10 numbered blocks): resets ~30+ GPU-only knobs (disable_cascade_attn, cpu_kvcache_space_bytes, mm_encoder_attn_backend, enable_layerwise_nvtx_tracing, max_num_partial_prefills, speculative quantization, kv_buffer_size, enable_permute_local_kv, 7 attention flags, flash_attn_version, ray_workers_use_nsight, numa_bind*, enable_dbo, ubatch_size, use_inductor_graph_partition). Notably it converts --numa-bind into --additional-config {"enable_cpu_binding": true}, routing the user’s intent to Ascend’s native CPU binding.
  4. init_ascend_config(vllm_config) → builds the AscendConfig singleton from additional_config.
  5. Compilation-mode rewrite: xlite graph, enforce_eager, splitting ops, HDK-950 graph-size pruning (prune_capture_sizes_for_950), SP-aligned capture sizes, encoder-decoder PIECEWISE fallback, ASCEND_LAUNCH_BLOCKING conflict.
  6. Worker selection: routes worker_cls to NPUWorker / NPUWorker310 / XliteWorker.
  7. Scheduler selection (see §4).
  8. Memory/affinity: sets PYTORCH_NPU_ALLOC_CONF=expandable_segments:True, enforces enable_mc2_hierarchy_comm XOR enable_fused_mc2.

Two directives are loaded at import-time before any method: VLLM_USE_BREAKABLE_CUDAGRAPH = False (platform.py:61) — DeepSeek V4’s PIECEWISE cudagraph is unsupported on Ascend.

3. Forward context extensions

ascend_forward_context.py extends vLLM’s per-step forward context with Ascend-specific MoE-comm dispatch and sequence-parallelism metadata.

MoECommType taxonomy

class MoECommType(Enum):
    ALLGATHER = 0   # no expert parallel, local-gather path
    MC2 = 1         # Matmul-Communication overlap v2 (fused dispatch/combine)
    ALLTOALL = 2    # standard expert-parallel dispatch/combine
    FUSED_MC2 = 3   # dispatch_{ffn,gmm}_combine MC2 variant

Why vanilla vLLM assumes CUDA-graph + AllGather: upstream vLLM’s MoE comm path is written around NCCL AllGather inside CUDA-graph-captured kernels, with a dense TP-shaped expert layout. Ascend has three divergences that force a runtime switch:

  1. Expert parallelism is first-class — A2/A3/A5 support EP with AllToAll / MC2; the comm pattern depends on device generation, world size, token count vs the MC2 capacity, quant type, and whether the node is a P or D.
  2. MC2 (Matmul-Comm-overlap) is an Ascend-specific fused “dispatch + matmul + combine” kernel with a bounded mc2_tokens_capacity; vLLM has no equivalent.
  3. Hierarchy — enable_mc2_hierarchy_comm and fused MC2 are mutually exclusive (platform.py:761).

Per-generation selection (select_moe_comm_method, ascend_forward_context.py:303-367)

select_moe_comm_method(num_tokens, vllm_config, is_draft_model)
  ├─ non-MoE  → None
  ├─ no EP    → ALLGATHER
  ├─ A2       → _select_a2  (MC2 if experts_per_device≤24 ∧ EP≥16 ∧ tokens≤capacity, else ALLGATHER)
  ├─ A3       → _select_a3  (FUSED_MC2 / MC2 / ALLTOALL by fused_decode_enable, EP≤32, w8a8)
  ├─ A5       → _select_a5  (MC2 / ALLGATHER / ALLTOALL)
  └─ 310P     → ALLGATHER

The type + a handle to the bound comm method (get_moe_comm_method(moe_comm_type)) are stamped onto the forward context (forward_context.moe_comm_type, .moe_comm_method), so each layer’s FusedMoE.forward reads a single attribute to dispatch.

set_ascend_forward_context — per-forward metadata

This context manager binds everything that’s execution-shape dependent into the current forward context so the ops compile/graph-capture see a stable set of flags:

  • mmrs_fusion = tp_world_size <= 8 (Matmul-AllReduce fusion kernel, currently disabled for TP>16).
  • flash_comm_v1_enabled — sequence-parallelism comm, enabled for MoE always, for dense only when num_tokens > 1000 (empirical concurrency threshold).
  • flashcomm_v2_enabled — when tp>1, with O-matrix TP group size from env.
  • pad_size / padded_length — padding num_tokens up to a TP multiple before the SP comm.
  • padded_num_tokens, mc2_mask — pre-baked boolean mask (reserved_mc2_mask) for MC2’s padded dispatch.
  • is_mtp_kv_consumer branch, prefetch_mlp_*, layer_idx, sinks, eplb_heat_collection_status.

The companion _ExtraForwardContextProxy exposes these via a v1/v2-model-runner-agnostic attribute accessor (_EXTRA_CTX), using additional_kwargs on the V2 path and setattr on the V1 path.

4. Core scheduler innovations (deep dive)

vllm-ascend ships four scheduler-side innovations in vllm_ascend/core/. These are not trivial tweaks; they address hardware-specific bottlenecks that the vanilla FCFS/chunked-prefill vLLM scheduler does not reason about.

4.1 Innovation taxonomy

File (class) Problem on Ascend Core algorithm Contrast with upstream vLLM
recompute_scheduler.py — RecomputeScheduler / AsyncRecomputeScheduler P-D disaggregation D-node: instead of P/D-stateful prefill-decode disaggregation, recompute the KV on the D side when memory pressure demands preemption. Avoids an expensive KV stream-back to P and keeps decode nodes stateless w.r.t. prefill position. Extends Scheduler with a RecomputeSchedulerOutput carrying preempted_reqs + recomputed_reqs. Preemption either offloads the KV (via connector.update_state_before_preempt hook → PreemptedRequestData) or frees and marks the request as “must-recompute” (RecomputeReqInfo). The output loop emits EngineCoreOutput(stop_reason="recomputed") so the request is sent back for recompute. MTP decode nodes fill spec_token_ids with PLACEHOLDER_TOKEN_ID to keep graph-friendly full-graphs while D pulls KV from P. Upstream Scheduler only ever pauses→resumes by reusing existing KV blocks. There is no concept of give up KV, send request back to be recomputed. This recompute policy is what allows Ascend D nodes to treat KV as disposable compute, not durable state. Activated only on kv_role='kv_consumer' (validated in platform.py:648-668).
short_request_first_scheduler.py — ShortRequestFirstRequestQueue + ShortRequestFirstSchedulerMixin Long-prompt prefill can monopolize the engine and starve short requests → tail-latency blow-up in mixed-length workloads. Priority waiting queue split into immediate / short / long lanes. _classify_queue routes by num_prompt_tokens <= threshold. Aging promotion forces a long request to the head after long_max_wait_ms, but only if short queue is empty-ish (avoids pathological promotion while shorts are pending). _preempt_request calls mark_force_immediate so preempted requests resume in the immediate lane. Only active under FCFS policy. Upstream waiting queue is a pure FCFS deque. vllm-ascend’s three-lane queue is a lightweight SRF scheduler: short jobs jump ahead, long jobs get anti-starvation aging, preempted/recovery jobs get immediate priority. Implemented as a mixin so it composes with both RecomputeScheduler and vanilla Scheduler.
scheduler_profiling_chunk.py — ProfilingChunkScheduler / profiling_chunk_predictor.py Fixed max_num_scheduled_tokens chunking wastes decode-time budget on Ascend (prefill latency scales non-linearly; a static chunk is either too small → many prefill steps or too large → decode stutter). Offline profiling → online closed-form control. At init it runs up to 65 real collective_rpc("profile_prefill_latency", chunk_size) forward passes, collects (seq_len, latency), then fits a quadratic model f(l) = a·l² + b·l + c. During scheduling, for each RUNNING/WAITING request it solves f(L+x)−f(L) = target_latency for the chunk x (quadratic formula), smooths it toward base_chunk_size, and aligns to max(page_size, 64). A running time_budget (target_latency derived from base chunk) gates scheduling. Supports online refinement via record_batch_execution_time with a history-aware f(C,H)=a·C·(C+H)+b·C+c·H model. Upstream Scheduler only has a fixed chunk = long_prefill_token_threshold / max_num_scheduled_tokens. Here the chunk is dynamic per request, per position, predicted from a measured hardware latency curve. This is arguably the most novel module.
single_type_kv_cache_manager.py — CompressAttentionManager Compressed-MLA (DSv4/DSA) groups store a quantized latent KV with a compress_ratio > 1; block accounting assumes full tokens, causing wrong prefix-cache hits, wrong max_memory_usage_bytes, and stuck long-input requests (vLLM #40863). Subclasses FullAttentionManager, divides token counts by compress_ratio in get_num_blocks_to_allocate, allocate_new_blocks, cache_blocks. find_longest_cache_hit uses a logical block size block_size * compress_ratio and recomputes block hashes accordingly, with DPC/PCP alignment (len * logical_block_size % alignment_tokens) and Eagle last-block-drop. get_manager_for_kv_cache_spec injects max_admission_blocks_per_request = cdiv(max_model_len // compress_ratio, block_size) + 1 to align admission with the pool’s startup-size bound. Upstream FullAttentionManager only understands raw token-counts. The manager is the place where the compressed latent and the sparse indexer/KV relationship is made block-consistent, and where the max_admission_blocks_per_request cap is ported from vLLM PR #40946.

4.2 Deep dive — ProfilingChunkScheduler (the most novel)

ProfilingChunkScheduler (scheduler_profiling_chunk.py:46) subclasses Scheduler and overrides only schedule() plus adds run_profiling_chunk_init. The upstream schedule() is forked almost verbatim, with deltas clearly marked # >>> PROFILING CHUNK >>> / # <<< PROFILING CHUNK <<<. The changes reduce to three concepts.

Concept 1 — startup profile the hardware curve. run_profiling_chunk_init (:108) drives real model forwards across chunk sizes:

base_chunk_size = self.profiling_chunk_manager.base_chunk_size
for i in range(num_samples + 1):
    chunk_size = int(base_chunk_size - (i - 1) * (base_chunk_size / num_samples))
    result = model_executor.collective_rpc("profile_prefill_latency", args=(chunk_size,), **rpc_kwargs)
    # ... collect (seq_lens, latencies) ...
predictor.fit(seq_lens, latencies)          # least-squares quadratic fit
predictor.set_target_latency(base_chunk_size)
predictor.is_ready = True

The quadratic model (profiling_chunk_predictor.py:36) is f(l) = a·l² + b·l + c. Its key predictive query (predict, :212) is: given already-computed tokens L and a target latency T, solve for next chunk size x:

# f(L+x)-f(L) = a*x^2 + (2a*L+b)*x = target_latency
A = self.quadratic_coeff_a
B = 2 * self.quadratic_coeff_a * num_computed_tokens + self.linear_coeff_b
C = -target_latency
discriminant = B * B - 4 * A * C
x = (-B + sqrt(discriminant)) / (2 * A)
smoothed = base_chunk_size + self.smooth_factor * (x - base_chunk_size)
chunk_size = max(int(smoothed), self.min_chunk)             # clamp
align = max(page_size, 64)
chunk_size = ((chunk_size + align - 1) // align) * align    # align

Why a quadratic and not a linear fit? Because prefill attention cost on Ascend grows super-linearly with sequence length (just like on GPU), so the marginal cost of adding a token grows with L. Solving f(L+x)−f(L)=T therefore yields a smaller chunk as L grows — exactly the behavior you want: early prefill steps are large (most KV still cheap), later steps shrink so you don’t overshoot the decode-time budget. The closed-form solution makes this O(1) per request at schedule time.

Concept 2 — dual constraint loop (token budget + time budget). The RUNNING and WAITING loops gain a time_budget counter initialized from predictor.target_latency:

target_latency = self.profiling_chunk_manager.predictor.target_latency
time_budget = target_latency if target_latency is not None else float("inf")
...
while req_index < len(self.running) and token_budget > 0 and time_budget > 0:
    ...
    if ... in prefill and model ready:
        predicted_chunk = self.profiling_chunk_manager.predict_chunk_size(
            num_computed_tokens=request.num_computed_tokens, target_time=time_budget)
        num_new_tokens = min(predicted_chunk, num_new_tokens)
    ...
    if request.num_computed_tokens < request.num_prompt_tokens:   # only prefill burns time
        time_budget -= self.profiling_chunk_manager.predict_time(num_new_tokens, ...)

The scheduler now reasons in time, not just token count. Decode requests (num_new_tokens == 1) have their latency skipped so they never starve. When time_budget hits zero the loop exits even if token_budget remains — it has used up its millisecond slice for this step.

Concept 3 — online refinement. record_batch_execution_time(request_chunks, elapsed_time) (:401) accumulates per-step (Σ(C+H)·C, Σ(C+H), N, elapsed_ms) points and refits the history-aware model f(C,H) = a·C·(C+H) + b·C + c·H once it has ≥5 points. After that the scheduler switches from predict to predict_with_history, which models how marginal cost depends on both chunk size and already-computed history. This lets the scheduler adapt to real batch-interference effects that the offline profile (run in isolation) cannot see.

Configuration (AscendConfig.profiling_chunk_config, ascend_config.py:647): enabled, smooth_factor (0,1], min_chunk, max_fit_chunk (≤30). Enabling it requires PP>1 (validated). It is incompatible with enable_balance_scheduling and is wired in platform.py:685-689:

if ascend_config.profiling_chunk_config.enabled:
    vllm_config.scheduler_config.scheduler_cls = \
        "vllm_ascend.core.scheduler_profiling_chunk.ProfilingChunkScheduler"
    import vllm_ascend.patch.platform.patch_profiling_chunk  # noqa

4.3 Deep dive — RecomputeScheduler (stateless KV via recompute)

recompute_scheduler.py:98 class RecomputeScheduler(ShortRequestFirstSchedulerMixin, Scheduler) is the scheduler that decouples decode-KV from prefill-KV. The docstring at schedule() (:195-204) restates the upstream “no prefill phase / no decode phase, just num_computed_tokens chasing num_tokens_with_spec” model, but Ascend adds a radical policy: if a running request can’t get its KV blocks, instead of just evicting an old request’s blocks for reuse, it may FREE the blocks and send the original request back to be recomputed.

The pivotal preemption branch (:320-374):

while True:
    new_blocks = self.kv_cache_manager.allocate_slots(request, num_new_tokens, ...)
    if new_blocks is not None:
        break
    # Cannot allocate → preempt.
    transfer_config = self.vllm_config.kv_transfer_config
    if transfer_config is not None and not transfer_config.is_kv_producer:
        recomputed_req = self.running.pop()              # victim
        recomputed_block_ids = self.kv_cache_manager.get_block_ids(recomputed_req_id)
        # Try to offload KV back to P via connector hook
        offloaded = bool(preempt_hook(recomputed_req, recomputed_block_ids, ...) ...
        if offloaded:
            preempted_req_data.append(PreemptedRequestData(...))   # KV streamed out
            self._preempt_request(recomputed_req, ...)
        else:
            self.kv_cache_manager.free(recomputed_req)            # just drop KV
            recomputed_reqs.append(RecomputeReqInfo(
                recomputed_req_id, recomputed_req.output_token_ids, ...))
    else:
        # upstream-style preempt reused blocks ...

Two flavors: - Offload (PreemptedRequestData): the KV blocks stay valid and are streamed to a P node via the connector; the request resumes later as-if pre-empted in vanilla vLLM. - Recompute (RecomputeReqInfo): the KV is freed; update_from_output (:918-928) emits an EngineCoreOutput with stop_reason="recomputed" so the request is scheduled for re-prefill. This is the “D node is stateless, P node owns the truth” P-D disaggregation mode.

MTP-KV-consumer bookkeeping (add_request :152, _update_waiting_for_remote_kv :184) fills spec_token_ids with PLACEHOLDER_TOKEN_ID so the decode graph stays full-shape while D is still pulling KV from P.

AsyncRecomputeScheduler (:1170) is a diamond-inheritance class (AsyncScheduler, RecomputeScheduler) that combines async scheduling with recompute.

4.4 ShortRequestFirst + CompressedKV notes

  • ShortRequestFirstRequestQueue (:58) is a three-lane deque queue. _classify_queue :179 routes by the configured threshold of num_prompt_tokens. _select_schedulable_queue :209 prefers immediate > short > long, but proactively promotes a long when it has waited longer than long_max_wait_ms and a short is waiting. It tracks _aged_long_promotions and warns (:252) if the scheduler is permanently degrading toward long-request priority. The Mixin (ShortRequestFirstSchedulerMixin :374) only installs itself under FCFS policy — a deliberate constraint that makes SRF a waiting-queue policy, not an arbitrary re-ordering. Preempted requests (_preempt_request :431) and recovery requests (_is_recovery_request :380, PREEMPTED / num_computed_tokens>0 / num_output_tokens>0) are marked force_immediate so they don’t re-contend in the long lane.

  • CompressAttentionManager (single_type_kv_cache_manager.py:32) has to undo the “compressed latent looks like fewer tokens than it really represents” illusion everywhere. cache_blocks (:160) caches at block_size * compress_ratio; find_longest_cache_hit (:196) walks block hashes at logical block size block_size * compress_ratio and trims to the alignment_tokens boundary required by PCP/DCP, dropping the last matched block when EAGLE spec decode is active. The registry hook register_ascend_kv_cache_specs (kv_cache_interface.py:208) maps these specs onto CompressAttentionManager / FullAttentionManager / SlidingWindowManager.

5. Supporting infra

  • envs.py — Ascend-specific env-var registry following vLLM’s __getattr__ lazy-env pattern. Key knobs worth knowing:
  • VLLM_ASCEND_ENABLE_MATMUL_ALLREDUCE — Matmul-AllReduce fusion TP kernel (A2 only, eager mode).
  • VLLM_ASCEND_ENABLE_FLASHCOMM1 / VLLM_ASCEND_FLASHCOMM2_PARALLEL_SIZE — FlashComm v1 (SP comm, concurrency≥1000) and v2 (O-matrix TP group size); the two sequence-parallelism comm protocols driving flash_comm_v1_enabled / flashcomm_v2_enabled in the forward context.
  • VLLM_ASCEND_ENABLE_MLAPO — MLAPO W8A8 optimization (default on, trades NPU memory for perf).
  • VLLM_ASCEND_ENABLE_FUSED_MC2 (0/1/2) — dispatch_ffn_combine / dispatch_gmm_combine_decode; the depth knob behind MoECommType.FUSED_MC2.
  • VLLM_ASCEND_ENABLE_NZ — weight cast to FRACTAL_NZ layout (0/1/2).
  • DYNAMIC_EPLB, VLLM_ASCEND_ENABLE_BATCH_MEMCPY (KV-offload async memcpy), VLLM_ASCEND_FUSION_OP_TRANSPOSE_KV_CACHE_BY_BLOCK.
  • cpu_binding.py — Ascend-native core affinity (replaces vLLM’s --numa-bind, auto-converted in _fix_incompatible_config). Two topology modes (:24-30): topo_affinity on A2/310P/A5 (threads bound to NUMA cores adjacent to their NPU) vs global_slice on A3 (board/chip topology from npu-smi info -m). A5 additionally omits IRQ cores (NO_IRQ_BINDING_DEVICE_TYPES). Handles ARM (aarch64) detection and disables itself on x86.
  • memcache_comm_fence.py — AttentionComputeStartGate synchronizes compute stream and MemCache transfer threads: the attention path records an NPU event just before submitting attention; MemCache worker threads wait() for that event before issuing H2D/L2G transfers, ensuring transfers start when the compute stream is truly at the attention boundary (not just after Python called the kernel).
  • batch_invariant.py — registers NPU/Triton batch-invariant addmm/bmm/softmax/mm/matmul/sum/rms_norm overrides so that an op’s output does not depend on batch shape. Required for training-inference consistency (deterministic FA3 path behind use_batch_invariant); sets HCCL_DETERMINISTIC=strict, LCCL_DETERMINISTIC=1, torch.use_deterministic_algorithms(True).
  • meta_registration.py — utilities to register Python meta implementations of custom ops under the "_C_ascend" namespace (so torch.compile / aclgraph tracing and shape inference work): register_meta_if_necessary(ns, op_name, fn) and example bgmv_expand_meta / sgmv_expand_meta for SGMV/BGMV LoRA expand ops (skipped on 310P).
  • profiling_config.py — writes service_profiling_symbols.{VLLM_VERSION}.yaml to ~/.config/vllm_ascend/ defining ms_service_profiler tap points (schedule, add_request, KV cache alloc/free, sampler, executor, MTP propose, rejection sample, DP-engine init). Safe no-op on failure, enabling atomic write.
  • utils.py (first 80 lines) — module constants (COMPILATION_PASS_KEY, quantization method names, ACL_FORMAT_FRACTAL_ND/NZ, shared-expert/CP comm stream placeholders) and the AscendDeviceType enum; exposes helpers used by platform/patch (is_310p, enable_sp, flashcomm2_enable, is_moe_model, refresh_block_size, update_cudagraph_capture_sizes, …). Shows the central helper surface that the platform/scheduler layers build on.

6. How the scheduler innovations compose

The scheduler features are not mutually exclusive alternatives — they compose via config in check_and_update_config:

additional_config:
  recompute_scheduler_enable:true   →  RecomputeScheduler (+ AsyncRecomputeScheduler if async_scheduling)
      └─ short_request_first_config.enabled
            └─ ShortRequestFirstSchedulerMixin injected into the waiting queue
  profiling_chunk_config.enabled    →  ProfilingChunkScheduler (PP>1, overrides scheduler_cls)
                                        + patch_profiling_chunk
  (neither)                        →  upstream Scheduler

recompute_scheduler_enable and profiling_chunk_config.enabled are on separate branches (platform.py:648 elif :685), so recompute + profiling-chunk are wired as mutually exclusive top-level schedulers, but SRF is a waiting-queue mixin that recompute can carry as a passenger.

On top of any scheduler, the KV layer is patched (single_type_kv_cache_manager + kv_cache_interface) so that compressed-MLA groups always route through CompressAttentionManager, and NPUPlatform.register_custom_kv_cache_specs ensures the Ascend MLA/SFA/sliding-window specs are registered before any scheduler runs.
