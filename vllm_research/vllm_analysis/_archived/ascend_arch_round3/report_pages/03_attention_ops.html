03 — Attention Backends, MLA, KV-Compression, Custom Ops, Triton Kernels, 310P, CANN Custom Ops

Scope: vllm_ascend/attention/, vllm_ascend/ops/, vllm_ascend/_310p/, vllm_ascend/device/device_op.py, vllm_ascend/_cann_ops_custom/. Analysis date: 2026-07-18. Repo: vllm-ascend (Huawei Ascend NPU plugin for vLLM v1).

1. Attention backend taxonomy

vLLM-Ascend plugs into vLLM v1’s AttentionBackendEnum.CUSTOM registry via @register_backend(AttentionBackendEnum.CUSTOM, "ASCEND") (and per-backend registrations in platform.get_attn_backend_cls). The backend is chosen by (use_mla, use_sparse, use_compress) in vllm_ascend/platform.py:784-810. Device mapping is by AscendDeviceType enum {A2=0 (910B, soc 220–225), A3=1 (soc 250–255), _310P=2 (soc 200–205), A5=3 (soc 260)} read from torch_npu.npu.get_soc_version() and matched against a baked-in build label (vllm_ascend/utils.py:788-836).

2. MLA on Ascend (深度重点)

This is the high-value path powering DeepSeek V3/R1 inference on Ascend.

2.1 Conceptual mapping

Where vanilla vLLM’s mla_decode_attention / mla_prefill run an explicit two-stage attention (project Q/K/V → paged-Softmax with separate K/V caches), Ascend MLA collapses the K side into a compressed latent c_kv stored in the KV cache and absorbs the up-projection W_UK into the query, never materializing full K/V heads in the hot path:

q_c = fused_qkv_a_proj(hidden_states).split([q_lora_rank, kv_lora_rank + qk_rope_head_dim])
q_c = q_a_layernorm(q_c)
kv_no_split = kv_a_layernorm-like projection, shape [B, N, kv_lora_rank + qk_rope_head_dim]
   └─ cached directly as (c_kv, k_pe) via npu_kv_rmsnorm_rope_cache

2.2 Decode path (_forward_decode, mla_v1.py:1370)

Preprocessing (mla_preprocess_decode, _q_proj_and_k_up_proj):

  1. q_nope, q_pe = self.q_proj(q_c).view(...).split([qk_nope_head_dim, qk_rope_head_dim]) — then absorb W_UK_T on the nope part via torch.bmm(q_nope, self.W_UK_T) → ql_nope (so the query is pre-baked into the latent space). This is the weight-absorption trick that enables the latent-space attention.
  2. RoPE applied on the q_pe rope part via npu_interleave_rope (rope_single).
  3. k_pe, k_nope = exec_kv_decode(kv_no_split, cos, sin, kv_cache, slots) writes the latent+rope cache with torch_npu.npu_kv_rmsnorm_rope_cache(..., cache_mode="PA_NZ" | "PA") — a single fused CANN op that does RMSNorm → RoPE → paged-cache write.

Core attention call (decode, non-quantized, mla_v1.py:1564-1570):

# shape knope/k_pe for npu graph mode should be:
# [num_blocks, num_kv_heads, block_size, kv_lora_rank / qk_rope_head_dim]
attn_output, _ = torch_npu.npu_fused_infer_attention_score_v2(
    q_nope, k_nope, k_nope,          # k_nope passed 3rd positional too (placeholder)
    query_rope=q_pe, key_rope=k_pe,  # <-- RoPE parts injected here
    num_query_heads=self.num_heads_padded,
    num_key_value_heads=self.num_kv_heads,
    input_layout="BSND_NBSD" | "BNSD_NBSD" | "TND_NTD",  # per sliding-window/spec
    softmax_scale=self.scale,
    block_table=decode_meta.block_table,
    block_size=block_size,
    actual_seq_qlen=actual_seq_lengths,
    actual_seq_kvlen=decode_meta.seq_lens_list,
    workspace=workspace,
)

The kernel is Fused Infer Attention v2 (npu_fused_infer_attention_score_v2), vLLM-Ascend’s single most important kernel. Unlike vanilla vLLM’s flash_attn / triton mla_decode, FIA-v2 natively takes the latent (q_nope, q_pe), (k_nope, k_pe) decomposition plus paged block tables — no explicit materialization of full K/V heads happens inside the kernel.

Output up-projection (_v_up_proj, mla_v1.py:880) — a fused batch-matmul that maps kv_lora_rank → v_head_dim:

x = x.view(self.num_heads, -1, self.kv_lora_rank)
x = torch_npu.npu_transpose_batchmatmul(x, self.W_UV, perm_y=(1, 0, 2))
x = x.reshape(-1, self.num_heads * self.v_head_dim)

2.3 Prefill path (_forward_prefill, mla_v1.py:1221)

Prefill materializes full k_nope, value = self.kv_b_proj(kv_c_normed) and runs a single FIA call with softmax_lse_flag=True, then _compute_prefill_context chunks the prefill context into chunked_prefill_workspace-sized pieces:

attn_output, attn_lse = torch_npu.npu_fused_infer_attention_score(
    query, key.contiguous(), value.contiguous(),
    num_heads=..., num_key_value_heads=self.num_heads, input_layout="TND",
    sparse_mode=3, scale=self.scale, actual_seq_lengths=actual_seq_lengths_q,
    actual_seq_lengths_kv=actual_seq_lengths_kv, softmax_lse_flag=True, ...)
# then chunked context via the same kernel + npu_attention_update to merge LSE
output_final, _ = torch_npu.npu_attention_update(tuple(lse_list), tuple(out_list), 0)

Each chunk loads its slice from the paged (c_kv, k_pe) cache via DeviceOperator.kv_cache_load → torch_npu.npu_gather_pa_kv_cache, runs the same npu_b_proj, and fuses via npu_attention_update.

2.4 MLAPO — fused MLA preproc (deep operator fusion)

enable_mlapo() gates a deeper fusion (mla_v1.py:942-1065, _process_weights_for_fused_mlapo & _process_weights_for_fused_mlapo_a5):

  • A2/A3: torch.ops._C_ascend.mla_preprocess(... wd_qkv, deq_scale_qkv, gamma1, beta1, wu_q, qb_deq_scl, gamma2, cos, sin, W_UK_T, k_nope, k_pe, slot_mapping, quant_scale0/1, ..., cache_mode="nzcache|krope_ctkv", quant_mode="per_tensor_quant_asymm") fuses Q/KV projections + LayerNorm + RoPE + cache write into one CANN custom op, then reorg_decode_q reshapes for FIA.
  • A5: npu_mla_prolog_v2 / npu_mla_prolog_v3 / npu_dynamic_quant family, casting to float8_e4m3fn / MXFP (npu_dynamic_mx_quant) — a separate fused prolog for FP8 decode.
  • Weights are pre-reformatted into FraCTAL_NZ (npu_format_cast(..., 29)) and trans_rope_weight + transdata are applied offline.

On KV consumer (PD-decode) instances, fused_qkv_a_proj.weight / q_proj.weight are set to None once MLAPO absorbs them, explicitly freeing VRAM (mla_v1.py:1053-1064).

2.5 How this differs from vanilla vLLM’s mla_decode_attention / mla_prefill

Graph capture CUDA graphs ACL task-graph replay: torch.npu.graph_task_group_begin/end + graph_task_update_begin/end with weak_ref_tensors parameter capture — FIA params, handles, events cached per token-count bucket

3. Context Parallel & KV-Compression attention

3.1 PCP (Prefill Context Parallel) + DCP (Decode Context Parallel)

Implemented across attention/context_parallel/{common_cp, attention_cp, mla_cp, sfa_cp, dsa_cp}.py. Two groups: get_pcp_group() and get_decode_context_model_parallel_group() (DCP). Key structures in common_cp.py:

  • AscendPCPMetadata — per-request PCP chunk pointers.
  • CPChunkedContextMetadata — cu_seq_lens, starts, seq_tot, max_seq_lens, workspace.
  • AscendMetadataForPrefill / AscendMetadataForDecode — decode-side PCP bookkeeping.

In attention_cp.py:108, AscendAttentionCPMetadataBuilder.build reads the all-rank context-lens tensor, partitions the PCP chunk load, and rewrites the attention mask accordingly. mla_cp.py:57-97 does the equivalent for MLA’s (c_kv, k_pe) cache by computing a virtual block size cp_local_block_size * dcp_size * pcp_size and realigning the block table with np.gcd so all ranks index a consistent paged view.

3.2 KV-compression attention — Hamming-sparse long-context (not GLA)

attention/kvcomp_attn/attention_utils.py implements a Hamming-distance hash-top-k sparse attention, not Mamba/GLA. Control gate: is_enable_hamming_sparse(enable_c8_quant) reads additional_config["enable_hamming_sparse"] and mutually-excludes C8 KV quantization (attention_utils.py:145-154).

Mechanism: 1. reshape_and_cache_kvcomp — every layer, compute hashk = hash_encoder.compute_hash(key), write into hashk_caches[layer] via torch.ops._C_ascend.npu_reshape_and_cache_bnsd. 2. get_kvcomp_decode_params — for decode, hash the query, then torch.ops._C_ascend.npu_hamming_dist_top_k(hashq, hashk_cache, topk, seq_lens, chunk_sizes, max_seq_len, sink, recent, ..., block_table, valid_query_mask, hamming_output) picks the top-K most-recent blocks by Hamming distance, returning a sparse block table and compressed seq_lens_from_hamming. 3. FIA is then called with the sparse block table — only attending to top_k * chunk_size tokens per request instead of full context. Metadata is plumbed via AscendMetadata.kvcomp_metadata: KVCompMetaData (worker/kvcomp_utils.py).

Memory/compute savings: context length collapses from L to top_k * chunk_size per request, with a sink (initial tokens) + recent (local window) always retained — a classic H2O/StreamingLLM-style long-context strategy implemented purely in custom ops.

The ops/gdn_attn_builder.py (and ops/gdn.py, ops/fla/) path is a separate linear-recurrent attention (Gated Delta Net / GDN) used by Mamba-style hybrid models, gated behind the flash_linear_attention ops; KV-comp is the other, independent long-context mechanism.

4. Fused MoE ops

All under ops/fused_moe/. The architecture mirrors vLLM’s EP-aware FusedMoE but uses Ascend-specific comm+compute fusion.

4.1 Gating + Top-K routing

select_experts (experts_selector.py) wraps either: - torch_npu.npu_moe_gating_top_k (fused gating kernel, via check_npu_moe_gating_top_k), or - a vLLM grouped_topk fallback, with zero_experts_compute (zero-expert bypass), AscendW8A8MXFP8DynamicLinearMethod & AscendW8A8LinearMethod quant hooks.

topk_ids is post-permuted by log2phy (EPLB logical→physical) before dispatch.

4.2 Comm-method stack (moe_comm_method.py)

MoECommType enum (ascend_forward_context.py:27): ALLGATHER = 0, MC2 = 1, ALLTOALL = 2, FUSED_MC2 = 3. Selected via _EXTRA_CTX.moe_comm_method and per-EP-size registered in setup_moe_comm_method (moe_comm_method.py:55).

4.3 MC2 — communication/computation fusion (token_dispatcher.py:102)

TokenDispatcherWithMC2.token_dispatch calls torch_npu.npu_moe_distribute_dispatch[v2]; combine uses npu_moe_distribute_combine. These are the MC2 primitives: they combine an all-to-all-style token scatter with the GMM compute on the cube (Cohen-McMullen-Cannon style comm/compute overlap on Ascend’s AI-Core). A5 vs A3 toggle extra args (a5_need_extra_args, need_extra_args), and hierarchical comm (is_hierarchical_communication_enabled) adds expert_scales.

4.4 Fused MC2 — single-op MoE decode

FusedMC2CommImpl.fused_experts (moe_comm_method.py:286-349) — the highest-fusion mode:

out = torch.ops._C_ascend.dispatch_ffn_combine(
    x=hidden_states, weight1=w1, weight2=w2, expert_idx=topk_ids,
    scale1=w1_scale, scale2=w2_scale, bias1=w1_scale_bias, bias2=w2_scale_bias,

Weight handling: w13/w2 are (E, N, K) pytorch shape → transposed to (E, K, N) and cast to FraCTAL_NZ (npu_format_cast(..., ACL_FORMAT_FRACTAL_NZ)) because dispatch_ffn_combine requires NZ (fused_moe.py:125-143). With dynamic_eplb, weights are unbinded into per-expert w13_weight_list/w2_weight_list to keep load-balancing movable.

4.5 MLP compute & experts

moe_mlp.py (unified_apply_mlp) and experts_selector.py route the per-token GMMs. prepare_finalize.py has four variants (PrepareAndFinalize{,WithAll2All,WithAllGather,WithMC2}) handling padding, shared-expert DP overlap, and replace-allreduce logic. gate_linear.py wraps the fused gate-up matmul.

The KDA / FLA Triton stack is the most interesting: these are linear-recurrent (GDN / GLA) kernels ported from the flash-linear-attention project for DSA/SFA-style sparse attention — they power Qwen3-Next-style mixed-attention on Ascend, where a CANN equivalent does not ship yet. The fla/ kernels are large, hand-tuned Triton (block-size 64, WY-factorized triangular solve via wy_fast.recompute_w_u_fwd), and run under PCP/DCP because their chunked layout maps to the split context-parallel dimension naturally.

6. 310P vs A2/A3

_310p/ is a separate code tree for the Ascend 310P (the “P” inference/edge series, e.g. Atlas 300I Duo / 310P soc 200–205). A separate tree is needed because 310P has:

  1. Operator-subset differences — _310p/attention/attention_v1.py:112-147 implements its own _flash_attention dispatching to _npu_flash_attention / _npu_flash_attention_v3 + a support_compressed_mask branch, and to _npu_paged_attention (forward_paged_attention, attention_v1.py:166). These are older CANN primitives than the FIA-v2 path used on A2/A3/A5 — 310P does not support npu_fused_infer_attention_score (FIA) at all, so the canonical Ascend MLA / FIA path cannot run. That is also why backend_map_310 only maps (False, False) → AscendAttentionBackend310 (platform.py:797-808): MLA/SFA/DSA are not implemented on 310P.
  2. KV-cache shape constraint — 310P requires 16-aligned last-dim tiling: (2, num_blocks, num_kv_heads*head_size//16, block_size, 16) and supports block sizes {128, 64} (get_supported_kernel_block_sizes → [128, 64]), whereas A2/A3/A5 mandate 128 only (attention_v1.py:141).
  3. Precision / quant — 310P has its own quantization/methods/{w8a8_dynamic, w8a8_static, w8a8sc, w8a8s, w8a8_base}.py registry and linear method (_310p/quantization/methods/w8a8_dynamic.py is a W8A8_DYNAMIC scheme registered to the 310P registry). It lacks A5’s FP8 (float8_e4m3fn) / MXFP8 path but retains W8A8 dynamic/static/scaled/sign schemes.
  4. DeviceOp — Ascend310PDeviceAdaptor (device/device_op.py:1925) overrides index_fill_ with a boolean-mask emulation because that op is unavailable on 310P; reshape_and_cache uses _npu_reshape_and_cache instead of the A2/A3 path.
  5. FLS/SSM path — 310P has its own GDN/Delta-net stack under _310p/ops/fla/ (gdn_310.py, chunk_gated_delta_rule.py, fused_gdn_gating.py, fused_recurrent_gated_delta_rule.py, l2norm.py, cumpute_causal_conv1d_metadata_310.py) calling _C_ascend.npu_recurrent_gated_delta_rule_310, chunk_gated_delta_rule_fwd_h, chunk_fwd_o, npu_causal_conv1d_310 — the _310 CANN ops are a 310P-specific subset distinct from the A2/A3/A5 Triton path.
  6. Fused MoE — _310p/fused_moe/ has its own fused_moe.py, moe_mlp.py, moe_comm_method.py, token_dispatcher.py, experts_selector.py — parallel implementation for the 310P operator set.
  7. Spec decode / sample — own spec_decode/llm_base_proposer_310.py and sample/{sampler, rejection_sampler}.py.

In short: 310P = a forked operator tree whose CANN backend predates (and sits below) the FIA-v2 + MLAPO stack that makes DeepSeek V3-class MLA fast on A2/A3/A5; it only supports vanilla MHA/GQA Flash-attn / paged-attn, W8A8 quant, and the linear-recurrent SSMs.

7. CANN custom ops

7.1 Registration mechanism

vLLM-Ascend has two custom-op surfaces:

(a) _C_ascend.* — bulk of Ascend-specific operators, registered into torch.ops._C_ascend via the global _CUSTOM_OP_REGISTERED flag and bootstrap_custom_op_env() (utils.py:774-781). vllm_ascend/_cann_ops_custom/ exists but contains only .gitkeep — the actual compiled operator library is loaded at import time from the CANN (torch_npu) runtime, not vendored. So “C custom ops” = the set of torch.ops._C_ascend and torch_npu.* functions exposed by the CANN/torch_npu install.

(b) register_custom_ops.py registers vLLM-namespaced PyTorch ops via vllm.utils.torch_utils.direct_register_custom_op so they’re reachable as torch.ops.vllm.{maybe_all_gather_and_maybe_unpad, maybe_pad_and_reduce, ...} and participate in vLLM’s FP8/all-gather/EP interplay.

7.2 How key CANN ops are exposed to vLLM’s graph

Most of the critical kernels are torch_npu.* native ops called directly in Python, not captured as opaque custom ops — meaning they integrate with vLLM’s v1 attention layer through normal torch dispatch. 7.3 Graph-capture interactions

ACL graph capture uses torch.npu.graph_task_group_begin/end + graph_task_update_begin/end with weak-referenced parameter tuples cached per token-count bucket (e.g. full_graph_fia, update_graph_params in attention_v1.py:452-750, mla_v1.py:766-878). weak_ref_tensors + a per-size graph_params.attn_params[num_tokens] / handles / events / workspaces lets each layer register its op into the captured task graph, with workspace pre-allocated from _npu_fused_infer_attention_score_get_max_workspace (and cached with _use_max_workspace_for_fia_graph) so mixed-attention models sharing one graph size don’t OOM.
