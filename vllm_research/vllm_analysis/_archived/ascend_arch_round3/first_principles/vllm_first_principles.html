vLLM 架构的第一性原理

基于 vllm-ascend（vllm-project/vllm-ascend, commit b266da5d）代码事实的反思，结合大模型推理系统的设计约束。
不是功能清单，而是“如果这条不成立，整个架构就要改写”意义上的第一性原理。
每条原理都给出代码证据与一句话可证伪条件。
P1 · 内存是硬约束 P2 · 服务即连续批调度 P3 · Python 控制面 + kernel 数据面 P4 · 算子融合 = 带宽倍增器 P5 · 硬件抽象必须零成本 P6 · 并行度是调度维度 综合推论

P1. KV 缓存是整个系统的一切 —— 内存容量是硬约束，算力不是

自注意力在序列长度 L 上是 O(L²) 计算、O(L) 内存（每个 token 产出一对 K/V 写入缓存）。这句话在训练侧只是一条复杂度注释，在服务侧却变成整个软件栈的塑造力。**服务系统面对的根本问题不是「算得多快」，而是「一块卡上能同时装多少个请求的 KV，同时不击穿延迟预算」**。

原理表述。若 HBM 无限大且带宽无限，则 PagedAttention、prefix caching、sleep mode、recompute scheduler、compressed KV、KV offload 都不会存在。这些模块存在的唯一原因是：在不丢吞吐的前提下，让有限 HBM 装进尽可能多的并发上下文，或者在「装不下」时以最小代价换出。 算力瓶颈通过 batch 即可缓解；内存瓶颈不能—— decode 单步已经是带宽受限（权重全量读入仅为 1 token），再堆 batch 也不会让容量线性扩大，因为 KV 线性吃掉 HBM。 若 HBM 单卡容量 ≫ 最大模型的权重 + 所有 in-flight 请求的 KV 总和，则内存调度层退化为 no-op，整条 PagedAttention 技术路线失效。当前 Hopper 80 GB 对 MoE 大模型恰恰不满足此条件。

代码证据

1. CompressAttentionManager 把 block 数除以 compress_ratio。这是一条直白的「内存即硬约束」推论：如果内存不是约束，缓存侧的任何压缩都不会带来收益，只会白白损失精度。

2. ProfilingChunkScheduler 不是「算力调度器」，而是「内存准入控制器」。它用闭式二次模型把 chunk 大小定下来，语义目标是 让 prefill 速率与 decode 速率匹配，不击穿时间预算——时间预算的本质是「在这个时间内能消化多少 KV 增量」。token_budget 与 time_budget 双循环（§Scheduler hot-loop）是在用时间约束约束内存增长速率。

    predicted_chunk = self.profiling_chunk_manager.predict_chunk_size(
        num_computed_tokens=request.num_computed_tokens, target_time=time_budget)
    num_new_tokens = min(predicted_chunk, num_new_tokens)
    ...
    if request.num_computed_tokens < request.num_prompt_tokens:   # only prefill burns time

P1 的推论

  • 调度器 ≠ 调度器；它是内存准入 + 延迟预算的联合优化器。
  • KV 块粒度（128 token/block）是「内存管理单元 (MMU) 页大小」的类比，选大浪费内存在 prefix caching，选小则 block table 内存和 kernel 索引成本爆炸。128 是工程甜点，不是理论最优。
  • 任何「在 HBM 里省内存」的技术路线（量化压缩、稀疏注意力、prefix 共享、跨层 KV 共享）的收益上限都由 P1 决定。

P2. 推理服务 = 连续批处理调度，不是「跑一个模型」

换个角度看同一个事实：LLM 单步 decode 是极端 memory-bandwidth-bound —— 读取全部权重只为产出 1 个 token。如果 batch = 1，H100 的利用率趋近于零。要吃饱算力必须 batch。可一旦 batch，请求就不再是"进来 → 跑完 → 退出"的静态形状，而是一个持续进出、长度互异、内存互不相同的动态集合。vLLM 架构从 Scheduler 往后的全部复杂性，是在服务这个事实：推理系统本质是一个持续接纳请求、持续在有限内存里腾挪的动态调度器，model executor 只是它的一个执行后端。

原理表述。如果 LLM decode 是 compute-bound（像训练那样），vLLM 只需要一个静态 graph + NCCL all-reduce。PagedAttention、continuous batching、chunked prefill、async scheduling 的合理性全部来自 decode 的带宽绑定性 + 请求的动态性。 roofline：decode 单 token 的 arithmetic intensity ≈ model_flops / model_params ≈ 6 (7B) ~ 60 (405B) FLOP/byte，落在 roofline 左侧（带宽受限区）。打破此状态的唯一方法是「把多个请求的 decode step 合并」，即 batch。 若未来 HBM 带宽 ~10×today（如光学/3D 互联成熟），decode 单 token 进入 compute 受限区 → batch 必要性下降，serving 架构退化为单请求流式执行，就像今天的图像分类服务。

代码证据

1. Compillation / fusion_pass 不是为训练设计的，而是为 batch 的连续解码服务。AscendCompiler 的 compute_hash 把 torch_npu_version、enable_npugraph_ex、enable_static_kernel 都放进去——这说明 compiled graph 是 per-ABI、per-batch-size、per-编译选项缓存的；decode 的图是按 capture size bucket 存好的。

P2 的推论

  • Chunked prefill 的存在 ≡ 承认 prefill 与 decode 争夺同一块 HBM 和 bandwidth；chunk 大小 = 对「prefill 吃掉多少带宽/时间」的预算。
  • Speculative decoding 把 K 个 token 压进同一个 forward，是在 decode 这个带宽受限区人为注入计算密度，把 batch 从「很多用户的 1 token」偷换成「1 个用户的 K token」。
  • Scheduler 的每次 schedule() 输出不是「选哪几个请求」，选 batch，选 chunk 大小，选抢占，选 KV 分配——这是一个微观的「OS 进程调度 + 内存管理」，不是 ML 里的 batch_size 超参。
社区文档对 ProfilingChunk 定位的纠正（重要）。原分析把 ProfilingChunkScheduler 归为「内存准入控制」（P1 工具）。官方设计文档 design_dcpp.md 给出更准确的主目的：消除 PP 流水线气泡。O(n²) self-attention 使同 size chunk 越往后越慢（prefix 历史增长）-> PP stage 间 bubble -> GPU 利用率下降。闭式求解的目标是让每 chunk 时间相等（不等 size、等时间），而非等内存增量。所以 ProfilingChunk 本质是 P6（并行度调度）的工具，借 P2 的 batch 调度回路实现，不是 P1 的工具。约束也印证：PP>1 必须、与 balance scheduling 互斥。借鉴 SGLang dynamic chunking + 增加 online calibration f(C,H)=a·C(C+H)+b·(C+H)+c。 docs/source/developer_guide/Design_Documents/dynamic_chunked_pipeline_parallel.md
社区演进：调度器家族在壮大（P2 的活体演化）。近期 PR 显示 vllm-ascend 的调度器不止 ProfilingChunk/Recompute/SRF 三种，还在扩张：
  • #12240 Batch-Job-Aware Scheduler（merged）— 离线批 LPT 调度，从已完成请求采样预测 job 的 decode 长度。
  • #12292 NonBSP（open）— 跨 DP rank load-aware 调度，打破 BSP 假设，优化通信气泡。
  • Balance scheduling（design_balance_schedule_refactor）— 语义明确为「leader-at-cap ⇒ 全局冻结准入」的全局准入门，不是主动重平衡（该语义被显式拒绝）。
这印证 P2：serving 系统的调度器是「请求作为动态物体的接纳控制」，不同工作负载（在线混合/离线批/PP/DP）催生不同调度策略，但都服务于「在有限内存+带宽下填饱算力」这一 P2 内核。

P3. Python 控制面 + kernel 数据面，Forward Context 是唯一的桥

vLLM 在两个世界之间做了一次职责切分，而且切得相当干净：所有与「下一步做什么」相关的逻辑跑在 Python（调度、batching、路由、抢占、工具调用解析）；所有与「这个 tensor 怎么算」相关的逻辑跑在 C++/CUDA/Triton/CANN kernel。这两个世界之间只有一种正经的通信方式：Forward Context。

这个切分之所以是一项「第一性原理」级的设计决策，是因为它要同时满足两件矛盾的事：

  • 控制面的逻辑（调度、抢占、工具调用、投机解码候选筛选）变化剧烈、模型相关、业务相关，因此必须用 Python 快速迭代。
  • 数据面的计算（attention、GMM、all-reduce）一个 kernel launch 只在微秒级，任何 Python-level 调度、字典查找、分支判断都是不可承受的—— 特别是 graph capture 要求整段 forward 没有任何 Python 副作用。
原理表述。LLM serving 系统的两层必须按「变化速率」而非「性能」来切分：控制面一周改十次，数据面一年代码不动但 ABI 跟着 CANN 大版本走。这两层唯一的桥必须是一个「per-layer capability descriptor」—— forward context。 如果 control plane 也用 C++ kernel 写（像 TF1 static graph），vLLM 两周一次的功能迭代速度直接死掉；如果 data plane 也用 Python/Triton 写（像 PyTorch eager），decode 延迟涨一个数量级。所以这不是性能分工，是「演化速率解耦」。 若 Python GIL + dispatch 开销 ≪ kernel launch 开销（比如未来 accelerator 自带 scheduling coprocessor），把 Scheduler 和 model forward 都下沉到 firmware 也可能成立——此时 Python 控制面会被整个旁路。

代码证据

P3 的推论

  • register_custom_ops / meta_registration 存在的理由是：C++ kernel 面需要 PyTorch meta function 做 shape inference，但 shape inference 本身又必须「不真跑 kernel」——这是控制面对数据面的另一类介入。
  • vllm-ascend 里 patch_profiling_chunk 要同时介入 EngineCore.__init__、Scheduler.update_from_output、Scheduler.schedule、子进程入口点，正是因为它要实现「Python 控制面 → kernel 数据面 → Python 控制面」的跨层反馈回路。这是 P3 的边界案例：反馈打破了单向桥。

P4. 算子融合 = 内存带宽倍增器 —— serving 优化的天花板由 roofline 左侧决定

在 LLM serving 这个 arithmetic intensity 极低的场景下，绝大多数 kernel launch 都是「读整张权重、写整张 activation」。在这种 roofline 左侧工作点上，唯一有效的性能手段是减少 HBM round-trip，即把多次「读 → 算 → 写」合并成「一次读 → 多次算 → 一次写」。

但融合不是免费的。融合的前提是「前后两个 kernel 之间的数据流形状是提前可知的（graph capture bucket 内）或者被 forward context 静态确定」。这就把 P4 锁死在 P1（内存约束）和 P3（forward context 提供静态属性）两个前提上：

  • 没有 P1（内存约束），没人需要融合——单独买一张更贵的卡就行了。
  • 没有 P3（forward context 静态化每个 layer 的 token/parallel 属性），融合 kernel 在 runtime 无法选择正确的 branch，融合反而要每次分支，得不偿失。
原理表述。在 roofline 左侧（bandwidth-bound）工作负载里，性能 = 可用带宽 / 每 token HBM transaction 数。算子融合是唯一能改变「每 token transaction 数」的杠杆。 roofline 简化：perf = min(peak_compute, peak_bandwidth × arithmetic_intensity)。decode 的 intensity 极低（≈ 6~60 FLOP/byte vs H100 峰值 ≈ 2000 FLOP/byte），故 perf ≈ bandwidth × intensity，唯一可优化项是 intensity。融合把相邻的 S 次 HBM read+write 合并为 1 次，强度 S 倍提升。 若 HBM 带宽 10×today 或 on-package HBM 像 SRAM 一样便宜，大部分 decode kernel 进入 compute 受限区 → 融合收益消逝，profiling 不再是关键路径。

    return fusion_pass_compile(graph, example_inputs, compiler_config, ...)

P4 的推论

  • CUDAGraph / NPUGraph / Xlite 是 P4 的极限：把整段 forward 融合成一次 launch。graph capture 的前提（P3：forward context 静态化）越不满足，capture 的 piecewise 越碎，退化为 eager，收益越小。这是 vllm-ascend 仍在填 patch_cudagraph、多种 piecewise fallback 的根本原因。
  • MLAPO 这类「只对特定 head_dim、特定 eps、特定 quant」生效的深度融合，验证了 P4 的一个重要边界：融合收益被其适用条件反向约束——越深的融合越窄，需要在代码里显式 guard（head_dim=128、eps∈{1e-5,1e-6}、compile_range.start > 512）。
P4 的 lifecycle 修正：优化技术会过时，不是永恒真理。本文 P4 范例里引用的 MatmulAllReduceAddRMSNormPass（matmul+AllReduce+AddRMSNorm 融合，内部 ReduceScatter+AllGather 减半流量）正在被 PR #12119 删除（"Remove matmulallreduce and pass optimization techniques"）。同期 #12120 删除 async_exponential 采样优化。这揭示 P4 的一个隐含边界：融合 pass 有生命周期--当 upstream 改了语义、或更优 kernel 出现、或维护成本超过收益时，融合 pass 会被移除。所以 P4 不是「融合越多越好」，而是「在当前 ABI + kernel 覆盖度下的临时最优」。读这份文档时应把每个 pass 视为「快照」而非「恒久设计」。 PR #12119 / #12120 + docs/.../performance_and_debug/optimization_and_tuning.md
P4 的全栈延伸：带宽优化不止在算子层。官方 optimization_and_tuning.md 性能手册显示，P4（减少 HBM round-trip / 带宽倍增）在 vllm-ascend 代码层之下还有几层：
  • HCCL AIV 模式（HCCL_OP_EXPANSION_MODE=AIV）- 通信由 AI vector core 直接用 RoCE 调度，而非 AI CPU。这是 P4 思想在通信栈的体现：把通信调度从慢路径（CPU）移到快路径（vector core）。
  • HCCL_INTRA_ROCE_ENABLE - 8P 间用 RDMA 替代 SDMA；HCCL_BUFFSIZE 控制 NPU 间共享缓存。
  • jemalloc / TCMalloc - 替换 glibc malloc，减少多线程锁竞争（CPU 侧带宽/延迟）。
  • OS 级 - CPU governor=performance、disable swap、disable NUMA balancing、sched_migration_cost_ns=50000。
P4 的内核是「带宽是瓶颈，减少无谓 round-trip」--这个原理从 NPU 算子一直延伸到 CPU malloc 和 OS 调度器，是一个全栈原理，不只是「写融合 kernel」。

P5. 硬件抽象必须零成本 —— 否则插件就是 fork 的另一个名字

vLLM 的「硬件可插拔 RFC」目标是为每种新后端写一个 Platform，无缝替换 CUDA。vllm-ascend 的工程现实揭示了这条路的真实成本结构：NPUPlatform 一个文件覆盖了 PlatformBase 大约 20 处 override，这之外却还有 ~60 个 monkey-patch。换言之，「干净 hook」只是入口；系统的真正语义散落在 scheduler、config validator、torch.distributed、torch.accelerator、FusedMoE 工厂、tool-call parser 等数十处没有 hook 的地方。

P5 不是承认失败，而是揭示了一条第一性原理：硬件抽象的「覆盖度」决定插件的产权维护成本。覆盖度 100% → 纯插件、升级 vLLM main 零成本；覆盖度 < 100% → 每个 upstream 版本都要重新审计 patch 兼容性。vllm-ascend 的 ~60 个 patch 就是这条原理的「技术债快照」。

原理表述。Any heterogeneous plugin for a fast-moving upstream = (clean hooks) + (pragmatic patches)。patch 数量 ≈ upstream 对硬件的隐式假设（CUDA-NCCL-FlashAttn-UDA）尚未被 hook 暴露的部分。 vLLM main 更新频繁（0.7→0.22 走了 18 个月），每个 main 版本都可能移动被 patch 的 symbol，导致 patch break。如果 hook 覆盖不足，每个版本需要人工审计数十个 patch —— 此时「插件」退化为「延迟 fork」。 若未来 VLLM 所有 config validator / scheduler / collective / torch.accelerator 都被 hook 化（即「Hardware Pluggable RFC」完成），patch 理想的命运是被删除。每一条带 "Future Plan: Remove this patch when upstream merges #XXXX" 注释的 patch，都是 P5 的一个活证据。

P5 的推论

  • XP（Expert Parallelism）的 redundant_expert + log2phy 是硬件抽象层没有覆盖的另一个例子：它既不是「CUDA→NPU 替换」问题，也不是「kernel 实现」问题，而是一个 vLLM 代码里尚未专家并行的纯算法调度问题。这条「P5 覆盖不到的算法层」是 vllm-ascend 最有独立的社区贡献价值的地方。
  • TPU / AMD / Intel 如果各自 fork，会在同样的 60 个位置撞墙；若都走 vLLM plugin，则能合力把这些 hook 向上游推进。这个「正向飞轮」正是 P5 的设计动力。
官方政策原文验证 P5（从「我的观察」升级为「项目政策」）。官方文档 docs/.../Design_Documents/patch.md 把 P5 写成明确的项目原则： "Patch is not the best way to make vLLM Ascend compatible. It's just a temporary solution. The best way is to contribute the change to vLLM to make it compatible initially." 三原则：(1) Less is more - 除非别无他法不要 patch；(2) 每个 patch 必须写移除计划（"describe the future plan for removing the patch"）；(3) 随时欢迎清理 patch。这把 P5 从「分析者的洞察」变成「上游-插件协作的契约」。
P5 的活体证据：patch 真在被偿还。近期 PR 直接展示 patch 的生命周期：
  • #11453 Remove obsolete parser patches - 删除 patch_glm47_tool_call_parser / patch_glm_tool_call_streaming / patch_tool_choice_none_content，因为 upstream vLLM 已合并 #45915（GLM streaming parser）+ #44105（empty tool_calls fix）。patch 被删除 = P5 债务被偿还。
  • vLLM 主仓 RFC #11162（Hardware Pluggable）已 closed（2025-06），插件系统落地（#7131 generic plugin Done），但 RFC 自承「what object can be pluggable is not fully defined」--这正是 ~60 patch 仍存在的根因，也是 P5 的边界。

P6. 并行度是调度维度，不只是执行模式

多数推理框架把 TP/EP/PP/CP 当成「执行层面告诉 runtime 怎么切分 tensor」的事情。vllm-ascend 的代码事实揭示了更深的耦合：并行度不是一个下游模块的输入，而是贯穿调度器、block table、forward context、attention backend、allocator 的一等维度。因为它同时影响：

  • KV 缓存的物理布局（PCP 交错、DCP 倍率、HMA 多类型组的块分配）；
  • 通信模式的选择（AllGather / MC2 / AllToAll / FusedMC2 是「device gen × token count × EP × quant」的 4D 查表）；
  • forward context 里的 pad / chunk / flashcomm 决策；
  • 编译 pass 的开/关（SP pass 的 fuse_allreduce_rms 需要 TP size）；
  • EPLB 的 expert-to-rank 映射。
原理表述。并行度在 LMLLM serving 里不是纯「runtime 后端」问题，在多请求动态 batch + 有限 HBM + 慢速 decode 场景下，它改变 KV cache 形状、权重分配形状、通信形状—— 因此必须作为一等调度维度贯穿整个栈。 对比训练：训练 batch 固定、sequence 固定，并行度可以静态确定后冻结；serving 里每步 request 集合不同、token 数随 chunk 动态变化、decode 触发 bandwidth 瓶颈 —— 同一个模型的最优并行配置在不同 token-count bucket 上根本不同（MC2 切换阈值 mc2_tokens_capacity 就是证据）。静态 bind 并行度 → 整体次优。 若出现「单卡装不下带宽但装得下模型」的场景（光学互联 / 3D 封装成熟），并行会从「带宽驱动」切换到「容量驱动」，通信量不再是首要指标 → MC2/AllToAll 选型逻辑退化为简单的 TP=1 全部 local。

MoECommType 的 4D 选择是 P6 最浓缩的代码。select_moe_comm_method 用 device gen、token 数、EP world size、quant type、spec/draft 五维做判断树 —— 这是把并行度当成调度维度的最显眼证据。

Forward context 把并行度「翻译」成每个 layer 能读懂的属性。pad_size = (tp_world_size - (num_tokens % tp_world_size)) % tp_world_size 是一次把「TP 并行度」翻译成「每层必须 pad 到 TP 倍数」的显式映射。没有这个翻译，attention kernel 拿到的 tensor shape 跟它的并行布局不匹配。

# ascend_forward_context.py:141-144

    pad_size = (tp_world_size - (num_tokens % tp_world_size)) % tp_world_size
    forward_context.pad_size = pad_size

P6 的推论

  • 未来要并入的并行维度（如 PP disaggregation、sequence parallelism 在 decode 上的应用）必须作为调度维度进入 forward context，而不是关在某个「parallel backend」里。
  • flash_comm_v2_enabled = flashcomm2_enable() and tp_world_size > 1 这条简单判断覆盖了所有未来可能出现的 sequence parallel 变体—— forward context 是 P6 的「维度注册中心」。
社区演进：并行度维度在主动扩张（P6 的活体证据）。近期 PR 不是「并行度配置选项变多」，而是在创造全新的并行维度，且每个都回溯进入调度层：
  • #11976 Virtual Pipeline Parallelism (VPP)（open）- V 形折回拓扑（even stage rank 0→N-1，odd stage N-1→0），纯 Python monkey-patch、零 upstream 修改、DeepSeek-V3.1 上 性能超过 CPP。这是一个被 P5（patch 机制）创造出来的全新并行维度，而不是上游 hook 化的结果。VPP 把「一个 logical forward 切 vp_size 个 virtual stage」注入 PP 调度回路。
  • #12292 NonBSP（open）- 打破 Bulk Synchronous Parallel 假设，跨 DP rank load-aware 调度，优化「通信气泡」。bubble_threshold=5.0、long_req_block_threshold=700。这是 DP 维度从「静态对等」演进到「动态负载感知」的信号。
  • #8497 Elastic EP Milestone 2（open）- EPLB 从「静态重平衡」升级到「动态专家并行扩缩容」：AscendElasticEPScalingExecutor、MC2 通信组从 GroupCoordinator 切到 StatelessGroupCoordinator、standby_state.py 待命通信组。EP world size 变成运行时可变量。
  • #8961 auto-scaling load balance proxy（open）- PD 分离演进成完整自动扩缩容系统（e-p-d / e-pd / p-d 部署模式自动检测、/instances/add|remove API、monitor_and_scale 0.5s 间隔），PD 比例成为运行时调度维度。
这四个扩张的共同模式：并行度不是「执行前静态选定」的超参，而是「运行时调度回路的一等输入」--这正是 P6 的核心主张。VPP 尤其值得注意：它证明 vllm-ascend 不只在「配置已有并行度」，而是在创造新的并行范式（V 形折回），且靠 P5 的 patch 机制零成本实现。 PR #11976 / #12292 / #8497 / #8961

综合推论：六条原理如何咬合成一个系统

六条原理不是彼此独立的公理，它们之间存在强约束关系——每一条的存在都以另外一条为前提，去掉任何一条其他几条都要崩塌。这张依赖图才是「第一性原理」真正该长成的样子：

  P1 内存是硬约束
   ├─ 需要 batch 来填饱算力 ──────────────► P2 服务 = 连续批调度
   │    ├─ batch 每种请求变化太快 ────────► P3 Python 控制面 + kernel 数据面
   │    │    └─ kernel 选择需要静态化 ───► P3 推论：forward context = capability negotiation
   │    └─ 带宽是唯一可用杠杆 ────────────► P4 算子融合 = 带宽倍增器
   │         └─ 融合条件静态化 ←─────────── P3 forward context
  P1 内存是硬约束
   └─ 多种后端不能各 fork ───────────────► P5 硬件抽象必须零成本
        └─ 覆盖度 < 100% 时退化为延迟 fork
  P1 × P2 交叉
   └─ 并行度是连续调度的调度维度 ───────► P6 并行度是调度维度
        └─ 每一层决策依赖 ←────────────── P3 forward context

推论 1：三个「架构级必然模块」

从六条原理可必然地推出 LLM serving 系统必须存在三个模块，vLLM 都实现了，而且三个模块彼此不重叠：

必然模块被哪条原理强制vLLM 的对应
Cache managerP1PagedAttention / block_table / CompressAttentionManager / prefix cache
Continuous batcher / schedulerP2 + P1Scheduler / SchedulerOutput / ProfilingChunk / Recompute / SRF
Operator fusion + kernel dispatch layerP4 + P3compilation passes / ACLGraph / forward_context capability negotiation
Hardware abstraction layerP5 + P6PlatformBase / NPUPlatform / register_* hooks

推论 2：「未来工作」的位置

每条原理的「可证伪条件」反向给出了未来扩张的优先级：

  • P1 证伪：HBM 单卡 1 TB+ → KV cache manager 可以简化，但在此之前 prefix caching / quantization / recompute / sleep mode 仍是主战场。
  • P2 证伪：当前不存在，故 continuous batching 在可预见的未来都成立。
  • P3 证伪：accelerator-driven scheduling coprocessor 不存在，故 Python 控制面会长期存在；但 torch.multiprocessing 与 patch 机制的摩擦可能以「kernel 下沉调度」方式局部打破 P3。
  • P4 证伪：HBM 成熟 → 消融实验显示 fusion 在 decode 上仍有 10-30% 收益的原因是权重本身要读，这部分省不掉。
  • P5 证伪：VLLM「Hardware Pluggable RFC」完成 → 这是 vllm-ascend 最直接的贡献方向。
  • P6 证伪：单卡大带宽成熟 → TP/EP/CP 的复杂选择树可以简化，全局化并行度配置。

推论 3：真正的工作不在 kernel，在调度与抽象

vllm-ascend 这个代码库给我最深的感受是：真正决定一个 LLM serving 系统是不是「能用、好用、通用」的，不是某个 kernel 跑多快，而是三件事有没有想清楚：(1) 内存的「所有权」如何分配与回收（P1）；(2) 请求作为「动态物体」如何被调度（P2+P6）；(3) Python 控制面与 kernel 数据面之间那条桥够不够静态、够不够窄（P3+P4）。 如果这三件事没想透，再好的 kernel 都会被调度层的二次污染吃掉；想透了，每个 kernel 的优化空间才是干净的。 vllm-ascend 的 8 个编译 pass、MLAPO prolog、四层 MoECommType、rew 三阶段 EPLB、CaMemAllocator sleep mode、ProfilingChunkScheduler 闭式 chunk solver —— 每一个都是某一组原理在一个特定硬件（Ascend NPU, A2/A3/A5/310P）上的具体投影。理解原理不会让你直接写出这些模块，但能让你判断在哪里写、为什么在这个位置写、不写会破坏哪条原理。

一句话总结

vLLM 架构的第一性原理是：在 HBM 容量这个硬约束下，用连续批化填满带宽、用 forward context 静态化控制面与数据面、用算子融合在 roofline 左侧叠加带宽利用率、用最小化的 Platform hook 把这一切向零成本硬件抽象方向推进——其中任何一条失效，整个工程现实都会从「可维护插件」退化为「持续审计的延迟 fork」。
