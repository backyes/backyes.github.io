1. 问题定义与分析边界

本报告聚焦三个递进问题：

  1. vLLM 如何把 token 位置映射到 HBM 上的物理 KV 块？—— block_table / slot_mapping 抽象。
  2. vLLM_Ascend 的 DSA（DeepSeek Sparse Attention）如何为每个 query token 算出"要 attend 哪些 KV block"？—— lightning indexer。
  3. DSA 算出的 index_id 最终如何驱动 paged KVCache 的查表（pageattention gather）？—— 这是本报告的核心。
为什么这个问题关键：Ascend 上不是"全量 SWA/MLA"，而是"HBM 上的 paged KVCache + DSA 稀疏索引"。只有理清 index_id → block_table → pageattention 这条线，才能理解 Ascend attention 的数据流本质——indexer 的输出不是一个 dense 的 token 列表，而是一组 block 级别的索引，它必须经过 block_table 的二次寻址才能落到 HBM 物理页。

3. slot_mapping → block_table → HBM 物理化

这是所有 paged attention 的基石。核心公式（两处等价实现）：

3.1 Triton kernel（vLLM 主路径，block_table.py:388-411）

virtual_block_indices = pos // virtual_block_size
block_indices = virtual_block_indices * BLOCKS_PER_KV_BLOCK
               + local_block_offsets // block_size
block_numbers = tl.load(block_table_ptr + row_offset + block_indices)  # 物理块号
slot_offsets = local_block_offsets % block_size
slot_ids = block_numbers * block_size + slot_offsets
slot_ids = tl.where(is_local, slot_ids, PAD_ID)

3.2 NumPy 路径（Ascend 310P，_310p/block_table.py:44-49）

logical_block_idx = positions // self.block_size
block_table_indices = self._get_block_table_indices(req_indices, logical_block_idx)
block_numbers = self.block_table.np.ravel()[block_table_indices]   # = block_table[req, logical_block_idx]
block_offsets = positions % self.block_size
np.add(block_numbers * self.block_size, block_offsets, out=self.slot_mapping.np[:num_tokens])
统一公式：
slot_mapping[token] = block_table[req, pos // block_size] * block_size + (pos % block_size)
即：逻辑块号 → block_table 查表得物理块号 → 乘 block_size 加 inblock offset = HBM 上的 flat slot。

3.3 block_table 矩阵本身

BlockTable.__init__（worker/block_table.py:97）分配 [max_num_reqs, max_num_blocks_per_req] int32 矩阵，第 i 行记录请求 i 被分配的物理 block id 列表，由 append_row（worker/block_table.py:114-130）写入。_get_block_table_indices（_310p/block_table.py:53-55）给出行内偏移：req_indices * row_stride + logical_block_idx。

5. DSA 全景：多层 cache 与多段压缩

DSA 的"稀疏"不是单一操作，而是 多层 cache + 压缩 + 索引 的组合。compress_ratio 决定层级：

6. 核心：DSA indexer 如何算出 index

indexer 的任务：为每个 query token 选出"最相关的 index_topk 个 KV block"。实现是 NPU 自定义 op npu_vllm_quant_lightning_indexer（lightning indexer）。

6.1 入口：indexer_select_qli（dsa_v1.py:2630）

def indexer_select_qli(self, x, qr, kv_cache, attn_metadata, cos, sin,
                       actual_seq_lengths_query, ...):
    # 1) 准备 indexer 的 q/kv + 写 indexer_state_cache
    q, kv, ik, isc, ifc, indexer_kv_state_meta, isc_meta, indexer_slot_mapping, wp = \
        self._indexer_qkv_prepare(x, qr, kv_cache, attn_metadata, ...)
    # 2) 学到的 indexer weights
    weights = self.weights_proj(x) * (self.indexer_softmax_scale * self.indexer_heads**-0.5)
    # 3) quant + scatter + 调用 lightning indexer
    return self._indexer_qli_finish(q, kv, weights, ik, isc, ifc, ...)

6.2 _indexer_qkv_prepare：indexer 的 Q/KV prolog（dsa_v1.py:2434）

这段做三件事：

  1. Q 投影：q = indexer_wq_b(qr) → RoPE → Hadamard 旋转（rotate_activation）。
  2. KV 压缩写入：调用 torch.ops._C_ascend.compressor 把当前 token 的 KV 压缩写入 indexer_state_cache（走 state_block_table 查表 scatter）。
  3. 返回 indexer_k_cache、indexer_scale_cache、indexer_slot_mapping 等供下一步用。

6.3 _indexer_qli：lightning indexer 调用（dsa_v1.py:2584-2628）—— 算出 index 的瞬间

def _indexer_qli(self, q, weights, q_scale, indexer_k_cache, indexer_scale_cache,
                 indexer_kv_scale_metadata, with_prefill):
    # 取 indexer 自己的 block_table 和 qli_metadata
    block_table = indexer_kv_scale_metadata.prefill.block_table   # ← indexer 独立 block_table
    qli_metadata = indexer_kv_scale_metadata.prefill.qli_metadata

    topk_idxs, _ = torch.ops._C_ascend.npu_vllm_quant_lightning_indexer(
        query=q_quant,           # indexer query (int8)
        key=indexer_k_cache,     # indexer 的量化 key cache
        weights=...,             # 学到的 block 级权重
        query_dequant_scale=...,
        key_dequant_scale=...,
        actual_seq_lengths_query=qlens,
        actual_seq_lengths_key=kvlens,
        block_table=block_table, # ← 关键：indexer 在 paged cache 上做 top-k
        metadata=qli_metadata,
        layout_query="TND",
        layout_key="PA_BSND",    # ← paged-attention block-slot-ND 布局
        sparse_count=self.index_topk,  # kv_budget，如 256
        sparse_mode=3,
        cmp_ratio=4,
    )
    return topk_idxs   # [num_tokens, index_topk] 的 block 级索引
关键洞察：lightning indexer 的输入 layout_key="PA_BSND" 和 block_table 表明——indexer 本身就是在 paged KVCache 上做评分。它不是输出 token 级别的索引，而是输出 block 级别的 top-k 索引（topk_idxs 的每个元素是"该 query token 应该 attend 的第 k 个 block 在 block_table 行内的位置"）。
关于 hash：DSA indexer 路径里出现的所有 hadamard/rotate_activation 都是 Hadamard 正交旋转（量化友好），不是 hash。代码库中另有基于真 SimHash + 汉明距离的独立稀疏路径（kvcomp_attn）以及与 KV 索引无关的 MoE 路由 hash——详见 §10 三种机制的完整辨析。

6.4 IndexCache：跨层复用 index

_get_indexcache_topk_indices / _update_indexcache_topk_indices（dsa_v1.py:1473-1491）在 skip_topk 或 use_index_cache 时让 DSA 跨层复用同一组 topk 索引，避免每层重算 indexer。decode 段占 buffer 前段、prefill 段占 [num_decode_tokens:]。

7. 关键转换：index_id → pageattention 查表

这是本报告的核心。DSA indexer 输出的 compress_topk_idxs（block 级 top-k 索引）如何最终驱动 paged KVCache 的 gather？

7.1 答案：index 直接作为 sparse attention op 的 cmp_sparse_indices 参数

在 _forward_prefill（dsa_v1.py:2081-2101）和 _forward_decode（dsa_v1.py:2390-2410）中：

7.3 index_id → pageattention 的转换语义

在 npu_sparse_attn_sharedkv 内部发生的融合操作：

对每个 query token t，对每个 top-k 位置 k： 
block_index = compress_topk_idxs[t, k]          # indexer 输出的 block 级 index_id
physical_block_id = cmp_block_table[req, block_index]  # ← 二次寻址：index_id → 物理块号
kv_block = cmp_kv_cache[physical_block_id]      # ← pageattention gather：物理块号 → HBM 上的 KV 页
然后对 gather 出的 index_topk 个 KV 块做 fused attention matmul。

也就是说：不存在独立的 gather kernel——cmp_sparse_indices + cmp_block_table + layout_kv="PA_ND" 三个参数一起，让 sparse attention op 在内部完成"index_id → block_table 查表 → 物理页 gather → attention"的融合。

7.5 与 vLLM GPU 侧 sparse path 的对照

vLLM GPU 侧 MLA sparse（vllm/v1/attention/backends/mla/flashmla_sparse.py）用 Triton kernel _convert_req_index_to_global_index_kernel（sparse_utils.py:12）做完全同构的转换：

9. 关键设计洞察（芯片/系统视角）

9.1 indexer 拥有独立的 block_table 是核心架构决策

DSA 不是"一套 block_table 打天下"。swa / compressor / indexer 各有独立 block_table，因为它们的物理分页不同：SWA 存原始 sliding-window token，compressor 存压缩后的 block，indexer_k_cache 存量化后的 indexer key。indexer 在自己的 paged cache 上评分，选出的是"compressor cache 的行内 block 索引"，再经 compressor 的 block_table 二次寻址到 HBM。两层 indirection 是 DSA 稀疏性的物理基础。

9.2 "index_id → pageattention"是融合语义，不是两步 kernel

与 GPU 侧 FlashMLA sparse 先 Triton 转全局 slot 再调 kernel 不同，Ascend 把 cmp_sparse_indices + cmp_block_table 作为 sparse attention op 的输入参数，在 kernel 内部完成"index_id → 物理页 → gather → matmul"融合。这减少了中间 tensor 的 HBM 往返，对 Ascend 的 HBM 带宽利用更友好。

9.3 slot_mapping 的 2D 格式化是 Ascend 特有

format_dsa_slot_mapping（device_op.py:903）把 slot 拆成 [block_idx, offset] 2D 形式，供 npu_scatter_nd_update_v2 直接做 scatter——这是 NPU DMA 的偏好格式（块+偏移分离），区别于 GPU 用 flat slot。

9.4 multistream 重叠：indexer 与 Q/KV 计算双流水

cv_indexer_select_qli（dsa_v1.py:2670）把 indexer 的 weights_proj 放在 aux_stream 与 main stream 的 q_quant/kv_scatter 并行（4-block 分段），是典型的 Cube/Vector 引擎重叠策略。_mla_prolog_multistream（dsa_v1.py:1730）也是同样模式。

9.5 compress_ratio 是架构可扩展的"旋钮"

通过 compress_ratio ∈ {1, 4, 128}，同一套代码覆盖：纯 SWA（轻量）、SWA+粗压缩（128）、SWA+细压缩+indexer（4）。indexer 只在 compress_ratio=4 时启用——这是 DeepSeek-V4 风格原生稀疏注意力的配置点。

10. 索引过程有 hash 计算么？三种选择机制的辨析

10.2 路径 A —— DSA indexer 的"Hadamard"不是 hash

DSA indexer 路径里出现的 rotate_activation / hadamard 容易被误认为是"hash"，但它是 Hadamard 正交旋转变换，不是 hash：

# attention/dsa_v1.py:82-84
def rotate_activation(x, hadamard):
    return hadamard_transform_ref(x, hadamard=hadamard, scale=hidden_size**-0.5)
    # 本质：out = x @ H * scale，H = scipy.linalg.hadamard 生成的正交矩阵

# 构造 (dsa_v1.py:432-454)：QR-orthogonalized Hadamard 矩阵，归一化到 bf16
AscendDSAMetadataBuilder.hadamard = torch.tensor(
    hadamard(dim_padded, dtype=float), ...).to(torch.bfloat16)

query 和 key 两侧都施加同一矩阵（dsa_v1.py:2482 q 侧、:2526 kv 侧）。正交变换保内积：(Hq)·(Hk) = q·k。它的作用是在 int8 动态量化评分之前旋转激活分布，使量化误差对点积的扰动最小——是量化友好旋转，没有任何二值化、降维、桶冲突等 hash 特征。

判定依据：哈希的核心是"把输入映射到有限桶空间并允许冲突"。Hadamard 旋转是正交可逆线性变换，输入输出维度相同，没有信息丢弃或桶映射 → 它不是 hash。

# worker/kvcomp_utils.py:456-460 compute_hash: sign(x @ W) 二值化 + bit-pack

10.5 为什么有两套稀疏选择机制？

13. batch prefill 查表：多请求如何共享 block_table

本章回答一个更底层的问题：batched prefill 中，不同请求的 index_id（各自 0-N 的 block 序号）怎么共用一套 block_table 查表结构？

13.1 核心结论：行=请求的二维 tensor + query_start_loc 请求寻址

不是"共用一套逻辑表"，而是"每个请求独立一行，打包进同一个二维 tensor，用 query_start_loc 做请求寻址 + index_id 做行内 block 寻址"——两级索引天然隔离。

13.2 block_table 是二维矩阵，行 = 请求

# worker/block_table.py:97
self.block_table = self._make_buffer(max_num_reqs, logical_table_size, dtype=torch.int32)
# shape: [max_num_reqs, max_num_blocks_per_req]
#  第 i 行 = 请求 i 被分配的物理 block id 列表（append_row 写入, :114-120）

每个请求在 block_table 里占独立一行，行号 = 请求在 batch 中的索引。不同请求的 index_id=0 指向的是各自行的第 0 列，物理块号完全不同——物理隔离靠的就是这个"行"。

13.3 query_start_loc：token → 请求 的映射表

kernel 拿到的是所有 prefill 请求 token 的拼接序列，它通过 query_start_loc（即 cu_seqlens_q）确定每个 token 属于哪个请求：

query_start_loc = [0, 128, 400, 1024]   # shape: [num_reqs + 1]，累积边界
                    ↑    ↑    ↑     ↑
                  req0  req1  req2  总 token 数

token [0, 128)   → 请求 0 → block_table 第 0 行
token [128, 400)  → 请求 1 → block_table 第 1 行
token [400, 1024) → 请求 2 → block_table 第 2 行

kernel 内部用 token 位置查 query_start_loc（二分/顺序）确定 req_idx，再取 block_table[req_idx] 这一行。这是 vLLM 标准的"packed sequence + cu_seqlens"抽象。

13.4 index_id 是"请求内的 block 序号"，不是全局物理块号

indexer 输出的 compress_topk_idxs[t, k] 语义：

"token t 应该 attend 的第 k 个 block，在它所属请求的 block_table 行内的列位置"

完整查表是两级：

req_idx         = query_start_loc 查 token t 属于哪个请求
block_idx       = compress_topk_idxs[t, k]                   # 行内的 index_id (0-N)
physical_block  = block_table[req_idx, block_idx]            # 最终物理块号
kv_data         = kv_cache[physical_block]                   # HBM 上的页 → pageattention gather

13.6 为什么这样设计？（系统视角）

设计选择原因
行=请求的二维 tensor，而非每请求一个独立 tensor单次 kernel launch 处理整个 batch，避免 per-request 小 kernel 调度开销；GPU/NPU 对规整二维访存友好
query_start_loc 做请求寻址变长请求（各请求 token 数不同）无需 padding 到等长，紧凑拼接 + 累积边界即可
index_id 是请求内序号indexer 评分天然是"请求内"的（每个请求的 KV 独立），输出不需要知道自己在 batch 中的位置，indexer 与 batch 解耦
block_table 行内是物理块号逻辑 block 序号 → 物理块号的映射由 block_table 完成，不同请求的逻辑 block 0 映射到不同物理页，隔离性由分配器保证

14. block_table.py 该放在 worker/ 下吗？职责与位置分析

BlockTable 是 paged KVCache 的索引核心，但它被放在 vllm_ascend/worker/ 下。从职责上看它更像核心数据结构，并不天然属于"worker 框架"。这是一个"概念上可争议、实践中有道理"的折中。

14.1 核心结论

BlockTable 类本身是硬件无关的纯数据结构 + 索引计算（无 NPU 专有 import、无状态、无调度逻辑）。从职责纯粹性看，它更适合放在 core/ 与 kv_cache_interface 同级。但放在 worker/ 下有与上游对齐 + InputBatch 生命周期的合理理由。短期保持现状（对齐价值高），长期下沉到 core/ 更干净。

14.3 为什么放在 worker/ 下？（站得住脚的理由）

理由 A：vLLM 上游也这么放（最强的理由）

vllm/vllm/v1/worker/block_table.py          ← 上游
vllm_ascend/worker/block_table.py           ← Ascend 镜像

vllm_ascend 作为 vLLM 的下游移植，目录结构主动与上游对齐——便于 upstream patch cherry-pick、贡献回上游时 diff 清晰、新同事从 vLLM 转来时心智模型一致。

理由 B：BlockTable 是 InputBatch 生命周期的一部分

这些 append_row/move_row/swap_row/clear_row 是调度器驱动的状态变更，BlockTable 被 InputBatch 持有，而 InputBatch 是 worker 的核心运行时对象。

理由 C：历史分层：worker 管状态，attention 管计算

vLLM v1 的架构分层：worker/ 管运行时状态（input batch, block table, slot mapping），attention/ 管计算（kernels, backends）。BlockTable 是"状态"而非"计算"，放在 worker/ 符合这个分层。

14.4 但批评确实有道理（放在 worker/ 的问题）

问题 A：worker/ 目录职责混杂

worker/
├── block_table.py          ← 纯数据结构（更像 core/）
├── npu_input_batch.py      ← 运行时状态容器
├── model_runner_v1.py      ← 框架编排
├── worker.py               ← 框架编排
├── kvcomp_utils.py         ← KV 压缩工具
├── pcp_utils.py            ← 并行工具
├── v2/                     ← 下一代

block_table.py 和 npu_input_batch.py 是被框架使用的组件，而 model_runner_v1.py / worker.py 是框架本身。它们混在一起，违反了"依赖方向单向"的原则。

问题 B：core/ 目录是更自然的归宿

core/
├── kv_cache_interface.py              ← KV cache 抽象接口
├── single_type_kv_cache_manager.py    ← cache 管理器
├── recompute_scheduler.py

core/ 已经存放了 KV cache 管理的核心抽象。BlockTable 作为 paged KVCache 的索引核心，与它们是同一类抽象，放在 core/ 下语义上更精确。

问题 C：Ascend 的 BlockTable 与上游完全解耦，"跟随上游"的理由弱化

# vllm_ascend/worker/block_table.py:14

# 对比 vllm_ascend/worker/v2/block_table.py:25

v1 的 BlockTable 是独立重写的（只借了 _compute_slot_mapping_kernel 一个函数），并不受 upstream 类层次约束。"放在 worker/ 是为了和上游对齐"这个理由对 v1 版其实不够强。反倒是 v2 的 AscendBlockTables(BlockTables) 才真正需要和上游同目录。

14.5 总结对比与判断

维度当前位置 worker/更合理位置 core/
职责匹配△ 是 InputBatch 的一部分✅ 与 kv_cache_interface 同级
依赖方向⚠️ 与框架编排混在一起✅ 被框架依赖，不依赖框架

判断：

  • 短期：保持 worker/ —— 与上游对齐的价值 > 职责纯粹性，且改动成本高（所有 import 路径要改）
  • 长期：如果要与上游适度解耦、建立 Ascend 自己的核心抽象层，把 block_table.py + npu_input_batch.py 下沉到 core/ 是更干净的选择。特别是 v2 已经在 worker/v2/ 里重建了一套，v1 的 BlockTable 未来可能被废弃，届时 core/ 版就是正统。

17.3 逐变量解读：为什么"写"和"读"必须分表

（1）compressor_attn_metadata — 读端：主 attention 如何消费压缩 KV

命名陷阱：听起来像"压缩器的 attention"，实际上是"消费压缩 KV 的主 attention"。这个 metadata 的 block_table 指向 compress_kv_cache（主 attention 层自己的 KV cache），驱动最终那个"同时 attend SWA + 压缩 KV"的主 sparse attention：

# dsa_v1.py:2081 —— 主 attention 读 compress_kv_cache
attn_output = attn_op(
    q, ori_kv=swa_kv_cache,
    cmp_kv=compress_kv_cache,                          # ← 读：主 attention 自己的 cache
    cmp_sparse_indices=compress_topk_idxs,
    ori_block_table=swa_prefill_metadata.block_table,
    cmp_block_table=compressor_prefill_metadata.block_table,  # ← 读端 block_table
    metadata=compressor_prefill_metadata.sas_metadata,        # ← 读端 sas_metadata
    cmp_mask_mode=3, ori_mask_mode=4, ...)[0]

读端 block_table 形状：行数 = num_prefill_tokens（原始 token 数 N）。主 attention 在"原始 token 粒度"上做 top-k 选 block——每个原始位置选 index_topk 个压缩 block 来 attend。读端请求边界 cu_seqlens_q = actual_seq_lengths_query（原始 token 的累积长度）。

compress_common_attn_metadata = compressor_attn_metadata（dsa_v1.py:1844）确认它是"common"（共享）attention 路径。

（2）compressor_kv_state_metadata — 写端：主 compressor 如何产生压缩态

这个 metadata 的 block_table 指向 state_cache（Compressor 的 AscendCompressorStateCache，物理上不同于 compress_kv_cache）。主 compressor 在 prefill 时逐步累加地把 token KV 压缩写入 state_cache：

# dsa_v1.py:2003-2022 —— 主 compressor 写入 state_cache（累加态）
compressed_kv = torch.ops._C_ascend.compressor(
    hidden_states,                     # 当前 token 的 hidden state
    self.compressor_wkv.weight,
    self.compressor_wgate.weight,
    state_cache.squeeze(-2),           # ← inout：累加态（kernel 内部读写 state_cache 做递归压缩）
    self.compressor_ape,               # 绝对位置编码参数
    self.compressor_norm.weight,
    compress_sin, compress_cos,
    state_block_table=compressor_state_prefill_metadata.block_table,  # ← 写端 block_table
    start_pos=common_prefill_metadata.start_pos,
    cmp_ratio=self.compress_ratio,     # 每 ratio 个 token 压缩成 1 个 entry
    cache_mode=1, ...)                 # cache_mode=1 = 累加模式

写端 block_table 形状：行数 = num_compressed_tokens ≈ N // ratio + R（_num_compressor_metadata_rows，dsa_v1.py:530-544）。这就是写/读分表的根本原因：写端在"压缩粒度"（行数少），读端在"原始粒度"（行数多）。

为什么"写"和"读"不能合并成一张表？——关键算法差异

维度写端 (compressor_kv_state)读端 (compressor_attn)
物理 cachestate_cache（AscendCompressorStateCache，head_size=state_dim 大）compress_kv_cache（主 attention KV cache，head_size=标准 head_dim）
block_table 行数N // ratio + R（压缩粒度，行数少）N（原始粒度，行数多）
block_size由 _dsv4_block_sizes() 决定（state cache 用更大的逻辑块存累加态）标准 cache_config.block_size（如 128）
slot_mappingcompress_slot_mapping（2D [block_idx, offset]，按压缩位置寻址）无独立 slot_mapping（读端走 paged block_table + paged attention gather）
操作语义累加写入：kernel 递归更新 state_cache，每 ratio 个 token 产生 1 个 entry选择性读取：按 top-k indices gather 特定 block 做 attention
访问模式顺序写入（prefill 逐 token 流）随机访问（按稀疏索引 gather 任意 block）

本质原因：压缩器是状态ful 的递归操作——state_cache 是累加态（类似 RNN hidden state），每个新 token 到来时 kernel 在 state_cache 上做读-改-写（cache_mode=1）。这个累加态必须单独存在——不能在原始 token 粒度的 compress_kv_cache 上直接累加，否则会破坏正在被 attention 读取的已完成压缩数据。写完后，再通过 dsa_kv_compress_scatter(compress_kv_cache, compressed_kv, ...)（dsa_v1.py:2040）把"压缩好的最终结果"scatter 到 compress_kv_cache 的对应位置，供主 attention 读取。state_cache 是草稿纸（写端），compress_kv_cache 是成品库（读端）。

（3）indexer.compressor.state_cache（indexer_state_cache） — indexer 的写端

indexer 自己有独立的 compressor（独立权重 indexcom_wkv/indexcom_wgate）。这个 compressor 把 token 压缩写入 indexer_state_cache（元组 [3]），作为 indexer 内部用于评分的累加态：

# dsa_v1.py:2502-2521 —— indexer 的内部 compressor 写入 indexer_state_cache
kv = torch.ops._C_ascend.compressor(
    x,
    self.indexcom_wkv.weight,        # ← indexer 专用压缩权重（与主 compressor_wkv 不同！）
    self.indexcom_wgate.weight,
    indexer_state_cache.squeeze(-2),  # ← inout：indexer 自己的累加态
    self.indexcom_ape, self.indexcom_norm.weight,
    state_block_table=kv_block_table, # ← indexer_state_metadata 的 block_table
    cmp_ratio=self.compress_ratio, cache_mode=1, ...)

为什么不能合并到 (2)：主 compressor 和 indexer compressor 是两次独立的压缩递归。主 compressor（compressor_wkv）给主 attention 提供压缩 KV；indexer compressor（indexcom_wkv）给 lightning indexer 提供用于 top-k 评分的 key 表示。两套权重训练目的不同、数值不同，各自维护独立累加态——合并会破坏递归状态，导致压缩结果错误。

（4）indexer.k_cache（indexer_k_cache） — indexer 的读端

这是 lightning indexer 的评分目标。indexer 从 indexer_state_cache（累加态）进一步派生出量化的 indexer_k_cache（元组 [4]）+ scale cache（元组 [5]），作为评分 key：

topk_idxs, _ = torch.ops._C_ascend.npu_vllm_quant_lightning_indexer(
    query=q, key=indexer_k_cache,            # ← 读端：indexer 的量化 key cache
    weights=...,                             # indexer 的 learned per-head weights
    block_table=block_table,                 # indexer_scale_metadata 的 block_table（读端）
    query_dequant_scale=q_scale,
    key_dequant_scale=indexer_scale_cache,   # ← 读端量纲（元组 [5]）
    metadata=qli_metadata,
    sparse_count=self.index_topk, sparse_mode=3, ...)

为什么 (3) 和 (4) 不能合并：(3) indexer_state_cache 是 indexer compressor 的累加态（fp32 高精度，head_size=2×coff×head_dim 大），在压缩过程中反复读写；(4) indexer_k_cache 是从累加态派生出的量化评分 key（int8/fp8 精度，head_size=标准 head_dim 小），只读。两者的物理存储格式、精度、block_table 行数都不同。混淆会让累加态被量化破坏，压缩递归无法正确进行。

18.3 Indexer cache 结构：物理分离

SFA 与 DSA 的关键结构差异：indexer cache 与主 MLA cache 物理分离：

# sfa_v1.py:1460-1513 —— _compose_sfa_kv_cache
def _compose_sfa_kv_cache(self, kv_cache):
    main_cache = kv_cache                       # 主 MLA cache
    indexer_cache = self.indexer.k_cache.kv_cache  # ← indexer 独立 cache（不同物理显存）
    if self.use_sparse_c8_indexer:
        # main: (packed_kv_cache,) + indexer: (indexer_k_cache, indexer_scale_cache)
        return (main_cache[0], indexer_cache[0], indexer_cache[1])
    # main: (k_cache, v_cache) + indexer: (indexer_k_cache,)
    return (main_cache[0], main_cache[1], indexer_cache[0])

这由 AscendSFAIndexerBackend（attention/indexer.py:14）管理。它是占位后端——仅让 cache planner 为 indexer cache 分配独立物理 tensor、同时共享主 MLA cache group 的 block id，真正的索引计算在 AscendSFAImpl 内完成。模型层的 Indexer 类（models/deepseek_v4.py:525-589）其 forward() 直接 return None，只是权重容器。

19. kvcomp_attn Hamming Sparse 算法起源（SimHash → HashEvict → 现代发展）

kvcomp_attn 的 Hamming sparse 选择机制不是孤立设计，而是SimHash（ locality-sensitive hashing）家族在 LLM KV cache 压缩/选择上的直接应用。

19.1 理论基石：SimHash 与 Hamming ≈ Cosine

kvcomp_attn 的 HashEncoder 实现的是经典 SimHash（Charikar 2002）：

# worker/kvcomp_utils.py:399-404 —— 投影权重的构造：高斯随机 + QR 正交化
random_weights = torch.normal(mean=0, std=2, size=(input_dim, hash_bits), ...)
Q, R = torch.linalg.qr(random_weights)
self.hash_weights = Q * torch.sign(torch.diag(R))

# worker/kvcomp_utils.py:456-460 —— compute_hash: sign(x @ W) → 二值化 → bit-pack
xW = torch.matmul(x_flat, self.hash_weights)        # [N, hash_bits]
packed = torch.ops._C_ascend.npu_sign_bits_pack(xW_flat, size=1)  # 8 bit → 1 uint8

数学理论基础（Charikar, "Similarity Estimation Techniques from Rounding Algorithms", STOC 2002）：

对随机投影向量 w，sign(x·w) 是一个随机超平面哈希。两个向量 x, y 的哈希位相等（即超平面同侧）的概率为：P[sign(x·w) = sign(y·w)] = 1 - θ/π，其中 θ 是 x 与 y 的夹角。由此：Hamming 距离 ≈ 角距离（angular distance）≈ 余弦距离。hash_bits 位中相同位的比例与余弦相似度单调相关。

这意味着：对 KV cache 中每个 block 预计算其 key 的 SimHash 签名；查询时只需计算 query 的 SimHash，与所有 block 签名做 Hamming 距离（XOR + popcount），最小的那些就是最相关的 block。一次 XOR + popcount 替代了一次完整的 q·k 点积。

19.2 算法祖先：HashEvict（与 kvcomp_attn 算法同构）

kvcomp_attn 的直接算法祖先是 HashEvict（arXiv 2412.16187, Liu et al. 2024）：

Minghui Liu, Tahseen Rabbani et al. · arXiv 2412.16187 · 2024-12

摘要核心："HashEvict uses locality-sensitive hashing (LSH) to compress the KV cache... computing the Hamming distance between binarized Gaussian projections of the current token query and cached token keys, with a projection length much smaller than the embedding dimension... makes these decisions pre-attention, thereby reducing computational costs. HashEvict is dynamic - at every decoding step, the key and value of the current token replace the embeddings of a token expected to produce the lowest attention score."

19.5 为什么随机投影 + Hamming 能 work？——信息论直觉

HashEncoder 的 hash_weights（worker/kvcomp_utils.py:399-404）的高斯随机投影 + QR 正交化有一个重要性质：

  • 高斯随机投影是 Johnson-Lindenstrauss 引理的实例——高维空间中的点集可以被投影到 O( log N / ε² ) 维而保留 ε-近似的成对距离
  • QR 正交化使投影各向同性（各方向方差均等），进一步提升 Hamming 距离与余弦相似度的相关性
  • npu_sign_bits_pack（kvcomp_utils.py:458）把每 8 bit 压成 1 uint8，使 Hamming 计算 = 1 次 XOR + popcount 覆盖 8 个超平面

直觉：想象每个 KV block 的 key 是 128 维空间的一个点。128 个随机超平面把空间切成 2^128 个"锥体"。Hamming 距离小的 block = 在同一锥体或邻近锥体 = 与 query 的方向接近。不需要真的计算 q·k，看它们被哪些超平面切成同侧就能近似排序。

19.6 HashEvict 论文的核心实验发现

根据 HashEvict (2412.16187) 的实验结果：

  • 在 reasoning、multiple-choice、long-context retrieval、summarization 任务上，压缩 KV cache 30%-70% 仍保持高精度
  • HashEvict 的压缩决策是 pre-attention（在计算注意力之前）完成——这正是 kvcomp_attn 在 decode-only 路径做的事
  • 动态驱逐：每步替换"预期产生最低注意力分数的 token"——对应 kvcomp_attn 维护 hash cache 并在每步更新

19.8 为什么是"pre-attention"选择？——计算节省的来源

kvcomp_attn 的关键设计：hashing 选择发生在真正计算注意力之前。

dense attention 的计算：对所有 N 个 KV token 做完整的 Q·K^T → softmax → V 加权，O(N) 次点积
HashEvict / kvcomp_attn：
  1. 投影 + 二值化 query：O(d × hash_bits) ≈ O(d)（d=嵌入维, hash_bits 通常 128，固定）
  2. 与 M 个 block 签名做 Hamming：O(M × hash_bits / word_size) = O(M × 16) if hash_bits=128（XOR+popcount 一条指令）
  3. Top-k 选 block：O(M log k)
  4. 仅对选出的 k 个 block 做完整 paged attention
节省来源：当 hash_bits ≪ d（如 128 ≪ 512）且 k ≪ M 时，Hamming 代理分数的计算成本远低于为所有 KV token 计算 full-precision attention score。代价是需要额外维护 hash cache（每 token/hash 存储 hash_bits 位二值签名，约 hash_bits/8 字节）。
