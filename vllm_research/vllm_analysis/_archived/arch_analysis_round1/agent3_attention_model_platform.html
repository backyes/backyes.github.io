Layer 3 — Attention Backends, ModelExecutor, Platform Device Abstraction, Compilation/Graph Capture

Companion note to the vLLM architecture series. Audience: inference architects, systems/parallel-computing engineers, DSA engineers. Scope: vLLM upstream vllm/v1/attention/, vllm/v1/worker/, vllm/platforms/, vllm/_custom_ops.py, vllm/compilation/, and the full vLLM-Ascend overlay.

1. The Attention abstraction stack

vLLM v1 splits attention into three collaborating interfaces, stripped of any op-code:

Interface File Responsibility
AttentionBackend (ABC) vllm/v1/attention/backend.py:56 Static factory for name, impl class, builder class, KV-cache shape, capability predicates (supports_head_size, supports_mla, indexes_kv_by_block_stride). State: none — it is resolved once and cached.
AttentionLayer (Protocol) backend.py:757 Carries per-layer Q/K/V scales (_q_scale, _k_scale, _prob_scale, float variants). The only thing the attention impl reads from a layer besides cache tensors.
AttentionImpl(AttentionImplBase) backend.py:871 Stateful per-layer kernel owner (forward(layer, q, k, v, kv_cache, attn_metadata, output, ...) → output). MLA uses MLAAttentionImpl (backend.py:954) with forward_mha / forward_mqa.
AttentionMetadataBuilder(ABC, Generic[M]) backend.py:607 Turns CommonAttentionMetadata (from the model runner) into backend-specific metadata M via build(). Also owns cudagraph support level (_cudagraph_support), batch reorder threshold, and update_block_table.

The registry (vllm/v1/attention/backends/registry.py) makes this hierarchy swappable at runtime without recompiling vLLM core:

  • AttentionBackendEnum (registry.py:34) maps every canonical backend to a fully-qualified class path string (FLASH_ATTN → vllm.v1.attention.backends.flash_attn.FlashAttentionBackend, …).
  • _ATTN_OVERRIDES: dict[AttentionBackendEnum, str] stores live rewrites. register_backend(AttentionBackendEnum.CUSTOM) (registry.py:233) is a decorator (@register_backend(...) class MyBackend) that writes cls.__module__.__qualname__ into the override table.
  • backend.get_class() (registry.py:141) resolves the override via resolve_obj_by_qualname and lazily imports the backend. Enums are __getitem__-patched (registry.py:21) so AttentionBackendEnum.FLASH_ATTN returns the loaded class.

1.1 How a backend is resolved (the init-time contract)

selector.get_attn_backend(...) (selector.py:69) → builds an AttentionSelectorConfig NamedTuple of all capability flags → _cached_get_attn_backend (selector.py:106) → current_platform.get_attn_backend_cls(backend, attn_selector_config, num_heads).

Key insight: the platform is the tie-breaker. Upstream CUDA's CudaPlatform.get_attn_backend_cls (platforms/cuda.py:396) does not return an enum member — it returns a class path string, optionally selected by capability (get_device_capability(), get_valid_backends, priorities) when the user did not pin --attention-backend. It also calls backend.get_required_kv_cache_layout() and pushes it into the global KV layout store. So selection is two-factor: enum hint + platform-side policy + backend self-validation (validate_configuration).

On the GPU runner, init_attn_backend (vllm/v1/worker/gpu/attn_utils.py:80) groups layers sharing the same (backend_name, KVCacheSpec, num_heads_q) tuple into AttentionGroups, creates a metadata builder per group, picks a common kernel block size per group (prepare_kernel_block_sizes), and reduces cudagraph support to the minimum across all groups. Per-layer backends come from attn_layer.get_attn_backend(), so heterogeneous attention (MLA decode + MHA prefill) co-exists.

1.2 Paged attention reads KV by block id

vllm/v1/attention/ops/paged_attn.py:15 (PagedAttention) — the only place a backend externalizes the block mapping. write_to_paged_cache(slot_mapping, ...) calls ops.reshape_and_cache(...) (a registered fake op on _C::reshape_and_cache), storing K/V keyed by slot. The "block table" is a [batch, max_pages] tensor of page indices carried inside common_attn_metadata.block_table_tensor. utils.py:276 pads/remaps it into a per-local-batch block_table_local; make_block_table lambda re-indexes. The trick: backends don't materialize the full gather; they hand the block table to a kernel that dereferences page_idx = block_table[b, p] to fetch the p-th page of request b.

For reshape-and-cache across devices, vLLM-Ascend backfills torch_npu equivalents via a per-op dispatch (see §5).

2. ModelExecutor / ModelRunner contract

GPUModelRunner (vllm/v1/worker/gpu_model_runner.py:446) implements the implicit runner Protocol surfaced by WorkerBase + mirroring classes:

Method File:line Role
__init__(self, vllm_config, ...) gpu_model_runner.py:449 Builds model, cudagraph dispatcher, sampling, async-output plumbing.
execute_model(scheduler_output, intermediate_tensors) gpu_model_runner.py:4098 The hot path: takes a SchedulerOutput (per-req token counts, KV state ids, connector metadata), returns ModelRunnerOutput \| IntermediateTensors \| None.
_update_states(scheduler_output) gpu_model_runner.py:1163 Refreshes input_batch from the scheduler — the "prepare model inputs" step.
_prepare_inputs gpu_model_runner.py:1931 Builds input_ids/positions, slot mappings, kv-shares.
_build_attention_metadata gpu_model_runner.py:2255 Iterates attn_groups, calls each builder's build(), populates cudagraph key dispatch.
_model_forward(input_ids, positions, intermediate_tensors, **kwargs) gpu_model_runner.py:3811 Pure self.model(...) wrapper — deliberately kept inspectable as a separate method.

get_model() gpu_model_runner.py:3276 Returns bare nn.Module.

The "model runner consumes a prepared input" pattern:

execute_model → _update_states (fills self.input_batch : InputBatch) → _prepare_inputs → _build_attention_metadata → _preprocess → _model_forward → _sample → _bookkeeping_sync. The model only receives position-aligned input_ids, positions, KV-cache references, and per-layer metadata tensors; all scheduling state lives in input_batch, never in the model. This is why the SUT (System Under Test — model architecture) stays device-agnostic.

The runner's only non-trivial device-specific responsibility is attention backend orchestration + cudagraph capture/scheduling, and those are both delegated to composable components.

3.3 Ascend's implementation of the interface

vllm_ascend/platform.py:128 class NPUPlatform(Platform) — 1255 lines, the single largest out-of-tree file. Notable wiring:

  • Device identity: _enum = PlatformEnum.OOT, device_name = "npu", device_type = "npu", device_control_env_var = "ASCEND_RT_VISIBLE_DEVICES", dispatch_key = "PrivateUse1" (the PyTorch device backend tag torch-npu registers). ray_device_key = "NPU" plugs the Ray resource manager.
  • Backend resolution replaces CUDA priorities. NPUPlatform.get_attn_backend_cls (platform.py:784) does not call into CUDA's priority machine; it picks from a static key → class-path map keyed by (use_mla, use_sparse, use_compress):
  • (False,False,False) → standard "vllm_ascend.attention.attention_v1.AscendAttentionBackend"
  • (True,False,False) → MLA "vllm_ascend.attention.mla_v1.AscendMLABackend"
  • (True,True,False) → sparse "vllm_ascend.attention.sfa_v1.AscendSFABackend"
  • (True,False,True) → compressed DSA "vllm_ascend.attention.dsa_v1.AscendDSABackend"
  • 310P silicon collapses to a single path (backend_map_310).
  • FA3 is opt-in via flash_attn_npu_v3 and gated by use_batch_invariant (_validate_fa3_backend, platform.py).
  • simple_compile_backend = "eager" (platform.py): a deliberate opt-out of torch.compile globally. This switches the driver to Ascend's own compilation stack (§4).
  • Memory: get_current_memory_usage uses torch.npu.max_memory_allocated; is_cumem_allocator_available → True (wires Ascend's CaMemAllocator from vllm_ascend.device_allocator.camem), so KV-cache/block allocation terminates in not the upstream cumem path but Ascend's own pinned-NPU allocator.

Design insight: NPUPlatform proves the Platform contract was right-sized: Ascend only reimplements (a) device identity, (b) backend policy, (c) compile-backend hooks, (d) memory accounting, and (e) kv-allocator availability. Everything above — scheduling, distributed comms flows, cudagraph timestamp formats, the execute_model structure itself — is inherited wholesale.

4. Compilation / Graph capture

4.1 Upstream CUDA Graphs + torch.compile

vllm/v1/cudagraph_dispatcher.py:15 class CudagraphDispatcher (350 lines) is the runtime CUDA-graph selector:

  • initialize_cudagraph_keys(...) builds a keys[CUDAGraphMode, set[BatchDescriptor]] table. A key encodes (num_tokens_padded, num_reqs_or_None, uniform, has_lora, num_active_loras). Two modes: FULL (whole model, fixed-shape) vs PIECEWISE (embedding+attention+inner-forward split, captures cudagraph per backward/forward segment — compilation/piecewise_backend.py).
  • dispatch(num_tokens, uniform_decode, has_lora, ...) (cudagraph_dispatcher.py:235) returns (CUDAGraphMode, BatchDescriptor). Decision tree: if num_tokens > max_cudagraph_capture_size → NONE; else look up FULL key, then PICEWISE with a relaxed batch descriptor (num_reqs=None, uniform=False); fallback NONE. LoRA count is "rounded up" to captured power-of-2 when cudagraph_specialize_lora.
  • Capture is driven by get_capture_descs() returning PIECEWISE-first (sorted large→small) so that bigger graphs get warm-up slots first. Inside the model forward, get_forward_context() carries the chosen mode+key for self.model(...) and the attention wrappers replay accordingly.
  • AttentionBackend._cudagraph_support: ClassVar[AttentionCGSupport] (backend.py:610) — one per builder — lets backends opt out of full/partial graphing, and init_attn_backend reduces to the min across groups, so a single non-graphing backend disables graphs for the whole GPU runner.

Upstream torch.compile passes CompilationConfig to torch._inductor.compile_fx; the Optane pass_manager_* use current_platform.pass_key to register custom inductor passes (Ascend rewires this to its own GraphFusionPassManager).

4.2 Ascend's ACL-graph substitute — why CUDA graphs are unusable

Ascend does not reuse CUDA graphs — the hardware command stream (ACL) is ISA-disjoint from CUDA, and torch.npu.NPUGraph has a different capture API. vLLM-Ascend's overlay therefore compiles a CUDAGraphMode-compatible shim:

vllm_ascend/compilation/acl_graph.py:

  • ACLGraphEntry (acl_graph.py:50) holds BatchDescriptor + torch.npu.NPUGraph | None + output + input_addresses. An NPUGraph is the NPU analog of torch.cuda.CUDAGraph.
  • ACLGraphWrapper (acl_graph.py:60) is the replay/capture proxy. On __init__ it asserts runtime_mode != NONE and acquires global_graph_pool from current_platform.get_global_graph_pool() — a device-level stream pool for graph capture without halting serving.
  • Dispatch mirrors upstream: wrapper trust the (runtime_mode, batch_descriptor) carried on forward_context; if mismatch → call runnable directly. Capture path builds a new ACLGraphEntry, replays existing. Notably no persistent buffers are managed by the wrapper — unlike CUDA graphs in upstream (which static-copy inputs), Ascend exposes dynamic batch-size inputs and leaves input staging to the outer runner. Input-addresses are traced for DEBUG-time stability check (_runnable_str, VLLM_LOGGING_LEVEL).
  • Error handling is hardware-aware: _is_stream_resource_capture_error and _is_old_hdk_capture_error (error code 207008, markers insufficient_stream_resources / alloc sq cq fail) swallow & recover from known NPU-side capture failures.

4.3 Ascend's Torch-compile fork

Ascend replaces the inductor path with two custom components (compiler_interface.py):

  • compile_fx (compiler_interface.py:39) calls aot_autograd(fw_compiler=inner_compile) where the inner compiler is compile_inner which pulls current_pass_manager = compiler_config[COMPILATION_PASS_KEY] and runs it (graph = current_pass_manager(graph)). Entry point: set_custom_compile_config wraps a torch.compile-style compile of the runnable.
  • AscendCompiler(CompilerInterface) (compiler_interface.py:226) is what the get_compile_backend (platform.py:813 → "vllm_ascend.compilation.compiler_interface.AscendCompiler") returns. It implements configure_compile_backend, _create_pass_manager, and compile(...) (compiler_interface.py:255).

Pass catalogue (compilation/passes/*.py, 70 classes in ~30 files):

Pass Pattern Why
QKNormRopeFusionPass QK-Norm + RoPE sequence Removes dtype round-trips between norm & rotary.
MatmulAllReduceAddRMSNormPass (middle + last layer variants) matmul → allreduce → add → rmsnorm Matches the classic pre/post-layernorm MoE chassis, fuses into a single CANN op.
AddRMSNormQuantFusionPass (+ SP / dynamic / MX variants) rmsnorm + quant → FP8/INT8 Backfills FP8 KV-cache path with CANN quant kernels.
SequenceParallelismPass / MulsAddFusionPass / NoOpEliminationPass / AllGatherChunkNoOpCleanupPass SP/T-COMM patterns Cleans all_gather/reduce-scatter scaffolding that HCCL emits.

Three mechanisms coexist on Ascend (contrast table):

Mechanism Upstream (CUDA) Ascend (NPU)
Full CUDA graph CUDAGraphDispatcher + torch.cuda.CUDAGraph ACLGraphWrapper + torch.npu.NPUGraph (same CUDAGraphMode enum, same BatchDescriptor key)
Piecewise torch.compile piecewise_backend + inductor AscendCompiler + GraphFusionPassManager (custom inductor pass manager via pass_key)
Custom Triton / CANN kernels _custom_ops.py on _C / _rocm_C (cuDNN/FlashAttention) torch_npu JIT kernels + _C_ascend.* (CANN-authored)

Design insight: the CUDAGraphDispatcher was explicitly designed as a key-dispatch proxy rather than a capture library, which is exactly why Ascend could slot in a parallel implementation. If vLLM had baked torch.cuda.CUDAGraph into the dispatcher itself, this overlay would have been a rewrite, not a subclass.

5. Ops & kernels layer

5.1 Upstream _custom_ops.py

vllm/_custom_ops.py is the Python-side registry of device-specific C++ kernels. Two styles co-exist:

  1. Global function registry with auto-generated wrappers: paged_attention_rocm, merge_attn_states, rotary_embedding, rms_norm, fused_add_rms_norm, cutlass_scaled_mm, silu_and_mul_per_block_quant, awq_dequantize, AWQ/GPTQ/T-Rocm kernels, etc. These build on torch.library op names like _C::awq_dequantize, _rocm_C::gptq_gemm_rdna3, torch.ops._rocm_C.*.
  2. Fake (abstract) implementations for torch.compile: @register_fake("_C::scaled_fp4_quant") (_custom_ops.py:97) returns the tensor shape/stride info inductor needs for opaque C calls. Every C kernel typically ships both a real impl (in _C.cpython-...so) and a fake registration so compile can reason about memory.

current_platform.import_kernels() (interface.py:363) calls import vllm._C then import vllm._moe_C_stable_libtorch with contextlib.suppress(ImportError). Overriding on a new device is just a matter of providing vllm/_C-compatible shims; the platform class itself does not register new ops.

5.2 Ascend's backfill of CANN kernels

Ascend cannot inherit vLLM's _C (NVIDIA CUTLASS/cuDNN/FlashAttention) and so mounts three parallel paths:

Path A — per-op PyTorch custom hook: vllm_ascend/ops/register_custom_ops.py.

Uses vllm.utils.torch_utils.direct_register_custom_op (a stable wrapper over torch.library.define + impl). Hooks:

  • _maybe_chunk_residual_impl — SP-aware residual TP-chunking (pads residual[pads:], torch.chunk(residual, tp_size)[tp_rank]).
  • _maybe_all_gather_and_maybe_unpad_impl — EP-aware all-gather that honors dp_metadata.num_tokens_across_dp_cpu (DP+EP overlap).
  • _maybe_pad_and_reduce_impl — replace allreduce with reduce_scatter under flash_comm_v1_enabled / enable_sp_by_pass. This is the comm/compute overlap path that mirrors upstream's allreduce+layernorm fusions but for HCCL.

Path B — torch_npu native ops: torch_npu.npu_fusion_attention, torch_npu.npu.distributed.all_reduce, torch.npu.NPUGraph. Called from attention impls and workers directly. ops/ops_initializer (the V0-compat layer) wires the dispatch table so DeviceOperator.reshape_and_cache, DeviceOperator.reshape_and_cache_flash, etc. route to the right NPU kernels per cache layout.

Path C — Triton-for-NPU: vllm_ascend/ops/triton/ packages Triton kernels (rms_norm.py, layernorm_gated.py, split_qkv_rmsnorm_rope.py, split_qkv_rmsnorm_mrope.py, split_qkv_tp_rmsnorm_rope.py) — these run on NPU via torch_npu's Triton JIT backend and let Ascend share kernel code with upstream's Triton backends where CANN coverage is thin.

Path D — CANN-authored C++ in vllm_ascend/C_ascend* (the _C_ascend torch op lib, registered with @register_fake on higher-level ops): torch.ops._C_ascend.npu_vllm_quant_lightning_indexer_metadata (dsa_v1.py:820) for DSA sparse attention, MoE fused kernels in ops/fused_moe/fused_moe.py, MLA kernels in ops/mla.py.

Design insight: Ascend's strategy is "put the four kernel sources behind one dispatcher" (DeviceOperator), so attention backends can call DeviceOperator.reshape_and_cache without knowing whether the target kernel is torch_npu, Triton, or _C_ascend. This mirrors vLLM's own wrinkle: _custom_ops.py serves the same role upstream.

6. Design insights for an inference architect (ranked by adaptation cost)

6.1 AttentionBackend — the raison d'être of vLLM's multi-hardware portability

Splitting "AttentionBackend / AttentionImpl / AttentionMetadataBuilder" was the single most important abstraction for multi-hardware. It lets vLLM swap flash_attn, triton, flashinfer, flex_attention, and NPU kernels at runtime while the heavy model body (model_runner._model_forward, sampling, scheduler) stays device-agnostic. Ascend demonstrates the cost asymmetry perfectly: implementing AscendAttentionBackend + AscendAttentionBackendImpl + AscendAttentionMetadataBuilder (~1408 lines) covers all non-MLA attention; separate AscendMLABackend + AscendSFABackend + AscendDSABackend variants add another ~1000 lines to serve DeepSeek-style MLA/DFA/DSA. Compare to the ~60k-line model runner. Attention is the 2–5% of code that must be rewritten per device.

Ranking extension surfaces by "cost of adaptation for a new DSA":

  1. MUST (no device boots without them) — Platform interface (identity, memory, attn-backend policy, device-set), at least one AttentionBackend with a working forward that reads KV via block_table, and the Op backfill for rms_norm/rope/reshape_and_cache or equivalents.
  2. SHOULD (for production quality) — the MLA/SFA/DSA backends if your DSA targets long-context / sparse attention; the custom-op registry (direct_register_custom_op equivalents) for quant/dequant GEMM; distributed communicator (get_device_communicator_cls) since HCCL vs NCCL semantics diverge on collectives ordering.
  3. OPTIONAL but recommended — compilation variant (ACLGraphWrapper is a production necessity for decode latency on Ascend; but a new DSA could start with eager-only). The Piecewise CUDA graph dispatcher is optional; Ascend forked it. torch.compile passes are optional; Ascend forked them aggressively because upstream inductor doesn't target CANN.
  4. Cosmetic - ViT backend selection, LoRA punica wrapper, integrated-GPU detection.

6.3 One subtle trap for DSA architects forwarded from Ascend's experience

The attention impl's can_return_lse_for_decode + lse_base_on_e (backend.py:806) and DCP supports_dcp, supports_mtp_with_cp_non_trivial_interleave_size are bit-structural contracts for decode/prefill context parallelism and spec decoding. A new DSA that sets lse_base_on_e = False (contrary to the FlashAttention default) will silently corrupt the cross-shard softmax denominator inside cp_lse_ag_out_rs / dcp_a2a_lse_reduce (vllm/v1/attention/ops/common.py). This is one line of metadata but a whole class of silent bugs. Any new backend that changes LSE semantics must also patch the DCP combine kernels.
