Agent 4 — Ascend 全栈适配图与通用 vLLM 硬件适配方法论

报告视角：vLLM-Ascend 作为非 GPU（华为昇腾 NPU）适配的完整样本 → 提炼任意硬件厂商适配 vLLM 的方法论。 受众：资深系统工程师 / 并行计算专家 / AI 算法研究员 / 推理架构师。 约定：文件引用均用 package/module.py:LINE，引文保留原文语言，关键 snippet ≤ 15 行。

概览一句话

vLLM-Ascend 不是 fork vLLM，而是一套 插件式 overlay：用 Platform + AttentionBackend 的注册式扩展点做主干，用 config injection 注入 Ascend 专属调度器，用 monkey-patch 层 处理 vLLM 未暴露 hook 的遗留/行为差异，并用 ACL Graph 替换 CUDA Graph 作为 NPU 图执行后端。整条适配线可拆为「platform shim → attention/ops kernel 替换 → worker/runner override → 分布式通信替换 → patch 补钉」——这条结构对 AMD / Intel Gaudi / Trainium / Groq / 任意 DSA 完全可复用。

A.2 架构深度解读（6 段核心创新 + 2 段基础创新）

以下 8 段不是文件罗列，而是 设计决策级 的解读——每段回答"为什么这样做、代价是什么、对其它硬件的启发"。

A.2.1 Platform 作为 CUDA-shim：伪装机制与断裂面

NPUPlatform(Platform) 把 NPU 呈报为 CUDA capable device：get_device_capability() 直接返回 None（platform.py:244）；get_device_name / get_device_uuid 走 torch.npu；set_device = torch.npu.set_device（platform.py:334）；num_compute_units 用 NPU 的 cube_core_num（≈ CUDA SM）回填（platform.py:283，platform.py:288 原文："This is the NPU equivalent of CUDA's multi_processor_count... consumed by vLLM's layernorm_guard.calc_rows_per_block"）。

被故意不实现的面： get_device_total_memory 直接 raise NotImplementedError（platform.py:275），并标注：

"DO NOT IMPLEMENT: Implementing it calls get_device_name() in advance and initializes torch_npu too early. torch_npu allows global initialization only once; duplicate initialization causes errors."

这台 NPU 必须等 vLLM 的初始化序列走完才能暴露 memory API——这是 shim 的 静态面 与 动态真正可用面 不对齐的经典断裂。

图执行裂缝：vLLM 的 compilation_config.cudagraph_mode 是强类型 CUDAGraphMode enum，Ascend 在 check_and_update_config（platform.py:481-545）逐项改写——FULL_DECODE_ONLY 允许，FULL_AND_PIECEWISE 不支持直接报错，PIECEWISE 要重写 cudagraph_capture_sizes 并禁用 DeepSeek V4 PIECEWISE（platform.py:63 top docstring："Breakable cudagraph is force disabled on Ascend because DeepSeek V4 PIECEWISE cudagraph is not supported yet."）。这暴露一个根本问题：vLLM 把图执行抽象成 enum + size list，但 CUDA Graph 仍是语义假设，ACL Graph 用户不得不拦截这一层才不被 CUDA-graph-only 路径烧伤。

对其它硬件的启发：任何非 CUDA 后端都需要一个 Platform shim 做「汇报 + 拦截改写」。但 get_device_total_memory 的 NotImplemented 模式必然带来内存预算路径的脆弱——建议 vLLM 显式提供 device_query_guard / lazy memory probe 接口，避免每个 vendor 各自 hack 初始化顺序。

A.2.2 调度器扩展：recompute + SRF + profiling_chunk

Ascend 对 vLLM 调度器的改动是 三层叠加，全部通过 config 替换而非改 upstream 代码实现：

  1. RecomputeScheduler（core/recompute_scheduler.py:98）= ShortRequestFirstSchedulerMixin + Scheduler，是 D 节点（kv_consumer）的主调度器，在 schedule()（L194）内把 preemption 换成 recompute，使跨 DP rank 的 token 数均衡——这是为 DSV4 / Finegrained TP 准备的"前置条件"：其跨-DP HCCL allgather/reduce-scatter 需要每 rank 同形 token（ascend_config.py:142-145 的 assert 原文）。
  2. ShortRequestFirstSchedulerMixin（core/short_request_first_scheduler.py:374）接在 SRF config 上，把 waiting queue 拆成"短请求优先 / 长请求带 max_wait_ms"两段，配合 recompute 的 preemption 让短请求快速进 schedule window。
  3. ProfilingChunkScheduler（core/scheduler_profiling_chunk.py:46）按 profiling latency 动态 chunk 大小（perf-based dynamic chunk sizing），要求 PP>1（ascend_config.py:66 assert）。

三者的 选择枢纽 是 additional_config + platform.py:686 的 scheduler_cls 字符串替换：

  • profiling_chunk_config.enabled=true → scheduler_cls = "...ProfilingChunkScheduler"（platform.py:686）+ 触发 patch_profiling_chunk。
  • recompute_scheduler_enable=true 且 kv_role=="kv_consumer" → 将 vllm_config.scheduler_config 整个对象替换为 RecomputeSchedulerConfig.initialize_from_output 返回的实例。
  • short_request_first_config 仅在 recompute 开启时生效（SRF mixin 挂在 RecomputeScheduler 类上）。

架构启发：Ascend 用 qualname 字符串（scheduler_cls）而非 Python class 做调度器解耦——这意味着"在 runtime 绑定一个非 upstream 子类"完全干净。这是 vLLM SchedulerConfig.scheduler_cls（vllm/config/scheduler.py:127）的最强设计点。任何新硬件想注入自定义调度器，应优先走此扩展点而非 patch。

A.2.3 Attention/ops 的 CANN kernel 替换

Ascend 的 Attention 用的是 注册式 backend 枚举——在 vLLM 的 AttentionBackendEnum.CUSTOM slot 注册 "ASCEND"、"ASCEND_MLA"、"ASCEND_SFA"、"ASCEND_DS_A"、"ASCEND_FA3" 五个 backend（核心入口 attention_v1.py:75 @register_backend）。每个 backend 暴露 get_impl_cls / get_builder_cls / get_kv_cache_shape / swap_blocks / copy_blocks —— vLLM 的 paged KV 生命周期完整注入 Ascend key。

值得注意的是 backend 名字 dirty hack（attention_v1.py:92）：

return "CUSTOM" if not envs_vllm.VLLM_USE_V2_MODEL_RUNNER else "FLASH_ATTN"

HACK 注释直接写明："vllm initialize_kv_cache method in model runner v2 make attention name assertion, we just set name to FLASH_ATTN to avoid assertion error."——这是 vLLM 内部断言泄漏 逼出来的 patch。

MLA 路径：AscendMLABackend（attention/mla_v1.py:72）的 get_kv_cache_shape 单独定义 MLA 的 cache 排布（单 cache，不拆 K/V），其 spec 来自 core/kv_cache_interface.py 的 AscendMLAAttentionSpec，使 vLLM 的 generic KVCacheSpec 机制被复用而非 replacement。

ops 替换机制分两档： - 一档用 torch Library("_C_ascend","IMPL") + direct_register_custom_op（register_custom_ops.py L4）注册 NPU kernel，并重写 fake/meta（meta_registration.py L48-58）让 torch.compile + ACL Graph 能追踪 shape 而不执行。 - 二档用 CustomLinearOp.apply() 替换 layer.forward（ops/linear_op.py L83/L114）：不是注册全局 op，而是在 process_weights_after_loading 阶段把 matmul+allreduce 序列整体替换为 NPU MC2 / matmul_allreduce / Flashcomm2 等特化实现（MatmulAllreduceRowParallelOp L409、Flashcomm2OProjRowParallelOp L298）。vLLM 没有 hook 到这里，所以 Ascend 用了 patch_process_weights_after_loading（patch 层）——这是 patch 层存在的典型案例之一。

对其它硬件：注册 custom backend（attention ops）是最干净的路径；但 forward-time op substitution 无 hook 是 vLLM 的明显空缺，迫使所有 vendor 都在补丁层处理。

A.2.4 HCCL 替换 NCCL + KV PD 拆解

Ascend 没有 NCCL。它在两个层级桥接：

进程通信层 —— PyHcclCommunicator（distributed/device_communicators/pyhccl.py:37）接手 ProcessGroup.HCCL： - L60-74：先校验 dist.get_backend(group) != Backend.HCCL（"HCCL group 不能自己绑自己"）。 - L87-104：rank0 调 hcclGetUniqueId，走 dist.broadcast（非 HCCL）分发——这是经典的"跨-hierarchy 引导握手"，与 PyTorch distributed 的 gloo-on-top-of-nccl 套路同构。 - 调用 hcclAllReduce / hcclBroadcast / hcclCommInitRank 全部通过 ctypes 直调 libhccl.so（pyhccl_wrapper.py:113 HCCLLibrary，L230 hcclAllReduce），用 aclrtStream 替换 CUDA stream 一一对应。

tensor-parallel op 层 —— register_custom_ops.py 的 _maybe_*_impl 族（_maybe_chunk_residual_impl、_maybe_all_gather_and_maybe_unpad_impl L55、_maybe_pad_and_reduce_impl L88）在 forward context 中选择 MC2 / Flashcomm v1 / AllGather / AllToAll 等 MoE 通信模式（ascend_forward_context.py:27 MoECommType 枚举：ALLGATHER / ALLTOALL / MC2 / FUSED_MC2）。这是 Ascend 最深的 opcode-level 改写，藏在 forward-context dispatch 里——vLLM 自身的 TP 路径并不感知 MC2 的存在。

KV PD 拆解 —— Ascend 为 prefill-decode 分体提供了 三套并行方案： 1. MooncakeConnector（mooncake_connector.py:1411）= batch hybrid P2P； 2. MooncakeLayerwiseConnector（mooncake_layerwise_connector.py:697）= layer-wise P2P（HBM→RDMA→HBM，线程内逐层 send/recv thread pair，KVCacheSendingLayerThread L204 / KVCacheRecvingLayerThread L530）； 3. AscendStore pool backend（kv_pool/ascend_store/）= 内存池共享路径。 三者通过 AscendMultiConnector（ascend_multi_connector.py:19）聚合，统一挂载在 vLLM 的 MultiConnector / KVConnectorBase_V1 接口上，走 mooncake TransferEngine RDMA。

启发：vLLM 的 KVConnector clean point 完全承载了 KV offload/disaggregation 创新；但 MoE 通信模式（MC2/AllToAll）无通用 dispatch hook，ascend_forward_context 的 _EXTRA_CTX 单例是 vendor 自行补上的"影子上下文"。

A.2.5 Patch 层：外科手术式扩展（技术 / 动因 / 风险）

Ascend 的 patch 层总计 60+ 文件，通过 patch/__init__.py 的 adapt_patch(is_global_patch=...)（utils.py:537）分两阶段触发： - is_global_patch=True → from vllm_ascend.patch import platform → 在 NPUPlatform.pre_register_and_update() 内触发，改平台期单例行为（cudagraph size、multiproc executor daemon、量化校验白名单、KV cache block size、MiniMax tool-call 等）。 - is_global_patch=False → from vllm_ascend.patch import worker → 在每个 worker.__init__() 触发，改 worker 期行为（fused moe 替换、triton 回退、Eagle speculator 初始化、routed experts capture、triton op 替换等）。

patch 文件的内部规范（patch/__init__.py:8-26 的 How Patched and how it works 文档块）强制每 patch 声明六项：What、Why、How、Related PR、Future Plan。以 patch_mamba_config.py（example 2）为例：Why = "block size is set to 16 in vLLM which is not supported by Ascend"，How = "Set block size to 128 on npu"，对应 upstream PR 明确。这种 "patch 文件即文档" 的实践保持了可维护性。

patch 的主要技术手段： 1. setattr(module, "func", wrapper)（patch/__init__.py:135）——直接改方法，处理大多数 hook。 2. rebuild_dataclass(Dataclass, force=True)（patch/__init__.py:121-127）——当 upstream 用 pydantic @config dataclass 做 init 校验时（如 SpeculativeConfig._verify_args），单纯改 function 不够，还需 rebuild_dataclass 让 validator decorator chain 重绑，连带 rebuild_dataclass(VllmConfig) 刷新嵌套校验。这是 Ascend 独创的 "pydantic dataclass 反射穿透" 技巧。 3. register_model("Alias", Cls) 补注册以绕过 whitelist（patch_minimax_m2_config.py 的 "4." 段）。

为什么必须 patch（而不是 subclass 干净扩展）：每一条 patch 的 Future Plan 都含一句 "Remove this patch when vLLM merges the PR"。它们全是 "upstream 已同意但尚未合并" 或 "vLLM 没有给该变更留 hook" 的临时补丁。没有 patch 层就无法在"不 fork"的约束下推进创新。

风险：patch 层对 vLLM 内部方法签名 / import 路径是强耦合的。vLLM v0→v1 迁移中这些隐式契约大面积断裂，Ascend 每主版本升级都需 patch 回归。这是 "plugin overlay" 架构的 固有税。

A.2.6 ACL Graph 替换 CUDA Graph

Ascend 没有 CUDA Graph。它用 ACLGraphWrapper（compilation/acl_graph.py:60）在 _BaseGraphRunner 同一语义层重建：

  • __init__（L85）接收 runtime_mode: CUDAGraphMode——(wrapper 注释原文) "the wrapper receives a runtime_mode and a batch_descriptor(key) from the forward context and blindly trust them for aclgraph dispatching." 这与 CUDA Graph wrapper 的契约完全对齐：capturable / replayable / 按 batch descriptor 缓存。
  • __call__（L133）按 batch_descriptor 字典缓存 ACLGraphEntry；未捕获 → torch.npu.NPUGraph() 捕获；已捕获 → replay。
  • 关键设计抉择（L59 wrapper docstring）：ACLGraphWrapper 不存 persistent buffer、不拷贝 runtime 输入——"as a trade-off for staying orthogonal to compilation logic... Nevertheless, tracing and checking the input addresses to be consistent during replay is guaranteed when VLLM_LOGGING_LEVEL == DEBUG." 这是一个语义折中：vLLM 的 graph replay 必须保证 input tensors 地址与捕获时一致，这是上层模型的状态管理职责。
  • inductor 侧 pass manager（graph_fusion_pass_manager.py:25 GraphFusionPassManager）在 compile 时注入 5 个 NPU 特定 pass：QK-Norm+Rope 融合、AllReduce+RMSNorm 融合、AllGather chunk noop 清理、NoOp 消除、norm-quant 融合。所有 pass 挂载在 pass_key = "ascend"（platform.py:159）的 inductor hook 下。

对其它硬件：ACL Graph 的存在证明 "图执行抽象"完全可替换。但 vLLM 仍把 CUDAGraphMode 用作 enum，导致 ACL Graph 用户要在 enum 上强 overlay——ideal 设计应是 GraphMode(Protocol) 让 ACL/NCCL/Metal 各自实现。

A.2.7 EPLB：Ascend 对 vLLM 的原生贡献

EPLB（Expert-Parallel Load Balancing）是 Ascend 团队主推的 纯原创子系统，遵循 vLLM 扩展点挂载：

  • EplbUpdator（eplb/eplb_updator.py:32）= 触发器，统计专家调用分布 → 决策 rebalance；
  • EplbWorker / EplbProcess（core/eplb_worker.py:29）= 进程内驱动；
  • policy 层（core/policy/policy_abstract.py）提供 4 种策略：RandomLoadBalance、DefaultEplb、SwiftBalanceEplb、FlashLB（policy_flashlb.py:515）——FlashLB 是生产级实现，内含 FlashTree 二分树（L326）+ numba 加速的匈牙利/树搜索 optimize_balanceness；
  • D2DExpertWeightLoader（core/eplb_device_transfer_loader.py:32）= 设备间权重搬运；
  • VllmEplbAdaptor（adaptor/vllm_adaptor.py:60）= 与 vLLM 引擎的回调 adaptor。

整条链走 vLLM 自己的 parallel_state + Worker callback，不 patch 改动。这是 "Ascend 把创新做成 vLLM 上游合入候选 / 内部 feature" 的范本——是对"别人为什么要在 vLLM 上做插件而不是 fork"的最好注脚。

A.2.8 meta_registration / xlite 支线 / fusion pass 的补充说明

  • meta_registration.py：定义 lib = Library("_C_ascend", "IMPL")（L48），并提供 register_meta_if_necessary（L52）：只在该 op 还没有 "Meta" dispatch key 的注册时才 lib.impl(op_name, fn, "Meta")。这是使 torch.compile + ACL Graph 能 custom op 追踪 shape 而不执行的最关键胶水——每新增一个 _C_ascend kernel 必须配一份 meta 函数，否则 torch.compile 会 fallback 或报错。
  • xlite：独立图运行时栈，XliteWorker(NPUWorker)（xlite/xlite_worker.py head）+ XliteModelRunner(NPUModelRunner)（xlite/xlite_model_runner.py），在 load_model 阶段把 vLLM model 包进 XliteWrapper（XliteModelRunner.load_model L46）并注册缓存。xlite 与 v1/v2 的 ACL Graph 是 第三套 路径，表明 Ascend 仍在探索"最贴合 NPU 内存节奏"的图管理方案。
  • gate_fusion inductor pass：graph_fusion_pass_manager.py 的 5 个 fs 融合 pass 把 matmul+allreduce（AllReduce RMSNorm 融合）这类 NPU 特化 micro-kernel 化——vLLM upstream 只为 CUDA/ROCm 做了少量 pass，Ascend 把这些 pass 挂到自己的 channel 下互不干扰。

Part B — 通用 vLLM 硬件适配方法论（the "how-to"）

本节是报告核心。把 Ascend case 提炼成任意硬件厂商（AMD / Intel Gaudi / AWS Trainium / Groq / 自研 DSA）都可照搬的流程，Ascend 仅作为 running example。

B.1 vLLM 扩展面分类（按适配成本 × 必要性分级）

分级 扩展面 适配方式 不做的后果 真实例子
MUST Platform stub（device count/name/memory/temperature） subclass Platform + 注册 进程起不来、memory profile 失败 Ascend NPUPlatform 全部 M；get_device_total_memory 选择 NotImplemented
MUST Attention backend（paged KV + custom kernel） @register_backend + AttentionImpl + AttentionMetadataBuilder fallback 到极慢 reference 路径 Ascend 5 个 backend（ASCEND/MLA/SFA/DSA/FA3）
MUST 核心 custom ops（matmul / attention / rmsnorm / act / rotary / embedding / logits） torch.library("myarch","IMPL") + meta torch.compile trace 失败、图执行 fallback Ascend _C_ascend + meta_registration.py 注册 bgmv/sgmv/rms_norm 等
MUST 分布式通信后端（collective 原语） torch.distributed custom backend + reduce-scatter/allreduce/broadcast wrapper TP/PP/EP 无法 cross-node Ascend HCCL/pyhccl；Intel Gaudi 走 habana 后端；AMD 走 RCCL
SHOULD 编译/图执行变体（CUDA Graph 替代） subclass graph manager / @register_graph_capture 每次 decode step 都 launch 大量细粒度 kernel、性能垮 Ascend ACLGraphWrapper + 5 个 inductor pass；Groq/Mythic 应有 own graph pipeline
SHOULD 量化方案（WA/WxAx 家族） subclass QuantizationConfig + get_quant_method 无法利用 vendor 的 INT8/FP8/FP4 特性 Ascend 10+ 量化方案
SHOULD 调度器微调（chunk / preemption / SRF / recompute） SchedulerConfig.scheduler_cls 子类化 能效/尾部延迟损失大 Ascend 三调度器；Groq（无 kernel launch 代价）可极简
NICE 推测解码 proposer subclass 各 proposer 草稿模型 失去 latency 关键路径优化 Ascend 7+ proposer 子类
NICE KV offload/disaggregation connector subclass KVConnectorBase_V1 长上下文 / PD 分体无法做 Ascend 三套 mooncake connector + ascend_store
NICE EPLB/EP 原生调度 独立子系统 MoE 模型专家不均衡 Ascend EPLB FlashLB
NICE Memory 特殊 allocator（sleep / pool） torch.cuda pluggable allocator 等价物 显存碎片、多租户损失 Ascend camem allocator

分级原则：MUST = "不做就起不来 / 性能差 10 倍以上"；SHOULD = "不做的代价是错过 vendor 30-100% 能效 / 吞吐"；NICE = "差异化价值，尤其服务 MoE / PD / 长上下文等场景"。任何新硬件应 按 MUST → SHOULD → NICE 顺序投入。

B.2 适配工作流（8 步可复制配方）

以下编号 = 推荐的严格顺序；每步附 vLLM 扩展点 + 反例警示。

Step 1 — 实现 Platform + 设备检测

  • 目标：让 LLM(model, device="npu") 在 device count / name / capability / memory 上不出错。
  • 动作：
  • 在 vLLM 的 _PLATFORM_ENVIRONMENT_REGISTRY 中 register_platform（v1 通常在 config 里靠 current_platform + pre_register_and_update 触发）。
  • 覆写：get_device_capability、get_device_name、get_device_total_memory、num_compute_units、set_device、inference_mode。
  • 在 check_and_update_config(vllm_config) 里强制改写该硬件不兼容的选项（例如 PIECEWISE cudagraph、不兼容的 sliding window）。
  • 反例警示：不要用 raise NotImplementedError 逃避 memory probe，否则 vLLM 的 memory profiling 路径崩溃——宁愿用 lazy init（参见 Ascend platform.py:275 教训）。
  • 验收：python -c "from vllm import LLM; LLM('tiny-model')" 能起进程。

Step 2 — 实现 Attention + paged KV

  • 目标：让 prefill 与 decode 都走自定义 NPU kernel，并按 block 化 KV cache。
  • 注册 MyArchAttentionBackend 到 AttentionBackendEnum.CUSTOM。
  • 实现 get_kv_cache_shape / swap_blocks / copy_blocks / get_supported_kernel_block_sizes / get_impl_cls / get_builder_cls。
  • get_impl_cls 返回的 Impl.forward 调用 vendor NPU kernel（Ascend = torch_npu.npu_fused_infer_attention_score.out；AMD = ck.atten_fwd；Intel Gaudi = habanas flash）。
  • 关键：get_kv_cache_shape 决定 KV 的物理排布；MLA 等 decoupled cache 在此自定义。
  • 验收：benchmark_serving p99 延迟偏差 <10% vs vendor native。

Step 3 — 注册自定义 ops 并注入 forward

  • 子步骤 3a：torch.library("myarch", "IMPL") 注册 C++ kernel；同步注册 meta function（否则 torch.compile trace 失败），写法参照 meta_registration.py:52。
  • 子步骤 3b：ops 替换策略自选：
  • 干净路线（推荐，将来能 upstream）：在 vLLM 的 LinearMethod.create_weights 中识别 arch，返回 MyArchLinearMethod。
  • patch 路线（当 vLLM 不给 hook）：在 process_weights_after_loading 阶段把 layer 包成 CustomLinearOp 或复制其 forward/kernels（Ascend patch/worker/patch_process_weights_after_loading.py）。
  • 关键：TP + MC2 / fused rs / fused ar 这类通信原语融合必须在 ops 层处理——vLLM 原生 TP 路径不会感知。
  • 验收：torch.compile(model, backend="myarch_inductor") 能 trace 完整 decode graph。

Step 4 — 覆盖 Worker / Runner

  • 子步骤 4a：subclass GPUModelRunner → MyArchModelRunner，覆写 execute_model、_prepare_inputs（metadata 重排）、load_model（注入自定义 infer graph wrapper）、capture_model（vendor 图捕获）。
  • 子步骤 4b：subclass WorkerBase → MyArchWorker，覆写 init_device / profile_num_available_blocks（调整 memory profiling 预算逻辑）/ determine_num_available_blocks。
  • 警告：v1 GPUModelRunner 体积 5000+ 行（Ascend 的 model_runner_v1.py 5188 行），override 成本极高，优先走 "only override hot methods" 策略，不要满抄。
  • 验收：端到端 decode loop 在 vendor 图后端上可复现。

Step 5 — 替换通信后端

  • 标准动作：实现 MyArchCommunicator，注册 torch.distributed backend；implement allreduce / reducescatter / allgather / alltoall / broadcast / send-recv group 接口。
  • 关键：rank0 的 unique-id bootstrap 通常跨 backend（Ascend 走 dist.broadcast over TCP/gloo），不要依赖 HCCL/NCCL 自带的 bootstrap 除非确认多机跨异构。
  • 进阶：按 mesh 维度注册 device-communicator（Ascend npu_communicator.py:23 _NPUAll2AllManager）。
  • 验收：TP/PP 多机跑通，PP microbubble 在 <5%。

Step 6 — 注入 config

  • 把 vendor 特化开关统一挂在 --additional-config '{"vendor_x": ...}' 的 VendorConfig 对象上（仿照 AscendConfig 模式）。
  • 选择器枢纽：在 Platform.check_and_update_config(vllm_config) 里统一改写，把 scheduler_cls / parallel / cache / quant / compiler / kv_transfer 全部在这一处收敛。这是 单点控制 的关键设计。

Step 7 — 用 patch 补齐剩余缺口

  • 只在 subclassing 无法覆盖时 patch。典型 patch 场景：
  • upstream 硬校验不让你模型/量化/推理路径过（_verify_* whitelist）；
  • upstream 强类型 enum 不承认你的模式（如 spec decode model_type 白名单）；
  • upstream 生命周期 hook 缺失（multiproc_executor daemon flag、per-layer op substitution）。
  • patch 文件规范（夺自 Ascend 最佳实践）：每 patch 一个 Python 文件 + 6 项条目（What/Why/How/Related PR/Future Plan/Remove condition）。
  • 把 patch 按 is_global_patch 分两批：平台期 patch（单例 / config 触发）vs worker 期 patch（进程内触发）。
  • 每 patch 必须绑定一个归零条件："Remove this patch when vLLM merges PR #xxxx"。

Step 8 — 对齐测试矩阵

  • 跑 vLLM 官方 tests/ 子集（core / attention / distributed / spec_decode / quantization），并补充 custom test：
  • API contract 测试：device-count、kv-cache-shape、swap-block；
  • Off-graph 与 Graph 差距 <3%；
  • 精度对齐：decode 128 token 与 vendor-native 相对 logit error <1e-3。
  • 跨 vLLM 主版本 升级要做全 patch 回归（参见 B.5）。

B.3 fork vs plugin vs patch 决策框架

决策拐点 用 subclass（clean） 用 monkey-patch（surgical） 用 fork（last resort）
vLLM 暴露了对应 registry / extension point ✅ 必选（Platform / AttentionBackend / KVConnector / SchedulerConfig.scheduler_cls / QuantizationConfig） ❌ ❌
改动是 add 一个新模型/新 backend ✅ @register_model / @register_backend ❌ ❌
改动是 tweak 一行 upstream 校验 / 默认值 ⚠️ 如果能通过 config 注入走 config ✅ setattr / _verify_* 替换 ❌
改动是新增 vendor-only dispatcher 不在 upstream 任何 hooks ⚠️ 可在 custom op + forward context 内吃掉 ✅ 在 worker_init patch 内替换 ❌
改动是大面积重写模型 forward / scheduler 整段逻辑 ❌ subclass 5000 行不现实 ❌ patch 不可读 ✅ fork / overlay + 定期 cherry-pick
改动是长期与 upstream 冲突且回归代价不可控 ❌ ❌ patch 脆弱 ✅ 内部 fork，做 B.5 所说的 release 同步
改动涉及 vLLM core loop / distributed core ❌ 不应 vendor 改 core ❌ 🟡 只有当 vLLM 明确 open 了 core 时才 fork

Ascend 的实例： - clean：NPUPlatform、AscendAttentionBackend、MooncakeConnector（KVConnector）、AscendQuantConfig-on-config。 - patch：patch_mamba_config (block size)、patch_minimax_m2_config (whitelist)、patch_process_weights_after_loading (op substitution)、patch_profiling_chunk (scheduler hook)。 - fork-level：core/recompute_scheduler.py 重写了 Scheduler 的核心 schedule()，这是 vLLM 未给点位的局部 fork。

黄金原则能上 subclass 绝不上 patch；能 patch 绝不 fork。升级成本呈数量级放大。

B.4 CUDA-compat shim 策略与断裂边界

"让 NPU / Gaudi / AMD GPU 看起来像 CUDA" 有 三个层次，每一层有明确的断裂边界：

层次 1 — device-id mapping（安全）：把 vendor device 映射到 torch.device("npu:X") / ("cuda:X") / ("xpu:X")。在 PyTorch 的 dispatch key 下实现 vendor tensor；vLLM 的 Platform shim 只负责 set_device、memory stats。Ascend 用 torch.npu.* 全替换。

层次 2 — runtime API emulation（部分安全）：stream、event、graph 原语。Ascend 用 torch.npu.Stream / torch.npu.Event / torch.npu.NPUGraph。关键的一致性保证： - wait-stream / record-event 语义必须严格对等； - NPU event 等待不支持 cross-stream 精细依赖时，ACL Graph wrapper 必须 fallback（参见 Ascend forced fallback to NONE on FULL_AND_PIECEWISE 路径）。

层次 3 — graph capture & kernel library（危险，高频断裂）： - CUDA Graph 的基本假设是 kernel 输入指针在 capture/replay 间恒定（Ascend ACL Graph 也只追踪地址一致性）。若 vendor 的 HBM allocator 在 replay 间换了基址，图会静默错——Ascend 显式在 debug 模式强校验 entry.input_addresses。 - cuDNN / cuBLAS / NCCL 这三个 CUDA-only 闭源 不能直接移植，每 vendor 必须提供等价：cuBLAS → vendor BLAS（CANN/rocBLAS/oneMKL）；cuDNN → vendor NNG；NCCL → vendor CCL。 - 通信 kernel 的 collective 语义存在细粒度差异（e.g. HCCL 的 HCCL_OP_EXPANSION_MODE=AIV 这种内部调度模式、kernel launch 的 async 含义），Ascend 这些选项在 ascend_config.py 中作为 additional_config 暴露，且需用 patch 注入（patch_minimax_m2_config.py 的 _verify_cuda_graph 改写 HCCL_OP_EXPANSION_MODE）。

断裂边界的两类： - Break-on-contact：编译期/启动期一用就挂——如硬编码 torch.cuda.current_stream(0)、cudnn.handle() 全局单例。解法：全局 grep torch.cuda、cudnn、nccl 词法。 - Break-on-load：运行时/模型加载期挂——如 memory profiling 误把 NPU 当前显存当可用、或 cudagraph pool 阻塞正常。解法：在 Platform shim 统一 lazy/check。

给其它厂商的经验：在 vLLM 路径下 torch.cuda.* 的 剩余 0 引用 是 shim 完成的充分必要条件——写脚本统计，0 前不发版。

B.5 测试与主版本同步策略

vLLM 以 "月更主版本" 节奏演进；v0→v1 是近年最大一次重构（Scheduler 重写、v2 model_runner、Config 对象化、cudagraph Mode enum 化）。Ascend 的策略：

  1. 把每主版本 align 锚定到 --additional-config 与 patch 两条线：ascend_config 是 feature switch 总栓，patch 是版本对齐归零的"债务清单"。
  2. patch 文件上标记 "Remove this patch when vLLM merges PR #xxxx"（见 patch/__init__.py 所有条目）。每主版本升级时扫描一次：已合入的 → 删 patch；未合入的 → rebase。
  3. 双 runner 并存：Ascend 同时维护 v1 NPUModelRunner 与 v2 NPUModelRunner（worker/v2/model_runner.py:58）——当 vLLM 主版本新增 v2 路径时 Ascend 不必追齐所有功能，而是 feature-gate 灰度（patch_v2/patch_use_v2_model_runner.py 控制）。
  4. xlite 支线是更激进的对齐策略：与 v1/v2 完全解耦的 runner/manger 堆，让 novel graph runtime 独立演进，不绑定 vLLM 内部 impl。

方法论原则： - 把 vLLM 当成"上游 Linux kernel"跟进：mainline-vendor 双轨；vendor 的实现尽量可插拔，vendor fork 的 core/ 子目录只放"上游拒绝"的 innovation。 - patch 层 size 作为 technical debt 度量：patch 文件持续增长 = vendor还没把 feature push 到 vLLM mainline。好的目标是每主版本净减 patch。

B.6 风险分析（此适配模型的脆弱面）

  1. vLLM 内部 API 迁移：v0→v1 把 Scheduler/worker/attention_meta/metrics 全面重命名；Ascend 的 60+ patch 文件每次大版本都要扫一遍。风险点：import path 改名 + dataclass field 重命名 + enum 新增。
  2. patch 层的脆弱性：当 upstream 把 verify_and_update_config 方法签名从 (cls, vllm_config) 改成 (cls, vllm_config, extra_arg)、或把 dataclass 从 attrs 换成 pydantic——patch 的 setattr wrapper 静默失败。归零条件（"vLLM 合入 PR"）在 v1→v2 迁移时代全部移位。
  3. CUDA-shim 泄漏：Ascend 在 v2 runner 路径下被迫把 MLA backend 名字写为 "FLASH_ATTN"（attention/mla_v1.py:80`）绕过 vLLM 内部断言——运行时名字不再反映语义，profile/日志/错误信息的 confusion。任何非 CUDA vendor 都会遇到类似泄漏。
  4. vendor kernel meta 注册遗漏：新 custom op 忘了配 meta（meta_registration.py），torch.compile 阶段的 graph capture 会 fallback 到 eager，decode 性能直接腰斩。需 CI 强校验"每个 _C_ascend 注册都有对应 meta"。
  5. config 爆炸：Ascend 的 ascend_config.py:897 已经有 40+ knob；其与 upstream VllmConfig 多对一映射使 "config 互斥判断" 膨胀（如 recompute_scheduler_enable 与 oproj_tensor_parallel_size 互斥被显式 assert）。维护需独立 test 文件做 pairwise exclusion 检查。
  6. dual/triple runner 维护成本：v1 / v2 / xlite 三套 runner 共存——新 feature 需三处同步，回归 matrix 3x。这是 Ascend 图执行长期演进的"甜蜜负担"。
  7. test gap：vendor-private test（model-based 端到端）与 vLLM public test 之间存在 gap；vendor-public 测试仅在 vLLM CI 跑一部分，其余 kernel-level / graph-level 行为漂移未被捕获。

Part C — 给推理架构师的设计洞察（~500 字）

Ascend case 给 vLLM 架构质量画像。

做得好、应当扩大的扩展点：

  1. Platform抽象——Ascend 用 subclass 就把 device discovery / memory / graph size 接管。这是 vLLM 最干净的扩展口（platform.py 接口 12 个类方法），AMD / Intel / Groq 都高度复用。建议：把 Platform.get_device_total_memory 改成 async/lazy 接口，消除 NotImple-mentedError hack；把 check_and_update_config 的 "config 改写"语义从目前"覆盖"变成"validator pipeline"（可堆叠校验器），降低每个 vendor 重写全段的风险。

  2. AttentionBackend 注册机制——@register_backend + get_impl/builder/kvcache_shape 协议把 paged KV 生命周期完整交给 vendor。Ascend 5 个 backend 共存证明了该机制的可扩展性。建议：把 backend name assertion（如强制写 "FLASH_ATTN" 才能绕过）改为无约束 string + capability traits，根除 vendor 绕路的动机。

  3. KVConnector 协议——MooncakeConnector / MooncakeLayerwiseConnector / AscendStore 三套并行挂在 KVConnectorBase_V1 上、通过 MultiConnector 聚合，证明 vLLM 的 KV offload/disaggregation 抽象够用。

  4. SchedulerConfig.scheduler_cls 字符串解耦——Ascend 注入 RecomputeScheduler / ProfilingChunkScheduler 完全不动 upstream scheduler 类。这是 vLLM 调度器扩展的范本场景。

暴露 gap、迫使 vendor 走 patch 的薄弱面：

  1. forward-time op substitution 无 hook：Ascend CustomLinearOp.apply() + patch_process_weights_after_loading 是 vLLM 不给"linear method 外围注入点"的副产物。应增加 ModelRunner.op_substitution_registry，让 vendor 用声明式替换 matmul+comm 序列。

  2. CUDAGraphMode enum 单一假设：ACL Graph / piecewise ACL Graph 用 CUDA Graph enum 强 overlay，每个模式要在 Platform shim 里人为 rewrite。应增加 GraphBackend Protocol，让 graph capture / replay / size 选择走 abstract base + vendor 注册，ACL/NCCL/Metal/GraphCore 各自实现。

  3. MoE 通信模式无 dispatch hook：Ascend ascend_forward_context 用 _EXTRA_CTX 单例注入 MC2 / FlashComm / AllToAll selection，本质是影子上下文 patch。应增加 MoECommStrategy extension point，让 vendor 在 config 层选择 MC2/FusedMC2/AllGather/AllToAll。

  4. patch 层是 "implicit contract debt"：Ascend patch/__init__.py 内 60+ 文件、每文件 6 项元数据、归零条件是 "vLLM 合入 PR"——这表明 vLLM 扩展面不够用，vendor 在用 "debt" 换 "不 fork"。量化指标：vendor patch file count 应作为 vLLM 健康度的负向 KPI。

给自研 DSA 的一条结论：vLLM 的 Platform + AttentionBackend + KVConnector + SchedulerConfig + QuantizationConfig 五大注册点 是够用来启动一个 "0 hack" 的非 CUDA 后端；但一旦进入 forward-time op substitution、MoE 通信模式、图执行三个深层，vendor 将被迫进入 patch 区。任何新 DSA 切入 vLLM 前，应先盘清自家 kernel 在以上 five + three 个区域的归属，再定 subclass / patch / fork 三档投入比。Ascend 的实践表明 patch 层占比 ~30%（60+/200+ 入口文件），这是该阶段的合理代价；但若想在 12 个月内把 patch 层压到 0，则需要与 vLLM 上游 共同设计 OpSubstitutionRegistry + GraphBackend Protocol + MoECommStrategy 三个新扩展点——这比任何单家 vendor 的 patch 自维护都更划算。
