1. 公理:软件模块化的第一性原理——"按变化速率切"

在讨论 vLLM 任何一个模块之前,先明确软件架构第一性原理(不是物理原理):

第一性原理 #0:一个模块的边界,应该切在"变化速率不同"的地方。
如果 A 和 B 总是同时变化,它们应该在同一模块;如果 A 每天变、B 每年变,它们必须分开,否则 A 的每次改动都要碰 B。

这条原理的推论:

变化速率vLLM 里对应所以被切到
每天——策略、参数、调度顺序recompute/SRF/chunked/FCFS/PriorityScheduler
每周——新模型、新模型变体200+ model defs (DeepSeek/Qwen/GLM/Gemma/Kimi...)models/ 目录
每月——新硬件、新 kernelFA3 / FlashInfer / Triton / CANN / ROCmAttentionBackend + ops/
每季——并行策略、通信拓扑TP/PP/DP/EP/CP 组合Worker + distributed/
每年——执行范式、图捕获CUDA Graph / torch.compile / ACL GraphCompilation
永不——推理的数学定义"一个 step = schedule → forward → sample"EngineCore.step

这就是 vLLM 模块分层的真正第一性原理:不是"功能分解",是"变化速率分解"。每一层只容纳一种变化速率;层与层之间的合约就是"变化慢的那一边的稳定接口"。

2. 全景:vLLM v1 模块职责地图

① EngineCore v1/engine/core.py

唯一职责:拥有时间线。
• 持有 Scheduler + Executor
• 驱动 step() 主循环
• 处理请求入队 / abort / pause / resume / sleep / wakeup
• 协调 DP rank 间同步
• 把 SchedulerOutput → Executor,把 ModelRunnerOutput → Scheduler.update_from_output
不碰:设备、模型、tensor、调度策略

② Scheduler v1/core/sched/

唯一职责:纯决策函数。
• 输入:当前请求状态 + KV-cache 可用块数
• 输出:SchedulerOutput(req→token 分配 + 新 block 分配 + 抢占列表)
• 策略:FCFS / Priority / SRF / recompute / profiling-chunk
• 持有 KVCacheManager(但只调用它的 allocate/free 接口)
不碰:设备、模型、kernel、通信

③ Worker v1/worker/

唯一职责:拥有设备。
• 持有 GPU/NPU device handle
• 持有 ModelRunner(或 v2 的 GPUModelRunner)
• 持有 KV-cache 物理张量 kv_caches: list[torch.Tensor]
• 负责 load_model / determine_available_memory / sleep / wakeup
• 负责 CPU binding / 设备初始化
不碰:调度策略、请求生命周期

④ ModelRunner v1/worker/gpu/model_runner.py

唯一职责:执行模型前向。
• execute_model(scheduler_output) → ModelRunnerOutput
• 把 SchedulerOutput 翻译成 InputBatch + block_tables + slot_mappings + attn_metadata
• 调用 model(**inputs) 或 cudagraph_manager.run_*
• 持有 Sampler / EPLB state / spec decode drafter
不碰:请求入队、调度策略、设备发现

⑤ AttentionBackend v1/attention/

唯一职责:把"通用 attention 元数据"翻译成"硬件 kernel 私有元数据"。
• Builder 把 common_attn_metadata → 本地 metadata
• Impl 执行 kernel
• 注册式:每个硬件一个 backend
不碰:模型结构、调度、KV-cache 管理

⑥ KV 管理层(四层)

KVCacheManager:请求生命周期 ↔ block 分配
KVCacheCoordinator:多 KV-cache group(MLA/GQA/SWA)的协调
BlockPool:物理 block 的 free-list + content-hash 缓存
KVConnector:跨 worker 的 KV 搬移(PD 分离 / offload)
不碰:调度策略、模型执行

⑦ Compilation compilation/

唯一职责:把"多次 kernel launch"变成"一次 graph replay"。
• CudagraphDispatcher 决定 FULL / PIECEWISE / NONE
• PiecewiseBackend 分段编译
• PassManager 做 kernel fusion
• 平台通过 get_static_graph_wrapper_cls() 注入 ACLGraphWrapper
不碰:模型语义、调度

⑧ Platform platforms/

唯一职责:把"设备相关假设"压成一个接口。
• 6 个必须实现的方法
• 持有 device capability / memory / stream / attn backend 选择
• 持有 graph pool / static graph wrapper 选择
不碰:一切上层逻辑

3. EngineCore:时间线的主人

vllm/v1/engine/core.py:101 class EngineCore

唯一职责

EngineCore 是"时间线"的唯一主人。它拥有 Scheduler(决策)和 Executor(执行),驱动 step() 主循环。整个推理系统的"心跳"就是 EngineCore.step()。

为什么必须独立

EngineCore 把三件事串行化到同一个线程:

def step(self):
    scheduler_output = self.scheduler.schedule()          # 1. 决策
    future = self.executor.execute_model(scheduler_output, non_block=True)  # 2. 派发(异步)
    model_output = future.result()                        # 3. 等待
    engine_core_outputs = self.scheduler.update_from_output(...)  # 4. 更新

这三步必须串行,因为调度决策依赖上一步的输出(哪些请求完成了、哪些 block 被释放)。如果把这些逻辑散落在 Worker 或 Scheduler 里,就需要跨模块的锁或消息队列——复杂度爆炸。EngineCore 把它们收拢到一个线程,用单线程事件循环消除了所有并发竞争。

EngineCore 还管什么(容易被忽略)

  • 请求生命周期:add_request / abort_requests / preprocess_add_request
  • 暂停/恢复:pause_scheduler / resume_scheduler / is_scheduler_paused
  • 睡眠/唤醒:sleep / wake_up(把 KV-cache 腾到 CPU 或关 NPU 电源)
  • LoRA 热插拔:add_lora / remove_lora / pin_lora / list_loras
  • 缓存重置:reset_prefix_cache / reset_encoder_cache / reset_mm_cache
  • DP 协调:step_with_batch_queue(多 DP rank 的 batch 队列同步)
  • spec decode 草稿收集:post_step → take_draft_token_ids

📐 这些看似杂乱的功能有一个共同点:它们都是"时间线上的事件"——请求来了、请求被取消了、系统要休眠了、LoRA 要换了。EngineCore 是"时间线主人"这个职责的自然推论。

EngineCore 不碰什么

EngineCore 不持有 device、不持有模型、不持有 tensor。它只持有 Scheduler(纯 Python 对象)和 Executor(跨进程 RPC 句柄)。这意味着 EngineCore 可以跑在 CPU 的一个线程里,完全不知道 GPU 的存在——这是"控制面与数据面分离"的极端形式。

4. Scheduler:纯决策函数

vllm/v1/core/sched/scheduler.py:418 def schedule()

唯一职责

Scheduler 是一个纯决策函数:输入当前请求状态 + KV-cache 可用块数,输出 SchedulerOutput(每个请求分多少 token、分配哪些新 block、抢占谁)。

为什么必须独立

调度策略是 vLLM 里变化最快的部分。v0 只有 FCFS,v1 加了 chunked prefill,Ascend 又加了 recompute / SRF / profiling-chunk。如果调度逻辑散落在 EngineCore 或 Worker 里,每次加策略都要碰时间线或设备代码。

Scheduler 的独立性体现在:

  • 不持有 device:它只操作 KVCacheBlock(纯 metadata),不碰 HBM
  • 不持有模型:它不知道模型结构、不知道 kernel
  • 不持有通信:它不知道 TP/EP/DP 拓扑
  • 输入输出都是纯 Python 数据结构:SchedulerOutput = (req_ids, num_scheduled_tokens, req_to_new_blocks, preempted_reqs, ...)

Scheduler 与 KVCacheManager 的关系

Scheduler 持有 KVCacheManager,但只调用它的 allocate_slots / free / get_computed_blocks 接口。Scheduler 是"请求生命周期"的主人;KVCacheManager 是"block 分配"的主人。两者分开,因为:

  • 调度策略变化时(比如加 SRF),block 分配逻辑不变
  • block 分配变化时(比如加 prefix cache / sliding window),调度策略不变

SchedulerOutput:跨边界的合约

vllm/v1/core/sched/output.py 定义的 SchedulerOutput 是 Scheduler 与 Worker 之间的合约:

@dataclass
class SchedulerOutput:
    scheduled_new_reqs: list[Request]          # 新请求
    scheduled_resumed_reqs: list[Request]      # 恢复的请求
    scheduled_running_reqs: list[Request]      # 继续运行的请求
    preempted_reqs: list[Request]              # 被抢占的请求
    req_to_new_blocks: dict[str, KVCacheBlocks]  # 请求→新分配的 block
    num_scheduled_tokens: dict[str, int]        # 请求→本步 token 数
    scheduled_spec_decode_tokens: dict[str, list[int]]  # spec decode 草稿
    scheduled_encoder_inputs: dict[str, list[int]]      # encoder 输入
    has_structured_output_requests: bool       # 是否有结构化输出请求
    finished_req_ids: set[str]                 # 本步完成的请求
    encoder_cache_not_empty: bool              # encoder cache 是否非空

📐 这个合约的设计决策:

  • 只传"决策",不传"数据":SchedulerOutput 里没有 tensor、没有模型输入,只有 block id 和 token 计数。这意味着这个合约可以跨进程、跨节点传输(PD 分离时 Scheduler 在 P 节点、Worker 在 D 节点)
  • 显式列出所有情况:new / resumed / running / preempted 四类,而不是一个通用状态。这让 Worker 可以用最简单的 if-else 处理,不需要再查请求状态
  • block 分配在 Scheduler 侧完成:Worker 只"消费" block id,不参与分配决策。这保证了 block 分配策略(比如 prefix cache 命中)集中在 Scheduler

5. Worker:设备的主人

vllm/v1/worker/worker_base.py + v1/worker/gpu_worker.py

唯一职责

Worker 是"设备"的唯一主人。它持有 GPU/NPU device handle、KV-cache 物理张量、ModelRunner。每个设备(每张卡)一个 Worker 进程。

为什么必须独立

设备相关代码是 vLLM 里变化第二快的部分(仅次于调度策略)。新硬件、新驱动、新 CANN/torch_npu 版本、新 CPU binding 策略——这些变化不应该碰到调度或模型代码。Worker 把它们全部收拢。

Worker 的职责清单

职责方法为什么归 Worker
设备初始化init_device需要 device handle、NPU 初始化、CPU binding
模型加载load_model / reload_weights需要把权重搬到 HBM
内存探测determine_available_memory需要实际分配 tensor 测 HBM 余量
KV-cache 物理存储kv_caches: list[torch.Tensor]HBM 里的张量
睡眠/唤醒sleep / wake_up需要操作 device power state
LoRA 加载add_lora / remove_lora / pin_lora需要改 HBM 里的权重
执行模型execute_model需要调 model forward
采样sample_tokens需要调 Sampler
健康检查check_health需要检查 device state

Worker 与 ModelRunner 的关系

Worker 持有 ModelRunner,但两者职责不同:

  • Worker:管"设备"——初始化、内存、加载、睡眠、健康
  • ModelRunner:管"模型执行"——把 SchedulerOutput 翻译成 kernel 调用

分开是因为:同一个设备可以跑不同的模型(换 model 不用换 Worker);同一个模型可以在不同设备上跑(换 device 不用换 ModelRunner 的核心逻辑)。Ascend 的 NPUWorker(NPUModelRunner) 是子类化,但接口不变。

Worker 不碰什么

Worker 不持有 Scheduler、不做调度决策、不持有请求状态。它只接收 SchedulerOutput(纯决策),翻译成 kernel 调用。这意味着 Worker 可以无状态地执行任何调度策略——调度策略变了,Worker 代码不动。

6. ModelRunner:模型的前向执行器

vllm/v1/worker/gpu/model_runner.py:1134 execute_model

唯一职责

ModelRunner 把 SchedulerOutput 翻译成 kernel 调用。它是"决策"到"执行"的翻译层。

execute_model 的数据流(为什么必须独立)

execute_model 做 5 件事,每件都是一种"翻译":

阶段输入 → 输出翻译类型
1. Request bookkeepingSchedulerOutput → 更新本地 req_states"调度决策" → "本地状态"
2. prepare_inputsreq_states → InputBatch{input_ids, positions, query_start_loc, seq_lens, logits_indices}"请求状态" → "模型输入张量"
3. prepare_attnreq_states + block_pool → block_tables[groups,reqs,blocks] + slot_mappings[groups,tokens]"block id" → "页表 + 物理偏移"
4. model_state.prepare_attncommon_attn_metadata → attn_metadata[backend]"通用元数据" → "硬件私有元数据"
5. model forward / cudagraph replaymodel_inputs → hidden_states / logits"张量" → "计算结果"

📐 为什么这 5 步必须在同一个模块? 因为它们共享一个中间表示(InputBatch + block_tables + slot_mappings)。如果把这些步骤散落在 Worker / AttentionBackend / Compilation 里,就需要在模块间传递这些中间表示——接口爆炸。ModelRunner 把它们收拢,对外只暴露 execute_model(scheduler_output) → output 一个接口。

ModelRunner 持有的"状态"vs"无状态"

ModelRunner 持有执行状态(req_states、block_tables、kv_caches、sampler),但不持有调度状态(哪些请求在运行、哪些被抢占)。这是它与 Worker 的关键区别:

  • Worker 状态:设备级——device handle、HBM 占用、模型权重
  • ModelRunner 状态:执行级——当前 batch 的 block 映射、attention 元数据、采样状态
  • Scheduler 状态:调度级——请求队列、优先级、block 分配策略

三层状态各自独立演化:换设备不改执行状态,换模型不改调度状态,换策略不改设备状态。

7. AttentionBackend:唯一硬件相关的算子

vllm/v1/attention/backend.py + v1/attention/backends/

唯一职责

AttentionBackend 把"通用 attention 元数据"翻译成"硬件 kernel 私有元数据"。它是 vLLM 里唯一一个"必须为每个硬件单独实现"的模块。

为什么必须独立

Attention 是 vLLM 里唯一"内存访问模式随序列长和 KV 布局变化"的算子。GEMM、Norm、MoE dispatch、RoPE 都是规整的 (N,d)→(N,d),只有 attention 是 (N,d) × KV(N,d) → (N,d)——它的性能取决于 KV 怎么排列(paged / ragged / compressed-MLA / sliding-window)。

这意味着:换硬件必须换 attention kernel;但换模型、换调度、换并行策略不应该换 attention kernel。所以 attention 必须单独抽层。

AttentionBackend 的三明治结构

层职责变化速率
AttentionBackendEnum + @register_backend字符串 → 实现类映射每硬件一次
AttentionMetadataBuilder.build(common)通用元数据 → 本地 metadata每 backend 一次
AttentionImpl.forward(layer, q, k, v, kv_cache, metadata)执行 kernel每硬件 × 每模型类型
KVCacheSpec描述 KV 张量形状 / block stride / 压缩每模型类型一次

📐 KVCacheSpec 是这个设计的关键。它把"KV 张量长什么样"从 ModelRunner 里抽出来:

  • FullAttentionSpec:标准 KV,block_size = 逻辑块大小
  • MLASpec:压缩 KV,storage_block_size ≠ block_size(一个逻辑 block 对应多个物理存储块)
  • RSWASpec:滑动窗口,需要 window_size
  • RaggedSpec:变长,用于 encoder-decoder
ModelRunner 只认 KVCacheSpec,不认具体模型——这是"模型结构变化"与"KV 管理"解耦的关键。

为什么不是"模型的一部分"

如果 attention kernel 嵌在模型代码里(比如 DeepSeekV3Attention 里直接调 FA3),那么:

  • 换硬件要改模型代码(200+ 模型 × N 硬件 = 爆炸)
  • 换 attention 实现(FA3 → FlashInfer → Triton)要改模型代码
  • prefix cache / sliding window / MLA 的 KV 布局变化要改模型代码
单独抽成 AttentionBackend 后,模型代码只调 self.attn(q, k, v),不关心底层是 FA3 还是 CANN。这是"变化速率分解"的又一个实例。

8. KV 管理层:为什么是四层而不是一层

v0 里 KV-cache 管理是一个 BlockSpaceManager。v1 拆成了四层:

① KVCacheManager

职责:请求生命周期 ↔ block 分配
• allocate_slots(request, num_tokens) → KVCacheBlocks | None
• free(request)
• get_computed_blocks(request)(prefix cache 命中查询)
• 持有 Coordinator + BlockPool
变化速率:调度策略级(每天)

② KVCacheCoordinator

职责:多 KV-cache group 协调
• 一个模型可能同时有 MLA + GQA + SWA 层
• Coordinator 把"逻辑 block id"翻译成"每个 group 的物理 block id"
• 子类:Unitary / Hybrid / NoPrefixCache
变化速率:模型结构级(每周)

③ BlockPool

职责:物理 block 的 free-list + content-hash 缓存
• get_new_blocks / free_blocks / touch / evict_blocks
• cached_block_hash_to_block:hash → block 映射
• 纯数据结构,不感知请求
变化速率:算法级(每月)

④ KVConnector

职责:跨 worker 的 KV 搬移
• start_load_kv / wait_for_layer_load / save_kv_layer / wait_for_save / get_finished
• 实现:PD 分离 / CPU offload / KV Pool
• 持有 KV-cache 物理张量的引用
变化速率:部署拓扑级(每季)

为什么必须拆四层

如果不拆问题拆开后
Manager + Coordinator 不分加新 KV 布局(MLA)要改 Manager 的 allocate 逻辑Coordinator 封装"多 group 翻译",Manager 只调 Coordinator
Coordinator + BlockPool 不分加 prefix cache 要改 block 分配算法BlockPool 封装"free-list + hash",Coordinator 只调 BlockPool
Manager + Connector 不分PD 分离 / offload 的搬移逻辑污染 ManagerConnector 独立实现搬移,Manager 只调 Connector 的 get_finished

📐 关键设计决策:allocate_slots 返回 None 而不是抛异常。KVCacheManager.allocate_slots 在 HBM 不足时返回 None,Scheduler 收到 None 后触发抢占。这个设计把"分配失败"从"异常"降级为"正常控制流",让 Scheduler 可以用统一的 while 循环处理:

while True:
    new_blocks = kv_cache_manager.allocate_slots(request, num_new_tokens)
    if new_blocks is not None: break
    preempted = self.running.pop()   # 抢占最低优先级
    self._preempt_request(preempted)
如果 allocate_slots 抛异常,Scheduler 就要 try/except,抢占逻辑和正常分配逻辑会混在一起——把"分配失败"设计成返回值而不是异常,是"控制流统一"的关键。

KVCacheBlocks:跨边界的 block 容器

vllm/v1/core/kv_cache_manager.py:30 KVCacheBlocks 是 Scheduler → Worker 之间传递 block id 的容器:

@dataclass
class KVCacheBlocks:
    blocks: tuple[list[KVCacheBlock], ...]   # 每个 group 一个 list
    def get_block_ids(self) -> tuple[list[int], ...]   # 返回每个 group 的 block id 列表

它只传 block id,不传 tensor。这意味着这个容器可以跨进程、跨节点传输(PD 分离时 P 节点分配 block、D 节点消费 block id)。

9. Compilation:把"录一次放多次"独立成一层的理由

vllm/compilation/ + v1/worker/gpu/cudagraph_utils.py

唯一职责

Compilation 把"多次 kernel launch"变成"一次 graph replay"。它解决的是kernel launch overhead问题:Python → CUDA driver → launch 一个 matmul ≈ 5-10μs,80 层 transformer forward 几十次 kernel = 录成 1 次 launch 整个 graph。

为什么必须独立

图捕获与回放是一种"执行范式",不是"模型逻辑"或"调度逻辑"。它独立于:

  • 模型:同一个模型可以 eager / piecewise / full graph 执行
  • 调度:同一个调度策略可以 eager / graph 执行
  • 硬件:CUDA Graph / ACL Graph / torch.compile 是不同的捕获后端
如果图捕获逻辑嵌在 ModelRunner 里,那么加一种新图模式(比如 piecewise)要改 ModelRunner——但 ModelRunner 已经有 1600 行。独立成 Compilation 模块后,ModelRunner 只调 cudagraph_manager.run_*(),不关心捕获细节。

三种图模式:为什么是三种而不是一种

模式动态维度显存TTFT适用
FULL完全固定大慢(首 token 要录)纯 decode 吞吐最大
PIECEWISE动态 shapes中快mixed prefill + decode
Eager (NONE)全动态小最快调试 / encoder-decoder

三种模式的存在是因为"图捕获的收益(消除 launch overhead)"与"图捕获的代价(显存 + 首 token 延迟 + 形状约束)"之间的 trade-off 没有单一最优解。Compilation 模块把这个 trade-off 暴露给用户,而不是硬编码一种。

平台注入点:get_static_graph_wrapper_cls()

vLLM 通过 NPUPlatform.get_static_graph_wrapper_cls() 返回 ACLGraphWrapper,让 Ascend 的图捕获逻辑(需要 update_graph_params() hook)可以注入,而不需要改 vLLM 的 CudagraphDispatcher。这是"平台差异通过接口注入,不改上层"的又一个实例。

10. 跨模块合约为什么必须这样切

合约 #1:SchedulerOutput(Scheduler → ModelRunner)

设计决策:只传"决策"(block id + token 计数),不传"数据"(tensor)。

为什么:Scheduler 可以跑在 CPU 的一个线程里,不持有 device;ModelRunner 跑在 GPU 上。如果 SchedulerOutput 里带 tensor,就需要跨进程传 tensor(昂贵)。只传 block id(整数)和 token 计数(整数),传输代价 O(1)。

合约 #2:block_tables + slot_mappings(ModelRunner → AttentionBackend)

设计决策:ModelRunner 把 SchedulerOutput 翻译成两张张量:

  • block_tables[groups, reqs, max_blocks]:每个请求的 block id 列表
  • slot_mappings[groups, tokens]:每个 token 的物理 KV 偏移
然后 AttentionBackend 的 Builder 把它们打包成硬件私有 metadata。

为什么:block_tables 是"页表",slot_mappings 是"页表 walk 后的物理地址"。这两张张量是"调度决策"到"kernel 输入"之间的唯一翻译。如果跳过这步,让 AttentionBackend 自己从 SchedulerOutput 算 slot_mappings,那么每个 backend 都要重复实现页表 walk——代码重复 N 次(N = 硬件数)。ModelRunner 做一次,所有 backend 共享。

合约 #3:ForwardContext(ModelRunner → model forward)

vllm/forward_context.py:132 class ForwardContext 是 model forward 的"执行上下文":

@dataclass
class ForwardContext:
    attn_metadata: dict[str, AttentionMetadata]   # 每层一个 metadata
    slot_mapping: dict[str, torch.Tensor]         # 每层一个 slot_mapping
    dp_metadata: DPMetadata | None                # DP 协调信息
    cudagraph_runtime_mode: CUDAGraphMode         # 当前图模式
    batch_descriptor: BatchDescriptor | None      # batch 描述(用于 LoRA dispatch)
    is_padding: torch.Tensor | None               # padding mask
    skip_compiled: bool = False                   # 是否跳过编译路径

为什么单独一层:ForwardContext 是"model 代码"与"执行框架"之间的防火墙。model 代码(200+ 模型定义)只读 ForwardContext,不直接碰 SchedulerOutput / block_tables / cudagraph_manager。这意味着:

  • 改调度策略:model 代码不动
  • 改图模式:model 代码不动
  • 改并行策略:model 代码不动
  • 加新模型:只需要从 ForwardContext 读数据,不需要理解调度/图/并行

合约 #4:KVCacheSpec(ModelRunner ↔ AttentionBackend ↔ BlockPool)

设计决策:KVCacheSpec 描述"KV 张量长什么样"(shape / stride / block_size / compression),由 ModelRunner 在 init 时计算,传给 AttentionBackend 和 BlockPool。

为什么:MLA 的 storage_block_size ≠ block_size(一个逻辑 block 对应多个物理存储块)。如果 BlockPool 不知道这个映射,就会分配错大小的 block。KVCacheSpec 把这个映射显式化,让 BlockPool 和 AttentionBackend 都按同一个 spec 工作。

11. 横切视角:具体场景如何穿过这些边界

P/D 分离:在 Worker 边界上切了一刀

P/D 分离没有引入新模块。它是在原有 Worker 边界上切了一刀:

节点拥有不拥有
P 节点(Prefiller)Scheduler + Worker(P 侧) + KVConnector(P→D)D 节点的 HBM、decode 循环
D 节点(Decoder)Scheduler + Worker(D 侧) + KVConnector(D←P)P 节点的 HBM、prefill 计算

关键设计决策:P 节点的 Scheduler 在 prefill 完成后,通过 KVConnector.request_finished() 把 KV 交给 D 节点,然后释放 P 节点的 block(但不释放请求状态)。D 节点的 Scheduler 收到请求后,标记为 WAITING_FOR_REMOTE_KVS,通过 KVConnector.start_load_kv() 拉取 KV,然后开始 decode。

📐 为什么 P/D 分离没有改 Scheduler 的核心逻辑:因为 Scheduler 只认 KVCacheBlocks(block id 列表),不认 block 在哪个节点。P 节点分配的 block id 通过 KVConnector 传到 D 节点,D 节点用同样的 block id 在自己的 HBM 里分配物理存储。Scheduler 的"请求生命周期"与"KV 物理位置"解耦——这是 KVCacheManager 独立于 Scheduler 的直接回报。

TP/EP/DP:为什么在 Worker 侧不在 Scheduler 侧

策略Scheduler 知道吗Worker 做什么
TP不知道——Scheduler 只认"一个请求",不认"一个请求被切到多张卡"Worker 在每层 forward 后调 all-reduce / all-gather,把部分结果合并
EP不知道——Scheduler 不关心 expert 分布Worker 在 MoE 层做 all-to-all dispatch / combine,把 token 路由到对应 expert 的卡
DP知道一点——Scheduler 知道 DP size,用于 prefill balancingWorker 在每 step 前后做 DP broadcast,同步 batch descriptor

为什么 Scheduler 不知道 TP/EP:因为 Scheduler 的决策粒度是"请求",不是"token 在哪个卡上"。如果 Scheduler 要知道 TP/EP,它就需要知道模型结构(MoE 有几层、每层几个 expert)——这违反了"调度策略与模型结构解耦"的原则。

Worker 怎么知道 TP/EP 而不污染 ModelRunner:通过 forward_context.dp_metadata 和 model.forward() 内部的 self.moe_router。ModelRunner 不直接碰 TP/EP 通信,它只调 model(**inputs);模型内部的 ColumnParallelLinear / RowParallelLinear / MoEDispatch 自己处理通信。这是"并行策略下沉到模型层,不进入执行框架"的设计。

算子融合:为什么下沉到 Compilation 而不是 ModelRunner

融合在哪例子为什么
Compilation PassManagerAllReduce+RMSNorm、QK-Norm+Rope、AllGather chunk noop融合是"把多个 kernel 合成一个",不改变模型语义——所以是"执行优化",不是"模型逻辑"
ops/ 层fused_moe、CustomLinearOp(Matmul+AllReduce)融合需要直接操作底层 API——所以是"算子实现",不是"模型逻辑"
不在 ModelRunner——ModelRunner 只调 model(**inputs),不直接调 kernel——所以融合逻辑放 ModelRunner 里无法触达 kernel

通算融合:为什么 comm_utils 独立演进

Ascend 的 ops/fused_moe/comm_utils.py 和 distributed/device_communicators/ 实现了通信与计算的 overlap(比如 AllGather 与 Matmul 的 pipeline)。这些代码独立于:

  • Scheduler:调度策略不关心通信怎么 overlap
  • ModelRunner:ModelRunner 只调 model(**inputs),不直接调通信
  • 模型:模型代码只调 self.moe_linears,不关心底层是 HCCL 还是 NCCL

为什么独立:因为通算融合是"通信拓扑"与"计算调度"之间的横向优化——它横跨多个模块,但不属于任何一个。把它放在 comm_utils 里,让通信专家和计算专家各自独立演化。

KV-cache 内存管理:从 v0 到 v1 的演化

版本KV 管理问题
v0单一 BlockSpaceManager请求生命周期 + block 分配 + prefix cache + swap 全耦合,改任何一项都碰其他
v1KVCacheManager + Coordinator + BlockPool + Connector 四层每层只负责一种变化:策略 / 多 group / 物理池 / 跨节点搬移

📐 v1 拆四层的直接动机:MLA 的出现。MLA 的 storage_block_size ≠ block_size,意味着"逻辑 block id"和"物理存储块"之间多了一层映射。如果还是 v0 的单一 Manager,这个映射逻辑会散落在 allocate / free / prefix-cache 各处。拆成 Coordinator 后,这个映射被封装在 Coordinator 里,Manager 只调 Coordinator 的接口——"多 group 映射"的变化被隔离在 Coordinator 一层。

12. 演化视角:v0 → v1 → v2 哪些边界被移动了

变化v0v1为什么移动
调度器与 worker 的关系Scheduler 在 Worker 进程里,与 model forward 共享线程Scheduler 在 EngineCore 进程里,与 Worker 跨进程消除调度与执行的相互阻塞:长 prefill 不再卡住调度器
调度循环prefill phase + decode phase 两阶段单一 token-budget 循环(无阶段)chunked prefill / spec decode / jump decode 需要统一循环
KV-cache 管理单一 ManagerManager + Coordinator + BlockPool + Connector 四层MLA / prefix cache / PD disagg 需要独立演化
图捕获在 ModelRunner 里直接调 CUDA Graph独立 Compilation 模块 + 平台注入ACL Graph / piecewise / breakable graph 需要独立于 ModelRunner
Attention嵌在模型代码里独立 AttentionBackend + Builder + Impl 三层200+ 模型 × N 硬件的 kernel 选择需要抽象

📐 演化方向总结:vLLM 的模块边界一直在"按变化速率切"的原则下移动:

  • 变化快的往里收:调度策略收到 Scheduler,kernel 收到 AttentionBackend,图模式收到 Compilation
  • 变化慢的往外推:执行范式(schedule→forward→sample)固化在 EngineCore.step,设备抽象固化在 Platform
  • 新变化速率出现就加新层:MLA → 加 Coordinator;PD disagg → 加 KVConnector;ACL Graph → 加 ACLGraphWrapper

13. 给架构师的设计洞见

判断框架:模块边界的"变化速率匹配度"

健康信号警告信号
模块内所有代码随同一个因素变化模块内部分代码随 A 变、部分随 B 变 → 应该拆
跨模块合约是"纯数据"(整数 / 枚举 / 不可变 dataclass)跨模块合约是"可变状态"或"回调函数" → 边界切错了
改策略不动执行代码,改执行代码不动策略代码改调度策略要改 Worker → 调度与执行没解耦
新硬件只需要实现 N 个接口,不改上层新硬件要改 Scheduler / ModelRunner → Platform 抽象不够

vLLM 做得最好的 5 个边界

边界为什么好
Scheduler ↔ ExecutorScheduler 是纯函数,可以单测、可以替换(Ascend 的 RecomputeScheduler 只改了 preemption 逻辑,没碰 Manager)
ModelRunner ↔ AttentionBackend200+ 模型 × N 硬件的 kernel 选择被压缩成"一个 Builder + 一个 Impl"
Platform 注入Ascend 只改了 Platform + ACLGraphWrapper + 5 个 attn backend,没改 vLLM 核心
ForwardContext 防火墙model 代码只读 ForwardContext,改调度/图/并行不动模型
KVCacheBlocks 容器只传 block id,不传 tensor → 可以跨进程、跨节点传输(PD 分离的基础)

vLLM 做得不够好的 3 个边界

边界问题建议
patch 层60+ patch 文件说明 vLLM 内部很多函数没有暴露 hook把 op-substitution / graph-backend / MoE-comm 做成正式扩展点
SchedulerOutput 膨胀随着 spec decode / multimodal / encoder-decoder 加入,字段越来越多考虑拆成"核心字段 + 扩展字段"(类似 protobuf 的 optional)
ModelRunner 1600 行execute_model 承载了太多职责(bookkeeping + prepare + attn + forward + sample)把 prepare_inputs / prepare_attn 抽成独立阶段对象(类似 Pipeline)

一句话总结

vLLM 软件架构的第一性原理:把"随 X 变化的部分"和"不随 X 变化的部分"切开。
• 调度策略(每天变) ↔ 执行范式(每年变) → EngineCore 把它们串行化
• 模型结构(每周变) ↔ 执行框架(每年变) → ForwardContext 把它们隔离
• 硬件 kernel(每月变) ↔ 模型语义(不变) → AttentionBackend 把它们隔离
• 部署拓扑(每季变) ↔ 请求生命周期(不变) → KVConnector 把它们隔离
• 图执行模式(每季变) ↔ 模型语义(不变) → Compilation 把它们隔离
• 设备差异(每年变) ↔ 上层逻辑(不变) → Platform 把它们隔离

每一个模块边界,都是"变化速率不匹配"的产物。 这就是 vLLM 软件架构的第一性原理——不是物理定律,是软件演化定律。
