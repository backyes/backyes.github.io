vLLM Architecture from First Principles

Physics-of-computation reading of vLLM (v1 era, commit 2db39c704). Audience: senior systems engineers, parallel-computing architects, AI-algorithm researchers designing inference fabrics. The claim of this note is that most of vLLM's structure is forced by three invariants — the transformer compute law, the memory-bandwidth wall, and hardware diversity — and the places where it is not forced are precisely the seams a port or a competing implementation may freely redesign. All path/file:line references are load-bearing pivot points, not décor.

Scope boundary. Request lifecycle, executor topology, the scheduler↔worker feedback contract, the Ascend overlay, the AttentionBackend/Platform swappable surfaces, and the full compilation story are each covered in dedicated notes elsewhere: - Engine / orchestration → agent1_engine_orchestration.md - Scheduler / KV-cache manager / AscendScheduler → agent2_scheduler_kvcache.md - Attention, model-runner, platform, compilation, Ascend passes → agent3_attention_model_platform.md

This note re-derives the same structure from invariants the other notes take as given. Where they answer "what does the code do", this note answers "what else could the code be, given physics".

The three invariants

Before any thesis, the three constraints that admit no engineering compromise:

  1. Transformer memory law. A live request of sequence length $S$ stores $\Theta(S \cdot L \cdot d)$ bytes of KV-cache ($L$ layers, head dim $d$, K+V). Memory is unbounded in $S$, but stale: a decoded token's KV never changes. This single asymmetry — grow monotonically, mutate never — is the root of paging.

  2. The bandwidth-flops wall at decode. Decode is memory-bandwidth-bound: one token → one matmul per layer with an $h \times h$ weight → arithmetic intensity $\approx h/2$ FLOPs/byte, well under the GPU ridge point. Prefill is compute-bound (many tokens × same weight). No single kernel can be optimal for both; the scheduler must interleave them by time-slicing, not by partitioning the device.

  3. Hardware is a distribution, not a singleton. NVIDIA/AMD/Intel/Huawei/Google/CPU each expose a disjoint dialect of the same math. If every #ifdef DEVICE_CUDA leaked into the hot path, the codebase would fork irreversibly. Every device-specific decision must be pushed behind an interface small enough to be re-implemented once.

Everything below is the consequence.

1. Paged KV-cache — from the memory law to the only correct design

First principle. A request's KV is an append-only, randomly-indexed-by-position tensor with no updates after write. Continuity in logical position is the only invariant; continuity in physical memory is irrelevant.

Given (1), two corollaries follow immediately:

  • Fragmentation is fatal if allocation is contiguous. Variable-length requests share a finite HBM pool. Contiguous alloc/free over a $O(n)$ live-request pool degrades to $O(n)$ first-fit compaction or, worse, leaves a Swiss-cheese of unusable gaps. The classic OS response is paging: fix-size pages + page-table indirection. vLLM names this KVCacheBlock + block_table.
  • Identical prefixes must alias. System prompts + few-shot examples make the prefix of a huge fraction of requests byte-identical across tenants and time. A paging scheme with a content-addressed page cache (virtual-memory page cache, Redis slab pages) turns dedup from "nice-to-have" to "the dominant term in capacity".

The data structure. The unit of paging is a metadata-only object:

vllm/v1/core/kv_cache_utils.py:117

@dataclass(slots=True)
class KVCacheBlock:
    block_id: int            # index into the flat HBM pool
    ref_cnt: int = 0         # live references (prefix-cache sharing)
    _block_hash: BlockHashWithGroupId | None = None
    prev_free_block / next_free_block: KVCacheBlock   # free linked list
    is_null: bool = False

The load-bearing insight: KVCacheBlock has no tensor. The actual block_size * num_kv_heads * head_size * 2 * dtype bytes live in the worker-side kv_caches: list[torch.Tensor] (flat HBM pool, model_runner.py:490, allocated by init_kv_cache, attn_utils.py:520). block_id is an integer index into that pool. This indirection is what makes the entire scheduler prefix-cache + swap + sliding-window + compressed-MLA stack coexist: all of them renumber/reweight the same block_ids without ever touching the pool.

The allocator is a doubly-linked free queue plus a content hash map:

vllm/v1/core/block_pool.py:162

self.blocks = [KVCacheBlock(idx) for idx in range(num_gpu_blocks)]
self.free_block_queue = FreeKVCacheBlockQueue(self.blocks)
self.cached_block_hash_to_block = BlockHashToBlockMap()   # hash -> KVCacheBlock

Allocation is $O(1)$ off the free list (get_new_blocks, :647); on a full free list, _maybe_evict_cached_block (:679) drops the lightest cached entry. When a request's prefix lands a hash hit, get_cached_block (:198) returns the same physical block to multiple Requests — ref-counted, freed only when all owners release it (touch :702, free_blocks :719). The hash is content-derived, so cross-tenant, cross-time prefix sharing "just works".

Physical data flow (the paging equation). The kernel never sees a logical token position. The worker translates every scheduled token (req, position) to a flat slot through the block table:

vllm/v1/worker/gpu/block_table.py:283

block_indices = positions // (block_size * CP_SIZE)
block_offsets = positions % (block_size * CP_SIZE)
block_numbers = tl.load(block_table_ptr + req_state_idx * stride + block_indices)
slot_ids = block_numbers * block_size + block_offsets

slot_ids is the single physical operand the paged-attention kernel reads. prepare_attn (model_runner.py:1041) builds exactly two dynamic tensors per step — block_tables $[groups,\; reqs,\; max_blocks]$ and slot\_mappings$ $[groups,\; tokens]$ — and these are the *only* scheduling-sensitive inputs to the attention kernel. Decode reads KV atslot_ids; the padding holes are filled withPAD_SLOT_IDso chunked-prefill and batched-decode share one CUDA-graph-capturable kernel. Prefill **writes** KV into freshly-zeroed blocks (new_block_ids_to_zero,scheduler.update_from_output`), the only mutation; thereafter the block is read-only and cacheable.

Paging is thus forced three ways at once: finite HBM, prefix dedup, and ragged-variable-length batching — all three collapse into one indirection.

2. The single-step scheduler — from the bandwidth wall to token-budget time-slicing

First principle. Decode is bandwidth-bound, prefill is compute-bound; they cannot simultaneously own the device, but they must appear to. The control plane must therefore carve each forward pass into a budget of tokens shared across phases, not partition requests into a "prefill phase" and a "decode phase".

Continuous batching = per-step scheduling. The old request-granular batch model wastes cycles in the gaps between requests of unequal length. Continuous batching schedules once per step, not once per request: every step, any set of in-flight requests may contribute tokens up to a per-step budget, so a long-running decode overlaps a short new prefill seamlessly. vLLM's Scheduler.schedule does exactly this:

vllm/v1/core/sched/scheduler.py:420

# There's no "decoding phase" nor "prefill phase" in the scheduler.
# Each request just has num_computed_tokens and num_tokens_with_spec…
# At each step, the scheduler tries to assign tokens to the requests so
# that each request's num_computed_tokens can catch up its
# num_tokens_with_spec. This is general enough to cover chunked prefills,
# prefix caching, speculative decoding, and "jump decoding."

The loop is one unified token-budget iterator over RUNNING, then WAITING (:462 / :1937):

vllm/v1/core/sched/scheduler.py:438

token_budget = self.max_num_scheduled_tokens      # hard per-step cap
...
while req_index < len(self.running) and token_budget > 0:
    num_new_tokens = (... catch-up to target ...)
    if 0 < long_prefill_token_threshold < num_new_tokens:
        num_new_tokens = long_prefill_token_threshold     # chunked prefill
    num_new_tokens = min(num_new_tokens, token_budget)
    ...
    if new_blocks is not None: break
    else: preempt lowest-priority running request    # reclaim budget

Chunked prefill (long_prefill_token_threshold, :500) is the analytic consequence of the bandwidth wall: without it, a compute-bound prefill monopolizes the device and starves all in-flight decodes (TTFT vs TTIT interference). Chunking hands decode a bounded share of each step.

Admission & preemption as a feedback controller. allocate_slots (kv_cache_manager.py:283) — whose 3-stage block layout | comp | new_comp | ext_comp | new | lookahead | is spelled out at :329-350 — returns None when required > free - reserved - watermark. None is the scheduler's pressure signal: STOP admitting, or preempt. The watermark (scheduler_config.watermark) reserves headroom so a last-block over-admission can't strand an in-flight prefill. This is a classic resource-feedback loop, not a policy: it is forced by the finite num_gpu_blocks which itself comes from a real HBM profile at engine init (engine/core.py:309, determine_available_memory).

Why v1 collapses v0's two phases. v0 kept prefill and decode in separate codepaths ("phases"). v1 realizes the two are the same operation under one token-budget loop — a prefill just has many tokens to catch up. The resulting single loop trivially absorbs chunked prefill, prefix-cache re-admission, speculative token lookahead, and cross-request priority as knobs on one budget. The code rewards this: schedule() is one function, not two. This is a rare case where the simpler model is also the more general one.

3. Python frontend / C++-CUDA-Triton kernel backend — the split forced by arithmetic intensity

First principle. LLM inference spends ~95 % of FLOPs in ~5 % of the code (matmul, attention, softmax, sampling) and ~5 % of FLOPs in ~95 % of the code (scheduling, tokenization, request management, API, connectors). The optimal language/execution regime for the two is disjoint:

  • The 5 % control plane changes often, needs fast iteration, and must integrate HuggingFace/tokenizers/Python ecosystem → Python, single-threaded, no GIL-fighting in the hot path.
  • The 5 % kernel plane is device-specific, must hit near-peak HBM bandwidth / MFU, and must compose inside CUDA graphs / custom ops → C++ / CUDA / Triton, registered as torch.library ops and called as opaque tensors from torch.compile.

Where the split lands. The ModelRunner.execute_model → _update_states → _prepare_inputs → _build_attention_metadata → _model_forward → _sample chain (gpu/model_runner.py, all pure Python orchestration with zero FLOPs of consequence) ends at two boundaries where Python hands off to kernels: - kv_caches bound into static_forward_context and read by the AttentionBackend.forward (flash_attn.py:811 → FA kernel at :1014). - sampler.forward_native / TopKTopPSampler (sampler.py:72, topk_topp_sampler.py:70) dispatching to FlashInfer, Triton, aiter, xpu, or pure-PyTorch by capability — the whole module is a capability dispatch, a pure-kernel surface.

_custom_ops.py is the registry of the device-specific kernels (paged_attention_rocm, rms_norm, cutlass_scaled_mm, awq_dequantize, :229, :491 …), each providing both a real impl and a @register_fake so torch.compile can reason about opaque C calls. The cost of the split — Python GIL + serialization in the multiprocess executor — is acceptable because Python touches only the 5 % control plane.

The price the split exacts (and why v1 tightened it). MultiprocExecutor.execute_model → collective_rpc (multiproc_executor.py:310, :343) fans a SchedulerOutput through a shared-memory MessageQueue and dequeues from response_mqs[output_rank] — IPC even within a single TP group. Two cross-process hops per step, cloudpickle on the method, GIL on the engine thread. v1's default hot path is UniProcExecutor (uniproc_executor.py:45) which calls straight into driver_worker synchronously and returns a true Future so the scheduler can overlap — supports_async_scheduling returns True (:146). The architectural lesson: the two-language split is correct, but its IPC tax must be minimized by keeping the scheduling process co-resident with at most one engine thread per device unless TP forces otherwise.

4. The AttentionBackend abstraction — the only data-dependent kernel is the only hardware-specific one

First principle. In a decoder transformer, every operation except attention is a regular batched matmul/projection/normalize whose memory access pattern is dense and independent of sequence length and KV layout: FFN, Q/K/O projection, RMSNorm, RoPE (pre-cache) — all regular GEMM. Attention is the ONLY operator whose access pattern depends on sequence_length, KV-cache layout, and block_size (paged, ragged, MLA-compressed, sliding-window). Therefore attention is the only kernel that must be rewritten per hardware. The rest of the stack is device-agnostic down to the GEMM (which cuBLAS/CUTLASS/HCCL each already solve).

This single-derivative fact forces an interface whose minimal contract is: (1) declare the KV-cache shape/dtype/layout this kernel consumes, and (2) build per-step metadata from the scheduler's request→block mapping, then (3) call the kernel.

The three-interface decomposition. vLLM factors attention into exactly these three objects:

vllm/v1/attention/backend.py:56

class AttentionBackend(ABC):
    # static factory: name / impl class / builder class / KV-cache shape
    def get_name() -> str: ...
    def get_impl_cls() -> type["AttentionImplBase"]: ...
    def get_builder_cls(): ...
    def get_kv_cache_shape(num_blocks, block_size, num_kv_heads, head_size, ...) -> tuple: ...
    # capability predicates (the spec the scheduler matches against)
    def supports_head_size / supports_mla / indexes_kv_by_block_stride / ...
  • AttentionBackend (factory + capabilities + shape) is resolved once and cached.
  • AttentionMetadataBuilder.build (backend.py:607, flash_attn.py:447) turns backend-agnostic CommonAttentionMetadata into backend-specific M.
  • AttentionImpl.forward (flash_attn.py:811) does the one real thing: flash_attn_varlen_func(..., block_table=block_table, ...) at :1027 — the block table is the only place the scheduler's paged view enters the math. The rest (Q scale, sliding window, R-SWA mask_mod, DCP context-parallel :920, quantization :900) is bookkeeping parameterizing this one kernel.

Registry dispatch via AttentionBackendEnum + _ATTN_OVERRIDES + @register_backend (backends/registry.py) lets this be swapped at init without touching core — a CUDA-type hint plus current_platform.get_attn_backend_cls (platforms/interface.py:373) is the device's turn to speak.

Multi-hardware as one consequence. Because attention is the only hardware-specific kernel, plugging a new device reduces to: subclass AttentionBackend/AttentionImpl/AttentionMetadataBuilder, override get_kv_cache_shape + forward + the block-layout contract (indexes_kv_by_block_stride, supports_mla), and register. Ascend proves it empirically — AscendAttentionBackend, AscendMLABackend, AscendSFABackend, AscendDSABackend (vllm_ascend/attention/{attention_v1,mla_v1,sfa_v1,dsa_v1}.py) reuse 100 % of the scheduler, KV paging, and cudagraph bookkeeping; they differ only inside .forward and in whether they use torch_npu / _C_ascend.* / Triton-to-NPU as the kernel source (ops/register_custom_ops.py, ops/triton/, C_ascend). The 5 % of code that is the kernel carries 100 % of the hardware personality — exactly as the first principle predicts.

5. Parallelism as composition — TP/PP/DP/EP are addends, not alternatives

First principle. A single accelerator cannot hold a 70B+ model nor sustain 10k req/s. Parallelism is therefore not a feature in vLLM; it is a composition law. The four axes each solve a distinct dimension of the scaling problem:

Axis Granularity Communication What it exposes to scheduler
TP (tensor parallel) intra-layer, per-linear all-reduce / all-gather, high bandwidth, on critical path invisible — one "device"
PP (pipeline parallel) inter-layer point-to-point, bubble-limited scheduler sees uniform capacity; bubbles hidden by async scheduling
DP (data parallel) whole-model replica request routing or KV sync independent schedulers, explicit coordination only for prefill balancing
EP (expert parallel, MoE) per-expert all-to-all invisible to scheduler; expert placement is a worker concern

The key architectural commitment: the scheduler is parallelism-agnostic. Scheduler.schedule deals in tokens, not devices; SchedulerOutput carries per-request num_scheduled_tokens and block ids; it has no notion of ranks, stages, or expert maps. The engine calls schedule → execute_model → update_from_output (core.py:576) and the executor is the only thing that knows how many ranks exist.

How composition is realized. Executor is abstracted as "run one TP group" (abstract.py): - MultiprocExecutor (multiproc_executor.py:103): N WorkerProc processes pre-born, holding the device + model; collective_rpc (:343) scatters one (method, args) over a shared-memory MessageQueue and funnels one response back from output_rank (_get_output_rank :498) — fan-out / single-reply. The scheduler need not know. - UniProcExecutor (uniproc_executor.py:45): single driver worker, in-process call — the degenerate "1-rank TP group", default hot path. - ExecutorWithExternalLauncher (uniproc_executor.py:150): overrides _init_executor to skip load_model and a determine_available_memory that hands control to the external launcher — the exact shape used by Ascend-style external harnesses.

The Future seam is how TP/PP get overlapped, for free. In core.py:588, execute_model(scheduler_output, non_block=True) returns a Future; the worker's reply is awaited after the scheduler emits the next SchedulerOutput. If vllm_config.max_concurrent_batches > 1, step is swapped to step_with_batch_queue (:617) — a deque of (Future, SchedulerOutput, grammar_future) triples. This is how pipeline bubbles are amortized: batch N executes while batch N+1 schedules. The contract is pure: your device must admit a true async future and you get overlap with zero extra engine code. That single seam decouples scheduling policy from collective latency.

6. The Platform abstraction — device diversity bounded to a fistful of methods

First principle. The set {NVIDIA, AMD, Intel, Huawei, AWS Inferentia/Trainium, Google TPU, CPU} is a stable fact of the market, and it grows. If CUDA-named assumptions leak pervasively, the repo forks. So vLLM defines device capability behind one base class and routes every hardware question to current_platform.*.

The contract. Platform (platforms/interface.py:134) is the single extension surface a new device subclasses. Its mandatory methods are deliberately minimal — identity, memory, and the attention-backend tie:

class Platform:
    _enum: PlatformEnum
    device_name: str
    device_type: str
    dispatch_key: str = "CPU"           # torch dispatch backend tag
    ray_device_key: str = ""            # Ray resource key
    device_control_env_var: str = "…"   # e.g. ASCEND_RT_VISIBLE_DEVICES

    def get_device_name / get_device_uuid / get_device_total_memory(device_id): ...
    def set_device(device): ...
    def get_attn_backend_cls(selected, cfg, num_heads) -> str: ...   # device's say
    def manual_seed_all(seed): ...

Everything else — pass_key (:177), get_device_capability (:420), get_current_memory_usage, inference_mode, is_cumem_allocator_available (:231), visible_device_id_to_physical_device_id — is optional with sane defaults. The Ascend port (NPUPlatform, vllm_ascend/platform.py) implements only: (a) device identity + dispatch_key = "PrivateUse1"; (b) a static (use_mla, use_sparse, use_compress) → class-path table replacing CUDA's priority policy in get_attn_backend_cls; (c) simple_compile_backend = "eager" to opt out of torch.compile and onto CANN; (d) memory accounting via torch.npu.max_memory_allocated. Everything above — scheduling, KV paging, cudagraph-mode keys, distributed comms, the execute_model skeleton — is inherited unchanged.

Why this interface is the right size. It is exactly the set of answers the generic code must ask a device to do work: "how much memory", "which RNG", "which backend", "how do I name you". It does not encode any particular model or scheduling policy. Consequently the seam has held across CUDA/ROCm/TPU/CPU/NPU. The corollary: a hardware adapter that implements Platform paged by the interface contract gets the entire scheduler, KV-cache manager, and orchestration stack for free, and only rewrites kernels (Th thesis 4) and a budget probe.

7. Design insights for an inference architect

Read the seven theses together and two meta-conclusions emerge about what in vLLM is load-bearing physics versus load-bearing engineering.

Forced, therefore stable. Three things are forced by the transformer-compute law, the bandwidth wall, and hardware diversity, and will not move under any reasonable future: - Paged KV-cache. Asymmetric append-only memory + finite HBM + prefix reuse makes indirection + content-hash the only sane design. Any high-performance inference engine will look like a page-cache OS for KV. vLLM's KVCacheBlock metadata / worker-pool split (kv_cache_utils.py vs gpu/block_table.py) is this pattern implemented literally. A new device that pages KV differently (e.g., hardware-tagged pages, compressed state) still surfaces an equivalent block-table + slot-mapping surface. - Attention as the only hardware-specific kernel. Because attention alone has length- and layout-dependent memory access, the AttentionBackend / Impl / MetadataBuilder triple is the natural minimum hardware contract. Any accelerator — NVIDIA, AMD, Huawei, TPU, CPU, a future DSA — will need exactly this metadata→kernel seam; only the body of .forward changes. - Device-memory + kernel-reporting interface behind which the rest of the stack is device-agnostic. This is the minimal-achievable Platform. It is small enough to implement once per silicon and expressive enough to carry every accelerator vLLM supports today. - Scheduler as a token-budget loop with pressure feedback. So long as decode is bandwidth-bound and prefill compute-bound, token-budget time-slicing + admission-gated-by-block-pressure is the canonical controller. The policy knobs (max_num_scheduled_tokens, chunk size, waiting order) vary; the loop skeleton won't.

Chosen, therefore evolvable. Three things are engineering choices vLLM happened to make, and a competitor or a v2 re-design may freely diverge: - The Python front-end. It optimizes for HuggingFace-ecosystem velocity and single-thread simplicity, and pays in GIL + IPC. A Rust/C++ front-end (or an LLM-native control plane) could collapse the process boundary MultiprocExecutor pays every step. Ascend's default use of ExecutorWithExternalLauncher + UniProcExecutor already shows the multi-process default isn't load-bearing. - Process-per-rank + shared-memory RPC. collective_rpc over shm message queues is a pragmatic TP default, but it serializes a SchedulerOutput per step and forces output_rank reduction. NCCL-process-group-first, or a registered-RDMA descriptor-passing scheme, are strictly better and require only a different Executor subclass — the engine loop doesn't change. - Sampling as capability-dispatched Python. TopKTopPSampler dispatch (flashinfer_sampler_supported → triton → aiter → native) lives entirely on the control-plane side; an end-to-end fused decode→sample kernel is plausible and would only move a boundary inside _sample.

The architectural invariant a hardware adapter should internalize. The cleanest way to predict "will my change fork core?" is to ask which tier it touches:

  1. PHYSICALLY FORCED (implement to exist): a KVCacheSpec + a paged KV surface that yields block_tables/slot_mappings; one AttentionBackend with a working .forward; a Platform reporting memory + attn-backend + identity. Below this floor the engine won't boot.
  2. POLICY / QUALITY (implement to compete): scheduler subclass + scheduler_cls string; a KVCacheSpec/SingleTypeKVCacheManager pair registered per attention-kind; an Executor model that matches your launcher. Here Ascend's overlay is the existence proof — RecomputeScheduler(Scheduler) + RecomputeSchedulerOutput, ShortRequestFirstSchedulerMixin + SJF-with-aging, ProfilingChunkScheduler, and per-platform single_type_kv_cache_manager — none of which touch the 600-line scheduling loop or the paged-attention pipeline.
  3. PERFORMANCE (implement to win): MLA/DSA/SFA backends if your DSA targets long context / sparse attention; a custom-op registry; a distributed communicator whose HCCL/NCCL/… semantics differ; a compile variant (ACLGraphWrapper + GraphFusionPass substituted for CUDA Graphs + inductor). These are optional for correctness and mandatory for parity — the exact place a new DSA earns its margin.

The deepest single lesson: vLLM's structure mirrors its physics with unusual fidelity. Where the code looks complex, usually a hard constraint is speaking; where the code looks simple (the step loop, the scheduler budget, the platform fields), it has converged to the unique stable point the constraints allow. A port or a rival should re-derive from the three invariants rather than copy the code mechanically — reproduce the physics, and the right architecture follows by necessity.
