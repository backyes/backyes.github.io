Layer 1 — Entrypoints, Engine Core, Executor, Orchestration

vLLM upstream repo: /vllm · vLLM-Ascend: /vllm-ascend · Focus: how a request enters vLLM, becomes scheduled work, runs a forward pass, and returns tokens. Audience: senior systems/engineer readers.

0. A note on v0 vs v1

The vllm/engine/, vllm/core/, vllm/worker/, vllm/attention/, vllm/spec_decode/ tree is the legacy v0 code. As of this checkout it is functionally empty:

  • vllm/engine/llm_engine.py (7 lines) — LLMEngine = V1LLMEngine, a plain alias to the v1 class.
  • vllm/engine/async_llm_engine.py — same, aliases vllm.v1.engine.async_llm.AsyncLLM.
  • vllm/core/, vllm/worker/, vllm/attention/, vllm/spec_decode/ — zero .py files.

All real logic lives under vllm/v1/. The v1/v0 distinction therefore matters less as "two engines" and more as two public surface shapes: the synchronous LLM API (offline, kept for back-compat) and the async-first AsyncLLM (the real serving path). Both now route to the same v1 core.

1. Request lifecycle (end-to-end)

Two top-level entry stacks share one engine core:

  • HTTP — vllm.entrypoints.openai.api_server (FastAPI) → AsyncLLM
  • Offline — vllm.entrypoints.llm.LLM (synchronous, batches prompts) → LLMEngine
  • gRPC / custom — vllm.entrypoints.grpc_server builds an AsyncLLM-delegating thread pool, so the engine-side path is identical and omitted below for brevity.

2. EngineCore architecture

EngineCore is the single inner loop. It owns: executor_class → model_executor, the Scheduler, a StructuredOutputManager, KV-cache derived state, a per-step futures deque (pipeline parallelism), and an aborts queue. Everything is hidden from the "serving" layer by the EngineCoreClient seam.

Key request state — Request

Defined at vllm/v1/request.py:59. Status is a RequestStatus enum (IntEnum) in the same file, line 343, partitioned into live (WAITING / WAITING_FOR_STRUCTURED_OUTPUT_GRAMMAR / WAITING_FOR_REMOTE_KVS / WAITING_FOR_STREAMING_REQ / RUNNING / PREEMPTED) and finished (> PREEMPTED): FINISHED_{STOPPED,LENGTH_CAPPED,ABORTED,IGNORED,ERROR,REPETITION}. Finishing is one-way: every status after PREEMPTED is terminal, mapped to a FinishReason via _FINISHED_REASON_MAP (request.py:376). Internally the scheduler tracks Request objects; the OutputProcessor tracks a mirror RequestState (engine/output_processor.py:129) — that pairing lets the wrapper stay detached from the scheduler's bookkeeping.

Wire formats (IPC)

  • EngineCoreRequest — msgspec.Struct(array_like=True, omit_defaults=True, gc=False) with request_id, prompt_token_ids, mm_features, sampling_params/pooling_params, arrival_time, data_parallel_rank, client_index, current_wave (__init__.py:88). The gc=False/array_like flags are chosen for zero-copy over ZMQ / shared memory.
  • EngineCoreOutput, EngineCoreOutputs — per-request token payload side, same struct policy.
  • Utility/control messages (UtilityOutput, EngineCoreRequestType, EngineCoreReadyResponse) ride the same socket stream, multiplexed by an enum tag.

Async boundaries — where concurrency actually lives

There are three concurrent surfaces a researcher must distinguish (a common source of bugs when porting to a new device):

  1. EngineCoreProc OS process — runs the Python busy loop (coroutine-free, thread-free in the hot path). Communication is +1 ZMQ ROUTER socket per client direction, + shared-memory message queues for model I/O. This is where the device lives.
  2. asyncio layer — AsyncLLM._run_output_handler + per-request generate() taskgraph. Pure bytes-in / bytes-out; never touches the device. It is allowed to block only in small slices (VLLM_V1_OUTPUT_PROC_CHUNK_SIZE) ; between slices it await asyncio.sleep(0).
  3. TP worker processes / the device — driven by MultiprocExecutor.collective_rpc. The GPU executes the forward; the engine loop is deliberately not inside that process.

The cleanest line to read: the engine loop is synchronous; concurrency is at the edges (ZMQ processes + asyncio). Hardware backends only ever see (1), via the Executor interface.

3. v0 vs v1 divergence

Concern v0 (now dropped) v1 (current)
Where the loop lived AsyncLLMEngine owned tokenization + scheduling + HTTP + metrics in one huge class. EngineCore is the inner loop; LLMEngine/AsyncLLM are thin wrappers; OutputProcessor handles detokenization.
Scheduling model Iteration-centric scheduler; speculative decode, chunked prefill bolted onto a sequential loop. Same SchedulerInterface, but the loop is continuous-decode-friendly; chunked prefill / streaming
are first-class.
Serving/RL coupling LLM subclassed AsyncLLMEngine; offline, HTTP, and research entry points shared a tangled class hierarchy. EngineCore is decoupled from any serving infra; AsyncLLM exposes it cleanly for orchestration/RL use.
Multi-process story Worker processes combined model execution and bootstrap; hard to swap launchers. MultiprocExecutor = "run one TP group"; EngineCoreProc = event loop in its own process; Ray-actor
(EngineCoreActorMixin) is a thin veneer.
State SequenceGroup/Sequence model living partly in core partly in worker. Request (core) + RequestState (wrapper) split so the core is agnostic to detokenization.
Public APIs LLM, AsyncLLMEngine divergent. Single v1 core + thin sync/async wrappers; legacy names are aliases.

Architectural motivations distilled

  1. Continuous-decode-friendly loop. EngineCore.step was factored out so the scheduler's work can overlap with model execution, enabling chunked prefill and speculative decode as first-class paths rather than branches bolted onto a sequential add_next_token loop. The StepFuture future allows schedule/forward overlap.
  2. Decouple inference from serving/RL. The v0 LLM simultaneously owned tokenization, scheduling, and HTTP — a dependency tangle that blocked RL researchers (who wanted the engine but not one specific HTTP shim). The v1 EngineCoreClient seam is the fix: swap InprocClient ↔ AsyncMPClient ↔ Ray actor without changing the inner loop.
  3. Single-process-oriented engine loop. v0 scattered work across worker processes; v1 pushes all the bookkeeping into the EngineCoreProc process and lets the executor only "do one TP group". This makes device adapters (e.g. Ascend) plug in once via the Executor interface.

The Executor interface (the contract the engine calls)

From abstract.py:146+:

  • get_class(vllm_config) — choose the impl:
  • explicit distributed_executor_backend → RayExecutorV2 (≥2 nodes),
  • RayDistributedExecutor (single node but >1 device),
  • MultiprocExecutor (multi-process, default),
  • UniProcExecutor (1 device),
  • ExecutorWithExternalLauncher (external launcher / Ascend-style).
  • execute_model(scheduler_output) → ModelRunnerOutput | None | Future (chain with sample_tokens) — the only call that actually touches hardware.
  • initialize_from_config(kv_cache_configs), determine_available_memory(), get_kv_cache_specs(), collective_rpc, execute_dummy_batch, profile, add_lora/remove_lora/pin_lora/list_loras, save_sharded_state, sleep/wake_up (for C10d sleep-level power mgmt).
  • supports_async_scheduling() — whether the executor can return a future and overlap scheduling.

Multiproc path (default)

MultiprocExecutor._init_executor (multiproc_executor.py:110):

  • reads tp_size × pp_size × pcp_size, stores self.workers: list[WorkerProc].
  • Allocates N WorkerProc processes (WorkerProc.make_worker_process, multiproc_executor.py:597+); each calls WorkerProc.worker_main, builds the device ModelRunner, and enters its own worker_busy_loop.
  • For I/O it uses shared-memory MessageQueue (self.rpc_broadcast_mq to scatter a batch of tasks; N response_mqs to gather results) — not sockets inside a single TP group.
  • execute_model → collective_rpc("execute_model", scheduler_output, unique_reply_rank=self.output_rank) (multiproc_executor.py:310): a fan-out to all workers, fan-in of a single reply. Index self.output_rank = the only rank that replies to minimize traffic (_get_output_rank, line 498).
  • Heartbeat via death_writers / monitor_death_pipe; on failure, register_failure_callback fires back into the engine loop.

Executor.execute_model is therefore decoupled from worker creation: the executor is only a scatter/gather primitive around a fixed pool of workers that already hold the device and model.

UniProc path (single device / external launcher)

UniProcExecutor (uniproc_executor.py:45) keeps a single WorkerWrapperBase (rpc_rank=0) and synchronously calls run_method(self.driver_worker, "execute_model", ...). ExecutorWithExternalLauncher (layer at line 150) overrides _init_executor to skip load_model (the external launcher owns it) and returns a determine_available_memory() that hands control to the launcher — this is exactly the shape an Ascend-style external-harness adapter would take.

Ray path

RayDistributedExecutor (ray_executor.py:64) runs model workers as Ray actors, builds a compiled Ray DAG (_compiled_ray_dag, line 527) for execute_model, and reinitialize_distributed (line 380) re-assigns a communication group when the set of actors changes. sample_tokens is a sibling collective_rpc("sample_tokens") so the TP/PP group proposal (line 409) lets the DAG carry tokens across without returning through the loop between steps.

So what exactly does the Executor isolate?

It isolates "how we run one TP group" from "how we orchestrate":

                SchedulerOutput          ModelRunnerOutput
  EngineCore ─────────────────────── Executor ──────────────────> device workers / ray
    (orch)       scheduler_output        (dispatch)                (device)
  • "how we orchestrate" = scheduler, KV-cache planning, multimodal, abort/token bookkeeping, output chunking, stop-string detection — all in EngineCore / EngineCoreClient / OutputProcessor.
  • "how we run one TP group" = allocation of N workers, the RPC primitive, failure handling, model loading — all inside Executor + the driver worker.

This split means the EngineCore never imports a torch device, a WorkerProc, or a Ray runtime.

5. Design insights for an inference architect

These are the patterns a hardware vendor or systems team must internalize when adapting vLLM (e.g. the Ascend port in /vllm-ascend) — not code trivia, but the load-bearing ideas.

(a) The engine loop is aggressively single-threaded; the author expects all device time to be

hidden behind the executor

Read EngineCore.step (core.py:576) again: future = execute_model(...) is the only non-trivial call inside the loop, and update_from_output is pure bookkeeping. vLLM assumes the scheduler can run every model step — including with zero requests, to hide stalls — and that the only non-deterministic latency is the device. Implication for a port: the device path must be throughput-bound and not futex-loop the engine thread; any device-side collective that doesn't return a future will directly stall the entire serving pipeline. Use supports_async_scheduling() to ask the engine to run schedule-and-execute concurrently (step_with_batch_queue, core.py:617) — the moment your device admits a future return, you get pipeline overlap for free.

(b) EngineCoreClient is the seam; pick it deliberately, and implement the right one

make_client (core_client.py:83) is chosen by two booleans — multiprocess_mode and asyncio_mode. The four combinations are:

multiprocess asyncio impl use
F F InprocClient offline LLM (tests / notebooks)
T F SyncMPClient the sync LLMEngine, useful when you control the loop
T T AsyncMPClient / DP variants production serving
F T unsupported –

Implication: a research/RL consumer wants sync_mp so it can call step() as a coroutine-free RPC loop. A production server wants async_mp. The Ascend port should not fork EngineCore; it should pick the right client class (or add a client_addresses-driven client whose sockets are injected by the launcher).

(c) Two-process separation keeps the device hot; treat it as invariant

EngineCoreProc (core.py:995) ≠ the device-worker process. The engine process runs Python + ZMQ; device workers run in WorkerProc (multiproc_executor.py:557+) and own the NPU/GPU. They are born before the engine accepts requests, talk over shared-memory MessageQueue, and never re-enter the engine loop. Implication: a port that wants the existing CUDA-free benefits must keep the same process topology. Changing the relationship between "engine proc" and "device proc" (e.g. merging them) breaks InputProcessor/test tooling that assumes the two can live apart.

(d) The scheduler is a pure decision function, not a state-of-the-world

SchedulerInterface (interface.py:37) promises: given vague throttle_prefills, return a SchedulerOutput; given a ModelRunnerOutput, return {client_idx: EngineCoreOutputs}. It owns the KV-cache bookkeeping, but it has no concept of the transport or the executor. The engine calls the scheduler before and after the executor call, and nothing in between. Implication for a port: you never need to touch the scheduler to support a new device. All that's required is a working Executor that understands SchedulerOutput and emits ModelRunnerOutput in the expected shape.

(e) Output processing is the one batch-wide pass — and it's deliberately detached

OutputProcessor.process_outputs (output_processor.py:584) is explicitly documented as "the only function that should loop over EngineCoreOutputs … if you need to touch every element of the batch, do it from within this loop." It lives in the wrapper, not in the engine core, precisely so a serving shim can stream tokens via asyncio queues while the heavy per-request detokenization and stats stay off the engine thread. Implication: device-specific post-processing (e.g. logits soft-cap, sampling adjustments) belongs either in the model runner (returns a built ModelRunnerOutput) or as a hook inside this batch-loop, never inside EngineCore.step.

(f) The futures-queue enables pipeline parallelism at the engine core level

If vllm_config.max_concurrent_batches > 1, EngineCore.step_fn is swapped to step_with_batch_queue (core.py:617) and the deque holds (Future, SchedulerOutput, grammar_future) triples. This is how vLLM removes pipeline bubbles: one batch executes while the next is scheduling. Implication for a port: when your hardware supports async scheduling, don't pretend to be synchronous. Return a true Future and let the queue overlap what would otherwise be idle stages.

(g) Control messages ride the same bus as data — the UtilityOutput pattern

_handle_client_request (core.py:1471) multiplexes ADD | ABORT | UTILITY | WAKEUP | EXECUTOR_FAILED. The UTILITY branch lets the client call an arbitrary EngineCore method through a lambda and receive the result bounded into a single EngineCoreOutputs. This is how stats, sharded-state save, distributed re-init, and pause/resume travel without dedicated sockets. Implication: a port that needs to add a new engine-level control knob (e.g. Ascend-specific warm-up, profiling toggle, or SWA weight sync) would often add a UTILITY method rather than a new wire type.
