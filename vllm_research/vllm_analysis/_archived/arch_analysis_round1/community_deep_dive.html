1. 性能优化手册——从 OS 到 kernel 的五层

来源：vllm-ascend/docs/source/developer_guide/performance_and_debug/optimization_and_tuning.md 完整手册 + performance_benchmark.md + service_profiling_guide.md + msprobe_guide.md。

层旋钮对推理的物理影响
1. OS 层CPU governor→performance; swap→0; 自动 NUMA 平衡关闭; kernel.sched_migration_cost_ns=50000-100000（防线程跳核）把 host-side jitter 从 ms 压到 μs；H2D 拷贝不被 swap 打断
2. allocatorjemalloc / TCMalloc（默认），多线程无锁分配峰值吞吐下碎片率降低，避免 multi-thread alloc 全局锁竞争
3. torch_npu 环境变量torch_npu 内存 / 调度优化（由 env 控制）华为 CANN 自身 toggle，厂商默认开启最佳
4. HCCL 通信AIV 模式（AI vector core 直接调度 RoCE，绕过 AI CPU）；HCCL_INTRA_ROCE_ENABLE（RDMA 替代 SDMA 8P mesh）; HCCL_RDMA_TC/SL（流量分级）; HCCL_BUFFSIZE（NPU 间共享缓存）这是 NPU 特有的一层：AI CPU 调度延迟高，绕过它可让集合通信延迟从 ms→sub-ms
5. kernel 融合70+ fusion pass 在 AscendCompiler / GraphFusionPassManager减少 kernel launch 数与中间张量 HBM 读写：AllReduce+RMSNorm、QK-Norm+Rope、AllGather chunk noop、norm-quant 等参见图编译
"性能瓶颈不在 model forward 里, 50% 在 host 与 NCCL/HCCL 的交界" —— 这是 NPU 推理社区被反复验证的经验。上面 §4 AIV 模式这条：把 comm 调度从 AI CPU（慢）移到 AI vector core（快），是 vLLM-Ascend 独立做出、尚未回馈 vLLM upstream 的重要优化。与此对应：CPU binding（§5）也是把 host-side 干扰减到最低。这两条说明 Ascend 适配的重心不只是"kernel 替换"，而是host ↔ device 边界重新平衡。

benchmark & profiling 工程实践

  • micro-benchmark：vllm_ascend/tests/ut（单测，可在 CPU 跑，mock device）+ tests/e2e（E2E 精度+吞吐回归）+ nightly CI（schedule_nightly_test_a2/a3.yaml）
  • profiling chain：msprobe（Ascend NPU 侧 trace 工具，NPU kernel timeline）→ service_profiling_guide.md 给出从 API 到 kernel 的全链路与 D2H 同步热点定位 → `service benchmark.md` 给出稳态吞吐/首 token 延迟的标准测法（建议至少 2-3× 设备数的 8K input / 1 output warm-up 热身——这是 HCCL one-sided comm mesh 建立的一次性开销）

2. ACL Graph 设计——为什么 Ascend 不能"照搬" CUDA Graph

来源：developer_guide/Design_Documents/ACL_Graph.md（官方设计文档，直接引用设计者语气）。

两条根本差异

差异 A：ACL图不能静态回放——attention 的运行时元数据必须靠 backend 主动 "update"。 官方原文："Full graph replay on Ascend has an extra problem that upstream generic documentation does not cover in detail: some attention operators need runtime metadata updates even when the overall graph is static."

机制：

  1. capture 阶段 attention backend 捕获 per-graph task handle / event / workspace + 待刷新的 tensor 弱引用
  2. replay 前 update_full_graph_params() 调 backend 的 update_graph_params()
  3. 后端在 update stream 上用 torch.npu.graph_task_update_begin()/end() 把底层 attn 算子参数刷新一遍
  4. torch.npu.ExternalEvent 保证 host-side update stream 与 replay stream 的顺序

📐 在 vllm_ascend/attention/*_v1.py、mla_v1.py、context_parallel/attention_py.py、mla_cp.py 里实现。关键设计事实：没有 update_graph_params hook 的 attn backend 无法做 full-graph replay——这是 Ascend 比 CUDA 更严的静态图条件，也 przykład 说明"硬件抽象不只 Platform 层, 还渗透到 attn kernel 内部"。

差异 B：ACL Capture 仍可能被运行时资源拒绝

官方原文："Unlike CUDA Graph on CUDA devices, ACL graph capture on Ascend can still fail when the selected graph sizes consume more runtime resources than the current backend can supply. Piecewise mode is the most sensitive case because it captures many subgraphs and the total capture cost scales with model depth and configured size coverage."

→ 新版(old update_aclgraph_sizes() 裁剪法已删除)改为运行时捕获失败 + 用户可读指引回落 eager，主调杆仍是 cudagraph_capture_sizes / max_cudagraph_capture_size。关键推论：ACL graph 的实际"大小上界"是实测出来的，不能按 CUDA 经验估。

模式与 strict normalization

  • encoder-decoder 必走 PIECEWISE
  • use_inductor 在 ACL graph 路径 强制禁用
  • ASCEND_LAUNCH_BLOCKING=1 与 ACL graph 不兼容
  • Xlite graph mode 可关闭 full / 回落 FULL_DECODE_ONLY
对适配工程师：ACL Graph 三原则
① NPUPlatform.get_static_graph_wrapper_cls() 返回 ACLGraphWrapper(这是唯一挂点,不改 dispatch policy)
② full-graph 需要 attn backend 提供 update_graph_params() hook
③ 捕获容量是runtime 决定,配置 cudagraph_capture_sizes 是主调杆,失败回落 eager

3. 量化生态——工程权衡

来源：quantization/ 在 90 天内 ~61 commits；官方 feature_guide/quantization.md + developer_guide/Design_Documents/quantization.md。

方案dtype 通路feature matrix 互斥定位
W4A4int4 weight, int4 activation⟷ EPLB/TP: ❌极限吞吐 EP off 场景
W4A8int4 weight, fp8 activation兼容 EPLB/TPMoE 吞吐主力
W8A8fp8 / int8全兼容LLM 通用

📐 这套多样性源自 CANN / torch_npu 链路"硬件原生 kernel 存在差异"：Ascend A2 的 cube core 偏好 fp16/bf16，A3 强化了 int8 路径，310P 是更低功耗 line (int4 wider)。一条简明规则：MoE 走 W4A8，dense 走 W8A8，极限吞吐(EP off)走 W4A4——这不是偏好,是硬件 kernel 覆盖度的现实。Ascend目前共 10+ 量化方案,见 `quantization/methods/`。W4A4 ⟷ EPLB 不兼容这条对应 feature matrix,原因：EPLB 在 expert 迁移时要做 int8/int4 格式转换,CANN kernel 在当前版本未支持。

4. 先进特性——KV Pool / PD 分离 / Context Parallel / 动态 PP

来源：feature_guide/epd_disaggregation.md·kv_pool.md·layerwise_kv_pool.md·Design_Documents/disaggregated_prefill.md·context_parallel.md·dynamic_chunk_pipeline_parallel.md。

PD 分离(P/D Disaggregation)——把"两段相反计算"拆到两类硬件

Connector 拓扑方向适用
MooncakeConnector (pull)D 节点主动从 P 拉 KVpull 控制在高带宽、大 KV
MooncakeLayerwiseConnector (push)P 逐层推 KV 到 D与 PP 组合; 低 push 开销; 配合 PCP

两类都走 Mooncake（Moonshot AI 的 RDMA 引擎，升腾上叫 HIXL ascend_direct），DC 之间走 RDMA 而不是 NCCL/HCCL——这是 vLLM-Ascend 独立演化出的路径,vLLM upstream 的 NIXL KV lease 是类似但实现独立。

KV Cache Pool (Ascend Store)——跨 instance 前缀共享

这是 vLLM-Ascend 对 vLLM upstream 的重大扩展（开源上游几乎没有等价物）：把多个 instance 计算出的 KV 投进一个分布式 key-value store，让后来的请求直接 GET 共享前缀。

后端机制
MooncakeMooncake Distributed Store + SSD offload（NVMe eviction → 1GB 对齐 fabric mem，bucket eviction policy=lru/fifo/none）
MemcacheMemFabric-backed；支持 device_rdma（A2 推荐）和 device_sdma（A3 HCCS 推荐）
YuanrongopenEuler 官方 etcd+HugeTLB huge page 共享内存；支持 Remote H2D（跨机 host→device）
KV Pool 调优 checklist（官方 FAQ 的精华）
① kv_load_failure_policy = recompute（recom 兜底,比 fail 稳,但 hybrid attn 模型如 DSV4 暂不支持）
② lease TTL：--default_kv_lease_ttl 必须 > ASCEND_CONNECT_TIMEOUT + ASCEND_TRANSFER_TIMEOUT
③ fabric mem 对齐 A3: global_segment_size 与 MOONCAKE_OFFLOAD_LOCAL_BUFFER_SIZE_BYTES 必须 1GB 整数倍
④ A3 + ASCEND_ENABLE_USE_FABRIC_MEM=1：开启统一直传(比 ASCEND_BUFFER_POOL 4:8 更稳)
⑤ PYTHONHASHSEED 所有节点一致（保证跨 worker 哈希命中）
⑥ warm-up：2-3× 设备数的 8K input + 1 output（等 HCCL mesh 建链）

Context Parallel(PCP / DCP)——沿 sequence 切分 KV

• PCP (Prefill-CP)：prefill 阶段沿 sequence 切分到多卡，KV 按 cp_kv_cache_interleave_size（默认 =1，token interleave）分布；需要独立通信域
• DCP (Decode-CP)：复用 TP 的通信域，在 decode/chunked prefill 阶段沿 sequence 切分 KV，消除 TP 域内的 KV 冗余副本（从而释放 HBM 给更多并发请求）
• 总 cp_size = pcp_size × dcp_size, cp_rank = pcp_rank × dcp_size + dcp_rank；block_size 必须整除 cp_kv_cache_interleave_size

DCP 这条的精妙之处：decode 阶段是 bandwidth-bound(I2),减少 KV 冗余 = 直接减少 HBM 需求 = 同样的 HBM 能并发更多请求 = 吞吐线性提升。PCP 这条更偏降低 TTFT(TFT)——这是 vLLM 把"decode vs prefill 是两种计算范式"这条 roofline 洞察投射到 parallel 策略上的直接体现。

Dynamic Chunked Pipeline Parallel (CPP)——profiling-driven

官方定义："profiling-based dynamic chunking strategy that optimizes prefill performance for long sequences in Pipeline Parallelism scenarios"。只在分离部署的 P 节点开，D 节点不用。关键：

  • PP 在多节点长序列下 pipeline bubble 严重；CPP 按 profiling latency 动态算 chunk——短序列 PP 不退化，长序列 PP 砍 bubble
  • 支持 eager + graph + prefix cache + chunked prefill 全兼容
  • 官方建议不在 PP 的 P 节点开 async-scheduling（实测 prefill 阶段 async 反而退化,好处小）
  • 推荐 MooncakeConnectorV1 作为 kv_connector（对 PP 支持最全）

→ backend 源码 core/scheduler_profiling_chunk.py·core/recompute_scheduler.py·patch_profiling_chunk.py。

5. 调度优化——roofline 视角

recompute vs swap：NPU 选"扔掉再算"

策略N 状态 HBM 不足时适用硬件
swap (CPU offload)把 KV 腾到 CPU，PCIe 搬进搬出GPU(bandwidth 高 / 搬得动)
recompute丢弃 KV，下次 token 算到时从 prompt 重算NPU(compute/bandwidth 高, 算得动搬不动)

这正是 I2 roofline 的 NPU 特有解。在 vLLM upstream（GPU后端）swap 是默认,Ascend 默认是 recompute。这个区别被写进 kv_load_failure_policy：upstream 是 fail,Ascend 推荐改为 recompute。

ShortRequestFirst(SRF)——反 head-of-line blocking

只在 PD 分离的 P 节点 + recompute 调度器 下启用。核心参数：

  • threshold (默认 256)：prompt ≤ threshold 算 short
  • long_max_wait_ms (默认 0)：长请求最长等 short 多久后被提升

官方 tuning 建议： threshold 取 P70-P85（prompt 分布两峰之间的 valley）；long_max_wait_ms 取 strict-short-first 下 long 请求 P90-P95 等待时间，但 shouldn't 从 short TTFT 推。这不是"吞吐优化",是公平性+短请求 SLO 的权衡。

profiling-chunk——roofline 的自动信号

用实际 profiling latency 动态决定 chunk size；配合 PP+async-scheduling 在 P 节点实测有效,见 scheduler_profiling_chunk.py。这条能看出：Ascend 在测"调度参数是否应该自动 profiling",而不是人工调。

CPU binding——host-side 的 roofline

vllm-ascend v0.18.0rc1 之后 ARM 服务器默认开启 CPU binding。收益：

  • 跨 NUMA 流量下降——worker 绑到本地 NUMA
  • 线程抢占减少——关键 runtime thread 固定核范围
  • 尾延迟稳定性——多 worker 隔离
  • Ascend 950 特殊：用 npu-smi info -t topo 选亲和 NUMA,每 worker 绑一 CPU cluster（lscpu Thread(s) per core: 8 CPU·core=1 / 16 CPU·core=2）

CPU binding 的存在证明 vLLM inference 的性能瓶颈已经从"GPU kernel"迁到"host-side 调度 + HCCL + D2H 拷贝"——GPU 侧 fusion 到极致后,host jitter 成了最后 10% P99 的决定因素。

Netloader 插件化——weight load 阶段的 optimization

load-format=netloader 走 HCCL send/recv 在已加载实例与新实例间直接传权重，避免从 storage 重新加载。这让 weight load 时间从分钟→秒级,且：

  • 减轻中心存储/网络负载
  • 故障恢复时新实例快速接新流量
  • 需要在每 worker 进程绑一个 listening port；HCCL buffer 需预留约 200 MB HBM（--gpu-memory-utilization 调低一格）
  • 推荐 VLLM_SLEEP_WHEN_IDLE=1 避免空闲时连接不稳

7. 第一性原理延伸——从社区活动读出正在固化的设计

判断框架：三层"物理强制"等级

等级含义vLLM 实例Ascend 实例
L1 implement-to-exist不做则系统不能工作paged KV / single-step scheduler / Platform ifaceHCCL 换 NCCL / Platform shim / ACL Graph
L2 implement-to-compete不做则竞争劣势prefix cache / chunked prefill / CPython 95/5recompute / KV Pool 后端 / SRF / quant 多样性
L3 implement-to-win不做则同级,做了则拉开PD disagg / spec decode / speculatorCPU binding / profiling-chunk / netloader / Flashcomm / AscendStore 后端三线

📐 按照这个框架：

  • PD 分离 正在从 L3 下沉到 L2（hyperscaler 标配）；KV Pool(Ascend Store) 正在从 L3 下沉到 L2（前缀共享成本从 "nice" 变 "must"）
  • Context Parallel 正在从 L2 下沉到 L1-L2 之间——长上下文(>1M token)没有 CP 就根本跑不起来
  • CPU binding 从 L3 → L2——host-side jitter 在极致 kernel 优化后已是主瓶颈
  • recompute-from-swap 这条会永远留在 L2，因为它在 NPU 和 GPU 上有完全相反的最优选择——不可能固化成 L1 通用

最关键的进阶物理（只在社区活动中显现,不在源码里直接看到）

定律 1："comp/bandwidth ratio 决定 swap vs recompute"—— NPU 计算力强,带宽紧张 → recompute;GPU 相反 → swap。这条在 spine 报告里有过表述,但 Ascend 把 kv_load_failure_policy=recompute 当默认值,以及 KV Pool Memcache 在 external backend 里也接受 recompute 兜底,让这条物理变成工程默认值,不再需要人工配置。

定律 2: "PD 分离是把 I2 roofline 的 'decode vs prefill 是两种计算' 从节点内推到节点间"。节点内用 chunked prefill 分时,节点间用 PD 分离空间分。两者都是同一个物理洞见的两种实现方式。当 TTFT SLO 与 TPOT SLO 冲突不可调和时,PD 分离从"优化选项"变成"必选项"。这就是为什么社区在 PD 分离上出现 Mooncake/Layerwise/ascend_store 三套并行后端——都在解同一个问题,只是网络层(拉/推/共享内存)不同。

定律 3: "KV-cache 从 cache(缓存)进化成 memory tier(内存层级)"。KV Pool 把 KV 从"单一 HBM 内的分页"扩展为"HBM → extended pool(Mooncake/Memcache/Yuanrong) → NVMe SSD offload"三级——这跟经典 OS memory hierarchy (cache → DRAM → swapfile) 同构。定律 (I1 transformer memory law) 再次发威：只要 KV 永不变且按位读,它就是内容寻址的、可 tiering 的——这是 vLLM-Ascend 对推理系统 memory model 的最大单项贡献。

定律 4: "host-side jitter 是极致 kernel 优化后的最后 10% P99"。这条只有在 CPU binding、msprobe、Netloader、scheduler_profiling_chunk 同时流行时才看得清楚。它是"kernel 侧已尽善尽美"的反面——真正的战场已经转移到"host 如何稳定地给 GPU/NPU 喂指令"。这条在 GPU 社区也被验证（NVIDIA 的 MIG / DPDK 都是同一主题）。

定律 5: "Python 95/5 的两端同时被压缩"。 kernel 侧 5% 已被各种 fusion + ACL graph 吞掉了大部分 launch overhead——但还不能 100%（attention 每步必须动 metadata）；调度侧 95% 已被 SRF + profiling-chunk + CPU binding 压低了 host jitter。还剩的关键余地：当 Ascend 把 host jitter 再压一级时,最终会逼迫 vLLM 把调度器从 Python 迁到 C++/Rust——回到 spine §5 "Python 前端是工程选择,不是物理必然"。这不会马上发生,但方向已明。
