0. Architectural landscape at a glance

Two independent planes interlock through a tight feedback contract:

Plane Packages Responsibility
Scheduler / KV-manager vllm/v1/core/sched/*, vllm/v1/core/{block_pool,kv_cache_manager,kv_cache_coordinator,single_type_kv_cache_manager}.py Policy + virtual block allocation (which request owns which KVCacheBlock)
Worker / Block-table vllm/v1/worker/gpu/{block_table,model_runner,input_batch}.py, vllm/v1/worker/gpu_worker.py Physical GPU tensor pool; turns block-ids into attention kernel operands (block_tables, slot_mappings)
Hardware-agnostic contracts vllm/v1/kv_cache_interface.py, vllm/v1/kv_cache_spec_registry.py How a device advertises a KV-cache format and plugs a new manager in

The central insight: the scheduler never touches GPU tensors. It allocates KVCacheBlock metadata objects; the worker maps those block_ids into the actual attention kernel input. This is the seam a new device substrate plugs into.

1. KV-cache data model: block → device memory

1.1 The KVCacheBlock object (the unit of paging)

vllm/v1/core/kv_cache_utils.py:117

@dataclass(slots=True)
class KVCacheBlock:
    """KV-cache metadata."""
    block_id: int            # 0 .. num_gpu_blocks-1
    ref_cnt: int = 0         # live references (prefix-cache sharing)
    _block_hash: BlockHashWithGroupId | None = None   # set only when full & cached
    _block_hash_num_tokens: int | None = None
    prev_free_block / next_free_block: KVCacheBlock   # doubly-linked free list
    is_null: bool = False    # block_id 0 placeholder, never cached

Why this matters: a KVCacheBlock is metadata-only. The actual block_size * num_kv_heads * head_size * 2 * dtype bytes live in the worker-side kv_caches: list[torch.Tensor] (worker/gpu/model_runner.py:494, allocated by init_kv_cache). The block_id is an index into that flat pool. This indirection is what makes prefix-cache dedup and swap possible.

Key point: allocation is O(1) off a free linked list. 2. Scheduler design: one loop, no phases

vllm/v1/core/sched/scheduler.py:70 class Scheduler(SchedulerInterface)

2.1 The scheduling loop (schedule, scheduler.py:418)

The design's headline — verbatim from the source — sets the philosophy (:420):

"There's no 'decoding phase' nor 'prefill phase' in the scheduler. Each request just has num_computed_tokens and num_tokens_with_spec … At each step, the scheduler tries to assign tokens to the requests so that each request's num_computed_tokens can catch up its num_tokens_with_spec. This is general enough to cover chunked prefills, prefix caching, speculative decoding, and 'jump decoding.'"

So continuous batching is realized at the token level: a request in the running set whose num_computed_tokens < num_tokens still needs compute is simply given a chunk of tokens up to the remaining budget — even if it is mid-prefill. Prefill and decode requests are interleaved freely in one list iteration.

Loop structure (schedule, scheduler.py:418-1008):

token_budget = max_num_scheduled_tokens          # per-step cap (config)
self.kv_cache_manager.new_step_starts()

# PASS 1 — RUNNING requests (preempt surviving, min-fragmentation first)
for req in self.running while token_budget > 0:
    clamp num_new_tokens by long_prefill_token_threshold   # chunked prefill chunk size
    clamp by (max_model_len - computed)
    if mamba "align" mode -> _mamba_block_aligned_split(...)   # hybrid alignment
    loop:
        new_blocks = kv_cache_manager.allocate_slots(req, num_new_tokens, lookahead)
        if new_blocks: break
        else: preempt lowest-priority running request (_preempt_request -> status PREEMPTED,
              num_computed_tokens=0, back to waiting queue :1184)
    record req_to_new_blocks, num_scheduled_tokens, token_budget -= n
    apply DP-prefill throttling (throttle_prefills)

# PASS 2 — WAITING requests (only if no preemption this step)
while (waiting or skipped_waiting) and token_budget > 0:
    if running is full -> break
    req = _select_waiting_queue_for_scheduling().peek()
    connector kv-offload gating, max_loras gating
    new_computed_blocks, num_computed = get_computed_blocks(req)   # prefix cache hit
       + connector.get_num_new_matched_tokens(...)                  # external KV prefetch
    num_new_tokens = min(remaining, token_budget)  [respects long_prefill_token_threshold]
    allocate_slots(..., new_computed_blocks=..., num_external_computed=...,
                   full_sequence_must_fit=reserve_full_isl)
    if None: break    # stop admitting; do NOT preempt for new admits
    pop & advance

2.2 Chunked prefill & token budgeting

  • max_num_scheduled_tokens (:111) → either explicit config or max_num_batched_tokens; the hard ceiling on tokens executed in one step.
  • long_prefill_token_threshold → the chunk size for long prefills; short prefills are scheduled whole.
  • When enable_chunked_prefill=False, waiting admission requires the whole prompt to fit in one step (break on budget exceeded, scheduler.py:844 in upstream).
  • Per-step preemption budget is reused: when a request is preempted, token_budget += num_scheduled_tokens.pop(...) (:578), so the freed capacity is immediately recyclable within the same step.

2.4 Scheduler output contract (SchedulerOutput, sched/output.py:191)

The scheduler emits one SchedulerOutput per step and the worker consumes it as the only cross-process payload:

Field Meaning
scheduled_new_reqs: list[NewRequestData] first appearance; carries full prompt + block_ids + sampling params (output.py:33)

Bandwidth-savings trick: cached requests send only new_block_ids (the diff) rather than the full block table, "so that we don't need to re-send it every scheduling step" (output.py:90).

3. Worker ↔ Scheduler feedback contract

The worker controls the physical memory; the scheduler controls policy. They meet on two handshakes:

3.1 Upstream: budget / block availability (pressure signal)

  1. Startup capacity handshake — engine/core.py:309-311: python kv_cache_configs = get_kv_cache_configs(vllm_config, kv_cache_specs, available_gpu_memory) scheduler_kv_cache_config = generate_scheduler_kv_cache_config(kv_cache_configs) vllm_config.cache_config.num_gpu_blocks = scheduler_kv_cache_config.num_blocks Workers (gpu_worker.py:701 get_kv_cache_spec) advertise their per-layer KVCacheSpec; the engine profiles available memory (determine_available_memory, core.py:286); generate_scheduler_kv_cache_config resolves the final num_blocks; the scheduler then constructs its BlockPool(num_gpu_blocks=...). This is the available-blocks feedback loop → the scheduler's num_gpu_blocks is literally the worker's measured capacity.

  2. Per-step admission gate — KVCacheManager.allocate_slots (kv_cache_manager.py:283) returns None when required_blocks > free_blocks - reserved_blocks - watermark_blocks. The scheduler treats None as "cannot fit, stop admitting" (or "must preempt a running request"). So block-pressure flows upstream as an allocate-failure signal that the scheduler reacts to. The watermark_blocks headroom (:sched.py ctor, scheduler_config.watermark) prevents the last blocks from being consumed by decodes.

  3. Prefill throttling feedback — throttle_prefills flag from EngineCore._should_throttle_prefills (core.py:609) for DP prefill balancing; scheduler defers new prefills when a non-cadence DP step already has in-flight decode work (:447-451).

3.2 Scheduler → worker (the SchedulerOutput dispatch)

worker/gpu/model_runner.py:

  • new requests (update_requests → :804): block_tables.append_block_ids(req_index, req_new_data.block_ids, overwrite=True) — writes the entire row (new request has no existing row).
  • cached requests (:827-836): block_tables.append_block_ids(req_index, req_new_block_ids, overwrite=False) — appends only blocks newly allocated this step.
  • zeroing new blocks (:848-850): kv_block_zeroer.zero_block_ids(new_block_ids_to_zero) before first use.
  • copy-on-write (:853): copy_kv_cache_blocks_inplace(...) for partial prefix-cache hits before attention reads them.

3.3 Worker → attention kernel (prepare_attn, :1040)

gather_block_tables(idx_mapping)    -> tuple[Tensor]  [groups, padded_reqs, max_blocks]
compute_slot_mappings(idx_mapping, query_start_loc, positions, num_tokens_padded)
                                    -> Tensor         [groups, padded_tokens]

These two tensors — block table (per-request block-id matrix) and slot mapping (per-token flat KV-cache offset) — are the only dynamic inputs the paged-attention kernel needs per step. This is the true "paged" surface.

3.4 Zeroing, prefix cache eviction, CoW lifecycle

  1. Scheduler emits new_block_ids_to_zero pulled from take_new_block_ids (scheduler.py:1103).
  2. Worker zeros GPU bytes → forward.
  3. update_from_output advances num_computed_tokens (scheduler.py:1218), then cache_blocks writes block hashes for prefix caching.
  4. On preemption, _free_request_blocks → block_pool.free_blocks decrements ref_cnt; with caching, hashes are retained and _maybe_evict_cached_block only removes the hash map entry (block data overwritten on next zeroing). Partial hits → take_pending_cow_copies + kv_cache_block_copies → copy_kv_cache_blocks_inplace on the worker.

Design insight (KV-cache): The split between BlockPool/KVCacheManager (scheduler, metadata+policy) and BlockTables (worker, GPU kernels) is the single most important architectural seam in vLLM. Everything that-hardware specific lives in the worker; everything that is scheduling/policy lives in the core. Paging is virtual: a request's logical positions are flattened to (block_id, slot) only when the worker builds the attention operands. This is what enables prefix-cache dedup (by block hash), swap, rolling sliding-window, and compressed-latent MLA to coexist — they all just change how block_ids get allocated and how the slot mapping is computed. A new hardware substrate that can satisfy the KVCacheSpec/KVCacheManager contract (page size, quantization, optional CoW) and produce a contiguous addressable KV tensor can be plugged into the entire scheduler/policy stack without touching scheduling code. The storage_block_size ≠ block_size split for MLA is the canonical example: the scheduler pages logical block_size tokens while the worker actually stores storage_block_size // compress_ratio entries; the attention kernel on the worker knows how to decompose a block-row into the right number of kernel-blocks (blocks_per_kv_block, worker/block_table.py:57).

6.2 How it hooks in — scheduler_cls string config pivot

Ascend does not monkey-patch upstream. It subclasses vllm.v1.core.sched.scheduler.Scheduler (:43) and the engine-core builds it from the config string scheduler_config.scheduler_cls (RecomputeSchedulerConfig.scheduler_cls, :59). Async selection (AsyncRecomputeScheduler vs RecomputeScheduler) happens in RecomputeSchedulerConfig.initialize_from_config (:62-72). This is the same pluggability slot upstream SchedulerConfig exposes — Ascend just provides its concrete subclass.

6.3 Scheduling policy changes (relative to upstream)

A. Recompute preemption — the central Ascend addition (RecomputeScheduler.schedule :194):

The scheduler still passes over RUNNING then WAITING, but the preemption path diverges (recompute_scheduler.py:307-350 vs upstream :543-634):

  • If a KV-transfer connector is a non-producer consumer, a preempted request is optionally offloaded: connector.update_state_before_preempt(...) returns offloaded; the block data is preserved and the request goes back to waiting as a PreemptedRequestData rather than wiped. Otherwise (offloaded=False) it falls back to upstream-style free + recompute + RecomputeReqInfo(recomputed_req_id, output_token_ids, client_index) (:346-359), a "recompute" token the PD-proxy can re-stream.
  • Notice the careful preservation of recomputed_block_ids / num_computed_tokens in the offload path (:327-338) — the scheduler retains enough to reconstruct state after re-admission.
  • Upstream-style PRIORITY preemption also retained (:361-398), so the scheduler degrades to FCFS running.pop() when policy isn't priority.

B. Short-request-first waiting queue (ShortRequestFirstSchedulerMixin + short_request_first_scheduler.py:58):

Three-lane queue ShortRequestFirstRequestQueue(RequestQueue) — immediate / short / long — selected via SteppedWaitingQueue Protocol (:36) methods: begin_step(token_budget), select_waiting_queue_for_scheduling(), has_schedulable_requests(). Priority: immediate → short → long. Aging/jump avoids long starvation: a long request whose head waited ≥ long_max_wait_ms while shorts exist is promoted (_select_schedulable_queue, :209-221), with a _consecutive_aged_long_promotions degradation streak + _maybe_warn_degradation :252. So it's SJF-with-aging, fully wired through the existing RequestQueue ABC and the _select_waiting_queue_for_scheduling dispatch (recompute_scheduler.py:440).

C. Profiling-based chunk sizing (scheduler_profiling_chunk.py:46 ProfilingChunkScheduler(Scheduler)):

On startup, run_profiling_chunk_init (:108) runs the real model forward at several chunk sizes, fits a quadratic latency model via ProfilingChunkManager (profiling_chunk_predictor.py), and uses it to choose the runtime chunk per waiting request. Stats-driven rather than static long_prefill_token_threshold. Ascend's profiling_config.py carries the knobs (smooth_factor, min_chunk, max_fit_chunk, page_size).

6.4 What Ascend adds to the output / handshake

RecomputeSchedulerOutput(SchedulerOutput) (:93) extends the upstream output with:

  • preempted_reqs: list[PreemptedRequestData] — (req_id, block_ids, num_computed_tokens) for the PD-proxy that must be notified when a request is preempted with KVs stranded on-device/offloaded.
  • recomputed_reqs: list[RecomputeReqInfo] — (req_id, output_token_ids, client_index) so the proxy can replay already-sampled output after a free-and-recompute preemption (vs. the offload-preserve case).

The upstream SchedulerOutput is reused 1:1 (cached-diff block_ids, new_block_ids_to_zero, CoW copies) → Ascend's recompute is purely a policy + accounting overlay on the same worker contract.

6.5 Ascend device KV manager

vllm_ascend/core/single_type_kv_cache_manager.py + vllm_ascend/core/kv_cache_interface.py — Ascend's own SingleTypeKVCacheManager impl per platform (910/310P), registering through the same KVCacheSpecRegistry. It overrides the per-platform attention shape (MLA on Ascend) and zeroing policy. This shows the vendor-extension point is the coordinator/manager + spec, not the scheduler loop.

Design insight (Ascend overlay): "Extension rather than fork" here has three concrete structural faces: (i) replace the policy object (RecomputeScheduler(Scheduler) + scheduler_cls config pivot) instead of editing schedule()'s 600-line loop; (ii) add extra state to SchedulerOutput subclasses (RecomputeSchedulerOutput) so the worker/PD-proxy can observe schedule-side decisions the kernel doesn't need; (iii) register a per-platform KVCacheSpec/SingleTypeKVCacheManager through the registry instead of branching kv_cache_manager.py. The result is a scheduler that customizes preemption semantics (recompute vs. free, offload-aware) and waiting order (SJF-with-aging + profiling-chunk) while reusing 100% of the block-table/slot-mapping/paged-attention path. For an inference architect this is the template for any accelerator-specific scheduling: plug at the policy + spec surfaces, never in the kernel/budget loop. The one thing Ascend cannot change without touching core is the paged-attention kernel operand layout — by design, that's the stable hardware abstraction.

8. Key takeaways for an inference architect

  1. Paging is the universal hardware abstraction. Block-id metadata (scheduler) ↔ contiguous GPU KV tensor (worker) is the seam. Every hardware feature — swap, prefix cache, rolling sliding window, MLA compression, CP/TP sharding — is expressed as "a different KVCacheSpec + a different SingleTypeKVCacheManager + (sometimes) a custom attention backend", not a new scheduler. If you are porting vLLM to a new device, the budget equation is: implement KVCacheSpec (page bytes / quant / ragged), one SingleTypeKVCacheManager, and a paged-attention kernel that reads the two operand tensors (block_tables, slot_mappings). The 600-line schedule() loop is then inherited for free.

  2. The block-pressure feedback loop is implicit but load-bearing. num_gpu_blocks comes from a real memory profile at engine init (core.py:309); per-step admission is gated by allocate_slots returning None. Watermarks (scheduler_config.watermark) + full_sequence_must_fit (reserve_full_isl) are the knobs that prevent over-admission + preemption thrash. When adding a device-specific admission policy, hook at allocate_slots args / SchedulerOutput, never inside the worker kernel path.

  3. Continuous batching = one-token-budget loop, no phases. This is what lets vLLM mix chunked-prefill chunks with decodes in a single step. The policy knob surface (max_num_scheduled_tokens, long_prefill_token_threshold, SchedulingPolicy, waiting-lane selection) is deliberately narrow. The Ascend overlay shows exactly how much scheduling personality you can inject (recompute preemption, SJF-with-aging, profiling-chunk sizing, PD-proxy restore tokens) by subclassing Scheduler and publishing subclass-specific output, without touching the per-token loop or the paged-KV backing pool.

  4. Extension points vs. stable surfaces. Stable across the stack: SchedulerOutput builder fields, KVCacheBlocks handoff, BlockTables operand layout, KVCacheSpec.{page_size_bytes, max_num_blocks_per_req}. Vendor-extensible: Scheduler subclass + scheduler_cls, KVCacheSpec/SingleTypeKVCacheManager via the registry, and the PD-proxy observed fields on SchedulerOutput subclasses. Ascend's recompute is the reference design for "customize preemption + admission, keep everything".
