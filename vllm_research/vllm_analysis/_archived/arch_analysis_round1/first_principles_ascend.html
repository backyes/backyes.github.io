第一性原理 Ascend

📑 L1 引擎/编排 · L2 调度器/KV缓存 · L3 注意力/模型/平台/编译 · L4 昇腾适配+通用方法论 · 第一性原理 vLLM · 第一性原理 Ascend

First-Principles Anatomy of vLLM-Ascend

Audience: senior systems engineers, parallel-computing experts, AI algorithm researchers, inference architects. Method: derive every significant design in vLLM-Ascend from a physical or structural constraint — "what MUST change because the hardware is different, what CAN stay because the physics is the same, and what pattern resolves the tension." Each thesis states the first principle, derives the design, and quotes the pivot code that embodies it. Scope boundary: this file does NOT re-engineer the executor / engine-core / scheduler mechanism — that ground is covered exhaustively in agent1_engine_orchestration.md and agent2_scheduler_kvcend.md. It focuses on the Ascend overlay: the seams, substitutions, and invention on top of the upstream v1 core.

0. The framing question: "porting an OS to alien silicon"

Running vLLM (a CUDA-native serving stack) on Huawei Ascend NPU is architecturally equivalent to porting an operating system to a new ISA that does not share the host's system-call ABI. The migration is NOT a translation of algorithms; it is a mapping of contracts:

Contract vLLM assumes Reality on Ascend (CANN)
Device id + memory stats (nvml) NPU device id + torch.npu runtime stats
CUDA streams / events torch.npu.Stream / torch.npu.Event
cuDNN / cuBLAS kernels CANN operators (torch_npu.*, ACL kernels)
NCCL collective comms HCCL comms (hccl*)
CUDA graph capture ACL graph capture / npugraph_ex / torchair
torch.compile → CUDA graph backend FX graph → torch.compile → AscendCompiler → NPU backend

The first-principles theses below walk this contract layer, in order of physical necessity.

1. The CUDA-compat shim — "present the alien as native"

First principle. vLLM's codebase — Platform, Worker, ModelRunner, the engine core, the executor, layout and quantization validation — is written against a CUDA-shaped world. torch.cuda.*, CUDA device ordinal semantics, nvml-based memory stats, multi_processor_count, and a CUDA-graph dispatcher are referenced from dozens of call sites. Rewriting all of them to be device-agnostic is a multi-year redesign. The constraint: an adapter must make the NPU quack like a CUDA device at every touchpoint vLLM already has, without rewriting vLLM's core.

Derived design: Platform as a feature-complete shim. Ascend subclasses upstream Platform into NPUPlatform and overrides every hardware-touching method to map CUDA-style semantics onto torch.npu / CANN equivalents:

# vllm_ascend/platform.py:128-139
class NPUPlatform(Platform):
    _enum = PlatformEnum.OOT
    device_name: str = "npu"
    device_type: str = "npu"
    simple_compile_backend: str = "eager"  # Disable torch.compile()
    ray_device_key: str = "NPU"
    device_control_env_var: str = "ASCEND_RT_VISIBLE_DEVICES"
    dispatch_key: str = "PrivateUse1"

This is the dispatch key declaration: tell PyTorch's dispatcher that "NPU" tensors belong to the PrivateUse1 backend (the OOT-device slot PyTorch reserves), so the dispatcher routes NPU-operator kernels instead of CUDA ones. Everything downstream — torch.compile decomposition, FX graph capture — keys off dispatch_key.

Where the shim maps cleanly, the override is a one-liner:

# vllm_ascend/platform.py:263-265, 334-335, 843-850
@classmethod
def get_device_name(cls, device_id: int = 0) -> str:
    return torch.npu.get_device_name(device_id)

@classmethod
def set_device(cls, device: torch.device):
    torch.npu.set_device(device)

@classmethod
def get_current_memory_usage(cls, device=None) -> float:
    torch.npu.reset_peak_memory_stats(device)
    return torch.npu.max_memory_allocated(device)

@classmethod
def num_compute_units(cls, device_id: int = 0) -> int:
    # NPU analogue of CUDA multi_processor_count (SM count) → Cube Cores
    cube_core_num = getattr(props, "cube_core_num", None)   # :298-305

And the CUDA-graph-equivalent surface and the comms-backend surface are swapped by returning Ascend class strings (vLLM's plugin convention avoids importing heavy CANN modules at platform-bootstrap time):

# vllm_ascend/platform.py:860-850
@classmethod
def get_static_graph_wrapper_cls(cls) -> str:
    return "vllm_ascend.compilation.acl_graph.ACLGraphWrapper"

@classmethod
def get_compile_backend(self) -> str:
    return "vllm_ascend.compilation.compiler_interface.AscendCompiler"

@classmethod
def get_device_communicator_cls(cls) -> str:
    return "vllm_ascend.distributed.device_communicators.npu_communicator.NPUCommunicator"

@classmethod
def get_attn_backend_cls(cls, selected_backend, attn_selector_config, num_heads=None):
    backend_map = {
        (True, False, False): "...attention.mla_v1.AscendMLABackend",
        (False, False, False): "...attention.attention_v1.AscendAttentionBackend",
        (True, True, False): "...attention.sfa_v1.AscendSFABackend",
        (True, False, True): "...attention.dsa_v1.AscendDSABackend",

Where the shim breaks down — the impedance mismatch cost. Not everything is swappable by a return-value override:

  1. get_device_total_memory is a NotImplementedError (platform.py:275-281). Comment: "DO NOT IMPLEMENT: Implementing it calls get_device_name() in advance and initializes torch_npu too early. torch_npu allows global initialization only once; duplicate initialization causes errors." The memory-stats path is therefore owned by the worker's determine_available_memory() handshake, not the platform class.
  2. CUDA graph capture cannot be emulated — it must be reimplemented as ACL-graph capture (Thesis 3).
  3. The cudagraph-mode config carries the CUDA term in its name, so check_and_update_config (platform.py:411-644) rewrites it to NPU semantics: it injects Ascend's default capture shape (min(max_num_seqs * decode_query_len, 512), without the CUDA-specific * 2, _get_default_max_cudagraph_capture_size :207-241), injects MLA/DSA into the splitting_ops list (:572-577), routes worker_cls to NPUWorker / NPUWorker310 / XliteWorker (:608-618), and disables ASCEND_LAUNCH_BLOCKING under graph mode (`:594-606**).

Insight for the architect. The Platform subclass is clean IF — and only IF — the difference is data (device name, SM count, comms class string). The moment the difference is a mechanism (graph capture, graph-mode semantics, per-CUDAGraphMode branching logic), the shim has to re-derive upstream policies inside check_and_update_config. That method is 230+ lines because Ascend's graph mode is NOT CUDA's graph mode with a renamed flag; the two diverge in capture shape, splitting ops, and dependency checks. The friction surface is the CUDAGraphMode enum itself — its name and state machine are CUDA-origin.

2. Custom ops as CANN kernel substitution — "the 5% hot path, reimplemented"

First principle. Inference cost follows the same 95/5 rule on NPU as on GPU: a handful of ops (matmul / GEMM, attention score, MoE dispatch+combine, norm, rotary embedding, fused activation) dominate latency and memory. PyTorch's generic kernels are written for CUDA and run poorly or abort on the NPU. The constraint: the compute-intense 5% must be CANN-native kernels, not PyTorch/cuDNN fallbacks — but the remaining 95% (indexing, reshapes, sampling, bookkeeping) can stay generic.

Derived design 1: the backfill-the-torch.ops.vllm.* registration. Ascend loads CANN custom ops into vLLM's private torch.ops.vllm.* namespace via torch.library, with dispatch_key="PrivateUse1" so the same call resolves to an NPU kernel when the tensor is NPU-located:

# vllm_ascend/ops/linear.py:69-95
direct_register_custom_op(
    op_name="unquantized_gemm",
    op_func=unquantized_gemm,
    fake_impl=unquantized_gemm_fake,
    mutates_args=[],
    dispatch_key="PrivateUse1",
)

class AscendUnquantizedLinearMethod(UnquantizedLinearMethod):
    def apply(self, layer, x, bias=None):
        return torch.ops.vllm.unquantized_gemm(x, layer.weight, bias)   # :95

The rest of vLLM's QuantizedLinearMethod.apply calls are unchanged — they go through quant_method.apply → torch.ops.vllm.* and silently hit the CANN kernel. The "envy" property: from vLLM's perspective, the upstream quant-method API is intact; under the hood the GEMM is an NPU kernel.

Derived design 2: attention = thin metadata builder + native fused kernel. The thickest compute op, paged attention, is implemented by dropping a CANN fused-inference-attention kernel into the MLA/DSA/SFA backends:

# vllm_ascend/attention/mla_v1.py:855-856
torch.npu.graph_task_update_begin(update_stream, handle)
torch_npu.npu_fused_infer_attention_score_v2.out(...)   # CANN fused kernel

# vllm_ascend/attention/dsa_v1.py:1771
qr, qr_pertoken_scale = torch.ops._C_ascend.npu_rms_norm_dynamic_quant(...)

Derived design 3: the parallel-aware op hierarchy (linear_op.py). Ascend's tensor-parallel GEMM is not just F.linear + a comm; it is a fused comm-compute pipeline with Ascend-specific EP/OTP/MC2 network topologies. The CustomLinearOp class hierarchy (linear_op.py:83-450) models these as separate compute graphs:

CustomLinearOp
 ├─ CustomColumnParallelOp (all-gather + gemm)
 ├─ CustomRowParallelOp    (split + gemm + reduce-scatter)
 │    ├─ MatmulAllreduceRowParallelOp        ← fused matmul+allreduce
 │    ├─ Flashcomm2OProjRowParallelOp        ← fused all-gather + oproj + reduce-scatter
 │    └─ DSV4OProjRowParallelOp              ← project -> transpose -> quant-bmm
 └─ CustomReplicatedOp

MatmulAllreduceRowParallelOp is the key innovation: a TP matmul whose partial-sum reduction is fused into one kernel. This matters because NPU inter-device bandwidth/latency characteristics reward kernel fusion more aggressively than typical GPU TP layouts.

Derived design 4: register_custom_ops.py — comm primitives as ops. Beyond GEMM, even the collective-comm building blocks are registered as custom ops: _maybe_chunk_residual_impl, _maybe_all_gather_and_maybe_unpad_impl, _maybe_pad_and_reduce_impl, _matmul_and_reduce_impl (ops/register_custom_ops.py:22-165). This lets torch.compile / ACL-graph capture consume comm ops as first-class graph nodes instead of opaque dist.* calls.

Insight for the architect. The Ascend ops layer is not a swap of cuBLAS for a vendor BLAS. It is a semi-rewiring of model topology — MatmulAllreduceRowParallelOp, Flashcomm2OProjRowParallelOp, and NPU-specific attention kernels restructure the per-layer compute graph to match the NPU network fabric. The registration trick (direct_register_custom_op + PrivateUse1) is the only clean part; the topology rewrites are forced by network-fabric physics, not by vLLM's preference.

3. ACL graph vs CUDA graph — "same idea, different capture"

First principle. A decode step is a fixed-shape, highly-repetitive computation graph — ideal for graph capture that amortizes kernel-launch overhead. CUDA graphs capture a stream of GPU kernels. The NPU has no CUDA. The constraint: the motivation for graph capture is hardware-universal (amortize launch cost), but the capture mechanism is hardware-specific and non-portable. Ascend cannot reuse vLLM's CUDA-graph dispatcher; it must re-implement "capture → key → replay".

Derived design: ACLGraphWrapper — same protocol, NPU capture. The wrapper implements exactly vLLM's FULL/PIECEWISE/NONE dispatch surface, but its capture/Replay primitives are torch.npu.NPUGraph and NPU streams:

# vllm_ascend/compilation/acl_graph.py:165-202
aclgraph = torch.npu.NPUGraph()
with ExitStack() as stack:
    if self.aclgraph_options.gc_disable:
        stack.enter_context(patch("gc.collect", lambda: None))
    forward_context.capturing = True
    try:
        with torch.npu.graph(aclgraph, pool=self.graph_pool):
            output = self.runnable(*args, **kwargs)   # capture
    except RuntimeError as exc:
        if _is_old_hdk_capture_error(exc): ...
        if _is_stream_resource_capture_error(exc): ...
entry.aclgraph = aclgraph        # cache
...
# replay path
entry.aclgraph.replay()

The same BatchDescriptor → ACLGraphEntry cache and "capture if absent, replay if present" dispatch that vLLM's CUDAGraphWrapper uses is reproduced here (acl_graph.py:133-267). The v2-only ModelAclGraphManager(ModelCudaGraphManager) (worker/v2/aclgraph_utils.py:41) subclasses the upstream CUDA-graph manager, so model-runner code dispatches through the same manager interface regardless of device.

Derived design: AscendCompiler — torch.compile to NPU-FX. For the graph-compilation direction (as opposed to direct capture), Ascend provides a torch.compile backend that converts PyTorch FX graphs to NPU-executable functions via npugraph_ex (preferred) or torchair (legacy):

# vllm_ascend/compilation/compiler_interface.py:161-223
import npugraph_ex as nge
torch.npu.set_compile_mode(jit_compile=False)
config = nge.CompilerConfig()
backend = nge.get_npu_backend(compiler_config=config)
compiled_fn = backend(graph, example_inputs)   # torchair fallback on ImportError

AscendCompiler(CompilerInterface) (:226-303) exposes the compile/load/compute_hash contract vLLM's compilation pipeline expects, with a hash keyed on torch_npu_version + enable_npugraph_ex + enable_static_kernel so NPU-build caches are invalidated correctly — the upstream CUDA hash ignores these.

Insight for the architect. Graph capture/non-capture is the biggest semantic divergence: CUDA and ACL graphs expose different capture constraints (NPU needs ASCEND_LAUNCH_BLOCKING=0, needs GC disabled per-subgraph, needs offloader-stream sync before capture, acl_graph.py:167-219). The wrapper faithfully reinstates the vLLM graph API on top of these non-portable primitives — another shim, but one that fully redraws the capture runtime.

4. HCCL replacing NCCL + KV PD disaggregation — "the comms layer, swapped"

First principle. Distributed TP/EP/PP inference needs all-reduce, all-gather, all-to-all, p2p. NCCL is NVIDIA-only. The constraint: the collective interface is universal; the backend implementation is vendor-specific and non-portable. Ascend therefore must provide an HCCL backend that matches the same rank/group/dtype/stream contract vLLM's communicators were written against.

Derived design 1: PyHcclCommunicator — HCCL via ctypes, NCCL-shaped API. This is the low-level NCCL-shaped binding: it creates an HCCL comm, exchanges hcclUniqueId via TCP/gloo, then issues hcclAllReduce/hcclBroadcast against that comm:

# vllm_ascend/distributed/device_communicators/pyhccl.py:77-152
self.hccl = HCCLLibrary(library_path)
with torch.npu.device(device):
    self.comm = self.hccl.hcclCommInitRank(self.world_size, self.unique_id, self.rank)
    # warmup
    self.all_reduce(data); stream.synchronize()

def all_reduce(self, in_tensor, op=ReduceOp.SUM, stream=None):
    self.hccl.hcclAllReduce(
        buffer_type(in_tensor.data_ptr()), buffer_type(out_tensor.data_ptr()),
        in_tensor.numel(), hcclDataTypeEnum.from_torch(in_tensor.dtype),
        hcclRedOpTypeEnum.from_torch(op), self.comm,
        aclrtStream_t(stream.npu_stream))              # :130-153

The device_identity assertion (assert in_tensor.device == self.device, "illegal memory access", :134-138) is the NPU version of NCCL's device-binding contract — HCCL comms are bound to a specific NPU and do not silently memcpy across devices.

Derived design 2: NPUCommunicator — drop-in replacement for CudaCommunicator. At the vLLM-communicator level, NPUCommunicator(DeviceCommunicatorBase) reuses upstream's gloo CPU groups for control, initializes self.device = torch.npu.current_device(), exposes the same all_to_all / all_gather / all_reduce interface, and adds Ascend-specific _NpuAll2AllManager fault tolerance (npu_communicator.py:40-86). Only get_device_communicator_cls → "....NPUCommunicator" (platform.py:849) differs from a CUDA install.

Derived design 3: KV PD disaggregation — Mooncake RDMA connector. Beyond 1:1 backend replacement, Ascend adds a first-party prefill-decode (PD) disaggregation transport via the Mooncake transfer engine — an RDMA-based KV-prefetch path. MooncakeLayerwiseConnector(KVConnectorBase_V1, SupportsHMA) is a clean-V1-style connector that implements the scheduler/worker KV-connector contract and shuttles KV between prefill-workers and decode-workers:

# vllm_ascend/distributed/kv_transfer/kv_p2p/mooncake_transfer_engine.py:11-29
from mooncake.engine import TransferEngine
self.transfer_engine = TransferEngine()
ret = self.transfer_engine.initialize(hostname, "P2PHANDSHAKE", "ascend", device_name)
def register_buffer(self, ptrs, sizes):
    self.transfer_engine.register_memory(ptr, size)

# connector (mooncake_layerwise_connector.py:697)
class MooncakeLayerwiseConnector(KVConnectorBase_V1, SupportsHMA):
    # get_num_new_matched_tokens / start_load_kv / wait_for_layer_load / save_kv_layer

The layerwise variant pipelines KV transfer layer-at-a-time with the forward pass (wait_for_layer_load, save_kv_layer), overlapping transfer and compute — the same technique DeepSeek's PD serving papers advocate.

Insight for the architect. HCCL-for-NCCL is forced and mechanical. Mooncake is original IP — Ascend's answer to the PD-disaggregated serving trend, built on the CANN-pluggable KVConnectorBase_V1 interface. That the Mooncake connector slots in via the upstream KV-connector contract (not a forked scheduler or engine) is the second proof-point (after the scheduler) that vLLM's connector abstraction is the right seam for disaggregated serving.

5. The patch layer — "surgical extension where subclassing fails"

First principle. Not all of vLLM's hot paths go through clean virtual-method hooks. Some critical behaviors live in module-level functions (not classes) or inside methods with no override point, or in package-__init__ bindings that model code imports before any subclass runs. The constraint: you cannot override what isn't a method; you cannot substitute what is imported by name elsewhere.

Derived design: import-time monkey-patching, partitioned into global vs. worker scope. utils.adapt_patch(is_global_patch=...) selectively imports patch modules, and importing them executes top-level code that rebinds module attributes:

# vllm_ascend/utils.py:537-541
def adapt_patch(is_global_patch: bool = False):
    if is_global_patch:
        from vllm_ascend.patch import platform   # noqa: F401  — scheduler/engine side
    else:
        from vllm_ascend.patch import worker      # noqa: F401  — device-worker side

Canonical example — patching a factory function, not a class. vllm.model_executor.layers.fused_moe.FusedMoE is a factory function, not a class. Models do from ...fused_moe import FusedMoE; FusedMoE(...). You cannot subclass a function; you rebind it:

# vllm_ascend/patch/platform/patch_fused_moe.py:56-58
def _ascend_FusedMoE(*args, runner_cls=None, runner_args=None, **kwargs):
    if runner_cls is None:
        runner_cls = _DefaultAscendMoERunner     # AscendMoERunner310 on 310P
    kwargs.pop("hash", None)
    return _original_FusedMoE(*args, runner_cls=runner_cls, runner_args=runner_args, **kwargs)

_fused_moe_layer.FusedMoE = _ascend_FusedMoE
_fused_moe_pkg.FusedMoE = _ascend_FusedMoE          # rebind both before model import

The file header comment (:19-28) documents the ordering constraint: patches must fire in worker init (adapt_patch) BEFORE from vllm_ascend import ops and BEFORE model import, or the rebound FusedMoE binding won't hold. This is the brittleness.

Canonical example 2 — patching a dispatcher method. ACL capture shapes require a NPU-specific padding rule, so CudagraphDispatcher._create_padded_batch_descriptor is replaced wholesale:

# vllm_ascend/patch/worker/patch_cudagraph.py:29-38
return BatchDescriptor(num_tokens=num_tokens_padded, num_reqs=num_reqs,
                      uniform=uniform_decode, has_lora=has_lora, num_active_loras=...)
CudagraphDispatcher._create_padded_batch_descriptor = _create_padded_batch_descriptor

Scale of the patch surface. The patch tree has ~50 files covering patch/platform/* (scheduler/engine-side: fused_moe, kv_cache_coordinator, distributed, mamba_config, profiling_chunk, mtp, …) and patch/worker/* (device-worker-side: cudagraph, deepseek_v2, fused_moe, triton, qwen3, weight utils, …). This is the largest subsystem in volume and the most frequent source of breakage across v0→v1 migrations.

Insight for the architect. The patch layer is a revelation of vLLM's missing extension points. Where a clean virtual method exists, Ascend subclasses (NPUModelRunner(GPUModelRunner), NPUPlatform(Platform)). Where only a free function or a non-overridable internal exists, Ascend must patch. Each patch is a small time-bomb: the upstream symbol it rebinds may move, rename, or be inlined in the next release. The v0→v1 migration evidently broke many (patch/worker/patch_v2/ has its own sub-tree). This is the adaptation-community's core technical debt, and the most actionable feedback item for upstream: convert the patched free functions into overridable methods.

6. AscendScheduler — "the scheduler as an extension point"

First principle. Scheduling policy (preemption semantics, admission order, chunk sizing) is not physically forced — it is an optimization choice over the same universal paging/KV-cache abstraction. The constraint: vLLM's scheduler already exposes a Scheduler class and a SchedulerOutput data contract; a new device should be able to change policy without touching the loop.

Derived design: subclass + config-string pivot. Ascend injects its scheduler by overriding the class string that the engine core builds:

# vllm_ascend/core/recompute_scheduler.py:58-75
@dataclass
class RecomputeSchedulerConfig(SchedulerConfig):
    scheduler_cls: str | type[object] = "vllm_ascend.core.recompute_scheduler.RecomputeScheduler"

    @classmethod
    def initialize_from_config(cls, vllm_config):
        if vllm_scheduler_config.async_scheduling:
            scheduler_config["scheduler_cls"] = "...AsyncRecomputeScheduler"
        else:
            scheduler_config["scheduler_cls"] = "...RecomputeScheduler"

RecomputeScheduler(ShortRequestFirstSchedulerMixin, Scheduler) (:98) subclasses upstream Scheduler directly. The extension surface is threefold: 1. Recompute preemption — on Ascend, a preempted request can either free+recompute (emit RecomputeReqInfo so the PD-proxy replays already-sampled output) or offload-KV-and-preserve (PreemptedRequestData + connector update_state_before_preempt, recompute_scheduler.py:307-398). This replaces upstream's "free and re-prefill" with two device-conscious alternatives. 2. SJF-with-aging waiting queue — ShortRequestFirstSchedulerMixin + ShortRequestFirstRequestQueue (immediate / short / long lanes, promote long after long_max_wait_ms, short_request_first_scheduler.py:209), optimizing NPU's memory-bandwidth-vs-compute ratio for short requests. 3. Profiling-based chunk sizing — ProfilingChunkScheduler (scheduler_profiling_chunk.py:108) runs real forward passes at startup to fit a quadratic chunk-latency model instead of the static upstream long_prefill_token_threshold. 4. Output contract extension — RecomputeSchedulerOutput(SchedulerOutput) (:93) adds preempted_reqs and recomputed_reqs lists; upstream's 600-line schedule() loop, the paged-KV pool, and the block-table operand path are reused verbatim.

The config pivot also includes two more optional adapters: ShortRequestFirstConfig (ascend_config.py:821) and ProfilingChunkConfig (:647), both default-off and injected through AscendConfig.check_and_update_config.

Insight for the architect. This is the cleanest Ascend extension and the strongest argument that Scheduler + SchedulerConfig.scheduler_cls was the right abstraction. Ascend rewrote the preemption semantics (recompute/offload), the waiting-order policy (SJF+aging), and the chunk-sizing policy (profiling), while inheriting 100% of the per-token loop, paged-KV path, and block-table/slot-mapping backend. This is the template any accelerator-specific scheduler should follow: subclass Scheduler, publish subclass-specific fields on SchedulerOutput, never touch the loop or the kernel.

7. EPLB — "Ascend's original combinatorial contribution"

First principle. In MoE expert-parallel serving under skewed expert routing, static expert-to-device assignment causes severe load imbalance → devices idle waiting for the hottest expert. The constraint: expert placement is a combinatorial optimization; static assignment is optimal only when routing is uniform. This is not device-specific — it's a property of MoE + EP — but the solution lives in the serving system, not the model, and Huawei contributed it back.

Derived design: offline-process rebalance + online D2D weight migration. EplbUpdator (eplb/eplb_updator.py) runs a dedicated EplbProcess (:29-212) that eats a moe_load tensor, calls a pluggable EplbPolicy.rebalance_experts (policy/policy_random.py:12), and produces an updated expert map. Across training iterations, the updator interleaves three phases gated by a shared memory dict:

# eplm_updator.py:129-152, 79-89
def forward_before(self):
    if self.get_update_info_flag():
        self.update_info_all = self.eplb_process.block_update_q.get()
    if self.update_expert_weight_flag():
        (expert_send_info, expert_recv_info, updated_expert_map, ...) = self.update_info_all.pop(0)
        self.eplb_loader.generate_expert_d2d_transfer_task(...)  # P2P copy plan
        self.eplb_loader.asyn_expert_weight_transfer(self.reqs)   # async D2D weight move

def compute_and_set_moe_load(self):
    local_load = self.adaptor.get_rank_expert_workload().unsqueeze(1)
    moe_load = self.comm_group.all_gather(local_load, dim=1).cpu()
    self.shared_dict["moe_load"] = moe_load              # feed the planner process

The cycle: (1) expert_heat_collection_interval iterations of workload accumulation → (2) wake planner, run policy → (3) per-layer async device-to-device expert-weight migration (D2DExpertWeightLoader.update_expert_map_and_weight, core/eplb_device_transfer_loader.py:32) → reset. warm_up_eplb (:157-183) does an all-rank P2P handshake to calibrate. A multi-stage variant (eplb_policy_type==3) is supported (:35,149).

The file's header Todo (:16) says "Once https://github.com/vllm-project/vllm/issues/22246 is merged in vllm, remove this updator" — acknowledging EPLB is intended to be upstreamed as an official vLLM mechanism, but Ascend needed it now rather than later.

Insight for the architect. EPLB is ascend's clearest original-IP block and the only subsystem here not forced by "the hardware is different"; it is forced by "MoE+EP+skew is a universal serving problem". The threaded-process + shared-dict + async-D2D design is portable to GPU serving as well — it is notan NPU-specific trick. For inference architects, EPLB's lesson is: expert load balancing is a systems problem, and the unit of rebalance is expert-by-expert weight migration, implemented as async D2P2P copies interleaved with the forward pass.

8. Plugin registration — "how Ascend injects itself without modifying vLLM's core imports"

First principle. vLLM is a library, not a monolith; its core intentionally never imports an accelerator name. The constraint: the adapter must inject itself at well-defined lifecycle hooks — platform, model, connector, weight engine, forward context — without rewriting vLLM's import graph. The hook contract is vLLM's plugin entry-point functions.

Derived design: five register_* entry points, each behind _ensure_global_patch. vllm_ascend/__init__.py exposes the exact functions vLLM's plugin loader calls:

# vllm_ascend/__init__.py:32-83
def register():
    """Register the NPU platform."""
    return "vllm_ascend.platform.NPUPlatform"

def register_connector():
    _ensure_global_patch()               # apply global (scheduler/engine) patches first
    from vllm_ascend.distributed.kv_transfer import register_connector
    from vllm_ascend.distributed.weight_transfer import register_engine
    register_connector(); register_engine()

def register_model_loader(): ...        # netloader + rforkloader
def register_service_profiling(): ...    # generate_service_profiling_config
def register_model():
    install_hunyuan_vl_processor_compat()
    from .models import register_model; register_model()

The _ensure_global_patch() guard (:12-30) matters: vLLM loads general plugins in engine-core subprocesses, where the e2e-test conftest hooks do NOT run. So global patches that affect scheduler/engine code must be re-applied here. register() is intentionally side-effect-free (returns only the platform string) — it runs before torch_npu is safe to initialize. All side-effectful setup is deferred to register_connector / register_model.

Derived design 2: model registration via ModelRegistry. vllm_ascend/models/__init__.py maps HuggingFace model-name strings to Ascend model classes through vLLM's ModelRegistry:

# vllm_ascend/models/__init__.py:4-8
from vllm import ModelRegistry
def register_model():
    ModelRegistry.register_model("DeepseekV4ForCausalLM",
                                "vllm_ascend.models.deepseek_v4:AscendDeepseekV4ForCausalLM")
    ModelRegistry.register_model("DeepSeekV4MTPModel",
                                "vllm_ascend.models.deepseek_v4_mtp:DeepSeekV4MTP")

Derived design 3: Python-side meta impls for torch.compile/tracing. meta_registration.py registers shape-inference ("meta") implementations of CANN custom ops into torch.library("_C_ascend") so that torch.compile and ACL-graph tracing can run without real tensors — only the impl(...) functions need live NPU kernels:

# vllm_ascend/meta_registration.py:17-33
lib = Library("_C_ascend", "IMPL")
def register_meta_if_necessary(ns, op_name, fn, overload=""):
    schema_to_find = ns + "::" + op_name
    if schema_to_find in torch._C._dispatch_get_registrations_for_dispatch_key("Meta"):
        return
    lib.impl(op_name, fn, "Meta")          # enables tracing/export/shape-inference

Insight for the architect. The registration path is layered in the right order: platform → (patch + connect) → model loader → model → compile-meta, with a single _ensure_global_patch barrier sequencing the global-vs-worker split. The reliance on def register() -> "ClassName" (a string-returning entry point) is the key decoupling mechanism: vLLM's plugin loader gets a locator without importing CANN. The cost: every new Ascend capability must be wired into one of these five register_* functions plus (if it needs a vLLM entry point) an override in NPUPlatform. A capability that doesn't fit any of the five — e.g. a new MoE preemption behavior — falls back to the patch layer (Thesis 5).

9. Xlite — "the third graph runtime" (a design observation)

Ascend ships a third inference path beyond eager + ACL graph: xlite. XliteModelRunner(NPUModelRunner) (xlite/xlite_model_runner.py:25) and XliteWorker(NPUWorker) (xlite/xlite_worker.py:22) swap the model for a whole-program Xlite-compiled graph at load_model time:

# xlite/xlite_model_runner.py:32-36
def load_model(self):
    from vllm_ascend.xlite.xlite import XliteWrapper
    super().load_model()
    self.model = self.xlite_model = XliteWrapper(self.model, self.vllm_config, device=self.device)

This implements the openEuler GVirt Xlite graph runtime. It is gated by ascend_config.xlite_graph_config.enabled (check_and_update_config platforms the worker_cls to XliteWorker at :614-617) and only hooks get_model / load_model / initialize_kv_cache — the upstream ModelRunner contract is otherwise untouched. This is an experiment in a "program-whole" graph backend alongside vLLM's native piecewise/Full ACL graph and torch.compile-via-AscendCompiler paths. It illustrates that Ascend treats the runner as the unit of inference-runtime-swappability.

Observation. Ascend therefore ships three parallel inference runtimes (eager, piecewise/full ACL graph, Xlite whole-program graph) behind one NPUModelRunner-derived hierarchy. This is a lot of runtime diversity; it exists because no single NPU graph backend dominates across all models and batch regimes on Ascend. For an architect this is a warning surface: every runtime has its own compile cache, warmup, and dispatch code path, and the selection happens via builder config rather than auto-tuned.

10. Design insights for an inference architect

10.1 What is physically forced vs. what is a design choice

Necessity class Subsystem Why Category
(1) Platform + memory stats NPUPlatform, set_device, get_current_memory_usage, num_compute_units vLLM's Platform/ModelRunner reference torch.cuda.* thermodynamics FORCED by CUDA-isms
(2) Custom ops ops/linear.py, ops/fused_moe/*, ops/layernorm.py, ops/rotary_embedding.py Generic PyTorch/CUDA kernels run poorly on NPU FORCED by CANN kernel gap
(3) Attention backends attention/{mla_v1,sfa_v1,dsa_v1,fa3_v1}.py MLA/DSA/SFA fused-kernel paths are required for the top models FORCED
(4) Comms backend PyHcclCommunicator, NPUCommunicator, pyhccl_wrapper.py No NCCL on NPU FORCED
(5) Graph capture acl_graph.py, npugraph_ex_compile, AscendCompiler NPU has no CUDA-graph FORCED
(6) Scheduler RecomputeScheduler, ShortRequestFirstSchedulerMixin, ProfilingChunkScheduler NPU favors recompute + SJF + profiled chunks CHOICE (high-value)
(7) EPLB eplb/* MoE EP skew under static placement CHOICE (original IP)
(8) PD disaggregation mooncake_*.py, layerwise_connector Disaggregated PD serving trend CHOICE (original IP)
(9) Patch layer patch/** vLLMissing extension points COST of misaligned seam

The "forced" half (1-5) is the unavoidable tax of non-CUDA hardware. The "choice" half (6-8) is Ascend's original IP — the adaptations that would improve GPU serving too. The patch layer (9) is the debt.

10.2 The "adaptation surface" ranked by first-principle necessity

  1. Platform subclass + dispatch_key — single mandatory root; all downstream dispatch keys off it (PrivateUse1).
  2. CANN custom-ops registry (ops/**) without which the hot path is unusable.
  3. ACL-graph capture / AscendCompiler — without real graph reuse, decode throughput collapses.
  4. HCCL communicator — without it, TP/EP/PP do not work.
  5. One AttentionImpl per attention variant (MLA/DSA/SFA/FA3) — without model-specific kernels, DeepSeek/Qwen models don't run well.
  6. KVCacheSpec + SingleTypeKVCacheManager (per-platform KV layer, core/kv_cache_interface.py) — slots through the existing registry, does not disturb scheduler.
  7. Scheduler subclass + SchedulerOutput extension — the template for accelerator scheduling.
  8. KVConnectorBase_V1 (Mooncake) — the template for disaggregated serving.
  9. EplbUpdator + process-per-rebalance — the template for MoE load balancing.
  10. Patches — the residual when (1-9) don't reach.

10.3 What this teaches about vLLM's architecture quality

Where vLLM's abstractions ENABLE clean extension: - Platform as pure-data shim (name/UUID/memory/comms-class strings) — clean swap for the easy half. - AttentionBackend registry keyed on (use_mla, use_sparse, use_compress) — NPUPlatform.get_attn_backend_cls (:784-810) is a configuration table, not a conditional tree that touches core. - Scheduler + SchedulerConfig.scheduler_cls string pivot — RecomputeScheduler proves you can rewrite preemption, admission order, and chunk sizing while inheriting 100% of the paged-KV path and the 600-line schedule() loop. - KVCacheSpec + KVCacheSpecRegistry + SingleTypeKVCacheManager — per-platform KV specs (MLA compression, 910/310P differences) plug in without touching the scheduler. - KVConnectorBase_V1 — Mooncake shows a brand-new PD-disaggregated transport slots into the scheduler/worker/connector contract without a forked engine.

Where vLLM's abstractions FORCE brittle workarounds (the patch layer): - Free functions imported as names (FusedMoE, dispatcher methods) can't be subclassed → patch_fused_moe.py, patch_cudagraph.py, and ~50 siblings. - CUDAGraphMode enum carries CUDA semantics in its state names and in its size rules (CUDA's * 2, CUDA's padding logic) → over half of check_and_update_config is re-deriving NPU graph rules. - Warmup/compile paths shared between CUDA and ACL graphs → patch_cudagraph.py, patch_triton.py, patch_npugraph_ex_triton.py patch at the warmup level.

10.4 Recommendations: how vLLM could better support non-NVIDIA hardware

  1. Convert patched free functions into overridable methods. The single highest-leverage change is to make FusedMoE a class (or an injectable factory) in upstream vLLM, and to make CudagraphDispatcher._create_padded_batch_descriptor a virtual method — each such change deletes an entire Ascend patch file.

  2. A formal ops registry / vendor-ops contract. Today custom ops are registered via direct_register_custom_op + dispatch_key="PrivateUse1", but the discovery between "Ascend wants a fused MLA kernel here" and "the model's attention layer wants torch.ops.vllm.mla_forward" is still positional. A vllm.attention / vllm.ops vendor registry (analogous to KVCacheSpecRegistry) would let Ascend publish "MLA → AscendMLAImpl" without touching NPUPlatform.get_attn_backend_cls.

  3. A comms abstraction layer. NPUCommunicator(DeviceCommunicatorBase) works, but every layer above it (EPLB's comm_group.all_gather, PyHcclCommunicator's device-binding assertions, flashcomm2/MLP TP groups) re-implements bandwidth/latency-conscious logic that upstream only provides via NCCL tunables. A DeviceCommunicator extension that exposes "fused matmul+reduce" and "fused all-gather+proj" primitives would eliminate the entire CustomLinearOp hierarchy (Thesis 2 design 3).

  4. Rename / device-neutralize CUDAGraphMode. Replace the CUDA-named enum with a device-neutral CaptureMode { NONE, FULL, PIECEWISE } and push the CUDA-specific padding * 2 rule into the CUDA platform. Then ACLGraphWrapper subclasses the same dispatcher as CUDAGraphWrapper, and check_and_update_config's 230-line NPU branch collapses to a handful of overrides.

  5. Treat "scheduler preemption policy" as a first-class plugin rather than a Scheduler subclass authored per silicon. Today RecomputeScheduler wrestles with the states PREEMPTED, recomputed_reqs, offloaded; a PreemptionPlugin hook inside the existing Scheduler would let Ascend ship its recompute/offload logic without a parallel Scheduler class.

  6. Upstream EPLB + moonsault PD. Both EPLB (eplb_updator.py:16) and Mooncake PD disaggregation are intended for upstream (the former explicitly references vllm-project PR #22246). Doing so is the single biggest lever to reduce Ascend's patch surface: the patches around patch_deepseek_mtp.py, patch_use_v2_model_runner.py, and patch_recompute_sampler.py exist in part because these innovations can't yet slot into the public engine.

10.5 One-sentence design principles for porting inference engines to a new device

  • The OS-port analogy holds: emulate the host's ABI at the seams (Platform, DeviceCommunicator, custom-ops registry), capture/replace the kernel layer (ACL graph, AscendCompiler), innovate on policy only where the old device's physics were suboptimal, not where they were wrong (RecomputeScheduler, EPLB, Mooncake).
  • Every patch file is a missing extension point; every subclass is a well-designed seam.
  • The "adapter surface" is bounded and rankable: platform/ops/comms/graph are forced; scheduler/EPLB/PD-disaggregation are IP; everything else is debt.
  • First-principles discipline pays: most Ascend code is either (a) a 1-line CUDA→CANN rename, (b) a structure-preserving subclass, or (c) deliberate MoE/comms innovation. The patches are the only part that resists this classification, and they are precisely where vLLM should invest in tighter seams.
