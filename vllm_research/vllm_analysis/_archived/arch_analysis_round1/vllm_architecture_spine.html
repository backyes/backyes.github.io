0. v0 已死，v1 的诞生——为什么 vLLM 必须重构

vLLM 在 2025 年 1 月正式发布 V1（vLLM V1: A Major Upgrade），并标注 V0 完全弃用（RFC #18571）。这不是功能升级，是一次基于计算约束的核心重写。

v0 为什么撑不住。v0 的 core/、worker/、attention/ 三层结构是功能分解：调度器一个团队、worker 一个团队、attention 一个团队。这看似干净，但当三个维度同时演进时（continuous batching、paged attention、chunked prefill、prefix caching、spec decode），改动必须跨三层协调，"新功能塞进旧分区"产生指数级耦合。实测现象：V0 代码 50% 在分支、30% 在修分支冲突的 bug。

v1 做了什么。v1 保留 v0 验证过的 ~200 个模型定义和 GPU kernel，重写调度器/KV-cache 管理器/worker/sampler 和 API server 五段核心（官方 V1 博文原文，见 docs/usage/v1_guide.md:4）：提供 simple, modular, easy-to-hack 的代码；组合 chunked prefill / CUDA graph / continuous batching 为统一架构；zero config——特性默认开；长上下文性能明显领先 v0。关键是：v1 把"分阶段"(prefill phase / decode phase)思维砍掉，换成了单步 token-budget——我们下文反复回到这一点的物理意义。

1. 三条物理不变量——vLLM 架构的"第一性原理"

不变量物理含义vLLM 里的直接后果
I1. Transformer 内存定律一个活请求存 Θ(S·L·d) 的 KV（S 序列长、L 层数、d head 维）；只追加、按位读取、永不修改Paged KV-cache（把 OS 页表搬到 HBM）；content-hash 前缀去重成为容量倍增器
I2. 带宽-算力墙（roofline）decode 1 token 扫整层权重 → arithmetic intensity ≈ h/2 FLOP/byte，远低于 GPU ridge point → memory-bandwidth-bound；prefill 多 token × 同权重 → compute-bound无任何单一 kernel 对 decode 和 prefill 同时最优；必须按步用 token-budget 分时复用；Ascend 上甚至引入 recompute
I3. 硬件多样性NVIDIA / AMD / Intel / Huawei / Google / 国产 DSA 是不兼容方言的同一个数学；#ifdef 进热路径会不可逆地 fork 仓库#ifdef 被压进 3 个接口：Platform（设备发现）、AttentionBackend（唯一的硬件相关算子）、Communicator（集合通信）
"Given physics, what MUST any correct high-performance inference system look like?"
—— vLLM 的大块结构不是某人拍脑袋的"架构"，而是 I1/I2/I3 被强制推出的解。

2. Paged KV-cache：推理系统的页表

为什么整块分配会死

I1 要求对每个活跃请求分配 Θ(S) 张量。请求长度动态变化（用户 prompt），张量不可变且长期占用。如果像传统 batch 一样连续分配，会在 HBM 里留下"瑞士奶酪"空洞。经典操作系统解法：paging——固定大小 page + 页表间接。vLLM 的 KVCacheBlock 就是 page metadata，block_table 就是 page table。

数据结构的解剖

vllm/v1/core/kv_cache_utils.py:117——block 是纯 metadata 对象，不持有张量：

@dataclass(slots=True)
class KVCacheBlock:
    block_id: int                 # 指向 HBM 平坦地址池的索引
    ref_cnt: int = 0              # 前缀共享的引用计数
    _block_hash: BlockHashWithGroupId | None = None   # content hash，作为 prefix-cache key
    prev_free_block / next_free_block: KVCacheBlock   # free 链表
    is_null: bool = False         # block_id=0 是占位 null page

📐 关键分离：scheduler 侧只碰 block metadata (KVCacheBlock)；真正的 block_size·num_kv_heads·head_size·2·dtype 在 worker 侧 的顶量 kv_caches: list[torch.Tensor](model_runner.py:490, 由 attn_utils.init_kv_cache 分配)。两层分离使 prefix-cache / swap / sliding-window / compressed-MLA 整个栈都能在同一组 block_id 上重编号、加权。

allocator：free-list + content hash map

vllm/v1/core/block_pool.py:162

self.blocks = [KVCacheBlock(idx) for idx in range(num_gpu_blocks)]
self.free_block_queue = FreeKVCacheBlockQueue(self.blocks)
self.cached_block_hash_to_block = BlockHashToBlockMap()   # hash → KVCacheBlock
  • 分配 O(1)：get_new_blocks(block_pool.py:647) 直接 free_block_queue.popleft_n(N)。
  • 淘汰 O(1)：free list 满时 _maybe_evict_cached_block(:679) 丢最轻 cached block。
  • 前缀去重：get_cached_block(:198) 按 content hash 查找，命中则不分配直接 touch(:702) 增加 ref_cnt——跨用户、跨时间的相同 sys prompt 共享同一物理页；free_blocks(:719) 只减 ref_cnt，归零才写回 free list。
  • partial-block entry(:445)：不同位置落在一个 block 内部边界也能注册更细粒度 hash，消除对齐浪费。

"页表 indirection"——这条 kernel 就是页表 walk

Worker 把 CPU 侧的 (req, position) 翻译成 HBM 单偏移量给 paged-attention kernel：

vllm/v1/worker/gpu/block_table.py:283（Triton kernel _compute_slot_mappings_kernel，上下文并行开时 CP_SIZE 有缩放）

block_indices  = positions // (block_size * CP_SIZE)
block_offsets  = positions %  (block_size * CP_SIZE)
block_numbers  = tl.load(block_table_ptr + req_state_idx * stride + block_indices)
slot_ids = block_numbers * block_size + block_offsets   # ← 物理地址
这一行 slot = block_id × block_size + offset 与 OS 物理地址 phys = ppn × PAGE_SIZE + page_offset 完全同构。block_table 是推理系统的 page table；block_id 是物理页框号(PFN)；slot_id 是线性物理地址。Cuda 上 PAD_SLOT_ID 对应"无效页表项"。这是 vLLM 对 GPU HBM 做"虚拟内存管理"的精确形式化。

每步只有两张动态张量进入 attn kernel：block_tables [groups, reqs, max_blocks] 与 slot_mappings [groups, tokens]（model_execute.py:1041 prepare_attn）。Decode 只读；prefill 写 KV 进新 block（Allocator 返回的 new_block_ids 预先清零，见 scheduler.update_from_output）；此后 block 只读、可被 cached。

免费附赠为什么 paging 是三条需求的同时解

需求若无 pagingpaging 之后
HBM 有限连续分配 → Swiss-cheeze 空洞，利用率长期 <60%固定大小 block 零外部碎片，利用 ~95%+
system prompt 重复每请求独立存储content hash 共享前缀 → 30-40% 峰值请求只耗 ~5% KV 显存
变长 batch必须 pad / 静态 buffer 浪费ragged batch 由 block_table 自描述，无需 padding 浪费算力

3. 单步调度器：带宽墙被迫的 token-budget

"没有 prefill 阶段，没有 decode 阶段"

vllm/v1/core/sched/scheduler.py:420——v1 调度器开头写的设计公理（是代码注释，但就是设计公理）：

# There's no "decoding phase" nor "prefill phase" in the scheduler.
# Each request just has num_computed_tokens and num_tokens_with_spec …
# At each step, the scheduler tries to assign tokens to the requests so
# that each request's num_computed_tokens can catch up its
# num_tokens_with_spec. This is general enough to cover chunked prefills,
# prefix caching, speculative decoding, and "jump decoding."

单循环 algorithm

scheduler.py:438

token_budget = self.max_num_scheduled_tokens     # 每步硬上限
while req_index < len(self.running) and token_budget > 0:
    num_new_tokens = (num_tokens + spec - num_computed)
    if 0 < long_prefill_token_threshold < num_new_tokens:
        num_new_tokens = long_prefill_token_threshold        # chunked prefill
    num_new_tokens = min(num_new_tokens, token_budget)
    new_blocks = kv_cache_manager.allocate_slots(request, num_new_tokens)
    if new_blocks is not None: break
    else: preempt lowest-priority running request          # reclaim budget

chunked prefill 是 roofline 的直接推论

阶段FLOP/byte瓶颈若不切会怎样
prefillhigh (多 token 扫同权重)compute-bound长 prompt 吃满 GPU 算力，decode 被饿死 → TTFT vs TTIT 的直接冲突
decode≈ h/2memory-bandwidth-bound调度器若不区分，一个长 prefill 卡 50ms+，decode 的 SLO 全部爆

v1 一次把这两个相反的计算范式塞进同一步（token_budget 每步 chunk），prefill 不卡 decode、decode 也不挨饿——这不需要特殊的 "prefill mode switch"，只是单一预算循环里"每个请求按权重直接领 token"。因此 chunked prefill / prefix cache / spec decode / jump decode 是同一循环的四张面孔，不是四个特殊分支。

admission / preemption 作为反馈控制器

KVCacheManager.allocate_slots(kv_cache_manager.py:283) 在 required_blocks > free − reserved − watermark 时返回 None。None 是调度器的压力信号：STOP 准入或 preempt 低优先级请求。reserved_blocks 和 watermark_blocks ——为 async KV connector（PD 分离）和 admission 留 headroom，是一个闭环反馈，不是配置驱动的 open loop 策略。num_gpu_blocks 本身由 engine init 时 HBM profile（model_executor.determine_num_available_blocks）实测得到，所以整个系统是实测硬件出来的闭环，不是一组想当然的参数。

工程师速算（H100 80GB 实装）：
80GB HBM ≈ 64GB 可用（16GB 模型 weight + 1-2GB workspace + CUDA context）；block_size=16 token × 80层 × 8 head × 128 d × 2(K+V) × 2byte(fp16) ≈ 52.4 MB/block → 64GB 可放 ~1200 block。每个 block 存 16 个 token 的 KV，即大概 ~18K token 的 KV 容量。128 并发 × 128 prompt ≈ 刚好打满。这就是 KV 容量封顶并发的由来——物理上限。

4. 心脏 execute_model：schedule → prepare → forward → sample

v1 整个系统的时序由三行代码驱动。vllm/v1/engine/core.py:576（EngineCore.step）：

scheduler_output = self.scheduler.schedule()
future = self.model_executor.execute_model(scheduler_output, non_block=True)
# ……
model_output = future.result()          # 或 sample_tokens
engine_core_outputs = self.scheduler.update_from_output(scheduler_output, model_output)
  • CPU 时间线: schedule() 在 CPU 算决策；future.result() 的 await 让 CPU 在 GPU 忙碌时做 bookkeeping 与 pre-fetch 下一步。
  • GPU 时间线: execute_model 被 executor fan-out 到 TP worker 进程的 GPU。
  • 异步调度(async_scheduling) 更进一步：把 sampled token 的 D→H 拷贝放到另一个 CUDA stream，GPU 与 CPU 进一步 overlap——这直接暴露"I2 的带宽墙也可以用 CPU overlap 变相抢回一点延迟"。

GPU ModelRunner.execute_model data flow（单步 1134-1374, GPUModelRunner）

5. Python 前端 + Triton/CUDA kernel：95/5 法则

Transformer inference 里，GEMM + attention + softmax + sample 吃掉 95% 的 FLOPs，但只占 5% 的代码；tokenization / scheduling / request management / API 吃掉 5% FLOPs 但占 95% 代码。v1 的选择：

  • Python 95%：HuggingFace 生态集成、调度策略、JSON grammar、profiler——开发速度远比性能重要。
  • Triton / CUDA / C++ 5%：vllm/_custom_ops.py 是 kernel registry；attention 走 FA3 / FlashInfer / Triton-for-NPU；sample 走 topk_topp_triton；fused MoE 用 modular kernel。

GIL 代价：Python 单线程 + GIL 阻塞，不能做 per-iteration 小 kernel 调度。这正是下面三种 CUDA Graph 模式存在的理由：把整个 forward录成一张静态 kernel 录播带，解除 GIL 对 kernel launch 序列化的约束。

6. AttentionBackend：唯一的硬件相关算子

📐 这是 vLLM 最干净的一条硬件抽象线，硬件适配团队的第一课。

为什么 attention 是唯一"必须换"的 kernel。GEMM、Norm、MoE dispatch、RoPE、residual add——全都是结构与长度无关的数学：(N, d) → (N, d)。唯有 attention 是 (N, d) × KV(N, d) → (N, d)——其 memory-access pattern 显式依赖 序列长、KV block 布局、sparse / sliding window / compressed-MLA。所以它是唯一一个"硬件换了 kernel 必须重写一项"的算子；其他 op 用 cuDNN / CANN / oneDNN / ROCm 通用库一条 GEMM 改写。

因此 vLLM 把 attention 拆成可插拔：

角色类 / 注册点责任
Backend enumAttentionBackendEnum + @register_backend(name, device)字符串 → 实现类映射
BuilderAttentionMetadataBuilder.build(common_attn_metadata)把 v1 的 common (q_start_loc, seq_lens, block_table, slot_mapping) 翻译成本地 metadata
ImplAttentionImpl.forward(layer, q, k, v, kv_cache, attn_metadata)执行 kernel，这是唯一直接操作 kv_caches 张量的地方
KVCacheSpecspec.KVCacheSpec / MLASpec / RaggedSpec / ...描述 KV 张量形状 / block stride / compression；决定 worker kv_caches 列表的 shape 布局

Dispatch 链：model_runner.init_attn_backend 按 group-by (backend, KVCacheSpec, num_q_heads) 把层分组，按组整组初始化 backend。(model_runner.py:442)

Ascend 套 5 个 backend：ASCEND (paged), ASCEND_MLA, ASCEND_SFA (sparse flash), ASCEND_DS_A (DSA sparse), ASCEND_FA3 —— 都注册到 AttentionBackendEnum.CUSTOM(core: attention_v1.py:75 @register_backend(AttentionBackendEnum.CUSTOM,"ASCEND"))。内核全部走 torch_npu.npu_fused_infer_attention_score——华为 CANN 专门为 paged-attention 做的融合 kernel。

6b. 并行计算：TP / PP / DP / EP 是如何合而不乱

策略切什么通信典型 bubble / 代价vLLM 位置
TP (tensor parallel)每层 matmul 沿 hidden / head 切每层 all-reduce (或 AG+RS)延迟低，但每 rank 通信量 = 2M，瓶颈 NVLink/交换带宽distributed/parallel_state.py, TP_GROUP；linear 列切 via ColumnParallelLinear
PP (pipeline)按层段切 L 层P2P send/recv [b,h,d] inter-layer tensorpipeline bubble = (p-1)/p 的 idle；bubble 随 batch 慢漂没PP_GROUP, IntermediateTensors(forward 中间回传), model_runner.update_pp_decode_requests
DP (data parallel)复制模型每 dp 权重一致；scheduler 跨 dp rank 平衡线性扩吞吐；需 同步 CUDA graph；load-balance 是金坑dp_utils.dp_broadcast、cudagraph_sync_dp, Disagg / PD
EP (expert parallel)MoE 层按 expert 切MoE dispatch + all-to-all 行路由路由决定一切；imbalance = expert 热点eplb/（昇腾原创！§9）、distributed/moe/

📐 这 4 种并行在 Compose，不是在选主——生产版 DeepSeek-V3 / Qwen3 同时 TP=8 + EP=64 + PP=4 + DP=32；vLLM 用 parallel_state.py 的 group cache 把 rank 分组做 composable，worker 只认 group 不认策略名。

7. Platform 抽象：device as an interface

设备多样性(I3)的集中解法是把"所有的设备相关假设压成一个接口"——vllm/platforms/interface.py:134 Platform。必须实现的 6 个纯虚方法：

get_device_name / get_device_uuid / get_device_total_memory
set_device / get_attn_backend_cls / manual_seed_all

可选面（有默认实现）：memory stats, device capability count, current_stream, profiler hooks, graph pool, ...。NPUPlatform（昇腾，platform.py）的 get_device_total_memory 直接 raise NotImplementedError（注释原句："DO NOT IMPLEMENT: ...torch_npu allows global initialization only once; duplicate initialization causes errors"）——这是一个典型的"接口承诺 vs 设备能力"失配——强烈建议 vLLM 上游做 lazy memory probe。

Platform 的另一个枢纽职责：get_attn_backend_cls(device, head_size, dtype, block_size, sliding_window, attn_type)。所有非 NVIDIA 设备的 attention 切换决策都在此做，是三级硬件选择（平台 → backend → kernel）的最上层。

8. 三种执行模式：FULL / PIECEWISE / eager CUDA Graph

📐 官方 CUDA Graph 设计图：wrapper_flow.png、current_design.png 与 previous_design.png。

vLLM 把 forward 录成 CUDA Graph 是为了消灭 kernel launch overhead：Python → CUDA driver → launch 一个 matmul ≈ 5-10μs；一个 80 层 transformer forward 80 × 几十 kernel = 录成 1 次 launch 整个 graph。代价是录播时与回放时输入 shape 必须一样——但推理每步 req 数、token 数、序列长每步都在变！三种模式是对此约束的 trade-off 家族：

模式动态维度代价适用
FULL CUDA Graphrequant 完全固定；最大 batch/token; pad 到 2 的整数倍显存大（录一个 2 步 shape 都覆盖）、TTFT 首 token 慢纯 decode 场景吞吐最大；serverless 长尾
PIECEWISE CUDA Graphdynamic shapes；注意力部分单独录制，GEMM 部分单独录制显存略小、TTFT 友好mixed prefill + decode；长上下文
Eager (NONE)全动态；每 kernel 单独 launchlaunch overhead 最大；但最灵活调试、encoder-decoder（cross-attn 每步变）、CPU/GPU 路径

Dispatch 入口：vllm/v1/worker/gpu/cudagraph_utils.py——CudagraphDispatcher 用 BatchExecutionDescriptor{num_tokens, num_reqs, cg_mode} 做 dispatch, candidate key (rounded_num_tokens, num_active_loras)。每一对 (token_num, lora_case) 在 warm-up 时录 graph，运行时按 min(num_reqs, batched_reqs) 查 nearest captured case。混合模式在 encode-decode（Whisper）上自动退 eager。Ascend 把 FULL-AND-PIECEWISE 直接禁用——因为 NPU graph capture 只有 ACL Graph，vLLM 的 CUDA-graph 语义假设在 CANN 上不成立。整个 Ascend § 最难的一段：ACL Graph 自己的 capture/replay 假设和 CUDA Graph 不一样（不保存 persistent input buffer、forward context 自管理）——这是 I3 最深的裂缝。详见 agent3 §4 三种机制对比表。

9. vLLM-Ascend：一条完整的非 GPU 适配线

Ascend 最醒眼的三个"原创"

  1. Schedule-level innovation (I2)：NPU 的 compute/bandwidth 比和 GPU 不同。RecomputeScheduler(recompute_scheduler.py:98)——把 preemption 从 "offload KV 到 CPU" 换成 "扔掉 KV、下一次需时recompute"——因为 NPU 计算力强于 PCIe 带宽，重算 > 搬运。这直接是 roofline 角度的硬件最优。ShortRequestFirst 三车道队列 + profiling-chunk 动态切割也都是 I2 的不同侧面。详见 agent2 § AscendScheduler vs upstream。
  2. EPLB—vLLM 项目里 Ascend 的唯一贡献：MoE EP 下 expert 热点 vs 路由均衡是组合优化，默认 load balance 造成严重 idle。Ascend 提出 FlashLB（匈牙利 + numba 树搜索）+ FlashTree + D2D loader，三层 updator→policy→D2D 构筑完整方案。详见 agent4 §EPLB。
  3. ACL Graph vs CUDA Graph：NPU 图执行和 CUDA Graph 语义差异——forward context 自管理、不保持 persistent buffer、ACL capture。Ascend 的 GraphFusionPassManager 做了 70 个 fusion pass。详见 agent3 §4。

patch 层——"暴露面不足"的诊断书

60+ patch 文件不是"工程债务的羞耻"，而是精确标注 vLLM 缺失扩展点 的 map：

  • patch_fused_moe → vLLM 没暴露 "FusedMoE factory 可换" 的接口
  • patch_cudagraph → CudagraphDispatcher 假定了 CUDA-graph-only 语义
  • patch_kv_cache_coordinator → KVCacheCoordinator 的一些内部 state 没暴露给用户改
  • patch_mla_prefill_backend → MLA prefill 路径的 backend switch 没接口
凡是 patch 层密集的 subsystem，就是 vLLM 下一版 "extension point gap"。建议 vLLM 上游重点把 op-substitution / graph-backend / MoE-comm 三个方向做成正式扩展面——Ascend 的 patch 债务会立刻清掉。这个论断在 agent4 §Part C 被严格论证。

关键 insight：meta_registration.py 里的 register_meta_if_necessary 把 Ascend custom op 注册为Meta 函数，以便 torch.compile 在 tracing 期能运行——这是 "PyTorch + 私有硬件" 在 dynamo/compile 路上的必要解法，AMD/Intel 也面临同样问题。

11. 给推理架构师的设计洞见

一组设计是 "被物理逼出来的" 还是 "工程选的"，决定它的命运。 我按此把 vLLM 的设计分成两类——只有后一类是你重写推理系统时真正能改的：

被物理"逼"出（难改，必须满足）工程选择（可选，可换）
Paged KV-cache（I1: 内存动态 + append-only）Python 前端（可换成 Rust/Go 以求单线程更低 overhead）
attention 是唯一硬件相关 kernel（I1 + 算子分析）shm-RPC executor（可换成 NCCL-.comm、NCCL-based actor、mp+shared memory）
Token-budget 单步调度（I2: roofline decode↔prefill）采样器实现（可换成 GPU 双精度 gumbel-max、tree-spec sampling）
PagedAttention / content-hash 共享（I1: 有限 HBM）chunked prefill 的 threshold 默认值
三层图执行模式（I3: kernel launch overhead vs 动态 shape）混合模式的优先级
TP/PP/DP/EP composable group（I4: 规模）pipeline bubble 的填充策略
Platform 接口 = I3 的重心默认 backend 选择树
给正在规划下一代硬件适配团队的一句话： "5 个注册点(platform, AttentionBackend, KVConnector, QuantConfig, scheduler_cls)已经够让一个非 NVIDIA 后端零 hack 启动——余下的 op substitution、MoE comm、graph backend 三个方向缺少正式接口，这就是 patch 层存在的根本原因。你适配工作的真正不是在写 patch，是为 vLLM 上游标出 patch 的位置——每个被 patch 的子系统都是 vLLM 下一版 "extension point gap"。

roofline 看 Ascend 的调度器为何那么激进的解释：NPU(compute/bw) 比高于 GPU，意味着 memory bandwidth 是更紧的瓶颈；offload KV 到 CPU 的 PCIe 代价远大于让 NPU 重算一次——因此 Ascend 选择 recompute 而不是 swap，选择与 GPU "保留优先" 相反的 "扔掉-重算" 策略，这不是"偏好不同"，是 roofline 的最优解不同。同理，ShortRequestFirst ——短请求尽快出 TTFT，把带宽腾给长 decode ——也是带宽墙 I2 的必然推论。

效率数字：在 v1 的 3 段式 step loop（schedule - execute - update）:

  • schedule 每步 ≈ Python 开销 + allocation ≈ μs
  • 执行（model forward + attention + sample）≈ ms——99%+ 的 wall-clock
  • 所以 v1 所有调度器创新（recompute / SRF / chunked / spec decode）必须不引入可观测的 schedule 延迟；只要 schedule 开销在 μs，它就不影响 SLO——这是调度器创新的"黄金操作空间"。
