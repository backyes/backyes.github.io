三大核心支柱（来自官方摘要原文）[1]：

  1. 半自回归生成（Semi-Autoregressive Generation）——用一个重并行骨干（parallel backbone）做大部分 draft 计算（T_draft ≈ O(1)，与 block size 无关），再叠加一个轻量串行头（Markov/RNN head）注入 block 内 token 依赖，解决纯并行 drafter 的 suffix decay（后置位置接受率快速衰减）。
  2. 置信度调度验证（Confidence-Scheduled Verification）——一个 confidence head 估计每位置前缀存活概率，配合 hardware-aware prefix scheduler，把 verify 长度选择形式化为全局吞吐最大化问题，按实时 GPU 负载动态裁剪低置信后缀。
  3. DeepSpec 开源基础设施——MIT 协议的统一训练/评测代码库，含 DSpark、DFlash、Eagle3 三种算法，附带 Qwen3-4B/8B/14B、Gemma4-12B 的 HuggingFace checkpoints。[2]
"Speculative decoding accelerates Large Language Model (LLM) inference by decoupling draft generation from target verification. While recent parallel drafters efficiently propose long token sequences in a single forward pass, they suffer from rapid acceptance decay due to a lack of inter-token dependencies. Furthermore, indiscriminately verifying these extended blocks wastes critical batch capacity on tokens with high rejection risks, severely degrading throughput in high-concurrency serving systems. We introduce DSpark... To maintain draft quality, DSpark utilizes a semi-autoregressive architecture—coupling a parallel backbone with a lightweight sequential module... To optimize system efficiency, DSpark employs confidence-scheduled verification, dynamically tailoring the verification length for each request based on estimated prefix survival probabilities and engine-specific throughput profiles." — DSpark 论文摘要原文，arXiv 未收录，PDF 见 github.com/deepseek-ai/DeepSpec/DSpark_paper.pdf

核心结论（一句话）：DSpark 用「重并行骨干 + 轻量 Markov 串行头」在 T_draft ≈ O(1) 下逼近自回归 draft 的 coherence，再用 confidence head + hardware-aware scheduler 把 verify 从「固定长度」变为「按存活概率与引擎负载动态分配」——算法层解决 suffix decay，系统层解决高并发下静态多 token verify 的吞吐塌陷。它在 DeepSeek-V4 生产环境将 per-user 生成速度提升 60%–85%（同吞吐），并拓展了严格 SLA 下的服务能力前沿（Pareto frontier）。[1][5]

⚠️ 数据解读提示：所有 +85%/+78% 等数字均为 DeepSeek 自测，无第三方独立验证。论文中 +661%（V4-Flash @120 tok/s/user）与 +406%（V4-Pro @50 tok/s/user）出现在 MTP-1 基线接近失效的严格 SLA 点，应理解为「DSpark 拓展了可行服务边界」而非「快了 6 倍」。日常对比应看 matched throughput 下的 60%–85%。[1][7]

二、MTP 算法基础：定义、训练目标与推理投机解码范式

2.1 投机解码（Speculative Decoding）基本循环

标准自回归解码每生成 1 个 token 需 1 次 target model 前向，延迟与输出长度近似线性。投机解码（Chen et al., 2023; Leviathan et al., 2023）将其拆为两阶段[1]：

  • Draft（草稿）：轻量 drafter 一次提出 γ 个候选 token x̂₁, …, x̂_γ
  • Verify（校验）：target model 单次前向并行校验，按拒绝采样规则从左到右接受最长一致前缀，并附赠 1 个 bonus token

单轮平均每 token 延迟[1]：

L = (T_draft + T_verify) / τ （τ = accepted length，含 bonus token）

三条加速杠杆[1]：

  • 降 T_draft（draft faster）——如并行 drafter 把 T_draft 压到 O(1)
  • 提 τ（draft better）——如 DSpark 的串行头提升后缀接受率
  • 降有效 T_verify（verify smarter）——如 DSpark 的 confidence scheduler 动态裁剪

2.2 拒绝采样（Rejection Sampling）：无损校验的数学

无论 draft 来自 DSpark、EAGLE-3、内置 MTP 还是 Ngram，校验逻辑都遵循同一套投机解码框架，区别仅在 draft 如何产生、verify 前是否动态截断[5]。对每个 draft 位置 k，target 计算自身分布 p_t^k，draft 分布为 p_d^k，接受 draft token x̂_k 的概率[1]：

α_k = min(1, p_t^k(x̂_k) / p_d^k(x̂_k))

校验从左到右[1][5]：

  • 以概率 α_k 接受 x̂_k，继续检查 k+1
  • 一旦在位置 k 拒绝，则 x̂_k, x̂_{k+1}, …, x̂_ℓ 全部丢弃（后缀再好也无效）
  • 在拒绝位置 k，从调整分布重采样 1 个 token 作为纠正：q^k(v) = norm(max(0, p_t^k(v) − p_d^k(v)))

该性质保证：最终输出 token 序列的分布与 target 逐步自回归采样完全一致（无损 / lossless）。Greedy 解码（temperature=0）退化为简单比对——若 x̂_k = argmax p_t^k 则接受，否则拒绝并取 argmax 纠正。[5]

Bonus token：Verify 结束后，除被接受的 draft 前缀外，target 额外产出 1 个 token——若 x̂₁,…,x̂_m 全被接受则 bonus 来自位置 m 的 target 分布；若在位置 k 拒绝则 bonus 为从 q^k 采样的纠正 token。因此单轮 τ = m + 1。[5]

DSpark 的关键理论洞察（无损性前提）：CRF-NAT 也在并行 hidden states 上放串行模块，但其全局归一化 partition function 阻止精确 per-token 概率计算；CTC-drafter 因对齐路径的 latent marginalization 只能 greedy verify。DSpark 通过保持串行修正局部化，使 per-token 概率保持精确 softmax 求值，这是无损拒绝采样的前提。 — DSpark 论文 Section 6 Related Work，DSpark_paper.pdf

2.3 DeepSeek-V3 的 MTP 模块：权威定义（论文原文公式）

MTP（Multi-Token Prediction）一词在 DeepSeek-V3 论文中既是训练目标，也可复用为推理投机解码的 draft。以下为 DeepSeek-V3 技术报告 Section 2.2 的原文定义[3]：

"Inspired by Gloeckle et al. (2024), we investigate and set a Multi-Token Prediction (MTP) objective for DeepSeek-V3, which extends the prediction scope to multiple future tokens at each position. On the one hand, an MTP objective densifies the training signals and may improve data efficiency. On the other hand, MTP may enable the model to pre-plan its representations for better prediction of future tokens... Different from Gloeckle et al. (2024), which parallelly predicts D additional tokens using independent output heads, we sequentially predict additional tokens and keep the complete causal chain at each prediction depth." — DeepSeek-V3 Technical Report, Section 2.2，arXiv:2412.19437

MTP 模块结构

DeepSeek-V3 的 MTP 用 D 个顺序模块预测 D 个额外 token。第 k 个 MTP 模块包含四部分[3][8]：

  • 共享 embedding 层 Emb(·)（与主模型共享）
  • 共享 output head OutHead(·)（与主模型共享）
  • 一个 Transformer block TRM_k(·)
  • 投影矩阵 M_k ∈ ℝ^{d×2d}

对于第 i 个输入 token t_i，在第 k 个预测深度，先组合第 (k−1) 深度的表示 h_i^{k-1} 与第 (i+k) 个 token 的 embedding Emb(t_{i+k})（公式 21）[3]：

h'_i^k = M_k [ RMSNorm(h_i^{k-1}) ; RMSNorm(Emb(t_{i+k})) ]

（k=1 时 h_i^{k-1} 为主模型给出的表示。）组合后的 h'_i^k 作为第 k 深度 Transformer block 的输入（公式 22）[3]：

h_{1:T-k}^k = TRM_k( h'_{1:T-k}^k )

再由共享 output head 算第 k 个额外预测 token 的概率分布（公式 23）[3]：

P_{i+k+1}^k = OutHead( h_i^k ) （OutHead = 线性映射到 logits + Softmax）

MTP 训练目标

每个预测深度计算交叉熵损失（公式 24-25）[3]：

ℒ_MTP^k = CrossEntropy( P_{2+k:T+1}^k , t_{2+k:T+1} ) = −(1/T) Σ log P_i^k[t_i]
ℒ_MTP = (λ/D) Σ_{k=1}^{D} ℒ_MTP^k
"MTP in Inference. Our MTP strategy mainly aims to improve the performance of the main model, so during inference, we can directly discard the MTP modules and the main model can function independently and normally. Additionally, we can also repurpose these MTP modules for speculative decoding to further improve the generation latency." — DeepSeek-V3 Technical Report, Section 2.2，arXiv:2412.19437

这是理解 DSpark 的关键起点：DeepSeek 的 MTP 模块在训练时是提升主模型性能的辅助目标，在推理时可选择性地复用为投机解码的 draft。DeepSeek-V3 论文明确指出其保持因果链的原则与 EAGLE 类似，但 EAGLE 的首要目标是投机解码，而 DeepSeek 用 MTP 主要是为改进训练[3]。

2.4 "MTP"一词的两种含义（对比 DSpark 时必须区分）

业界"MTP / Multi-Token Prediction"常指两类不同设计，对比 DSpark 时必须分开[5]：

类型代表Draft 来源是否需额外训练
内置 MTP 头DeepSeek-V3/V4、Qwen3.5、GLM-5目标模型内嵌的 MTP 层预训练/后训练阶段已训好
外置 Draft ModelDSpark、DFlash、Eagle3独立小模型或附加模块需单独训练（DeepSpec 提供流程）

DSpark 属于外置 draft model路线，论文中的生产基线 MTP-1 则是 DeepSeek 内置的单 token MTP（静态 verify 2 token），二者不应混为一谈[5]。

The research line became more explicit in 2022. Heming Xia, Tao Ge and co-authors introduced SpecDec... 三、DeepSeek MTP 演进：V3 内置 MTP → MTP-1 → DSpark

3.1 第一阶段：DeepSeek-V3 内置 MTP（训练目标 + 可选 draft）

如第 2.3 节所述，DeepSeek-V3（671B 总参 / 37B 活跃 MoE）在预训练时设置 MTP 目标，主要 densify 训练信号、提升数据效率与表征 pre-plan 能力。推理时既可丢弃 MTP 模块让主模型独立运行，也可复用为投机解码 draft[3]。AMD ROCm 教程实测在 SGLang + 8×MI300X 上，DeepSeek-V3 启用 MTP（NEXTN 算法，lmsys/DeepSeek-V3-NextN draft，num-steps=2, num-draft-tokens=3）后，延迟与吞吐提升 1.2 到 2.1 倍[8]。

3.2 第二阶段：MTP-1 生产基线（保守的静态单 token）

尽管 V3 的 MTP 模块支持多 token draft，DeepSeek 在生产环境长期使用的是 MTP-1——仅 draft 1 个 token（+静态 verify 共 2 token）。DSpark 论文 Section 5.4 给出了原因[1]：

"This single-token setup was historically maintained in production because deploying a static multi-token drafter (e.g., MTP-3/5) strictly degrades aggregate throughput under high concurrency due to excessive verification overhead." — DSpark 论文 Section 5.4，DSpark_paper.pdf

即：静态多 token MTP（MTP-3/5）在高并发下因 verify 开销过大严格降低聚合吞吐，故生产长期保守用 MTP-1。MTP-1 并非"劣化版 DSpark"，而是 DeepSeek 在吞吐与延迟权衡下选的保守基线[5]。

3.3 第三阶段：DSpark 替换 MTP-1

"MTP-1 represents the former production setup, having been superseded by DSpark two weeks following the DeepSeek-V4-preview release." — DSpark 论文 Section 5.4，DSpark_paper.pdf

DSpark 在 DeepSeek-V4-preview 发布两周后即上线替换了 MTP-1。社区评论指出：若效果没有显著差距，DeepSeek 不会在生产环境做这种替换[7]。DSpark 的价值在于同时提高 acceptance 并控制 verify 浪费——在保持吞吐前提下解锁长 block（生产 γ=5，低并发扩到 4–6 token verify）[1]。

四、DSpark 深度拆解：半自回归 + 置信度调度

DSpark 单轮解码循环（来自论文 Figure 1 描述）[1][5]：

Target Model → 生成 anchor token D（上一轮 bonus token）
      ↓
[Parallel Backbone (DFlash 风格)]
   单次前向 → 全 block 隐状态 h₁…hγ + 基础 logits U₁…Uγ
      ↓
[Sequential Head (Markov / RNN)]
   自左向右采样，注入 token 间依赖 → 修正分布 p_k(·|x₀,x<k)
      ↓
[Confidence Head] → 每位置条件存活概率 c₁…cγ
      ↓
[Hardware-Aware Prefix Scheduler] → 动态截断 verify 长度
      ↓
Target Model 并行 verify 被保留的前缀 → 接受/拒绝 + bonus

4.1 半自回归生成（Semi-Autoregressive Generation）

4.1.1 动机：Parallel Drafter 的 Suffix Decay

纯并行 drafter（如 DFlash）在一次前向内独立预测 block 内各位置。当上下文存在多种合理续写（如 "of course" vs "no problem"）时，各位置对所有可能前驱做边缘化，易产生 multi-modal collision（如 "of problem"），导致 block 后部 acceptance 快速衰减[1][5]。

"实际数字：DFlash 在对话任务上，第 1 个位置猜对率大概 72%，到第 7 个位置掉到 63%。前面猜错一个，后面全部连带作废。" — wiselychen 技术解析，ai-coding.wiselychen.com/.../dspark-...（转述论文 Figure 2 数据）

纯自回归 drafter（如 Eagle3）依赖建模强，但 T_draft ∝ γ，只能用浅层网络和小 block。DSpark 的折中：重并行 + 轻串行[5]。

4.1.2 Parallel Stage（并行骨干）

基于 DFlash 架构，主要机制（论文 Section 2.2 + 3.1）[1]：

  • 从 target 若干层 {l₁,…,l_m} 抽取 hidden states，拼接投影后作为 KV 注入 draft 各层：H_ctx = RMSNorm(W_c[H(l₁);…;H(l_m)])
  • 注入方式：K_i = [W^K_i H_ctx ; W^K_i H_d], V_i = [W^V_i H_ctx ; W^V_i H_d]（沿 sequence 维拼接）
  • 全 block 双向注意力 + target context 注入
  • 共享 target 的 embedding 与 LM head（冻结）
  • 输入：anchor token embedding + (γ−1) 个 mask token embedding

相对原版 DFlash 的小改动：将 anchor 本身作为 block 第 1 个预测位，γ 个输入 token（anchor + γ−1 masks）产出 γ 个 draft logits，略减 draft 计算量[1]。

4.1.3 Sequential Stage（轻量串行头）

在并行骨干输出的 base logits U_k 上，叠加前缀相关的 transition bias B_k(x₀, x<k, x_k)，通过自回归因子分解得到 block 内因果分布（论文公式 4）[1]：

P(X|x₀) = ∏_{k=1}^{γ} p_k(x_k | x₀, x<k)
p_k(v|x₀,x<k) = exp(U_k(v) + B_k(x₀,x<k,v)) / Σ_{u∈V} exp(U_k(u) + B_k(x₀,x<k,u))

其中 x₀ = 上一轮 anchor，U_k = 并行骨干在位置 k 的 base logit，V = 词表。推理时按 p_k(·|x₀,x<k) 自左向右采样；因采样过程本身是序列的，串行模块必须计算轻量（T_sequential ≪ T_parallel）以保证整体 draft 延迟仍由并行阶段主导[1]。

4.1.4 Markov Head（默认，生产用）

B_k 仅依赖前一 token：B(x_{k-1}, x_k)。原理上是完整 V×V 矩阵，用低秩分解近似（论文公式 5）[1]：

B = W₁ W₂ , W₁ ∈ ℝ^{V×r}（embedding lookup）, W₂ ∈ ℝ^{r×V}（logit projection）
B(x_{k-1}, ·) = W₁[x_{k-1}] W₂ ∈ ℝ^V （r = 256 默认）

低秩分解使存储与每步计算都很小，大词表也可高效。论文举例：一旦位置 1 采样 "of"，Markov head 在位置 2 提升 "course" 抑制 "problem"，缓解 cross-mode collision[1]。

额外开销：约 0.2%–1.3% 相对 DFlash baseline[5]；wiselychen 转述论文实测为整体延迟仅增加 0.6%–1.3%[7]。

4.1.5 RNN Head（可选）

Markov head 超过一步即无记忆，RNN head 通过维护循环状态 s_k 累积 block 内完整前缀历史（论文公式 6）[1]：

输入: z_k = [s_{k-1} ; W₁[x_{k-1}] ; h_k] ∈ ℝ^{2r+d}
门控更新: s_k = σ(W_g z_k) ⊙ s_{k-1} + (1−σ(W_g z_k)) ⊙ tanh(W_c z_k)
B_k(x<k, ·) = W₂^T tanh(W_o z_k) （s₀ 初始化为零）

RNN head 在长 block 上 acceptance 略优，但部署复杂，论文默认 Markov[5]。值得注意的是，论文 Section 6 提到并发工作 Domino 引入的 CausalEncoder 概念上类似 DSpark 的 RNN Head[1]。

4.1.6 效果与关键对照实验

效果（Qwen3-4B）：DSpark 在 position 1 继承 deep parallel drafter 的高 acceptance（Math 约 0.93），同时在 position 2–7 抑制 DFlash 的快速衰减，曲线接近 Eagle3 的稳定性[5]。

💡 关键对照（论文 Figure 3）：2 层 DSpark 已优于 5 层 DFlash——加一个轻量修正的效果，比把并行模型堆厚 2.5 倍还好。这证明串行建模的参数效率高于堆叠并行层。 — DSpark 论文 Figure 3，转述自 知乎技术解析 与 wiselychen

4.2 置信度调度验证（Confidence-Scheduled Verification）

4.2.1 问题：固定长度 Verify 的浪费

即使 draft 质量好，盲目 verify 整个 block 在高并发下会浪费 target batch 容量：尾部低置信 token 几乎必被拒，却占用与其他请求共享的算力。领域差异也显著：Code/Math acceptance 高，Chat 低（论文 Table 1：Qwen3-4B 上 Math τ ≈ 5.57，Chat τ ≈ 3.49）[5]。

论文将瓶颈归因于两个相互作用因素（Section 3.2）[1]：

  • 数据侧：draft acceptance 跨域变化——结构化文本（code）acceptance 高，开放式 chat 显著低
  • 系统侧：verify 额外 token 的实际成本严格依赖引擎负载——轻负载下额外 verify 即使被拒也近乎免费；高并发下每个不必要 verify 都占用本可服务其他请求的 target batch 容量

4.2.2 Confidence Head

对每个 draft 位置 k 输出标量 c_k ∈ (0,1)，表示在前缀全部已接受条件下，位置 k 的 token 能通过 target verify 的条件概率（论文公式 7-8）[1]：

c_k = σ( w^T [h_k ; W₁[x_{k-1}]] ) （轻量线性投影 + sigmoid）
监督标签: c_k* = 1 − (1/2) ‖ p_d^k − p_t^k ‖₁ （draft 与 target 分布的总变差距离）

其中 h_k = 骨干 hidden state，W₁[x_{k-1}] = 上一 draft token 的 Markov Embedding。[1]

4.2.3 Sequential Temperature Scaling (STS) 校准

因为 hardware-aware scheduling 需要累积接受概率的绝对幅度（而非仅排名）来计算期望 τ，而神经置信度估计常过自信，直接使用会扭曲吞吐估计。STS 引入逐位温度校准（论文 Section 3.2.1）[1]：

  • 因 c_k 建模条件概率，链式法则下前缀被接受的联合概率 = 累积乘积 ∏_{i≤k} c_i
  • 用 held-out 验证集，自左向右逐位做 1D grid search 找最优温度，最小化累积乘积的 Expected Calibration Error (ECE)
  • 已校准的前序位置分数保持固定
  • 温度缩放是保序变换：修正概率幅度匹配经验接受率，但不破坏 confidence head 学到的相对 token 排名

知乎解析指出：STS 将 ECE 从 3%–8% 压到约 1%[5]。

4.2.4 Hardware-Aware Prefix Scheduler（Algorithm 1）

将 verify 长度选择形式化为全局吞吐最大化问题（论文 Algorithm 1）[1]。对请求 r，前缀存活概率与系统指标定义如下：

前缀存活概率: a_{r,j} = ∏_{i≤j} c_{r,i} （累积乘积，随 j 单调非增）
总 batch token 数: B = Σ_r (1 + ℓ_r)
期望接受 token 数: τ = Σ_r ( 1 + Σ_{j=1}^{ℓ_r} a_{r,j} )
引擎吞吐曲线: SPS(B) = steps/s（引擎初始化时 profile 一次，存为 cost table）
目标: 最大化 Θ = τ · SPS(B)

贪心解法（论文 Section 3.2.2）[1]：因为 a_{r,j} 随 j 单调非增，扩展请求 r 从 j−1 到 j 的边际增益恰为 a_{r,j}。这一单调性保证了全局按 a_{r,j} 降序排序候选 token 自然尊重 block 内前缀依赖。算法按 a_{r,j} 全局降序贪心扩展各请求的 verify 前缀，每次 O(1) 查表更新 Θ，在 Θ 不再上升时 early stop。

无损性保证：lossless speculative decoding 严格要求 non-anticipating property——admission 决策不依赖未来候选 token[1]。被 scheduler 保留并送入 target 的前缀仍按 §2.2 拒绝采样规则校验——调度只影响「验多少」，不改变「怎么验」，因此输出分布仍然无损[5]。

4.2.5 生产适配（Section 5.2 工程妥协）

线上 SPS 曲线非平滑、与 Zero-Overhead Scheduling (ZOS) / CUDA Graph 冲突，论文做了工程适配[5]：

  • SPS 曲线非平滑 → 去掉 early stop，做全局搜索
  • 与 ZOS / CUDA Graph 冲突 → 异步调度：用 2 步之前的 confidence 预测近似下一 step 的 batch 容量 K，当前 step 仍按实时 a_{r,j} 排序，形成因果屏障
"对话任务的验证通过率从 45.7% 提到 95.7%。不是猜得更准了，而是把没把握的猜测在送去验证之前就砍掉了，不浪费大模型的计算资源。" — wiselychen 转述 DSpark 论文，ai-coding.wiselychen.com/.../dspark-...

八、系统与芯片视角：batch capacity、SPS 曲线、硬件适配

本节面向系统与芯片研究人员，从 GPU 算力-访存-kernel launch 角度解析 MTP/投机解码的硬件影响，以及 DSpark 的系统级设计如何与硬件特性耦合。

8.1 为什么投机解码能加速：算力-访存的不对称

LLM 自回归解码的瓶颈通常不是算力而是访存带宽。每次生成 1 个 token 都需把整个模型权重（和 KV cache）从 HBM 读到 SM，而每个参数只参与极少 FLOPs——GPU 算力大量闲置。投机解码的本质是[6]：

"GPU 算得很快，但大部分时间都在等内存搬数据，像是一台跑车被困在红绿灯，油门踩不下去... 大模型一次验证 7 个字跟生成 1 个字的成本差不多（都是一次 forward pass），所以只要猜对率够高，速度就能翻好几倍。" — wiselychen 技术解析，ai-coding.wiselychen.com/.../dspark-...

即：verify γ 个 token 的单次前向，与生成 1 个 token 的单次前向，其权重访存成本几乎相同（都是一次模型 forward），但前者可产出最多 γ+1 个 token。只要 acceptance 够高，单位访存换取的有效 token 数大幅提升——这是投机解码在带宽受限场景下高效的根因，也是 MTP/DSpark 对芯片设计的核心启示：解码阶段的瓶颈是带宽而非算力，任何能"用算力换带宽利用率"的机制都有价值。

8.2 DSpark 的系统级核心：把 verify 长度变为调度变量

传统投机解码的 verify 长度是固定的（num_speculative_tokens 静态）。DSpark 的关键系统创新是把 verify 长度从"超参数"变为"在线调度变量"，并形式化为全局吞吐最大化[1]：

Θ = τ · SPS(B) （期望接受 token 数 × 引擎 steps/s）

其中 SPS(B) 是引擎吞吐曲线（给定 forward batch size B 时的 steps/s），在引擎初始化时 profile 一次存为 cost table[1]。这是一个硬件感知的设计：scheduler 直接读取该硬件/引擎组合的实际 SPS 曲线来做决策，而非假设理想线性扩展。

SPS 曲线的硬件含义

SPS(B) 本质反映了硬件的 batch 扩展特性[1]：

  • 小 B（轻负载）：算术强度低，SM 利用率不足，SPS 近似常数——此时额外 verify token 几乎免费（边际成本趋零），故 scheduler 把 budget 从 2 token 扩到 4–6
  • 大 B（高并发）：接近算力或显存带宽饱和，SPS 开始下降——此时每个 verify token 都有真实机会成本（占用可服务其他请求的 batch 容量），故 scheduler 收缩 budget

这意味着 DSpark 的收益高度依赖硬件 SPS 曲线的形状——曲线越平缓（小 B 区间越长），可激进扩展 verify 的空间越大；曲线越早饱和，scheduler 越早收缩。不同 GPU 代际（A100 vs H100 vs H800）、不同芯片（MI300X、国产芯片）的 SPS 曲线不同，DSpark 的实际收益会随之变化。

8.3 高并发下的 batch capacity 竞争

论文反复强调的核心矛盾[1]：高并发下，verify token 与活跃请求共享 target 的 batch 容量。

"Under high-concurrency deployments, every unnecessary verification occupies target model batch capacity that could otherwise serve other active requests." — DSpark 论文 Section 3.2，DSpark_paper.pdf

这是为什么静态 MTP-3/5 在高并发下严格降低聚合吞吐——每个请求盲目 verify 3–5 个 token，会把 target 的 batch 填满低质量候选，挤掉其他请求的 admission。DSpark 的 confidence scheduler 通过在 verify 前剪掉低置信后缀，把宝贵的 batch 容量留给"最可能被接受"的 token，等效于一个准入控制（admission control）机制[1]。

8.4 与 continuous batching / KV cache 的交互

  • Continuous batching：投机解码的 verify 阶段在 continuous batching 框架下表现为"一个请求一次前向处理 ℓ+1 个 token"。verify 长度动态化意味着 batch 内不同请求的 token 数不同，需要引擎支持变长 verify 的批处理。DSpark 的 scheduler 显式管理总 batch token 数 B = Σ(1+ℓ_r)[1]，与 continuous batching 的 token-level 调度天然契合。
  • KV cache：draft model 的 KV cache（尤其 DFlash 式多层 KV 注入）需要额外显存。DSpark 用 5 层 parallel backbone + Markov head，显存开销中等。hybrid 模型（如 Qwen3.5 MoE + GDN）在特定硬件上 DFlash 可能触发 KV cache page size 对齐问题，需调 --max-num-seqs[5]。
  • Tree attention：EAGLE 系支持 tree-based verification（多路径并行 verify），但论文指出"大量 verify token 数降低整体服务吞吐"[1]。DSpark 论文 offline 评测统一用 chain-based drafting（单链、无树）以公平对比[5]。

8.5 与引擎优化（ZOS / CUDA Graph）的冲突与妥协

现代推理引擎广泛使用 Zero-Overhead Scheduling (ZOS) 与 CUDA Graph 来消除调度开销、提升 GPU 利用率。但 DSpark 的 dynamic scheduler 与之冲突[5]：

  • CUDA Graph 要求静态 shape/计算图，而 DSpark 每步 verify 长度可变 → 直接冲突
  • ZOS 期望调度决策在 kernel 启动前确定，而 DSpark 的 confidence 依赖当前 step 的 draft 输出 → 时序冲突

论文的工程妥协：异步调度——用 2 步之前的 confidence 预测近似下一 step 的 batch 容量 K，当前 step 仍按实时 a_{r,j} 排序，形成因果屏障[5]。这是一个算法动态性 vs 系统静态性的典型权衡，对芯片/引擎协同设计有启示：未来硬件若能原生支持变长 batch 的高效 CUDA Graph 或动态调度，DSpark 类方法的工程妥协成本将大幅降低。

8.7 模型规模与收益关系

"模型越大收益越高。Speculative decoding 的原理是'大模型验证的边际成本低'，所以 target model 越大、drafter 跟 target 的速度差距越大，收益越明显。在小模型（7B 以下）上，drafter 本身的开销可能吃掉大部分收益。" — wiselychen 技术解析，ai-coding.wiselychen.com/.../dspark-...

这解释了 DSpark 在 DeepSeek-V4-Pro（1.6T/49B 活跃）与 V4-Flash（284B/13B 活跃）这类大 MoE 上收益显著——target 验证的单步成本远高于 draft 生成，加速比被放大。也意味着中小模型（≤7B）部署 DSpark 需谨慎评估 draft 开销占比[7]。

九、局限、未来方向与对开源生态的影响

9.1 论文自述局限（Section 5.4 Limitations）

"Although the prefix scheduler minimizes wasted target-model verification, DSpark still incurs a fixed draft-side cost to generate the initial γ-token block via the parallel backbone. For complex queries with inherently low acceptance rates, this upfront drafting compute is unrecoverable. Future optimizations could introduce difficulty-aware early exiting within the draft model, enabling such requests to bypass full-block generation." — DSpark 论文 Section 5.4，DSpark_paper.pdf

即：低 acceptance 的复杂查询，parallel backbone 的前期 draft 计算是固定且不可回收的浪费。未来方向是 difficulty-aware early exit，让难题跳过完整 block 生成[1]。

9.4 对开源生态与产业的影响

"过去 speculative decoding 的问题不是理论不成熟，而是工程门槛太高... 结果就是：论文很多，但真正在生产环境用起来的主要就是 Google（Gemini）跟 DeepSeek。DeepSpec 把大部分工作打包开源了，而且默认支援的就是社群最常用的 Qwen3 跟 Gemma4。落地门槛从'需要一个推理团队'降到'下载 checkpoint + 改配置'。" — wiselychen 技术解析，ai-coding.wiselychen.com/.../dspark-...

核心影响[6][7]：

  • 跨模型通用：DSpark 不绑 DeepSeek 自家模型，Qwen3、Gemma4 均有现成 checkpoint；企业可为自己托管的开权重模型（Llama、Mistral、Granite、Command 等）训练 DSpark-style draft[6]。
  • 开源 vs 闭源：DSpark 适用于企业控制权重和 serving 栈的场景；对纯 API 客户（无法访问 token verify loop、logits、batching、scheduler）无法从外部启用[6]。这强化了开权重/自托管 AI 基础设施的理由。
  • 推理优化成新战场：VentureBeat 评述"decoding efficiency is becoming another major battleground"——Faster generation 意味着更低用户延迟、更高 provider 吞吐、更好经济性[6]。
  • 填平护城河："推理加速过去一直是闭源厂商的护城河。DSpark 把护城河填平了一块。"[7]
  • 生态集成预期：若 vLLM/SGLang 集成 DSpark（考虑 MIT 协议，"只是时间问题"），开源模型推理速度将集体提升[7]。

9.5 未来研究方向（综合论文 + 社区）

  1. Difficulty-aware early exit：低 acceptance 请求跳过完整 block 生成（论文明确）[1]
  2. Scheduler 近似改进：异步 2-step 延迟、非平滑 SPS 曲线的工程妥协，理论最优与线上实现存在 gap[5]
  3. 多级 proposer：Ngram 命中 → 回退 DSpark/MTP，需框架支持[5]
  4. thinking mode 适配：reasoning 模型需重新 fine-tune draft[2]
  5. draft tree + scheduler 融合：DSpark 论文用单链公平对比，生产可探索 draft tree 与 scheduler 结合[5]
  6. 芯片协同设计：原生支持变长 batch CUDA Graph / 动态调度的硬件，可消除异步调度妥协（见 §8.5）
