That means it is still easy to end up in situations like:\n\nprefill is saturated while decode is relatively idle\ndecode is saturated while prefill is underutilized\ndecode-side KV capacity becomes the dominant bottleneck even when compute is still available\nremote KV transfer backlog creates hidden stalls between the two pools\n\nSo from my perspective, PD solves mixed-pool interference, but not yet dynamic imbalance between the pools.\n\nProposal\n\nIntroduce a bottleneck-aware expansion layer for PD disaggregation.\n\nA simple first version could classify runtime pressure into three categories:\n\nPrefill-bound\nPrefill-side compute or memory pressure dominates.\n\nDecode-bound\nDecode-side token throughput or scheduling pressure dominates.\n\nDecode-KV-bound\nDecode-side KV holding capacity dominates, even if decode compute is not yet saturated.\n\nThen the runtime (or a future policy layer built on top of it) could choose the most effective expansion action:\n\nPrefill-bound -> add prefill capacity\nDecode-bound -> add decode capacity\nDecode-KV-bound -> add decode-side KV capacity / tier\n\nThe key point is that this would make PD more adaptive without changing the core PD architecture.\n\nSuggested First Scope\n\nA practical first step might be:\n\nexpose structured runtime pressure signals\nclassify bottlenecks into Prefill-bound, Decode-bound, or Decode-KV-bound\nsupport one minimal expansion hook or policy target\n\nEven just that would already make PD more adaptive, without requiring full runtime elasticity.\n\nNon-Goals\n\nThis is not proposing:\n\nreplacing PD disaggregation\ngeneric runtime GPU hot-plug\nruntime TP/NCCL topology mutation within an active worker group\nfull cluster autoscaling inside vLLM\nRelated Work\n\nThis seems closely related to, but different from, the following threads:\n\nPD disaggregation roadmap: [Roadmap]: PD Disaggregation with NixlConnector
