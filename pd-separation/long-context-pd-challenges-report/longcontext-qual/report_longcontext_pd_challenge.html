核心判断：这是一个尚未被充分研究的新问题

结论先行："1M+ tokens × PD disaggregation"在 2026 年中仍是一个新浮现、未被系统化解的交叉问题域。

  • 过往 PD 论文（DistServe 2024、Splitwise 2024、Tetrathlon 2025、Mooncake 2024）默认研究的是短上下文（<128K），其核心假设是 prefill compute-bound + decode memory-bound 的二元对立在长上下文下发生根本变异。
  • 2026 年 Q1–Q3 才出现首批显式讨论"long context + PD intersection"的学术工作（Lynx, SAC, 3DLS, PrfaaS, Dual-Pool Routing），这表明该交叉方向进入学术视野的时间窗口不到 12 个月。
  • 工业界态度：SGLang 和 vLLM 把 1M+ context 视为 PP (Pipeline Parallelism) 问题而非 PD 问题设计，暗示业界尚未在 PD 范式内找到 1M 的自然解法。
深层洞察：传统 PD 分离的理论基础——"prefill 是 compute-bound、decode 是 memory-bound"这个二分法——在 1M+ 上下文下崩溃。当输入序列长达 1M tokens 时：
(1) Prefill 同时成为 memory-bounded（KVCache 在 prefill 阶段就开始膨胀）；
(2) Decode 在读取 KV 时面临长序列二次注意力的带宽-容量双杀；
(3) KV 在 P→D 之间的传输本身成为系统首个 GB 级瓶颈。

关键研究列表（arXiv 论文纪要）

A. 直接研究"long context + PD disaggregation"交集的工作

arXiv ID标题核心贡献 + 与 PD + 长 context 的关联
2604.15039 PrfaaS: Prefill-as-a-Service
Ruoyu Qin et al., Moonshot AI, 2026-04		 直接指出：在 Long-context 场景下，传统 dense attention 模型的 KV Cache 量使 prefill 与 decode 被迫紧耦合在同一高带宽域内。PrfaaS 通过 hybrid attention 压缩 KV 后，将 prefill offload 到独立集群，经以太网传给本地 decode。关键洞察：长 context PD 跨 DC 部署的瓶颈不在 KV 大小本身，而在 selector 调度 + 带宽感知 + cache 感知 placement 的协同。
2607.01831 Lynx: Progressive Speculative Quantization for KV Transfer
Wenchen Han et al., SIGCOMM'26 submission, 2026-07		 直接切中 PD 长上下文最大瓶颈——KV cache 在 P→D 网络传输的 TTFT。Lynx 挑战"KV cache 是一个不可分割的整体"假设，将 KV 拆为 Anchor(MSB)+Residual(LSB) 双流：先收 Anchor 就开始 speculative decode，Residual 并发传、最后验证。在 disaggregated inference 下，TTFT 接近 4bit 量化水平，精度等同 BF16。
2606.19746 SAC: Disaggregated KV Cache System for Sparse Attention LLMs with CXL
Ruiyang Ma, Teng Ma et al., 2026-06		 揭示传统 RDMA-based disaggregated KV pool 对 sparse attention 根本浪费——长 context 下只有 top-k KV entries 有效，但 RDMA 要把整块 prefix KV 取回。SAC 用 CXL 细粒度 load/store 只在需要时取 top-k 条目。对 PD 长 context 架构有范式启示：CXL 让 decode 端按需读取成为可能。
2607.01617 3DLS: 3D Logic-Stacked Architecture for Disaggregated LLM Serving
Jaehun Lee et al., IEEE CAL 2026, 2026-07		 首次从芯片层面揭示 PD 长上下文的核心冲突：P→D KV transfer 与 decode TP collectives 共享 die-to-die 互连，混合流量抢占 decode critical path。3DLS 用 3D 堆叠将 KV 转移走垂直互连，decode TP fabric 独立。提示：长 context PD 的瓶颈不仅在系统层，已在封装互连层显现。
2604.08075 Dual-Pool Token-Budget Routing for Cost-Efficient LLM Serving
Xunzhuo Liu, Huamin Chen et al., 2026-04		 显式指出传统 PD production fleet 为最坏上下文配置资源导致 4-8x 吞吐浪费、OOM、preemption 频发。提出按 token budget 路由到 short-context pool / long-context pool，与 PD disaggregation 无缝组合。洞察：长 context 请求的容量规划本质是让 fleet 承受 80-95% short + 5-15% long 的 mix 调度。

B. 长上下文推理系统基础设施层

arXiv ID标题核心贡献 + 与 PD 交集的关联
2511.00321 Scalable PNM for 1M-Token LLM Inference (CXL)
Dowon Kim, Jungwook Choi et al., 2025-11		 针对 1M token 推理设计 CXL-PNM 系统：将 token page selection offload 到 CXL 内存内的 PNM 加速器，消除 KV recall 成本。PNM-KV 和 PnG-KV 在 405B 1M token 下取得 21.9x 吞吐提升。对 PD 架构的启示：如果 P 端能把 KV 压缩/筛选后才传 D，传输和内存瓶颈同时缓解。
2407.00079 Mooncake: A KVCache-centric Disaggregated Architecture
Ruoyu Qin et al., Moonshot AI, 2024-06 (v4 2025-09)		 Kimi 生产平台，首个提出 overload early rejection 政策。长 context 下 Mooncake 的 Conductor 必须预测 future KV block usage + 未来 generation length 来决定拒绝/接受。chunked pipeline parallelism (CPP) + layer-wise prefill 是针对长 context TTFT 的专门机制。基准文献：Mooncake 是 PD 长 context 研究绕不开的参考点。
2512.03324 Cache What Lasts: Token Retention for Memory-Bounded KV Cache
Ngoc Bui et al., 2025-12		 针对 long-horizon LLM 推理的 KV cache 容量有界场景，提出 token retention 策略。与 PD 的关联：当 decode 端 HBM 无法容纳 1M token 全量 KV 时，该工作给出 eviction policy 的原则性框架。
2601.22001 Heterogeneous Computing: Key to AI Agent Inference
Aaron Zhao (Imperial), Junyi Liu (MSR), 2026-01		 明确定义 Capacity Footprint (CF) 指标——每个请求在 DRAM 中需要的 bytes 数——指出 AI agent 推理已撞上 memory capacity wall，PD disaggregation 的部署必须按 OI/CF 二维配置异构加速器。

工业界公开讨论与产品实践

① SGLang 社区视角

关键信息源：Pipeline Parallelism for Long Context 文档（直接打开） + PR #8846（Support PD disaggregation with Prefill PP，2025-10 合并）

PPT 文档明确指出：

"KV cache techniques effectively mitigate redundant computation, but they cannot circumvent the prohibitive TTFT inherent in ultra-long sequences with extremely large initial Input Token Length (ITL)."

核心链条：

  • SGLang 认为 1M+ context 的问题首先是 PP 问题——TP 跨节点有通信瓶颈，PP 只需 stage 边界通信，更适合长上下文。
  • 但 PP+PD 的组合直到 2025-10 的 #8846 才首次支持（"Support PD disaggregation with Prefill PP"），且 Decode PP 至今未支持（"Decode PP will be supported in the next PR"）。
  • 为了减小 pipeline bubble，SGLang 提出 Dynamic Chunking 机制——因为 attention 复杂度随 prefix 长度增长，chunk 大小需要动态递减，这揭示长 context prefill 的非均匀计算分布加剧了 PD 调度的难度。
  • Micro-batching Event Loop + async P2P通信是现阶段解法，说明P→D KV 转移的延迟隐藏是工程瓶颈。

观察：SGLang 从未在文档中声称 PD disaggregation 是长 context 的天然解法；相反，社区对 1M context 的讨论几乎全部在 PP 范式下进行。PD 分离在 1M 场景下为何需要 PP 帮忙？这本身就说明 PD 架构面对长 context 时的原生瓶颈。

② vLLM 社区视角

关键信息源：GitHub Issue #11286 [Performance]: decoding speed on long context（2024-12 开）

Issue 实测数据揭示了长 context decode 的断崖式下降（单 A800 80G, bs=90）：

Prompt length0-100100-200500-1k1k-2k2k-4k4k-8k8k+
Decoded tokens/s53645500281524841627782273

vLLM 维护者 noooop 的官方解释（直接译文）：

"Most of the time, the GPU meets bandwidth bottleneck rather than computing bottleneck. As the prompt length increases, the size of the kv cache that needs to be read even exceeds the size of the model. This is why inference speed decreases. Inference latency increases linearly to the context size, primarily due to the time needed to access cached tokens."

对 PD 架构的启示：

  • 上述数据是共置引擎的观测，PD 分离下 D 端的 KV 同样遇到该带宽瓶颈，且更严重——因为 KV 还要从 P 端跨网络传输而来。
  • chunked_prefill 在长输入下更优，但用户报告"仍无用"——暗示vLLM 默认 scheduler 在长 context 下 bubble 问题主导。
  • vLLM 目前没有针对长 context 的 decode 端 HBM capacity planning 专门的 issue 或文档，--block-size 仍是主要调优参数，未发现类似 SGLang PP 的专门配套。

vLLM Semantic Router（arxiv:2604.dualpool 同源工作）：xunzhuo 系列的 Token-Budget Routing 指明 fleet 中 80-95% 是短请求，为最坏长 context 配置 fleet 会导致 4-8x 吞吐浪费。

③ Mooncake / Kimi 视角

关键信息源：Mooncake HTML 全文 + PrfaaS (arxiv:2604.15039) + SGLang #8846

Mooncake 生产视角下长 context PD 的独有挑战：

  • Overload-oriented Scheduling：Mooncake 论文明确指出"existing research assumes all requests will be processed"——只有 Kimi 因用户量暴涨 facing severe overload。长 context 加剧 overload：每个请求占据 decode slot 期间还要持续消耗 HBM 容量。Early rejection policy 必须预测 generation length。
  • Prefill + 长 context 的相互恶化：Conductor 要尽可能复用远程 KV Cache 以减少计算，但"waiting for KVCache stored on lower-tier storage may violate TTFT SLO"；同时 "high demand on KVCache server can lead to network congestion"。
  • Decode VRAM 硬上限："aggregation 受 TBT SLO 与 decode 端 VRAM 可容纳 KV 总量双重限制"。当 batch size × context length × KV/ token > VRAM 时 decode 端直接饱和。
  • Chunked Pipeline Parallelism (CPP) + Layer-wise Prefill：Mooncake 专为长 context TTFT 设计的机制，将 KV 流式传输到 decode，与 SGLang 后来的 PP for Long Context 不谋而合。
  • PrfaaS (2026-04)：Moonshot 近作进一步承认——长 context dense attention 下的 KV 迫使 P/D 紧耦合在同一个 RDMA 域；hybrid attention 使跨 DC 部署成为可能，但non-trivial："real workloads remain bursty, request lengths highly skewed, prefix caches unevenly distributed"。
Kimi/Mooncake 的核心观察：长 context PD 的真正瓶颈不在单次请求内部，而在请求分布偏斜 + overload 交互。当 fleet 面对 5-15% 的 1M+ 请求 + 85% <1K 请求混合时，传统 PD 的静态配置必然崩溃。

④ AWS / Meta / NVIDIA 视角

  • NVIDIA Dynamo (路由层)：目前公开的 Dynamo 讨论聚焦 KV-aware routing（cache overlap + active load 复合指标），但未发现专门针对 1M+ context 的 PD 路由策略公开文档。Rubin CPX 是 NVIDIA 首个"prefill 专用加速器"（2025 台北展），隐含的叙事是长 context prefill 的算力需求已需要专用芯片。
  • Meta：PyTorch Blog 2025-09-12 披露的 Service Proxy at Decode + cache-aware LB，upstreaming 到 vLLM；同样未公开长 context 专项讨论。
  • Alibaba PAI-EAS：三种静态/动态 PD + EP 模式可配，但无 1M+ context 专项资料。
  • AWS：Dynamo-style router = control plane + token threshold 路由；未见到类似 Mooncake early rejection 的公开材料。

六大核心挑战的定性剖析

挑战 1：D 端 HBM 的容量压力 vs 共置推理的差异

PD 分离相对于共置推理的独特挑战：

共置（chunked PD 同机）PD 分离（跨机）
KV 生存空间 P 和 D 共享同一 GPU 的 HBM，prefill 结束后 KV 本地可见 KV 必须跨网络传输到另一台 GPU；传输后的 KV 占 D 端 HBM，且 D 端 无法释放 直到请求结束
连续 batching 操作简单，KV local 自取 必须对抵达的 KV 预分配 decode 缓冲（prefetch-aware admission）
容量浪费 KV pool 被所有请求共享 每台 D 节点按最大上下文预留 KV pool——长 context 导致 pool 5-10x 膨胀
Overload 表现 OOM 后拒绝即可 P→D 传输中才 OOM → +network 浪费 + decode slot 占用

判定：PD 分离下长 context 的 HBM 容量压力不仅是绝对量的放大，更是调度时序问题——KV 在传输中不可被 decode 使用，但已在目标节点被预留。

挑战 2：KV 随序列长度线性膨胀在 PD 二元下的失配

传统 PD 理论基础——prefill 是 compute-bound，decode 是 memory-bound——在长 context 下在两个阶段同时失效：

  • Prefill 端：attention 是 O(N²) 计算，1M token prefill 的 compute 与 memory 同时爆炸（即使 chunked prefill，prefill 依然需要暂存中间 KV）；
  • Decode 端：曾经每个 batch 只需 attention 到当前 token 的固定成本，现在必须 attention 到 1M+ 历史 token——decode 不再是简单的 memory-bandwidth-bound；它变成了长序列 read-bandwidth-bound + capacity-bound 双杀；
  • 系统层面：the ratio of compute-to-memory capacity（OI/CF）在长 context 请求间变化 10-100x，令 fleet 配置困难 (arXiv:2601.22001)。
核心洞察：长 context 下 prefill 和 decode 都从 compute/memory 二维平面的不同象限向"高 compute + 高 memory"象限坍缩，PD 分离本应带来的"各取所长"优势被压缩。

挑战 3：P→D KV 传输——GB 级阻塞

以 Llama-3-70B（80层, 128 head dim=128, GQA 8 组）为例估算：每个 token 的 KV ≈ 80 × 8 × 128 × 2 × 2 bytes ≈ 320KB。

  • 1M token → 320GB KV 共（超过任何单机 HBM，必须分片）。
  • 即便是 100K token，KV 也 ≈ 32GB，单节点 HBM 即被占满。
  • Lynx (arxiv:2607.01831) 直面此问题：challenge 了 "KV cache 不可分割" 的隐含假设，但这是 2026-07 才出现在 SIGCOMM 的新工作。

PD 分离特有现象：KV 不仅要跨网络传输，还要按 batch + layer 顺序组织以便 decode attention 直接读取。这进一步增加了 decode 端的 buffer 管理复杂度。

挑战 4：prefix cache 效率随 context 长度升高而退化

  • SGLang 的 RadixAttention 在 short context 下 hit rate >80%，但在 long context 下，prefix 块极易超过 DRAM/SSD 容量被驱逐——每个 long request 占据大块 radix tree，子节点极少被复用。
  • Mooncake 三级存储（GPU HBM → CPU DRAM → SSD）中，long context 占用 SSD tier 后，即便 HBM 空闲，KV 仍需从 SSD->DRAM->HBM 多跳读取。
  • SAC (arxiv:2606.19746) 是首个针对该问题从 CXL 语义出发的 disaggregated KV 系统，暗示 RDMA 时代 PD 的 prefix cache 假设已不可扩展。

挑战 5：fleet 级长尾调度崩溃

Dual-Pool Token-Budget Routing (arxiv:2604.08075) 直接指出：

  • 生产 fleet 按 worst-case context 配置 → 80-95% <1K 请求都在"长上下文模式"下运行，浪费 4-8x 并发能力。
  • 长请求的 KV pool 占满 decode 节点后，短请求被 preemption → 5.4x preemption rate 提升。
  • PD disaggregation 假设 prefill / decode 解耦后各自独立调度，但长 context batch 实际上让 decode 变成了 queue-length-dominated 系统。

挑战 6：early rejection 的副作用

Mooncake (arxiv:2407.00079) 观察到一个反直觉现象：

"A straightforward implementation of such an early reject policy surprisingly leads to fluctuations in the overloads."

原因：长 context 请求在 prefill 完成后才知是否能在 decode slot 中被接受。prefill 是 expensive compute，如果被 reject，已消耗的计算被浪费、网络带宽被占用、decode 端的期望 batch 被破坏。

前沿解决路径与范式判断

路径 1：CXL 原生 disaggregated KV 系统（SAC 范式）

SAC (arxiv:2606.19746) + PNM-1M (arxiv:2511.00321) 代表一条路径：让 KV 留在远端，只在 decode attention 需要时 cache-line 粒度读取。对 PD 的影响：

  • D 端不再需要全量 KV 驻留 HBM——HBM 从"KV pool"降级为"KV cache"。
  • 代价：每次 attention 都要 CXL 读取延迟（~100ns vs HBM ~50ns），仅对 sparse attention 可行。

限制：CXL 生态成熟度、dense attention 无解。

路径 2：KV 压缩+流式量化（Lynx 范式）

Lynx (arxiv:2607.01831) + Token-Budget Routing (arxiv:2604.08075) 代表系统层压缩路径：

  • Anchor + Residual 双流让 decode 在 10% KV 到达时就开始工作——TTFT 接近 4bit 量化但精度等同 BF16。
  • Token-Budget Routing 按预算路由让长 context 进专用 pool，避免fleet 层污染。

限制：仅解了 P→D 传输瓶颈，D 端 HBM 容量瓶颈仍存。

路径 3：3D 芯片隔离（3DLS 范式）

3DLS (arxiv:2607.01617) 代表硬件层路径：将 P→D KV 转移走垂直互连，decode TP 留在 lateral fabric，避免混合流量竞争。

  • 这是首个承认"PD 长 context 瓶颈已到达封装互连层"的工作。
  • 吞吐比共享 fabric 基线高 1.49x，比 workload-aware priority 基线高 1.17x。

限制：尚未流片，2-3 年内不可生产。

路径 4：hybrid attention 架构（PrfaaS 范式）

Moonshot 的 PrfaaS (arxiv:2604.15039) 承认：dense attention 下的 PD 长 context 是无解的——必须依赖 hybrid attention（如 Jamba、DeepSeek-V3.2 的 sparse attention）压缩 KV 才可能。

范式判断：工业界正在从"PD 系统层面解决 long context"向"模型架构 + 系统协同"视角迁移。

路径 5：overload-aware 全局调度（Mooncake → PrfaaS 连贯路径）

Mooncake 的 Conductor + early rejection + 预测未来 generation length，演进到 PrfaaS 的跨 DC KV 选择性 offload——这是唯一一条被商业产品验证的路径。

内在限制：overload 预测依赖 workload pattern，面对 bursty long context 流量（agentic 系统）预测准确率下降。

SGLang + vLLM 的工程层配套现状

  • SGLang：PP for Long Context 文档承认 TP 跨节点不适合长上下文，PP 是正确答案；但 PP + PD 组合直到 2025-10 的 #8846 才初步支持 Decode-side PP 至今 pending。Dynamic Chunking 部分缓解 pipeline bubble，但无法突破单 request KV 容量墙。
  • vLLM：Semantic Router（Dual-Pool Routing 同族）+ NIXL push/pull KV transfer，没有 1M+ context 专项优化路径。--block-size=128 是当前仅有的长 context 参数。
  • NV Dynamo：KV-aware Smart Routing 是行业标准路由，但未公开 1M+ 专项策略。
