超长序列下的 P/D 分离架构挑战

📅 2026-07-15 🎯 学术 + 工业双视角 📎 子报告：📚 资料全索引 · D 端内存瓶颈 Issue · vLLM D 端架构源码 · 超长序列定性研究 · 工业界引述
本报告核心问题：当 LLM 推理的序列长度达到 128K → 1M tokens，P/D 分离（Prefill-Decode Disaggregation）架构面临哪些研究上尚未完全解决的行业挑战？
调研发现这不是单一问题，而是互相缠绕的三重压力： ① D 端 KV Cache 膨胀导致的容量瓶颈（Decode-KV-bound，vLLM RFC 已正式命名）
② P→D 超长 KV 传输带来的时间压力（TTFT 主导项从计算变成传输）
③ 长序列 attention 自身带来的计算图变化（retraction、chunked prefill、低效的 attention kernel）

1. 问题框架：为什么"长序列 × PD 分离"是当前研究焦点

1.1 背景：PD 分离已是 LLM 推理事实标准

DistServe (OSDI'24) 确立 P/D 分离范式后，SGLang、vLLM、Mooncake、NVIDIA Dynamo 等开源 / 商业系统全面跟进，Meta / Alibaba PAI / AWS / Volcengine / Moonshot Kimi 均已落地部署。PD 分离的常态输入长度已从 2024 年的 2-4K tokens，上升到 2025 下半的 32-128K tokens，1M 级别已列入 2026-2027 年产品规划（Kimi K2.5、Claude、Gemini 系列在 2025 年先后宣布 1M 上下文）。

1.2 新矛盾：PD 分离 + 长序列 = 压力叠加而非简单相加

长序列在共置（colocated）推理中已带来挑战；但在 PD 分离下，这类挑战会被放大，因为：

  • Prefill 端的巨大 KV 必须完整搬到D 端 —— 传输路径成为新的主导延迟项。
  • D 端必须为每个长请求预留大量 HBM给 KV，但 D 端的高利用率靠的就是小 KV batch；长序列让这个前提失效。
  • 共置下可以通过"长请求 pair 短请求"让 P 和 D 错峰；PD 分离后两者资源池独立，长请求无法被"错峰缓冲"。

1.3 三重压力的关系

压力 ① D 端 KV 容量压力 ② P→D KV 传输压力 ③ 长序列 Attention 计算
在 PD 分离下被放大机制D 端 HBM 必须同时存放所有活跃请求的 KV，长请求会让 batch size 急剧压缩128K+ 的 KV 经网络搬运，传输时间可能超过 prefill compute 本身Decode 每一步都要对完整 KV 做 attention，decode 时长随 sequence 线性增长
关键定性阈值≈ 128K（GQA 模型 D 端从并行退化为串行）≈ 64K（传输开始主导 TTFT）≈ 32K（标准 FMHA kernel 效率开始退化）
是否有研究未覆盖区有（异构 KV pool sizing 是 open 问题）多数已有工作计算 + 路由联合优化是 open 问题

2. 第一重压力 — D 端 KV 容量瓶颈（Decode-KV-bound）

2.1 学术圈与工业界的"官方命名"

vLLM 社区 RFC #43470 "Bottleneck-aware expansion for PD disaggregation"（2026-05-22, author: zsn6493）首次将 PD 运行时的压力分为三类：

  • Prefill-bound：P 端 compute/memory 压力主导 → 加 P 容量
  • Decode-bound：D 端 token 吞吐 / 调度压力主导 → 加 D 容量
  • Decode-KV-bound：D 端 KV holding capacity 主导，即使 compute 未饱和 → 加 D 侧 KV capacity / tier

关键原文："decode-side KV capacity becomes the dominant bottleneck even when compute is still available; remote KV transfer backlog creates hidden stalls between the two pools"

协作方 robertgshaw2-redhat（llm-d 组织，与 NVIDIA Dynamo 同源）回应：该能力应放在编排层（llm-d flow-control / Dynamo 路由），vLLM 作为执行层只需"暴露压力信号"即可 —— 这一分工共识说明KV 容量调度已从系统问题提升为编排层问题。

源：github.com/vllm-project/vllm/issues/43470

源：#30010（2026-06）

2.3 SGLang GB300 Long-Context Blog：长序列 decode 被认为是 KV-dominated workload

SGLang 在 GB300 NVL72 长上下文博客（2026-07-08）中明确写道：

"Long-context decode quickly becomes KV-dominated and memory-bound: each new token repeatedly reads the full KV history, so both KV capacity (how many sequences can stay resident) and HBM bandwidth become the primary bottlenecks."

团队列出的 key techniques 直接针对 D 端容量：

  • Wide-EP Scaling：Expert Parallelism + DP attention 把 KV 分布到更多 GPU（最多 32 个），"reducing memory pressure per GPU and allowing larger decode batch sizes without triggering retraction"（retraction = 因为容量不足把 KV 丢弃再重算）。
  • GB300 的 288 GB HBM3e（vs GB200 192 GB）直接给 D 端更多"长序列预算" — 但团队指出"mem_fraction_static=0.75 对比下 GB300 的大内存允许设更高"。即：HBM 容量仍然是长序列 PD 的硬约束，即使每台 GPU 有 288 GB，设置时仍要被 KV pool 吃掉 ~75%。

源：Deploying DeepSeek on GB300 NVL72: Big Wins in Long-Context Inference（LMSYS 2026-07-08，此前已抓取归档为 blog_gb300_longctx.md）

3. 第二重压力 — P→D KV 传输的时间成本

3.1 为什么 P→D 传输在长序列下成了 TTFT 主导项

在 PD 分离下，D 端必须"收到完整 KV Cache"才能开始 decode，而这个等待时间直接加在 TTFT 上。共置推理没有这个成本（KV 就在本地显存里）。

NetKV（2606.03910）论文证明：

"Disaggregated LLM inference forces the KV cache to traverse the datacenter network before decoding begins, so transfer time enters directly into the Time to First Token (TTFT) budget."

更进一步的结论：随着上下文长度增长，仅靠 cache-aware（KV 亲和性）和 load-aware（负载均衡）调度会"arbitrarily suboptimal" —— 必须加入网络拓扑感知（network-aware）的代价函数。这说明在长序列 + PD 分离下，P→D 的数据路径本身从"完美网络假设"退化为"实际拓扑相关性"，这是共置推理不需要面对的挑战。

源：NetKV (2606.03910)

3.2 学术研究前沿：长序列 P→D 传输缓解三件套

路径代表工作关键思路长序列下适用性
KV 渐进传输Lynx (2607.01831)把 KV 拆为 Anchor（高位，优先）+ Residual（低位，投机 decode），D 端收到 Anchor 就开始 decode直接针对长序列 decode 不能等完整 KV 的痛点

3.3 跨 DC P→D 的极端情况（1M+ tokens 直接跨城）

Prefill-as-a-Service (PaaS, 2604.15039) 把 P 端和 D 端物理拆分到不同 DC，之间走 commodity Ethernet：

"... PD disaggregation has become the standard architecture for large-scale LLM serving, but in practice its deployment boundary is still determined by KVCache transfer. ... real workloads remain bursty, request lengths are highly skewed, prefix caches are evenly distributed, and inter-cluster bandwidth fluctuates."

对长序列研究者的启示：长序列下的 P→D 路径已经从"机内 NVLink" → "机架 RDMA" → "DC 内 RoCE" 一路退化到 "跨 DC Ethernet"。这是极端但现实的场景，在研究长序列 PD 时必须把网络距离作为一等参数，而不是 off-band 假设。

源：Prefill-as-a-Service (2604.15039)

4. 第三重压力 — 长序列 Attention 自身的反客为主

4.1 为什么长序列下"retraction"成为 D 端的隐性成本

在序列长度远大于 GPU cache 容量时，D 端只能保留最新的 K 个 token（sliding window），一旦 decode 窗口滑出 KV pool，需要丢弃旧 KV 并在重新需要时 recompute——这就是 retraction。

GB300 博客原文明确提及这一危险：Wide-EP 的目标之一是 "without triggering retraction"。说明即使 288 GB HBM 的 GPU，retraction 在 128K 工作负载下仍是一个现实风险。

SGLang #31252 内也讨论了 HiCache 方案——Prefill 端用根号的 KV cache 表示（Lora-style），Decode 端只在需要时 reconstruct —— 这是对 retraction 问题的一个重构方案。

源：#31252（HiCache discussion） · GB300 博客

4.2 SGLang 对长序列的"PP prefill + 动态 chunking"路径

在 128K prefill 下，TTFT 从 ~3.2s 被压缩到 ~8.6s（GB300 + 32K dynamic chunking）。核心逻辑是

  • 把 128K prefill 切成 32K 的 chunk，在流水线（pipeline parallel）内 overlap
  • Dynamic chunking vs static chunking 8-17% 提升，因为消除了固定调度下的 pipeline bubble

这说明：即使 P 端计算和网络传输都已经优化，prefill 本身的调度粒度在 128K+ 下仍是一个开放优化空间。

4.3 芯片级方案：3DLS 对长序列的启示

3DLS（2607.01617）指出，在长序列 PD 分离（每 layer KV 都要搬到 D 端）下，KV 传输与 TP 的 all-to-all 集体通信在同一 2D 平面互连上混跑，decode critical path 被拉长。

3DLS 的解法：把 KV 传输走垂直（3D TSV）通道，让 decode-side TP 留在平面互连上。1.49× throughput 与 60.2% latency 下降，给出方向：长序列 PD 的芯片互连拓扑需要为 KV 传输单独开道。

源：3DLS (2607.01617)

4.4 Conversation-level 调度：避免每 turn 都重新搬 KV

长序列不仅是"单请求长 prompt"，更是"多轮对话累积长 history"。

  • ConServe（2606.01839）：把调度单元从"turn"提升到"conversation"，Turn 1 走 PD，之后 pin 在同一个 D 节点，不再反复搬。p95 first-effective-token ↓51.08%，能效 +7.75%。
  • PPD MultiTurn（2606.13358）：区分 full prefill 与 append-prefill，后者直接在 D 端完成，KV 不再跨节点 —— Turn 2+ TTFT ↓68%。

这两项工作共同指向一个洞察：长序列 PD 的真实压力不在"长 prompt"本身，而在"长 history 反复被搬动"。缓解策略的核心是"稳定 D 端 residency"——一旦 KV 落地 D 端，就尽量别动。

5. 行业头部玩家如何公开讨论这一挑战

5.1 SGLang（LMSYS）

  • GB300 博客明确说长序列 decode 进入"KV-dominated and memory-bound"状态，关键不是 compute 而是KV 容量与HBM 带宽。
  • EPD（2026-01）讨论弹性 Encoder-Prefill-Decode 分离 —— 128K+ 图像 token 也会压垮固定 D 端。
  • Heterogeneous EPD（2026-06）把 CPU 也纳入 EPD 资源池 —— 长序列下纯 GPU 端吃紧，必须加入异构存储层。

5.2 Moonshot Kimi / Mooncake（FAST'25）

  • Mooncake 是业界第一家在论文中明确写"highly overloaded scenarios"作为 PD 分离主要挑战的工作。其核心设计选择——prediction-based early rejection——正是针对"D 端装不下所有到达的 KV"这一长序列瓶颈。
  • 三级 KV 存储（GPU HBM → CPU DRAM → SSD / 网络）把"D 端容量"从 HBM 单一一层扩展为三层存储，是唯一商业系统中长序列存储分层的明确实现。

5.3 vLLM

  • RFC #43470 正式把 Decode-KV-bound 列为 PD 三大瓶颈类之一 —— 社区共识已形成。
  • vLLM 的 NIXL lease renewal（lease=30s+heartbeat）是长序列 PD 下的一种"租约式容量预留"方案：D 端必须持续 heartbeat 才能保持 KV，失败则 KV 兜底清理 —— 长序列下 heartbeat 压力显著上升。

5.4 NVIDIA Dynamo

  • 沿用 KV-aware Smart Routing 把KV 热度纳入 D 端选择打分 —— 长序列 D 端的热门 KV 会持续占据 pool，如何避免"热 KV 偏见"是一个 open sub-problem。
  • GTC 2026 上 Dynamo 把 PD 分离 + Large Expert Parallelism 并推（参考 SGLang GB300 博客的致谢列表）。

5.5 Meta / Alibaba / AWS

  • Meta（PyTorch Blog 2025.09）：强调"D 端独立扩缩容"，但没明确写长序列 D 端容量瓶颈。
  • Alibaba PAI-EAS：提供静态 / 动态 / 共置三种 PD 模式，并开放 EP+PD 联合部署（DeepSeek MoE）。
  • AWS：Bedrock + Dynamo + HyperPod，提供 token-threshold 决定是否走 PD 分离 —— 长序列用户接受"走分离"或"走共置"的配置型取舍。

7. 研判：已解 / 部分解 / 未解 问题地图

7.1 已解（有明确工程或学术共识）

  • Decode-KV-bound 作为瓶颈类 已被 vLLM RFC 正式命名，学术界广泛认可。
  • P→D KV 渐进传输（Lynx、SplitZip、SCD）在理论和技术路径上都相对成熟，128K+ 场景可组合使用。
  • Cross-turn KV 复用（ConServe、PPD MultiTurn）在对话级有成熟解法。
  • 硬件趋势：从 H100 80GB → H200 141GB → GB200 192GB → GB300 288GB，HBM 容量在快速增长，给 D 端 128K+batch 提供物理空间。

7.2 部分解（有方案但仍有显著开放子问题）

  • D 端 KV pool sizing 的理论与实证模型：目前尚无统一的"长序列 PD 容量规划教科书"。GB300 博客实测数据 + SGLang Issue #24523 的 SWA bug 说明：软件栈跟不上硬件，容量利用率仍低于理论值。
  • 长序列下 KV 压缩 vs 精度取舍：SpectrumKV 发现 Qwen2.5-7B 在 int4 下"catastrophically fails"，而 Mistral-7B 稳定 —— 长序列对更激进的 KV 压缩是双刃剑。
  • CXL / 异构 KV pool 的工程化：TraCT（CXL）和 HMA-Serve（异构）是原型，尚未在头部云大规模部署。
  • D 端高并发 + 长序列 + CUDA-graph 三者的稳定性：SGLang #30010 死锁问题仍 open，尚无修复。

7.3 未解（仍是学术前沿）

  • 1M+ tokens 下的 PD 分离"可行性边界"建模：目前最大公开部署约 128K-256K；1M 级别的 PD 分离尚无系统层面的研究报告。
  • 多租户下长序列 PD 隔离：长序列 D 端 KV 池容易被单一用户独占（"KV 饥饿"），公平调度仍在 RFC 阶段。
  • KV 精度在长序列 PD 下逐 token 自适应：SpectrumKV 的 per-token mixed precision 还在早期，没有"按序列位置动态调整"的工作。
  • PD 分离在 1M+ 下是否仍是最优架构：有学者（HMA-Serve、CrossPool）在质疑"PD 分离"是否是长序列的最优解，或者某种"长序列共置"混合架构更适合 1M。
  • 长序列 attention 与 PD 传输联合优化：retraction、chunked prefill、PP 长 context prefill 与 PD KV 传输目前割裂优化，缺乏联合理论框架。

7.4 补充前沿工作（2026-07-15 调研新增）

以下为 2026-07-15 后续调研新发现的高阶论文，进一步佐证上述判断：

  • SAC (2606.19746) — 首个为 稀疏注意力 + CXL 优化的 disaggregated KV 系统。RDMA 方案在稀疏注意力下"fetch 整块 KV 但只用 top-k"是严重浪费；SAC 利用 CXL load/store 语义按需取 KV 条目，在 DeepSeek-V3.2（稀疏注意力）+ SGLang 上 2.1× 吞吐、9.7× TTFT、1.8× TBT。关键启示：长序列 PD 的未来不在"压缩 KV 走 RDMA"，而在"让稀疏注意力直接跑在 CXL 内存上"。
  • CXL-1M PNM (2511.00321) — 1M-token + 405B 参数级 LLM 推理的 CXL-Processing-Near-Memory 系统。PNM-only offloading (PNM-KV) 和 GPU-PNM 混合 (PnG-KV) 实现最高 21.9× 吞吐与 60× 能耗下降。关键启示：1M 级长序列 PD 一旦脱离"GPU HBM 是唯一 KV 容器"这一假设，性能天花板会打开数量级级别。
  • ZCube (Z.ai 智谱, 2026-05-20 公开博客) — 明确树立 PD + 长上下文是 网络层结构性挑战："cross-node KV Cache transfers make inference traffic highly asymmetric"，100G→200G 升级实测 TTFT ↓22%、整体吞吐 ↑19%。这种扁平化网络拓扑创新是面向长序列 PD 的基础设施层新方向。
  • vLLM MiniMax M3 博客 (2026-06-12) — MiniMax M3 是"为百万 token context 设计"的模型，采用 MSA block-sparse attention + MXFP8 + chunked prefill + block-size 128。关键启示：模型侧稀疏注意力与系统侧 PD 分离已开始协同设计。
  • vLLM Issue #11286（实测）— DeepSeek decode 速度从 5364 tok/s（<100 token）断崖下降到 273 tok/s（8K+ token），显示长序列 decode 性能退化剧烈，目前 vLLM 无 1M+ 专项优化。
  • SGLang PR #8846 — SGLang 把 1M+ 视为 PP 问题设计，PP+PD 组合 2025-10 才"初步支持"，Decode-side PP 至今 pending。工业界领袖把 1M+ 调出 PD 框架处理，间接印证 PD 范式在 1M+ 下的局限性。

7.5 vLLM 源码级 D 端内存承诺与容量护栏（2026-07-15 新增）

对 vLLM main 分支（2026-07-15）源码级抓取，揭示 PD 分离下 D 端 KV 容量的内生矛盾：

  • 关键架构决策：无论 pull/push 模式，D 端的 block 都是预分配的——Prefill 端把 KV 推到 D 端"事先预留好的 logical block ids (preallocated)"（引自 nixl_kv_push_connector.md）。这意味着 D 端必须在收到 KV 之前就承诺内存；当 PD 规模较大时，这种"预分配 + lease"机制会在 D 端形成大量的"内存承诺"。
  • 默认参数（来自 vllm/config/cache.py）：
    block_size=16 / gpu_memory_utilization=0.92 / cache_dtype="auto" / enable_prefix_caching=True
  • 唯一护栏：vLLM v1 只有 RECOMPUTE preemption，watermark_blocks ≈ 1% 是 D 端 KV pool 几乎唯一的硬护栏。kv_lease_duration=30s + heartbeat 续约（~5s 周期、延 ~20s）把"承诺"变成可撤销，但续约本身增加长序列下 D 端高并发心跳压力。
  • D 端容量公式：allocate_slots → determine_available_memory() 做一次完整 model run 量 peak memory 后反推出 KV 空间；KVCacheManager → BlockPool → KVCacheTensor 三层结构。这是长序列 PD 调参的"原始旋钮"。
  • kv_cache_dtype 极端案例：最激进的 nvfp4/per-token-head 配置下，D 端 KV 容量放大倍数可达 5.3×（相比 fp16），但引入精度与容量的取舍。
