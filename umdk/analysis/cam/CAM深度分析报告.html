1. 执行摘要与核心发现

UMDK/CAM 是华为 Ascend NPU 上面向 MoE Expert Parallelism (EP) 通信加速的软件栈，核心使命是在 Ascend 910_93 (A3) 芯片上为 MoE 模型提供高性能的 EP 通信算子，并进一步将通信与计算融合为单一算子。

🎯 6 大核心发现
#发现意义
1控制面 HCCP + 数据面自研 的混合模式NDA 层复用 HCCP 连接管理，但直接从 ChannelEntity 提取硬件寄存器地址，在 kernel 中直接操作 RDMA WQE/CQE 硬件队列，绕过 HCCL 软件栈
21C2V 异构融合架构FusedDeepMoE 将 Dispatch + GMM1 + SwiGLU + GMM2 + Combine 五阶段融合为单 kernel，AIC 做 GMM 计算，AIV 做通信调度，中间数据全在片上流转，消除至少 8 次 GM 访问
3Magic+Value 无锁同步notify_dispatch 不依赖 HCCL 集合通信，自管理 Ping-Pong DMA 实现 All-to-All，高 32 位 magic 防 ABA 问题
4编译时多态后端选择aclshmemi_backend_traits<B> traits 系统在编译期选择 IN_DIE/XSCALE 后端，零运行时开销
5INT8 动态量化内置Dispatch 阶段内置 per-token 动态量化，节省约 50% Window 带宽
6测试体系存在缺口无集成测试套件、无 Fuzz、无 ASAN、无代码覆盖率门禁，与 URMA 的完善测试体系形成鲜明对比

3. NDA 传输抽象层深度分析

NDA (Network Device Abstraction) 是 CAM 的"操作系统网络栈"，分为 host 端 和 device 端 两部分。

3.1 Host 端架构

RdmaTransportManagerV2 使用 HCCP 的 HcommEndpointCreate/HcommChannelCreate 建立 QP 连接，但从 ChannelEntity 内部结构体提取硬件寄存器地址（SQ head/tail pointer, doorbell address）。

┌─────────────────────────────────────────────────────────────┐ │ Host Side │ │ RdmaTransportManagerV2 │ │ ├── OpenDevice() → 获取 phyId, deviceIp │ │ ├── CreateEndpoint() → HcommEndpointCreate (RoCE) │ │ ├── RegisterMemoryRegion → HcommMemReg (注册 HBM 为 MR) │ │ ├── Connect() → HcommChannelCreate + FillRdmaInfo│ │ └── GetQpInfo() → 返回 device-visible 的 QP 信息 │ ├─────────────────────────────────────────────────────────────┤ │ Device Side │ │ AIV 核直接操作 XSCALE Diamond RDMA 引擎 │ │ ├── WQE (Work Queue Element) 提交 │ │ ├── CQE (Completion Queue Element) 轮询 │ │ ├── Doorbell 寄存器写入 │ │ └── 编译时多态: aclshmemi_backend_traits<B> │ │ ├── IN_DIE 后端 (片内互联) │ │ └── XSCALE 后端 (跨节点 RoCE) │ └─────────────────────────────────────────────────────────────┘

CommArgs (utils/op_kernel/comm_args.h) — 整个通信层的核心参数结构体：

3.3 为何需要 NDA 层

核心洞察MoE EP 通信的模式是细粒度的、动态的、不规则的 P2P 数据传输，HCCL 的抽象层级过高，无法满足延迟敏感、计算-通信重叠、远程原子操作三大需求。

4. EP 通信算子深度分析

CAM 提供 4 个核心 EP 通信算子，遵循 Ascend C 三层编程模型 (op_api → op_host → op_kernel)：

4.1 dispatch_layout

纯计算算子，在 dispatch 前按 expert 粒度重排 token。输入 topk_idx，输出 num_tokens_per_expert 和 send_token_idx。

4.2 moe_dispatch_notify (notify_dispatch)

自管理 IPC 架构notify_dispatch 不依赖 HCCL 集合通信，自己管理 Ping-Pong DMA 实现 All-to-All。使用 Magic+Value 无锁同步机制。

4.3 moe_dispatch_normal

参数类型说明
xbf16/fp16 [bs, hid]输入 token
topk_idxint64 [bs, topk]目标 expert ID
num_tokens_per_expertint [num_experts]来自 dispatch_layout
send_token_idx_smallint [bs, topk]来自 dispatch_layout
group_epstrHCCL 通信组名
use_quantbool是否启用量化

4.4 moe_combine_normal

Combine 阶段的加权求和，将各 expert 输出按 topk_weights 聚合。精度标准：非量化场景平均相对误差 < 0.5%，量化场景 < 1%。

4.5 关键性能优化点

优化实现收益
双缓冲 (Ping-Pong)CpGM2GMPingPong UB 交替通信-计算重叠
地址对齐所有 Window 访问对齐到 512B避免跨 cache line
增量 Flag 检查checkedFlagNum += i跳过已确认 flag
状态位批量 ReduceSum快速判断所有数据是否到达减少轮询开销
原子操作聚合AscendC::SetAtomicAdd跨 core 聚合
三元组嵌入每个 token 附加 (srcRank, srcTokenIdx, topkIdx)combine 无需额外通信

5. FusedDeepMoE 融合算子深度分析

FusedDeepMoE 是 CAM 中最复杂的融合算子，将 MoE Decode 阶段的完整计算流水线融合进单一 kernel。

5.1 融合策略

标准小算子组合 (5次kernel launch + 多次GM读写): Dispatch → GMM1 → Swiglu → GMM2 → Combine FusedDeepMoE (1次kernel launch, 全在片上完成): Dispatch → GMM1 → Swiglu → GMM2 → Combine (全部融合)
核心动机消除中间数据的 Global Memory 往返。标准组合中 X1、SwigluOut、X2 需至少 8 次 GM 访问，融合后全部在片上 workspace 流转。

5.2 1C2V 异构架构

处理器职责说明
AICGMM 计算执行 GMM1/GMM2 矩阵乘法
AIV-SendCore发送调度发起 RDMA write
AIV-RecvCore接收调度轮询 CQE
AIV-CompCoreSwiglu+量化执行 SwiGLU 激活和动态量化

5.3 三条执行路径

路径条件行为
非深度融合moeExpertNumPerRank==1独立 Dispatch → GMM1 → GMM2
深度融合moeExpertNumPerRank>1GMM1 内完成 Dispatch，GMM2 内完成 Combine
共享专家share expert enabled额外处理 share expert 的 GMM1/GMM2

5.4 Swiglu + 量化融合

GMM1 的 int32 结果在 UB 中直接完成 PerTokenDequant → Swiglu → DynamicQuant 三步流水线，输出 int8 到 workspace，不经过 GM。

5.5 GEMM 子模块 (catlass 集成)

grouped_matmul_slice_m_per_token_dequant_multistage_workspace.hgemm/kernel/Per-token 反量化多阶段 workspace

9. 设计 Trade-off 深度剖析

9.1 用户态 RDMA vs HCCL 集合通信

方案优势劣势
用户态 RDMA ✓极致低延迟、可流水线化、支持 inline 操作需管理 QP/CQ 生命周期、代码复杂
HCCL AllToAll抽象简单、自动处理拓扑无法 kernel 内融合、调度开销大

决策理由：MoE EP 通信模式高度动态，HCCL 的静态规划 AllToAll 无法高效处理。

9.2 编译时后端选择 vs 运行时调度

方案优势劣势
编译时 traits ✓零运行时开销、编译器可优化每个后端需单独编译
运行时 switch单 binary 适配多后端分支预测开销、无法内联

9.3 控制面 HCCP + 数据面自研

方案优势劣势
HCCP 控制面 + 自研数据面 ✓复用成熟连接管理、极致数据面性能需从 HCCP 内部结构体提取信息
完全自研无外部依赖QP 状态机、错误处理工作量大
纯 HCCL最简单无法满足 kernel 内 RDMA 需求

9.4 FusedDeepMoE 的融合粒度

方案优势劣势
Dispatch+GMM1+SwiGLU+GMM2+Combine ✓最大程度减少 HBM 读写和 kernel launch编译复杂度高、需定制 epilogue
