CAM (Communication Acceleration for Matrix) 架构深度分析报告

1. 整体架构设计哲学

1.1 定位与使命

CAM 是华为 Ascend NPU 上面向 MoE Expert Parallelism (EP) 通信加速的软件栈。其核心使命是：

在 Ascend 910_93 (A3) 芯片上，为 MoE 模型提供高性能的 EP 通信算子（Dispatch/Combine），并进一步将通信与计算（GEMM/SwiGLU）融合为单一算子（FusedDeepMoE），以最大化 HCCS/RoCE 互联带宽利用率。

1.2 设计哲学总结

CAM 的设计体现了以下核心哲学：

哲学维度 具体表现
硬件原生 直接操作 RDMA QP/SQ/CQ 硬件队列，绕过 HCCL 抽象层以获得极致延迟
垂直融合 FusedDeepMoE 将 Dispatch + GMM1 + SwiGLU + GMM2 + Combine 融合为单算子，消除多次 kernel launch 和 HBM 读写
编译时多态 通过 C++ template traits 系统（aclshmemi_backend_traits）在编译期选择 IN_DIE / XSCALE 后端，零运行时开销
SOC 世代解耦 operator_registry.json + select_ops.py 实现 SOC 世代与算子版本的声明式注册
分层传输抽象 NDA (Network Device Abstraction) 层分离 host 端资源管理（RdmaTransportManagerV2）与 device 端 RDMA 原语
Ascend C 编程范式 严格遵循 op_host（tiling 计算）/ op_kernel（计算 kernel）/ op_api（aclnn 接口）三层模型

2.1 关键设计决策：为何需要 NDA 层

传统 HCCL 通信库提供高阶集合通信原语（AllReduce/AllToAll 等）。但 MoE EP 通信的模式是 细粒度的、动态的、不规则的 P2P 数据传输（每个 token 发往不同 expert 所在的 rank），HCCL 的抽象层级过高，无法满足：

  1. 延迟敏感：需要直接 post RDMA WQE 到硬件 SQ，避免 HCCL 的软件调度开销
  2. 计算-通信重叠：需要在 kernel 内部发起 RDMA 操作后立即进行 GEMM 计算
  3. 原子操作：Combine 阶段需要 remote atomic 操作更新 token 计数器

因此 NDA 层本质上是一个 用户态 RDMA 子系统，在 Ascend 芯片上直接操作 RoCE 硬件。

3.2.2 关键数据结构

AiQpRMAQueueInfo（host 侧镜像，最终拷贝到 device）：

struct AiQpRMAQueueInfo {
    uint32_t count;
    AiQpRMAWQ* sq;     // Send Queue 信息 [rankCount]
    AiQpRMAWQ* rq;     // Receive Queue 信息 [rankCount]
    AiQpRMACQ* scq;    // Send Completion Queue [rankCount]
    AiQpRMACQ* rcq;    // Receive Completion Queue [rankCount]
    RdmaMemRegionInfo* mr;  // Memory Region keys [rankCount]
};

每个 AiQpRMAWQ 包含： - wqn — QP 编号 - bufAddr — SQ ring buffer 的 HBM 地址 - wqeSize / depth — WQE 大小和队列深度 - headAddr / tailAddr — producer/consumer index 的 HBM 地址 - dbAddr — doorbell register 地址 - atomicAddr / atomicLkey — 原子操作的本地缓冲区

3.2.3 内存注册策略

Host 端管理两类内存注册：

  1. 用户 HBM 缓冲区（RegisterMemoryRegion）：通过 HcommMemReg 注册，区分 COMM_MEM_TYPE_DEVICE（HBM）和 COMM_MEM_TYPE_HOST（DDR）。
  2. 原子操作共享内存（RegisterAtomicMemory）：大小为 ATOMIC_MAX_NUM(128) × sizeof(uint64_t) × rankCount，512 字节对齐，用于存储原子操作的 fetch-before-write 结果。

3.3.3 XSCALE 后端详解（rdma_device_backend_xscale.hpp）

XSCALE 后端直接操作 DDRM Diamond RDMA 引擎，这是 Ascend 910_93 跨机通信的硬件核心。

WQE 格式（XSCALE v2）：

struct aclshmemi_xscdv_wqe_ctrl_seg_v2 {
    uint8_t msg_opcode;       // 操作码（RDMA_WRITE=1, RDMA_READ=2, ATOMIC_CAS=26, ATOMIC_FA=27...）
    uint8_t with_imm : 1;
    uint8_t csum_en  : 2;
    uint8_t ds_data_num : 5;  // data segment 数量
    uint16_t rsv1;
    uint32_t msg_len;
    uint32_t opcode_data;
    uint32_t se : 1;          // solicited event
    uint32_t ce : 1;          // 是否产生 CQE
    uint32_t in_line : 1;     // inline 模式
    uint32_t fence_mode : 2;
    uint32_t mask : 2;
    uint32_t rsv : 5;
    uint32_t wqe_id : 20;     // WQE ID
};  // 共 16 bytes

CQE 格式（Completion Queue Entry）：

struct aclshmemi_xscdv_diamond_cqe_v2 {
    uint32_t error_code : 8;
    uint32_t qp_id      : 15;
    uint32_t se         : 1;
    uint32_t has_pph    : 1;
    uint32_t type       : 1;
    uint32_t with_imm   : 1;
    uint32_t csum_err   : 4;
    uint32_t imm_data;
    uint32_t msg_len;
    uint32_t vni;
    uint32_t ts_l;            // 时间戳低 32 位
    uint32_t ts_h;            // 时间戳高 16 位（共 48 位）
    uint32_t msg_opcode : 8;
    uint32_t wqe_id     : 20;
    uint32_t owner      : 1;  // ownership bit
};  // 共 32 bytes，cqe64 补齐到 64 bytes

关键操作流程（以 RDMA WRITE 为例）：

// 1. 检查 SQ 剩余容量，不足则 poll_cq 等待 WQE 完成
if ((cur_head + XSCALE_POLL_CQ_THRESHOLD) % depth == (depth + cur_tail) % depth) {
    aclshmemi_roce_poll_cq<XSCALE>(pe, qp_idx, cur_head, ...);
}

// 2. 获取 remote/local memory keys
wr.rkey = remote_mem_info->rkey;
wr.lkey = local_mem_info->lkey;

// 3. 填充 WQE（ctrl seg + data seg）

// 4. 刷 cacheline 到 HBM

// 5. Ring SQ Doorbell（更新 head pointer + 写 doorbell register）

Poll CQ 流程：

while (cur_tail != target_idx) {
    // 检查 owner bit 判断 CQE 是否有效
    while (!aclshmemi_roce_xscale_check_cqe_owner(cqe_addr, cur_tail, depth) &&
           run_cycles < ACLSHMEMI_XSC_POLL_CQ_TIMEOUT_CYCLES) {
        run_cycles++;
        dcci_cachelines((__gm__ uint8_t *)cqe_addr, sizeof(aclshmemi_xscdv_cqe64_t));
    }
    // 检查 error_code
    status = cqe_addr->cqe.error_code;
    cur_tail++;
}
// 更新 CQ tail 并 ring CQ doorbell
aclshmemi_roce_ring_cq_doorbell<XSCALE>(...);

超时机制：5 分钟超时（ACLSHMEMI_XSC_POLL_CQ_TIMEOUT_DURATION = 5ULL * 60 * 1000000 微秒），超过后返回 ACLSHMEMI_XSC_POLL_CQ_TIMEOUT_ERROR = 0x10000。

3.4 NDA 如何抽象 HCCL/HCCP

NDA 并非完全绕过 HCCL/HCCP，而是 分层使用：

层级 组件 作用
初始化 HcommEndpointCreate / HcommChannelCreate 使用 HCCP 协议栈建立 QP 连接，获取 SQ/CQ/MR 句柄
内存注册 HcommMemReg / HcommMemUnreg 通过 HCCP 注册内存区域，获取 lkey/rkey
QP 信息提取 ChannelEntity → AiQpRMAQueueInfo 从 HCCP 内部结构体提取硬件寄存器地址
运行时数据面 直接操作 WQE/doorbell 绕过 HCCL/HCCP 软件栈，AIV 核直接写硬件 SQ 队列

这种设计的好处： 1. 控制面复用 HCCP：利用其成熟的拓扑发现、连接建立、QP 状态机管理 2. 数据面自研：针对 MoE EP 的 P2Gather 模式做极致优化（inline write、masked atomic、pipeline sync）

4. Operator Registry 与 SOC 世代机制

4.1 operator_registry.json 结构

{

设计要点： - soc_versions 定义每个 SOC 世代支持的算子列表 - operator_meta 提供算子级元数据（如 requires_shmem、family、variant） - 算子身份 = 源码目录名（ascend_kernels/ 下的子目录）

4.2 select_ops.py 解析逻辑

文件位置：build/cam/comm_operator/select_ops.py

def resolve(registry, soc_list, shmem, quant, user_ops):
    # 1. 确定候选集：全量 SOC 列表 or 用户指定的 -a 列表
    candidates = user_ops if user_ops else soc_list

    # 2. fused_deep_moe base/w4a8 互斥规则
    if quant:
        # -q 选择 w4a8 变体，移除 base
        candidates = [o for o in candidates if o != BASE]
    else:
        # 无 -q 时 base 优先，移除 w4a8
        candidates = [o for o in candidates if o != W4A8]

    # 3. SHMEM 过滤：未安装 SHMEM 时跳过 requires_shmem 的算子
    if not shmem:
        candidates = [o for o in candidates if not requires_shmem(o)]

    # 4. 保持 registry 顺序（确定性构建）
    return [o for o in soc_list if o in candidates]

互斥规则的本质：fused_deep_moe（base）和 fused_deep_moe_w4a8 共享完全相同的源文件名（fused_deep_moe.cpp），区别仅在于编译时宏定义（-DW4A8_QUANTIZATION），因此不能同时编译进一个 run package。

4.3 构建系统集成

build/cam/build.sh 通过以下流程调用 select_ops.py：

5. op_host / op_kernel / op_api 三层编程模型

5.1 模型概述

5.2.2 op_kernel 层

dispatch_layout.h — Kernel 类定义：

template <typename T>
class DispatchLayout {
public:
    __aicore__ inline void Init(...) {
        // 从 tilingData 读取配置
        numTokens_ = tilingData->dispatchLayoutInfo.numTokens;
        // 计算每个 AIV core 负责处理的 token 数
        aivNum_ = numTokens_ <= maxAivNum ? numTokens_ : maxAivNum;
        // 分配 UB buffer
    }

    __aicore__ inline void Process() {
        // 1. 从 HBM 加载 topk_idx 到 UB
        DataCopyPad(topkIdxTensor, topkIdxGM_, ...);
        // 2. 统计每个 expert/rank 的 token 数
        for (int i = 0; i < tempTokens_; ++i) {
            for (int j = 0; j < numTopk_; ++j) {
                int64_t expert_idx = topkIdxTensor.GetValue(i * numTopk_ + j);
                numTokensPerExpertTensor.SetValue(expert_idx, per_expert_num);
            }
        }
        // 3. 写回 HBM
        DataCopyPad(numTokensPerExpertGM_, numTokensPerExpertTensor, ...);
        // 4. 跨 core 同步
        SyncAll<true>();
    }
};

7. 关键设计决策与 Trade-offs

7.1 Trade-off 1: 用户态 RDMA vs HCCL 集合通信

方案 优势 劣势
用户态 RDMA (CAM 选择) 极致低延迟、可流水线化、支持 inline 操作 需要管理 QP/CQ 生命周期、代码复杂度高
HCCL AllToAll 抽象简单、自动处理拓扑 无法做到 kernel 内融合、调度开销大

决策理由：MoE EP 的通信模式是高度动态的（每个 token 的目的地不同），HCCL 的静态规划 AllToAll 无法高效处理。用户态 RDMA 允许在 kernel 内部按需发起单次 RDMA 操作，实现真正的计算-通信重叠。

7.2 Trade-off 2: 编译时后端选择 vs 运行时调度

方案 优势 劣势
编译时 traits (CAM 选择) 零运行时开销、编译器可优化 每个后端需要单独编译
运行时 switch 单 binary 适配多后端 分支预测开销、无法内联

决策理由：RDMA 操作在关键路径上，每次操作节省几个时钟周期在百万次调用下累积显著。且 Ascend 芯片的后端（IN_DIE vs XSCALE）在部署时即确定，无需运行时适配。

7.3 Trade-off 3: 三层编程模型 vs 单层 kernel

方案 优势 劣势
op_host/op_kernel/op_api (CAM 选择) 符合 CANN 生态规范、支持 GE 图编译、自动并行 代码分散在多个文件
单层 kernel 代码集中 无法利用 CANN 的调度优化、不兼容 ACLGraph

决策理由：Ascend C 三层模型是华为 CANN 生态的标准约定。op_host 的 tiling 计算允许运行时根据输入 shape 动态调整数据切分策略，op_kernel 的模板化支持多种数据类型，op_api 层则保证了与 torchair 图编译模式的兼容。

7.4 Trade-off 4: 注册式 SOC 管理 vs 硬编码

方案 优势 劣势
JSON registry + Python resolver 新增 SOC 世代只需编辑 JSON、算子与 SOC 解耦 引入外部依赖
硬编码 CMake 条件 无额外依赖 每加 SOC 需修改构建脚本

决策理由：CAM 未来需要支持 A2、A3 等多代芯片，注册式设计使得算子开发者无需了解构建系统细节。select_ops.py 的互斥规则（base/w4a8）和 SHMEM 过滤逻辑也更容易维护。

7.5 Trade-off 5: 控制面 HCCP + 数据面自研

方案 优势 劣势
HCCP 控制面 + 自研数据面 (CAM 选择) 复用成熟的连接管理、同时获得极致数据面性能 需要从 HCCP 内部结构体提取信息
完全自研 无外部依赖 QP 状态机、错误处理、重传机制需要大量工作量
纯 HCCL 最简单 无法满足 kernel 内 RDMA 需求

决策理由：这是 NDA 层最精妙的设计。控制面（连接建立、QP 状态机、内存注册）使用 HCCP 的 HcommEndpointCreate / HcommChannelCreate API，数据面则直接从 ChannelEntity 结构体提取硬件寄存器地址（SQ head/tail pointer, doorbell address），在 kernel 中直接操作硬件。

7.6 Trade-off 6: FusedDeepMoE 的融合粒度

方案 优势 劣势
Dispatch+GMM1+SwiGLU+GMM2+Combine (CAM 选择) 最大程度减少 HBM 读写和 kernel launch 编译复杂度高、需要定制 epilogue
仅 Dispatch+Combine 融合 实现简单 GEMM 仍需独立 kernel

决策理由：MoE decode 阶段的 FFN 计算（GMM1 + SwiGLU + GMM2）受限于 HBM 带宽。融合后，dispatch 的输出直接在 UB/Cache 中进入 GMM1，GMM2 的输出也直接在 UB 中进入 combine 阶段，避免多次 HBM round-trip。从 fused_deep_moe/gemm/ 和 fused_deep_moe/epilogue/ 目录结构看，华为为融合算子定制了： - block_mmad — 矩阵乘累加 tile - block_mmad_preload_async_with_callback_resident_a — 异步预取 + 回调 - block_epilogue_per_token_dequant_swiglu — SwiGLU + 反量化融合 epilogue

8. 总结

CAM 是一个 面向 MoE EP 通信的垂直优化软件栈，其架构设计体现了以下核心洞察：

  1. MoE EP 通信不适合传统集合通信抽象 — 需要用户态 RDMA 获得极致延迟
  2. 计算-通信融合是性能关键 — FusedDeepMoE 将 5 阶段流水线融合为单 kernel
  3. 硬件多样性需要编译时多态 — traits 系统零开销适配 IN_DIE/XSCALE 后端
  4. 生态兼容性需要分层设计 — 控制面复用 HCCP，数据面自研，三层编程模型符合 CANN 规范

NDA 层是 CAM 的"皇冠上的明珠"，它实现了一个轻量级用户态 RDMA 子系统，在 Ascend 芯片上直接操作 RoCE 硬件，同时保持与 HCCP 控制面的兼容性。这种设计模式对理解国产 AI 芯片的软件栈演进具有重要参考价值。
