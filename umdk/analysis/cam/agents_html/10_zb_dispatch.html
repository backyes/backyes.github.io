1. ZB (Zero-Buffer) 设计哲学

1.1 什么是 Zero-Buffer？

ZB (Zero-Buffer) 是 CAM 库在 cam_feature 分支上引入的下一代 MoE Dispatch 通信范式。其核心思想是：

完全摒弃传统的 HCCL Window 通信模式，改用 CANN SHMEM（用户态 RDMA）API 直接在卡间对称内存（symmetric memory）上进行数据搬运和同步。

1.2 传统 HCCL Window 模式的问题

在传统的 MoE Dispatch 流程中（参见 07_hccl_window_deepdive.md），通信依赖 HCCL Window 机制：

维度 HCCL Window 模式 Zero-Buffer 模式
内存占用 每卡预分配 ~200MB HCCL Buffer 作为通信窗口 零额外内存，直接使用对称内存布局
通信路径 数据 → HCCL Window → RDMA → 对端 HCCL Window → 用户 Buffer 数据 → RDMA → 对端用户 Buffer（直达）
同步机制 HCCL Collective + 信号量 SHMEM flag 轮询 + Magic 机制
延迟 多次内存拷贝 + HCCL 软件栈开销 单次 RDMA + 用户态 flag 同步
灵活性 窗口大小固定，需预分配 按需使用，无固定窗口限制

1.3 Zero-Buffer 的核心优势

  1. 零额外内存: 不需要预分配 HCCL Window Buffer，节省 ~200MB/卡
  2. 单次 RDMA 直达: 数据直接从源卡写入目标卡的用户可见内存，无需中间拷贝
  3. 用户态同步: 通过 SHMEM flag + Magic 机制实现轻量级跨卡同步，避免重量级 HCCL Collective
  4. 对称内存布局: 所有卡上的 flag 区域布局完全一致，通过 shmem_ptr(base, rankId) 即可定位对端内存

2. 共享基础设施分析

2.1 zero_buffer_sync_flag.h — 核心同步机制

这是整个 ZB 体系最关键的共享组件。ZeroBufferSyncFlag 类实现了基于对称共享内存的跨卡 flag 同步。

内存布局

gva_gm + FLAG_AREA_BASE (100KB):
┌──────────────────────────────────────────────────────────┐
│ Magic Area (24 KB)                                       │
│   slot[coreIdx] at coreIdx × 512B                        │
│   每核单调递增，每次 IncrementMagic() +1                  │
├──────────────────────────────────────────────────────────┤
│ Barrier PingPong Buffer 0                                │
│   slot[srcRank] (32B each), epWorldSize slots            │
├──────────────────────────────────────────────────────────┤
│ Barrier PingPong Buffer 1 (同上)                          │
├──────────────────────────────────────────────────────────┤
│ Flag PingPong Buffer 0                                   │
│   flag[srcRank][eventID] (32B each)                      │
│   srcRank ∈ [0, epWorldSize)                             │
│   eventID ∈ [0, slotsPerRank)                            │
├──────────────────────────────────────────────────────────┤
│ Flag PingPong Buffer 1 (同上)                             │
└──────────────────────────────────────────────────────────┘

Total = Magic(24KB) + 2×BarrierBuf + 2×FlagBuf

Magic 机制

// 高 32 位存储 magic，低 32 位存储 phase
__aicore__ inline int64_t MakeFlagValue(int32_t magic, int32_t value)
{
    return (static_cast<int64_t>(static_cast<uint32_t>(magic)) << 32) |
           static_cast<int64_t>(static_cast<uint32_t>(value));
}
  • 每个 kernel 入口调用 IncrementMagic() 一次
  • Magic 单调递增：kernel K 得到 M，kernel K+1 得到 M+1
  • Ping-pong buffer 选择：magic % 2
  • 关键不变量: 单调 magic 保证任何旧 kernel 的 stale flag 都会被拒绝

同步协议

Writer (rank A, core i):
  1. 完成数据写入（DMA 到远端 shmem）
  2. SetFlag(destRank=B, eventID=i, phase)
     → MTE3 fence（flush 所有先前的写入）
     → 写 flag 到 B 的本地 shmem slot [A][i]

Reader (rank B, core j):
  1. WaitFlag(srcRank=A, eventID=i, phase)
     → 轮询本地 shmem 直到 flag 匹配（magic, phase）
  2. 返回后，rank A 的数据保证可见

同步粒度

模式 slotsPerRank 使用场景
Per-core blockNum 每个 core 向所有 rank 发信号
Per-slot N (任意) 细粒度事件同步
Per-rank 1 粗粒度 rank 间同步

2.2 zero_buffer_api.h — SHMEM API 封装

shmem.h 是 CANN 工具链提供的外部头文件，提供： - aclshmem_ptr(baseAddr, rankId) — 获取对端 rank 的远端内存指针 - aclshmem_half_put_nbi() — half 类型非阻塞 RDMA put - aclshmem_bfloat16_put_nbi() — bf16 类型非阻塞 RDMA put - aclshmem_float_put_nbi() — float 类型非阻塞 RDMA put - shmem_get_int32_mem_nbi() — int32 类型非阻塞 RDMA get

SyncCollectives 是一个更早期的同步实现，使用双内存区域设计： - Memory A（basicSyncAddr）: 卡内同步 flag - Memory B（blockOuterSyncAddr）: 卡间同步 flag

部分 ZB 算子（如 lowlatency_zb）仍混合使用 SyncCollectives 和 ZeroBufferSyncFlag。

3. moe_dispatch_layout_zb — 布局算子

3.1 算子概述

moe_dispatch_layout_zb 是 MoE Dispatch 流程的第一阶段，负责： 1. 统计每个 rank 和每个 expert 接收的 token 数量 2. 计算 isTokenInRank 标记矩阵 3. 计算 sendTokenIdxSmall（token 在目标 expert 内的局部序号）

关键特征: 此算子不涉及跨卡通信，纯本地计算。它使用 SyncAll<true>() 进行卡内多核同步。

3.4 Kernel 实现 — A1 版本 (dispatch_layout.h)

核心算法：两阶段 token 计数

Phase 1: CountTokensInRound() — 分轮次统计

// 每轮处理 tokensPerRound 个 token
// UB 预算: 190KB max
uint32_t fixedSize = numTokensPerRank32AlignIntLen_ + numTokensPerExpert32AlignIntLen_ +
    Ceil(numRanks_ * sizeof(T), UB_32_ALIGN) * UB_32_ALIGN;

uint32_t perTokenSize = Ceil(numTopk_ * sizeof(int64_t), UB_32_ALIGN) * UB_32_ALIGN +
    Ceil(numRanks_ * sizeof(T), UB_32_ALIGN) * UB_32_ALIGN +
    Ceil(numTopk_ * sizeof(T), UB_32_ALIGN) * UB_32_ALIGN;

uint32_t tokensPerRound = (UB_MAX_SIZE - fixedSize) / perTokenSize;

每个 token 的处理逻辑：

for (uint32_t j = 0; j < numTopk_; ++j) {
    int64_t expertIdx = topkIdxLocal.GetValue(i * numTopk_ + j);
    expertLocal.SetValue(expertIdx, expertLocal.GetValue(expertIdx) + 1);
    int rankId = expertIdx / expertsPerRank;
    if (!seenLocal.GetValue(rankId)) {
        rankLocal.SetValue(rankId, rankLocal.GetValue(rankId) + 1);
        isTokenLocal.SetValue(i * numRanks_ + rankId, 1);
        seenLocal.SetValue(rankId, 1);
    }
}

Phase 2: CalcSendTokenIdxInRound() — 计算发送序号

for (uint32_t i = 0; i < roundTokens; ++i) {
    for (uint32_t j = 0; j < numTopk_; ++j) {
        int64_t expertIdx = topkIdxLocal.GetValue(i * numTopk_ + j);
        T val = expertLocal(expertIdx);
        sendIdxLocal(i * numTopk_ + j) = val;
        expertLocal(expertIdx) = val + 1;  // 递增，下一个 token 的序号
    }
}

跨核原子累加

Phase 1 中，每个 core 计算完自己的 numTokensPerExpert 后，通过 SetAtomicAdd 将结果写入后续 core 的临时区域，实现跨核累加：

AscendC::SetAtomicAdd<T>();
for (uint32_t i = coreIdx_ + 1; i < aivNum_; ++i) {

AscendC::SetAtomicNone();

3.5 Kernel 实现 — A2 变体 (dispatch_layout_a2.h)

A2 变体（DispatchLayoutA2）是为 Ascend910B (A2) 芯片设计的增强版本，引入了服务器层级（server-level hierarchy）概念。

关键差异

维度 A1 (DispatchLayout) A2 (DispatchLayoutA2)
处理模式 分轮次 (round-based) 单次全量 (single-pass)
层级 Rank 级别 Server + Rank 级别
新增输出 无 localTokenServer* 系列
同步 SyncAll (多轮) 单次 SyncAll

4. moe_dispatch_normal_zb — 标准分发算子

4.1 算子概述

moe_dispatch_normal_zb 是 MoE Dispatch 流程的第二阶段（数据搬运阶段），负责： 1. 从 layout 算子获取 put_offset（目标偏移）和 send_tokenIdx（发送序号） 2. 通过 SHMEM RDMA 将 token 数据直接写入对端 rank 的 expandXOut 缓冲区 3. 支持可选的 DynamicQuant（INT8 动态量化）

关键特征: 使用 ZeroBufferSyncFlag 进行跨卡同步，使用 shmem_ptr() 获取远端指针。

核心执行流程 (Process)

template <CamTypeClass>
__aicore__ inline void MoeDispatchNormalZeroBuffer<CamTypeFunc>::Process()
{
    if ASCEND_IS_AIV {
        // Step 1: IncrementMagic — 获取 magic M+1
        syncFlag_.IncrementMagic();

        // Step 2: BarrierAll — 确保所有 rank 的 Notify 结果可见
        //         (putOffset, recvData 等已被 Notify 计算完成)
        syncFlag_.BarrierAll();

        // Step 3: 通过 shmem 写 token 数据到远端 rank
        InputToDstOutput();

        // Step 4: BarrierAll — 确保所有 rank 完成写入
        //         后续 GMM kernel 可以安全消费 expandXOut
        syncFlag_.BarrierAll();
    }
}

template <TemplateMC2TypeClass>
__aicore__ inline void MoeDispatchLowlatencyZeroBuffer<TemplateMC2TypeFunc>::Process()
{
    if ASCEND_IS_AIV {
        // Phase 1: 本地统计每个 expert 的 token 发送数量
        LocalLayout();
        SyncAll<true>();

        // Phase 2: 通知所有 rank 本地统计完成
        SetLayoutStatus();
        SendCountNotify();  // 等待所有 rank 的统计结果
        SyncAll<true>();

        // Phase 3: 处理接收到的全局统计信息
        WaitNotify();
        SyncAll<true>();

        // Phase 4: 实际数据搬运
        InputToDstOutput();

        // Phase 5: 同步确保数据写入完成
        SetSyncFlag();
        WaitSyncFlag();
        SyncAll<true>();

        // Phase 6: 清理
        CleanUp();
    }
}

Phase 1: LocalLayout — 本地统计

__aicore__ inline void LocalLayout()
{
    // 1. 加载 expert_ids
    DataCopyPad(expertIdsTensor_, expertIdsGMTensor_, expertIdsCntParams, expertIdsCntCopyPadParams);

    // 2. 按 core 分片统计每个 expert 的 token 数
    for (int32_t i = startTokenId; i < endTokenId; ++i) {
        int32_t expertIdx = static_cast<int32_t>(expertIdsTensor_(i));
        int32_t curCnt = sendCountLt.GetValue(expertIdx) + 1;
        sendCountLt.SetValue(expertIdx, curCnt);
    }

    // 3. 原子写入共享元数据区域
    AscendC::SetAtomicAdd<int32_t>();
    DataCopyPad(LocalNotifyDataTensor_, sendCountLt, sendCountDataCopyParams);
    AscendC::SetAtomicNone();
}

Phase 2: SetLayoutStatus — 通知统计完成

__aicore__ inline void SetLayoutStatus()
{
    // 每个 AIV 负责通知一部分 rank
    for (uint32_t targetRankId = startRankId; targetRankId < endRankId; targetRankId++) {
        // ★ 使用 aclshmem_float_put_nbi 写入对端 rank 的 layoutStatus
        aclshmem_float_put_nbi(layoutStatusTensor_[epRankId_], statusTensor_, 1, targetRankId);
    }
}

Phase 3: SendCountNotify — 等待并获取对端统计

__aicore__ inline void SendCountNotify()
{
    for (uint32_t targetRankId = startRankId; targetRankId < endRankId; targetRankId++) {
        // 1. 轮询等待对端 rank 的 layout flag 匹配当前 magic
        while ((curVal < minFlagVal) || (curVal > maxFlagVal)) {
            DataCopy(layoutStatusFp32Tensor_, layoutStatusTensor_[targetRankId], intriParams);
            curVal = layoutStatusFp32Tensor_.GetValue(0);
        }

        // 2. ★ 使用 shmem_get_int32_mem_nbi 从对端 rank 读取 send count
        shmem_get_int32_mem_nbi(allExpertTokenNumsGMTensor_[targetRankId * moeExpertNum_],
            LocalNotifyDataTensor_, moeExpertNum_, targetRankId);
    }
}

Phase 4: InputToDstOutput — 数据搬运

__aicore__ inline void SendToMoeExpert()
{
    for (int32_t index = startTokenId; index < endTokenId; ++index) {
        int32_t tokenIndex = index / axisK_;
        uint32_t dstExpertId = expertIdsTensor_(index);
        int32_t dstRankId = dstExpertId / moeExpertNumPerRank_ + sharedExpertRankNum_;

        // 计算当前 token 在目标 expert 内的序号
        CalTokenSendExpertCnt(dstExpertId, index, curExpertIdx);

        // 计算目标偏移
        int32_t dstExpertOffset = sendCountsGlobal(offsetIdx - 1);

        // ★ 通过 shmem_ptr 获取对端 rank 的远端内存指针
        auto dstPtr = reinterpret_cast<__gm__ uint8_t *>(shmem_ptr(expandXOutGM_, dstRankId));
        dstWinGMTensor.SetGlobalBuffer(
            (__gm__ ExpandXOutType *)(dstPtr + hOutSizeAlign_ * (dstExpertOffset + curExpertIdx)));

        // 写入数据
        if constexpr (DynamicQuant || StaticQuant) {
            // 量化路径
            QuantProcess();
            DataCopyPad(dstWinGMTensor, xOutTensor_, expandXCopyParams_);
            DataCopyPad(dstScaleGMTensor, xOutFp32Tensor[hOutSizeAlign_ / sizeof(float)], scaleCopyParams_);
        } else {
            // 直接写入
            DataCopyPad(dstWinGMTensor, xTmpTensor_, expandXCopyParams_);
        }
    }
}

Phase 5: SetSyncFlag / WaitSyncFlag — 数据写入同步

__aicore__ inline void SetSyncFlag()
{
    // 每个 core 向负责的 rank 写入 flag
    for (uint32_t targetRankId = curBlockStartRankId_; targetRankId < curBlockEndRankId_; ++targetRankId) {
        auto ptr = GetSyncFlagAddrByRankId(targetRankId) + epRankId_ * DISPATCH_SYNC_STATE_OFFSET;
        gmRemoteStatusGT.SetGlobalBuffer((__gm__ int32_t *)(ptr));
        DataCopy<int32_t>(gmRemoteStatusGT, statusTensor[...], 8UL);
    }
}

__aicore__ inline void WaitSyncFlag()
{
    // 轮询等待所有对端 rank 的 flag
    while ((sumOfFlag < minTarget) || (sumOfFlag > maxTarget)) {
        DataCopy(statusFp32Tensor, statusFp32TensorGT[...], intriParams);
        ReduceSum(statusSumOutTensor, statusFp32Tensor, gatherMaskOutTensor, mask, rankNumPerBlock_, 1);
        sumOfFlag = statusSumOutTensor.GetValue(0);
    }
    // 清理 flag
    Duplicate<int32_t>(cleanStateTensor, 0, ...);
    DataCopy(statusFp32TensorGT[...], cleanStateTensor.ReinterpretCast<float>(), intriOutParams);
}

AllGatherSendData — 跨卡数据收集

__aicore__ inline void AllGatherSendData()
{
    // 按 core 分片负责一部分 rank
    SplitCoreCal(epWorldSize_, rankNumPerBlock, startRankId, endRankId);

    for (uint32_t targetRankId = startRankId; targetRankId < endRankId; targetRankId++) {

        // GM→GM 拷贝（支持类型转换）

            recvDataGt_[targetRankId * numExperts], COPYONLY);
    }
}

CpGM2GMPingPong — Ping-Pong GM→GM 拷贝

template <typename K, typename U>
__aicore__ inline void NotifyDispatchZeroBuffer<T>::CpGM2GMPingPong(
    int64_t dataSizeRemain, const GlobalTensor<U> &sendDataInputGt,
    const GlobalTensor<K> &recvDataOutputGT, int op)
{
    // 双 buffer ping-pong: UB_HEAD_OFFSET 和 UB_MID_OFFSET
    __ubuf__ U *inputUB[2] = {UB_HEAD_OFFSET, UB_MID_OFFSET};
    __ubuf__ K *outputUB[2] = {inputUB[0], inputUB[1]};

    SetAtomic<K>(op);
    for (int64_t i = 0; dataSizeRemain > 0; i++) {
        event_t eventId = (i & 1) ? EVENT_ID0 : EVENT_ID1;
        CpGM2UB(inputUB[i&1], input + inputOffsetNum, size);  // GM→UB
        // 类型转换（如需要）
        CastImpl(outputUB[i&1], inputUB[i&1], RoundMode::CAST_NONE, size / sizeof(K));
        CpUB2GM(output + outputOffsetNum, outputUB[i&1], size);  // UB→GM
        dataSizeRemain -= size;
    }
    UnsetAtomic(op);
}

BuildMaxBs — 计算最大 batch size

__aicore__ inline void BuildMaxBs()
{
    int32_t maxBsNum = 0;
    for (uint32_t srcRankId = 0; srcRankId < epWorldSize_; srcRankId++) {
        DataCopy(tokenPerExpertDataLt, recvDataTensor_[numExperts * srcRankId], numExperts);
        Cast(floatExpTokenCntLt, tokenPerExpertDataLt, RoundMode::CAST_NONE, numExperts);
        ReduceSum(floatExpTokenSumCntLt, floatExpTokenCntLt, sharedTmpBuffer, numExperts);
        int32_t curRankBsNum = static_cast<int32_t>(floatExpTokenSumCntLt(0));
        maxBsNum = curRankBsNum > maxBsNum ? curRankBsNum : maxBsNum;
    }
    maxBsGt.SetValue(0, maxBsNum / topkNum_);
}

BuildTotalRecvCount — 计算前缀和偏移表

__aicore__ inline void BuildTotalRecvCount()
{
    for (uint32_t rank = startRankId; rank < endRankId; ++rank) {
        // 转置并计算前缀和
        ReorderRecvDataOutput(rank, recvTokenLt, true);  // isCumSum = true
        DataCopyPad(recvCntGt[rank * numExperts], recvTokenLt, copyParams);
    }
}

ReorderRecvDataOutput — 矩阵转置 + 前缀和

__aicore__ inline void ReorderRecvDataOutput(int32_t rankId, LocalTensor<int32_t> &transLt, bool isCumSum)
{
    uint32_t moeExpertPerRankNum = numExperts / epWorldSize_;
    uint32_t startExpId = rankId * moeExpertPerRankNum;
    uint32_t endExpId = rankId * moeExpertPerRankNum + moeExpertPerRankNum;

    int32_t prefixSum = 0;
    for (uint32_t expId = startExpId; expId < endExpId; ++expId) {
        for (uint32_t srcRank = 0; srcRank < epWorldSize_; ++srcRank) {
            uint32_t index = (expId - startExpId) * epWorldSize_ + srcRank;
            uint32_t pairIdx = srcRank * numExperts + expId;
            int32_t curRecvCount = recvDataTensor_(pairIdx);
            transLt(index) = isCumSum ? prefixSum : curRecvCount;
            prefixSum += curRecvCount;
        }
    }
}

8. 性能优势分析

8.1 内存占用对比

组件 HCCL Window 模式 Zero-Buffer 模式
HCCL Buffer ~200MB/卡 0
Flag 区域 N/A ~100KB/卡 (Magic + Barrier + Flag)
通信窗口 预分配固定大小 按需使用，无固定窗口
总计 ~200MB/卡 ~100KB/卡

内存节省: 每卡节省约 200MB，对于大规模 EP（如 256 卡）集群，总计节省约 50GB 内存。

8.2 通信延迟对比

阶段 HCCL Window 模式 Zero-Buffer 模式
数据写入 用户 Buffer → HCCL Window 用户 Buffer → RDMA → 对端用户 Buffer
同步信号 HCCL Collective SHMEM flag 轮询
数据读取 对端 HCCL Window → 对端用户 Buffer 直接读取（已在用户 Buffer）

延迟降低原因: 1. 消除中间拷贝: 数据直接从源写入目标，无需经过 Window 中转 2. 用户态同步: SHMEM flag 轮询避免 HCCL Collective 的软件栈开销 3. 硬件直通: RDMA 引擎直接访问用户内存，无需 CPU 干预

8.3 同步机制优势

Magic + Ping-Pong 机制: - 每个 kernel 调用获取唯一 magic 值 - magic % 2 选择 ping-pong buffer，避免与上一次调用的 flag 冲突 - 单调递增 magic 保证 stale flag 被自动拒绝 - 无需显式清理 flag 区域（ping-pong 自动覆盖）

BarrierAll 协议:

SyncAll (卡内) → Write flags to all ranks → Wait all ranks' flags → SyncAll (卡内)
  • 卡内同步确保所有 core 完成计算
  • 分 core 策略：每个 core 负责一部分 rank，避免热点
  • 无环依赖保证：DAG 结构，不会死锁

8.4 量化加速

normal_zb 和 lowlatency_zb 支持 INT8 动态量化： - 将 BF16/FP16 数据量化为 INT8 传输，带宽需求减半 - 在目标卡上反量化恢复精度 - 量化/反量化使用向量单元并行处理

8.5 AIV 并行策略

ZB 算子充分利用 Ascend 910 的 AIV（AI Vector）架构： - 最多 48 个 AIV core 并行工作 - 每个 core 负责一部分 token 或一部分 rank - 分片策略：SplitToCore() / SplitCoreCal() 均匀分配负载
