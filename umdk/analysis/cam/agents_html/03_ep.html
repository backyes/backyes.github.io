1. 架构总览与设计哲学

1.1 CAM 的定位

CAM (Communication Acceleration for Matrix) 是华为 Ascend NPU 面向 MoE (Mixture-of-Experts) Expert Parallelism 场景的通信加速库。其核心设计目标是：

  • 消除 All-to-All 瓶颈: MoE 中 token 需要跨卡路由到对应 expert 所在的 rank，传统 All-to-All 通信开销巨大
  • 零拷贝 (IPC): 利用 HCCL Window 机制实现卡间直接内存访问，避免显式收发
  • 计算通信重叠: 通过状态位同步 + 双缓冲 (ping-pong) 实现流水线隐藏延迟

2.2 SyncCollectives (utils/op_kernel/sync_collectives.h)

这是卡间同步原语的核心实现，使用 HCCL Window 上的 flag 实现跨卡同步：

同步标志结构：

constexpr int64_t FLAG_UNIT_INT_NUM = 4;      // 每个 flag 占 4 个 int64
constexpr int64_t SYNC_UNIT_SIZE = 4 * sizeof(int64_t);  // 32 Bytes
constexpr int64_t MAGIC_OFFSET = 32;          // magic 占高 32 位

核心机制 — Magic + Value 编码：

__aicore__ inline int64_t MergeMagicWithValue(int32_t magic, int32_t value) {
    return (static_cast<int64_t>(static_cast<uint32_t>(magic)) << MAGIC_OFFSET) 
           | static_cast<int64_t>(value);
}

设计洞察: 高 32 位 magic 用于区分不同轮次（避免 ABA 问题），低 32 位 value 表示具体事件。这种无锁设计允许跨卡原子操作。

内存布局 (每个 rank 的 shareAddr)：

┌──────────────────────────────────────────────────────┐
│ 段 A: 卡内同步 flags (blockNum × FLAG_UNIT_INT_NUM)   │
├──────────────────────────────────────────────────────┤
│ 段 B: 卡间同步 flags (blockNum × FLAG_UNIT_INT_NUM)   │
├──────────────────────────────────────────────────────┤
│ Chunk Flags: IPC 传输完成标记 (per destRank)            │
└──────────────────────────────────────────────────────┘

WaitOneRankPartFlag 实现 — 核心等待循环：

do {
    DataCopy(localWait, globalWait[...], ...);  // GM→UB 拷贝
    WaitFlag<MTE2_S>();                          // 等待 DMA 完成
    for (i = 0; i < remainToCheck; ++i) {
        v = localWait.GetValue(i * FLAG_UNIT_INT_NUM);
        if ((v & MAGIC_MASK) != (checkValue & MAGIC_MASK) || v < checkValue) {
            isSync = false;
            checkedFlagNum += i;  // 跳过已确认的 flag
            break;
        }
    }
} while (!isSync);

优化点: 增量检查（checkedFlagNum）避免重复检查已完成的 flag，减少 UB 读取开销。

3. dispatch_layout 算子

3.1 功能定位

dispatch_layout (aclnnDispatchLayout) 是 MoE 路由的前置分析算子，纯计算、无通信。它根据 topkIdx 计算：

  • 每个 rank 需要接收的 token 数 (numTokensPerRank)
  • 每个 expert 需要处理的 token 数 (numTokensPerExpert)
  • 每个 token 是否在某个 rank 内 (isTokenInRank)
  • 用于 notify 的发送数据 (notifySendData)
  • 小 sendTokenIdx (sendTokenIdxSmall)

3.2 op_api 层

aclnnStatus aclnnDispatchLayout(void *workspace, uint64_t workspaceSize, 
                                 aclOpExecutor *executor, aclrtStream stream) {
    if (NnopbaseSetHcclServerType) {
        NnopbaseSetHcclServerType(executor, NNOPBASE_HCCL_SERVER_TYPE_MTE);  // 关键: 使用 MTE 模式
    }
    return aclnnInnerDispatchLayout(workspace, workspaceSize, executor, stream);
}

关键设计: 强制设置 HCCL Server Type 为 MTE (Memory Transfer Engine)，绕过 AICPU 中转。

3.3 op_host 层 (Tiling 逻辑)

输入/输出定义： - Input: topkIdx (int64, [numTokens, numTopk]) - Attrs: num_tokens, num_ranks, num_experts, num_topk, local_ranksize - Outputs: numTokensPerRank, numTokensPerExpert, isTokenInRank, notifySendData, sendTokenIdxSmall

核心参数校验： - numRanks ∈ (0, 384] - numExperts ∈ (0, 512] - numTopk ∈ (0, 16] - numTokens ∈ (0, 8000] - 约束: numExperts % numRanks == 0（expert 必须均匀分布）

Workspace 计算：

workSpaces[0] = SYSTEM_NEED_WORKSPACE + KERNEL_USE_WORKSPACE;  // 16MB + 1MB

Tiling Key 固定为 23。

3.4 op_kernel 层 (核心算法)

并行策略: 按 token 维度切分到多个 AIV core：

aivNum_ = numTokens_ <= maxAivNum ? numTokens_ : maxAivNum;
// 每个 core 处理 tempTokens_ 个 token

核心算法 — Process()：

Phase 1: 统计 (每个 core 独立)
  for each token i in [startToken, endToken):
    for each topk j:
      expert_idx = topkIdx[i][j]
      numTokensPerExpert[expert_idx]++
      rank_id = expert_idx / experts_per_rank
      if !seenRank[rank_id]:
        numTokensPerRank[rank_id]++
        isTokenInRank[i][rank_id] = 1
        seenRank[rank_id] = 1

Phase 2: 全局聚合 (原子加)
  SetAtomicAdd<T>()
  for each other core i:
    DataCopyPad(notifySendData[i * numExperts], numTokensPerExpert)
  SetAtomicNone()

Phase 3: SyncAll 全卡同步
  SyncAll<true>()

Phase 4: 计算 sendTokenIdxSmall
  for each token i:
    for each topk j:
      expert_idx = topkIdx[i][j]
      sendTokenIdxSmall[i][j] = aggregatedNumTokensPerExpert[expert_idx]++

算法洞察: Phase 2 使用 SetAtomicAdd 而非显式 reduce，利用硬件原子操作实现跨 core 聚合，避免额外的同步开销。

UB 内存规划：

topkIdx32AlignIntLen_ = Ceil(tempTokens_ * numTopk_ * sizeof(int64_t), 32) * 32;
numTokensPerRank32AlignIntLen_ = Ceil(numRanks_ * sizeof(T), 32) * 32;
numTokensPerExpert32AlignIntLen_ = Ceil(numExperts_ * sizeof(T), 32) * 32;
isTokenInRank32AlignIntLen_ = Ceil(tempTokens_ * numRanks_ * sizeof(T), 32) * 32;

4. notify_dispatch 算子

4.1 功能定位

notify_dispatch (aclnnNotifyDispatch) 是元数据 All-to-All 通信算子。它在实际 token 数据传输前，先让每张卡知道：

  • 其他卡要发送给自己多少 token (recvCount)
  • 从哪个 offset 开始存放 (recvOffset)
  • 本卡要给其他卡发送的 offset (sendDataOffset)
  • 全局最大 batch size (maxBs)
  • 每个本地 expert 接收的 token 总数 (recvTokensPerExpert)

4.2 op_api 层

参数列表 (极复杂，反映通信元数据交换的本质)：

aclnnNotifyDispatchGetWorkspaceSize(
    sendData, tokenPerExpertData, sendCount, numTokens,
    commGroup, rankSize, rankId, localRankSize, localRankId,
    sendDataOffset, recvData, totalRecvTokens, recvCount, recvOffset,
    maxBs, recvTokensPerExpert, ...
)

4.3 op_host 层

HCCL 通信配置：

uint32_t opType1 = OP_TYPE_ALL_TO_ALL;  // 8 = AlltoAll
std::string algConfigAllToAllStr = "AlltoAll=level0:fullmesh;level1:pairwise";

通信拓扑: level0 使用 fullmesh（适合机内全互联），level1 使用 pairwise（适合跨机逐步传输）。这是经典的两级 All-to-All 策略。

Tiling Key 选择（基于数据类型）: - Float16 → 20 - BFloat16 → 21 - Float → 22 - Int → 23 - Ascend910B 额外 +100

4.4 op_kernel 层 (核心算法)

NotifyDispatch 是 CAM 中最复杂的算子，它实现了自管理的 All-to-All（不依赖 HCCL 集合通信）：

四阶段流水线：

Process():
  if (blockIdx < 1):
    AssembleSendData()         // Phase 0: Core 0 组装发送数据
  SyncAll<true>()
  if (blockIdx < coreNumPerStageX):
    InputToShareSlice()        // Phase 1: 将自己的数据写入其他卡的 Window
  if (blockIdx < coreNumPerStageY):
    ShareToShareSlice()        // Phase 2: 从其他卡的 Window 读取数据
  SyncAll<true>()
  ReorderOutput()              // Phase 3a: 重排接收到的数据
  BuildTotalRecvTokens()       // Phase 3b: 构建 totalRecvTokens (blockIdx==0)
  BuildRecvCount()             // Phase 3c: 构建 recvCount (blockIdx==1)
  BuildRecvOffset()            // Phase 3d: 构建 recvOffset (blockIdx==2)
  BuildMaxBs()                 // Phase 3e: 构建 maxBs (blockIdx==3)
  BuildRecvTokenPerExp()       // Phase 3f: 构建 recvTokensPerExpert (blockIdx==4)

AssembleSendData 详解：

// 每个 expert 的 sendPerGroup = 3 个字段:
// [numTokensExpert, prefixSum, numTokens]
for (int i = 0; i < numExperts; ++i) {
    int numTokensExpert = tokenPerExpertUB[i];
    sendDataUB[i * sendPerGroup] = numTokensExpert;       // 该 expert 有多少 token
    sendDataUB[i * sendPerGroup + 1] = prefixSum;          // 偏移量
    sendDataUB[i * sendPerGroup + 2] = numTokens;          // 总 token 数
    sendDataOffsetUB[i] = prefixSum;
    prefixSum += numTokensExpert;
}

InputToShareSlice — 核心 IPC 写入：

// 每个 core 负责对 copyLen 个 rank 进行写入
CpGM2GMPingPong<T>(copyLen * perRankDataNum * sizeof(T), 
                    readGt, writeGt[copyOffset * perRankDataNum], COPYONLY);
// 写入完成后设置 flag 通知对端
uint64_t v = MergeMagicWithValue(magic, 1);
CpUB2GM(shareAddrs[i] + rank * FLAG_UNIT_INT_NUM, inputUB, sizeof(uint64_t));

设计亮点: 使用 Ping-Pong 双缓冲 GM→GM 拷贝，UB_HEAD_OFFSET (96B) 和 UB_MID_OFFSET (96B + 95KB + 32B) 交替使用，实现流水隐藏。

ShareToShareSlice — 读取对端数据：

// 等待本卡数据已被写入对端
WaitSyncFlag(magic, 1, copyOffset, rank, copyLen);
// 从对端 Window 读取属于本卡的数据
for (int i = 0; i < copyLen; i++) {
    CpGM2GMPingPong<T>(perRankDataNum * sizeof(T), 
                        readGt1[i][rank * perRankDataNum],
                        recvDataOutputGt[checkRank[i] * perRankDataNum], COPYONLY);
}

CpGM2GMPingPong — 核心 DMA 引擎：

// 双缓冲交替:
// 偶数轮: 用 inputUB[0] + outputUB[0]
// 奇数轮: 用 inputUB[1] + outputUB[1]
for (int64_t i = 0; dataSizeRemain > 0; i++) {
    event_t eventId = (i & 1) ? EVENT_ID0 : EVENT_ID1;
    WaitFlag<MTE3_MTE2>(eventId);        // 等待上一轮写完成
    CpGM2UB(inputUB[i&1], input, size);  // GM→UB
    CpUB2GM(output, outputUB[i&1], size); // UB→GM
    SetFlag<MTE3_MTE2>(eventId);         // 通知下一轮可开始
    dataSizeRemain -= size;
}

4.5 同步机制

Magic 值获取：

__aicore__ inline uint64_t NotifyDispatch<T>::GetMagicValue(void) {
    // 从 localWindowsExp + STATE_WIN_OFFSET + blockIdx * UB_ALIGN_SIZE 读取
    magic = selfDataStatusTensor(blockIdx * UB_ALIGN_SIZE);
    if (magic <= 0) magic = 1;
    selfDataStatusTensor(blockIdx * UB_ALIGN_SIZE) = magic + 1;  // 自增防 ABA
    return magic;
}

Window 地址计算：

shareAddrs[rank] = GetWindAddrByRankId(rank, ctxIdx) 
                   + (magic % PING_PONG_SIZE) * (IPC_BUFF_MAX_SIZE + IPC_DATA_OFFSET);

Ping-Pong 双 Window: 交替使用两个 102MB+2MB 的 buffer，避免读写冲突。

5.4 op_kernel 层 (核心算法)

模板参数化：

template <typename XType, typename ExpandXOutType, 
          bool dynamicQuant, bool IsSmoothScaleExist, bool IsShareExpertRank>
class MoeDispatchNormal { ... };

四阶段流水线 (Process)：

InputToShare()    →  将 token 数据写入目标 rank 的 Window
SetStatus()       →  设置目标 rank 的状态位 (通知数据已到达)
WaitStatus()      →  等待所有源 rank 的数据到达
ShareToOutput()   →  从 Window 读取数据到本地输出

InputToShare 详解：

// 每个 core 负责 [startTokenId, endTokenId) 的 token
for (tokenIndex = startTokenId; tokenIndex < endTokenId; ++tokenIndex) {
    dstExpertId = expertIdsTensor(tokenIndex - startTokenId);
    curExpertCnt = sendTokenIdxTensor(tokenIndex - startTokenId);
    dstExpertOffset = sendOffsetTensor(dstExpertId);
    // 计算目标 Window 地址
    rankGM = shareGM + hOutGMAlignSize * (dstExpertOffset + curExpertCnt);

    if (DynamicQuant) {
        // 量化路径: BF16 → FP32 → Abs → ReduceMax → Scale → INT8
        QuantProcess();
        FillTriple(xOutTensor, tokenIndex / topK, tokenIndex % topK);
        DataCopyPad(dstGT, xOutTensor, hCommuCopyOutParams);
    } else {
        // 非量化路径: 直接拷贝 + 填充三元组
        FillTriple(xTmpTensor, tokenIndex / topK, tokenIndex % topK);
        DataCopyPad(dstGT, xTmpTensor, hCommuCopyOutParams);
    }
}

FillTriple — 元数据嵌入：

__aicore__ inline void FillTriple(LocalTensor<ExpandXOutType> &xOutTensor, 
                                   uint32_t tokenIndex, uint32_t k) {
    LocalTensor<int32_t> xOutTint32 = xOutTensor.ReinterpretCast<int32_t>();
    xOutTint32(expandIdxStartIdx) = epRankId;      // 源 rank ID
    xOutTint32(expandIdxStartIdx + 1) = tokenIndex; // 源 token index
    xOutTint32(expandIdxStartIdx + 2) = k;          // topk index
}

设计洞察: 在每个 token 的 hidden state 后面附加三元组 (srcRank, srcTokenIdx, topkIdx)，这样 combine 阶段无需额外通信即可知道数据来源。

SetStatus — 状态位同步：

// 将 status 写入目标 rank 的 Window 状态区
for (uint32_t i = startExpId; i < endExpId; ++i) {
    targetRankId = i / moeExpertNumPerRank;
    offset = stateOffset * (epRankId + i % moeExpertNumPerRank * epRankSize);
    rankGM = GetWindStateAddrByRankId(COMM_EP_IDX, targetRankId) + offset;
    DataCopy(dstStatusGT, statusTensor[(i - startExpId) * 8], 8UL);
}

WaitStatus — 等待数据到达：

while (sumOfFlag != compareTarget) {
    DataCopy(statusFp32Tensor, windowInstatusFp32Tensor[startStatusId * stateOffset / 4], ...);
    ReduceSum(statusSumOutTensor, statusFp32Tensor, gatherMaskOutTensor, ...);
    sumOfFlag = statusSumOutTensor.GetValue(0);
}
// 清理状态
Duplicate<int32_t>(cleanStateTensor, 0, ...);
DataCopy(windowInstatusFp32Tensor[...], cleanStateTensor.ReinterpretCast<float>(), ...);

性能洞察: 使用 ReduceSum 在 Vector 单元上快速求和所有状态位，当 sum == statusNumPerCore 时说明所有数据已到达。

ShareToOutput — 从 Window 读取：

for (uint32_t i = startStatusId; i < endStatusId; ++i) {
    fromRank = i % epRankSize;
    count = recvCountTensor(i) - preCount;
    for (uint32_t j = 0; j < count; ++j++) {
        // 从 Window 读取 token + expandIdx
        DataCopyPad(xTmpTensor, srcTokenGT, hCommuCopyOutParams);
        // 提取 expandIdx (三元组)
        DataCopyPad(expandIdxOutGT[(targetOffset + j) * 3], xTmpTensorInt[expandIdxStartIdx], ...);
        // 如果是量化模式, 提取 scale
        if (DynamicQuant) {
            DataCopyPad(dynamicScalesOutGT[targetOffset + j], xOutFp32Tensor[hUBAlignSize / 4], ...);
        }
        // 拷贝 token 到输出
        DataCopyPad(dstTokenGT, xTmpTensor, expandXCopyParams);
    }
}

5.5 量化机制 (QuantProcess)

__aicore__ inline void QuantProcess() {
    Cast(floatLocalTemp, xInTensor, CAST_NONE, h);        // BF16 → FP32
    Abs(floatLocalAbsTemp, floatLocalTemp, h);              // |x|
    ReduceMaxInplace(floatLocalAbsTemp, h);                 // max(|x|)
    dynamicScale = 127.0 / (floatLocalAbsTemp.GetValue(0) + 1e-12f);
    Muls(floatLocalTemp, floatLocalTemp, dynamicScale, h);  // x * scale
    Cast(int32LocalTemp, floatLocalTemp, CAST_RINT, h);    // FP32 → INT32 (四舍五入)
    Cast(halfLocalTemp, int32LocalTemp, CAST_ROUND, h);    // INT32 → FP16 (dequant)
    Cast(xOutTensor, halfLocalTemp, CAST_TRUNC, h);        // FP16 → INT8
    // 存储 scale 供 combine 反量化
    floatLocalTemp.SetValue(hUBAlignSize / sizeof(float), 1.0 / dynamicScale);
}

ReduceMaxInplace 优化：

// 两级 reduce: 先两两比较 (stride=8), 再 WholeReduceMax
if (repsFp32 > 1) {
    Max(srcLocal, srcLocal[64], srcLocal, 64, repsFp32 - 1, {1,1,1,0,8,0});
}
WholeReduceMax(srcLocal, srcLocal, mask, 1, 8, 1, 8);

6.4 op_kernel 层 (核心算法)

两阶段流水线：

CopyBufferToShareAndSetStatus()  →  将 expert 输出写入源 rank 的 Window，设置状态
ReadBufferFromRemote()           →  等待数据到达，读取并按 weight 加权求和

CopyBufferToShareAndSetStatus 详解：

// 每个 core 负责 [startTokenId, endTokenId) 的 token
for (tokenIndex = startTokenId; tokenIndex < endTokenId; tokenIndex++) {
    srcRankId = srcInfoLocal(index + 0);  // 源 rank
    srcTokenId = srcInfoLocal(index + 1);  // 源 token
    srcTopkId = srcInfoLocal(index + 2);   // topk index

    // 拷贝数据到源 rank 的 Window
    CopyBufferToShare(srcRankId, srcTokenId, srcTopkId, tokenIndex);
    // 设置状态位
    SetStatusBySrcInfo(srcRankId, srcTokenId, srcTopkId);
}

CopyBufferToShare — Window 写入：

GM_ADDR dstGM = GetBufferAddrByRankId(srcRankId) 
              + (srcTokenId * axisK_ + srcTopkId) * h512AlignRecvXLen_;
// 512 字节对齐存储，每个 slot 存一个 topk 结果
DataCopyPad(dstWindow, localCopyTensor, xOutCopyParams);

SetStatusBySrcInfo — 状态更新：

GM_ADDR stateGM = GetStateAddrByRankId(srcRankId) 
                + (srcTokenId * axisK_ + srcTopkId) * UB_32_ALIGN;
// 写入 1.0 (0x3F800000) 表示该 slot 数据已就绪
DataCopy<uint32_t>(stateGMTensor, statusTensor, FLOAT_NUM_PER_ALIGN);

WaitBuffCopy — 等待数据到达：

target = axisK_ * FLOAT_NUM_PER_ALIGN;  // 所有 topk 数据都到达
while (current != target) {
    DataCopy<float>(stateTensorLocal, stateGMTensor, calCount);
    Sum(tempStateTensorLocal, stateTensorLocal, sumPerKParams);
    current = tempStateTensorLocal(0);
}
// 清零状态
Duplicate<float>(tempStateTensorLocal, 0.0, calCount);
DataCopy<float>(stateGMTensor, tempStateTensorLocal, calCount);

ReadBufferAndWeightedSum — 加权求和：

Duplicate(sumFloatBufLocal, 0, axisH_);  // 初始化累加器
for (topkId = 0; topkId < axisK_; topkId++) {
    scale = topkWeightsLocal.GetValue((tokenIndex - startTokenIndex) * axisK_ + topkId);
    // 从本卡 Window 读取该 topk 对应的结果
    localTokenAddr = localRankGM_ + (tokenIndex * axisK_ + topkId) * h512AlignRecvXLen_;
    DataCopyPad(tmpToken, localTokenTensor, xOutCopyParams);
    // 类型转换 + 乘权重 + 累加
    Cast(tokenFloatLocal, tmpToken, CAST_NONE, axisH_);     // BF16 → FP32
    Muls(weightedMulBufLocal, tokenFloatLocal, scale, axisH_); // * weight
    Add(sumFloatBufLocal, sumFloatBufLocal, weightedMulBufLocal, axisH_); // sum
}
// 最终结果转回 BF16
Cast(xOutLocal, sumFloatBufLocal, CAST_RINT, axisH_);
DataCopyPad(xOutGlobal_[tokenIndex * axisH_], xOutLocal, xOutCopyParams);

数值精度洞察: 累加使用 FP32 精度（避免 BF16 累加误差），最终输出时转回 BF16/FP16。这是混合精度计算的标准做法。

8. 通信模式全局分析

8.1 通信模式总结

算子 通信类型 HCCL 操作 数据量
dispatch_layout 无 无 纯本地计算
notify_dispatch All-to-All 自管理 IPC 极小 (元数据)
moe_dispatch_normal 间接写入 Window IPC 大 (hidden states)
moe_combine_normal 间接写入 Window IPC 大 (expert 输出)

8.2 自管理 All-to-All 的优势

CAM 的 notify_dispatch 不直接使用 HCCL 的 All-to-All API，而是自己管理 IPC：

  1. 精确控制: 直接操作 Window 地址，避免 HCCL 额外开销
  2. 流水线: Ping-Pong 双缓冲实现计算通信重叠
  3. 同步优化: Magic+Value 编码实现无锁同步
  4. 内存效率: 原地更新，无需额外 buffer

9. 量化机制深度分析

9.1 量化模式

enum { NO_SCALES = 0, DYNAMIC_SCALES = 2 };
  • quantMode=0: 无量化，BF16/FP16 直通
  • quantMode=2: 动态 INT8 量化（per-token 动态 scale）

9.2 量化流程 (Dispatch 阶段)

BF16 Token → FP32 → |x| → max(|x|) → scale = 127/max → x*scale → INT8
                                                         ↓
                                                  存储 1/scale 供反量化

9.3 反量化流程 (Combine 阶段)

// 如果 combine 需要反量化 (当前代码中 combine 直接操作 recvX)
Cast(tokenFloatLocal, tmpToken, CAST_NONE, axisH_);  // INT8 → FP32 (如果量化)
Muls(weightedMulBufLocal, tokenFloatLocal, scale, axisH_);

9.4 量化对内存的影响

非量化模式 (per token):

[h * 2B (BF16)] + [32B (scale)] + [12B (expandIdx)] = h*2 + 44 Bytes

量化模式 (per token):

[h * 1B (INT8)] + [32B (scale)] + [12B (expandIdx)] = h + 44 Bytes

带宽节省: 当 h=7168 时，非量化 14380B → 量化 7212B，节省约 50% Window 带宽。

10. 关键性能优化点

10.1 双缓冲 (Ping-Pong)

NotifyDispatch 的 CpGM2GMPingPong：

__ubuf__ U *inputUB[2] = {UB_HEAD_OFFSET, UB_MID_OFFSET};
for (int64_t i = 0; dataSizeRemain > 0; i++) {
    event_t eventId = (i & 1) ? EVENT_ID0 : EVENT_ID1;
    WaitFlag<MTE3_MTE2>(eventId);     // 等待写完成
    CpGM2UB(inputUB[i&1], input, size);
    CpUB2GM(output, outputUB[i&1], size);
    SetFlag<MTE3_MTE2>(eventId);      // 通知可开始下一轮
}

Combine 的 Magic 双 Window：

winDataSizeOffset_ = magic_ * (totalWinSize / 2UL);  // 交替使用两半

10.2 地址对齐

所有 Window 访问都严格对齐到 512B (WIN_ADDR_ALIGN)：

hOutGMAlignSize = Ceil(hScaleIdxSize, WIN_ADDR_ALIGN) * WIN_ADDR_ALIGN;
h512AlignRecvXLen_ = Ceil(hRecvXTypeLen_, WIN_512_ALIGN) * WIN_512_ALIGN;

原因: HCCL Window 的跨卡访问需要 512B 对齐以达到最佳 DMA 性能。

10.3 增量 Flag 检查

避免重复检查已完成的 flag，减少 GM→UB 读取。

附录 C: 关键洞察总结

C.1 设计范式

  1. Window-based IPC: 所有跨卡数据交换通过 HCCL Window 直接内存访问，避免显式 Send/Recv
  2. Flag-based Synchronization: 使用 Magic+Value 编码的 flag 实现无锁同步
  3. Two-phase Pipeline: 先交换元数据 (notify)，再传输实际数据 (dispatch)
  4. Self-managed All-to-All: notify_dispatch 自己管理 IPC，不依赖 HCCL 集合通信

C.2 性能关键路径

Token 路由延迟 = notify_dispatch (All-to-All 元数据) 
               + dispatch_normal (Window 写入 + 状态同步)
               + expert_compute
               + combine_normal (Window 读取 + 加权求和)

C.3 与竞品对比洞察

特性 CAM (华为) NCCL (NVIDIA) 差异
All-to-All 自管理 IPC NCCL 原生 CAM 更精细控制
同步 Magic+Flag NCCL 内部 CAM 显式可见
量化 内置 INT8 需外部实现 CAM 端到端
双缓冲 Magic 翻转 无 CAM 避免冲突

C.4 潜在优化方向

  1. 量化 Combine: 当前 combine 阶段未利用 INT8 反量化，可在 Window 读取时直接反量化
  2. TP 支持: 当前 TP 仅支持 ring AllGather/ReduceScatter，可探索 2D-HMesh
  3. 动态 Topk: 当前 topk 固定，可探索动态 topk 路由
  4. Sparse Expert: 当前 expert 均匀分布，可探索负载均衡路由
