SHMEM 框架与 NDA (Network Device Abstraction) 层深度分析

代码仓库: UMDK/CAM — Communication Acceleration for Matrix on Ascend NPU 分析分支: master (SHMEM/NDA 框架) + cam_feature (ZB 算子消费端) 分析日期: 2026-07-21 分析范围: 从 host QP 建立到 device RDMA 操作的全链路基础设施

0. 关键发现：SHMEM 框架与 ZB 算子的分支分布

重要: SHMEM/NDA 框架代码位于 master 分支，而 ZB (Zero-Buffer) 算子位于 cam_feature 分支。cam_feature 分支本身不包含 NDA 框架源码，它通过外部依赖方式引用 shmem 包（从 gitcode.com/cann/shmem 克隆，commit 6a5f4b09）。

2. 核心数据结构

2.1 aclshmemi_rdma_info — 设备端 RDMA 全局信息

// shmemi_device_rdma.h
struct aclshmemi_rdma_info {
    uint32_t qp_num;   // 每个连接的 QP 数量 (multi-QP 支持)
    uint64_t sq_ptr;   // Send Queue 上下文数组 [PE_NUM][qp_num]
    uint64_t rq_ptr;   // Receive Queue 上下文数组
    uint64_t scq_ptr;  // Send Completion Queue 上下文数组
    uint64_t rcq_ptr;  // Receive Completion Queue 上下文数组
    uint64_t mem_ptr;  // Memory Region 信息数组 [MAX_PE_NUM]
};

该结构体驻留于 device HBM，由 host 端 FillRdmaInfo() 通过 AclrtMemcpy 拷贝上去。设备端通过 aclshmemi_qp_info_fetch() 获取指针。

2.2 SQ/CQ 上下文

struct aclshmemi_rdma_sq_ctx {

    uint64_t tail_addr; // Consumer Index 地址 (host/device 共享)

    uint64_t db_addr;   // doorbell 寄存器地址
    uint32_t sl;        // Service Level
    uint64_t amo_addr;  // 原子操作结果回写地址
    uint32_t amo_lkey;  // 原子操作内存 lkey
};

struct aclshmemi_rdma_cq_ctx {
    uint32_t cqn;       // Completion Queue Number
    uint64_t buf_addr;  // CQ ring buffer 基址
    uint32_t cqe_size;  // 每个 CQE 大小 (XSCALE: 64B, 含 32B padding)
    uint32_t depth;     // ring buffer depth
    uint64_t head_addr; // Producer Index 地址
    uint64_t tail_addr; // Consumer Index 地址
    aclshmemi_rdma_db_mode_t db_mode;
    uint64_t db_addr;   // CQ doorbell 寄存器地址
};

};

2.4 atomic_params — 16 种变体联合

struct aclshmemi_rdma_atomic_params {
    union {
        struct { uint64_t compare_add; uint64_t swap; };           // 基础 CAS/FA
        struct {                                                   // 通用 masked
            uint64_t swap_add_data;    uint64_t compare_data;
            uint64_t swap_add_mask;    uint64_t compare_mask;
        } masked_common;
        struct { uint64_t swap_data; uint64_t compare_data;        // masked CAS64
                 uint64_t swap_mask; uint64_t compare_mask; } masked_cas64;
        struct { uint64_t swap_data; uint64_t compare_data;        // masked CAS32
                 uint32_t swap_mask; uint32_t compare_mask; } masked_cas32;
        struct { uint64_t add_data; uint64_t field_boundary; }     // masked FA64
                 masked_fa64;
        struct { uint32_t add_data; uint32_t field_boundary;       // masked FA32
                 uint64_t _reserved; } masked_fa32;
    };
};

设计洞察: 使用 union 覆盖所有原子操作的参数变体，通过 masked_common 统一接口传入 swap_add_data/compare_data/swap_add_mask/compare_mask，后端根据 IS_MASKED 和 sizeof(T) 选择正确的硬件 WQE 格式。

3. API 调用链

3.1 RDMA Write 完整路径

用户代码: aclshmemx_roce_put_nbi(dst, src, buf, elem_size, pe)
    │
    ▼
aclshmemi_roce_write<T, B>(dst, src, pe, 0, len, ub64, ub32, sync_id)
    │  构建 wr = {remote_addr=dst, local_addr=src, message_len=len}
    ▼
aclshmemi_roce_post_send<B, OP_RDMA_WRITE>(wr, pe, qp_idx, ...)
    │
    ▼ (XSCALE 特化)
aclshmemi_xscale_post_send_read_write<OP_RDMA_WRITE>(wr, pe, qp_idx, ...)
    │
    ├── 1. 读取 SQ head/tail (dcci_cachelines 从 HBM)
    ├── 2. 检查 SQ 满: (cur_head + 16) % depth == cur_tail → poll_cq
    ├── 3. 获取 rkey/lkey (从 rdma_info->mem_ptr)
    ├── 4. 计算 WQE 地址: sq_base_addr + wqe_size * (cur_head % depth)
    ├── 5. fill_wqe → 构造 128B WQE
    │       ├── fill_wqe_ctrl_seg: msg_opcode=RDMA_WRITE(1), ds_data_num=2
    │       └── fill_wqe_data_seg: [rkey|remote_addr] + [lkey|local_addr]
    ├── 6. dcci_cachelines(wqe_addr, 128)  // flush WQE 到 HBM
    ├── 7. cur_head++
    └── 8. ring_sq_doorbell:
            ├── 写 cur_head 到 sq_context->head_addr (HBM)
            └── 写 doorbell 寄存器: {next_pid=cur_head<<3, qp_id=wqn}

3.2 Atomic Fetch-and-Add 完整路径

用户代码: aclshmemx_roce_atomic_fetch_add(dst, value, pe)
    │
    ▼
aclshmemi_roce_amo_add<T, IS_MASKED>(dst, src, pe, 0, add_val, 0, ...)
    │  if constexpr (XSCALE) → aclshmemi_roce_atomic_fetch_and_add
    ▼
aclshmemi_roce_atomic_fetch_and_add<T, IS_MASKED, B>(dst, src, pe, ...)
    │
    ├── 获取 sq_context, remote_mem_info
    ├── 构建 wr:
    │     remote_addr = dst
    │     local_addr = sq_context->amo_addr  (原子结果回写地址)
    │     atomic.masked_common.swap_add_data = add_val
    │     atomic.masked_common.swap_add_mask = boundary
    │     rkey = remote_mem_info->rkey
    │     lkey = sq_context->amo_lkey
    ▼
aclshmemi_backend_traits<B>::atomic_op_traits<T, IS_MASKED>
    ::post_send<OP_ATOMIC_FA>(wr, ...)
    │
    ▼ (XSCALE)
aclshmemi_rdma_xscdv_post_send_atomic<T, IS_MASKED, OP_ATOMIC_FA>(wr, ...)
    │
    ├── 检查 SQ 满 → poll_cq
    ├── aclshmemi_roce_xscale_fill_wqe_atomic<T, IS_MASKED, OP_ATOMIC_FA>
    │       └── aclshmemi_rdma_xscdv_fill_wqe_fa<T, IS_MASKED>
    │             ├── fill_wqe_ctrl_seg: opcode=8B/4B_MSK_FETCH_AND_ADD
    │             ├── fill_wqe_data_seg: [rkey|remote] + [lkey|amo_addr]
    │             └── fill_fa_atomic_seg: [add_data|field_boundary] (big-endian)
    ├── dcci_cachelines(wqe_addr, 128)
    └── ring_sq_doorbell

3.3 Quiet (同步等待完成)

aclshmemx_roce_quiet(pe, buf, sync_id)
    │
    ▼
aclshmemi_roce_quiet(pe, qp_idx, ub64, ub32, sync_id)
    │
    ├── 遍历所有 qp_idx ∈ [0, qp_num)
    ├── 读取 sq_context->head_addr → cur_head
    ▼
aclshmemi_roce_poll_cq<XSCALE>(pe, qp_idx, cur_head, ...)
    │
    ├── 读取 CQ tail (HBM)
    ├── while (cur_tail != target_idx):
    │       ├── 计算 CQE 地址: cq_base + cqe_size * (cur_tail % depth)
    │       ├── 检查 owner_bit == (cur_tail / depth) & 1
    │       ├── dcci_cachelines(cqe_addr, 64)
    │       ├── 超时检测: 5 分钟 (按 cycle 计数)
    │       └── 检查 error_code
    ├── cur_tail++
    └── ring_cq_doorbell: 写 CQ tail + doorbell

4. 编译时多态 — aclshmemi_backend_traits 系统

4.1 设计理念

采用 CRTP 风格的特化模式（无需虚函数），通过模板特化实现零开销后端分发：

// 基类模板 (未定义, 强制特化)
template <aclshmemi_rdma_backend_t B>
struct aclshmemi_backend_traits {
    template <typename T, bool IS_MASKED>
    struct atomic_op_traits {
        template <aclshmemi_rdma_atomic_op_t OP>
        static ACLSHMEM_DEVICE uint32_t fill_wqe(...);
        template <aclshmemi_rdma_atomic_op_t OP>
        static ACLSHMEM_DEVICE void post_send(...);
    };
};

// XSCALE 特化 (完整实现)
template <>
struct aclshmemi_backend_traits<aclshmemi_rdma_backend_t::XSCALE> {
    template <typename T, bool IS_MASKED>
    struct atomic_op_traits {
        template <aclshmemi_rdma_atomic_op_t ATOMIC_OP_CODE>
        static ACLSHMEM_DEVICE uint32_t fill_wqe(...) {
            return aclshmemi_roce_xscale_fill_wqe_atomic<T, IS_MASKED, ATOMIC_OP_CODE>(...);
        }
        template <aclshmemi_rdma_atomic_op_t ATOMIC_OP_CODE>
        static ACLSHMEM_DEVICE void post_send(...) {
            aclshmemi_rdma_xscdv_post_send_atomic<T, IS_MASKED, ATOMIC_OP_CODE>(...);
        }
    };
};

5.3 CQE 格式 (32 bytes + 32 bytes padding = 64 bytes)

struct aclshmemi_xscdv_diamond_cqe_v2 {
    union { uint32_t flags_qp_id_err_code; struct {
        uint32_t error_code : 8;
        uint32_t qp_id      : 15;
        uint32_t rsv        : 1;
        uint32_t se         : 1;
        uint32_t has_pph    : 1;
        uint32_t type       : 1;
        uint32_t with_imm   : 1;
        uint32_t csum_err   : 4;
    }; };
    uint32_t imm_data;
    uint32_t msg_len;
    uint32_t vni;
    uint32_t ts_l;
    uint32_t ts_h;

    union { uint32_t owner_rsv; struct {

5.6 Owner Bit 机制

XSCALE 使用 owner bit 实现 CQE 生产-消费同步：

bool aclshmemi_roce_xscale_check_cqe_owner(cqe64, cur_tail, depth) {
    bool owner_bit = cqe64->cqe.owner & 1;
    bool expect_bit = (cur_tail / depth) & 1;  // 每轮翻转
    return owner_bit == expect_bit;
}

原理: owner bit 在每轮 CQ 环遍历后翻转，硬件写入 CQE 时设置当前轮的 owner 值，软件通过比较判断 CQE 是否已更新。

5.7 字节序转换

XSCALE 硬件要求 big-endian 格式：

// 手动实现 htobe64 (无标准库依赖)
ACLSHMEM_DEVICE uint64_t aclshmemi_htobe64(uint64_t host_val) {
    return ((host_val >> 0) & 0xFF) << 56 |
           ((host_val >> 8) & 0xFF) << 48 |
           ...  // 逐字节反转
}

注意: 4B fetch/swap 操作在 XSCALE 上获取的数据为 little-endian，需要额外 aclshmemi_htobe32 转换（见 aclshmemi_roce_get_atomic_fetch_data）。

6. 原子操作 — 16 种变体的 IS_MASKED 设计

6.1 变体矩阵

操作 类型 IS_MASKED 硬件 OPCODE WQE 结构
FA 64B false FETCH_AND_ADD(27) ctrl + 2 data + atomic_seg(16B)
FA 64B true 8B_MSK_FETCH_AND_ADD(32) ctrl + 2 data + masked_fa_seg(16B)
FA 32B true 4B_MSK_FETCH_AND_ADD(34) ctrl + 2 data + masked_fa32_seg(16B)
FA 32B false 不支持 static_assert 报错
CAS 64B false CMP_AND_SWAP(26) ctrl + 2 data + atomic_seg(16B)
CAS 64B true 8B_MSK_CMP_AND_SWAP(31) ctrl + 2 data + 2×masked_cas_seg(32B)
CAS 32B true 4B_MSK_CMP_AND_SWAP(33) ctrl + 2 data + masked_cas32_seg(16B)
CAS 32B false 不支持 static_assert 报错

6.2 高级 API 到硬件原语的映射

// atomic_set → CAS with mask=UINT64_MAX, comp_mask=0
aclshmemi_roce_amo_cas<T, true>(dst, nullptr, pe, 0, value, 0, UINT64_MAX, 0, ...);

// atomic_swap → CAS with comp=0, mask=UINT64_MAX, comp_mask=0
aclshmemi_roce_amo_cas<T, true>(dst, nullptr, pe, 0, value, 0, UINT64_MAX, 0, ...);

// atomic_fetch → FA with add=0, boundary=0
aclshmemi_roce_amo_add<T, true>(dst, nullptr, pe, 0, 0, 0, ...);

// atomic_and → CAS: swap_mask=~value, comp_mask=0
aclshmemi_roce_amo_cas<T, true>(dst, nullptr, pe, 0, value, 0, ~value, 0, ...);

// atomic_or → CAS: swap_mask=value, comp_mask=0
aclshmemi_roce_amo_cas<T, true>(dst, nullptr, pe, 0, value, 0, value, 0, ...);

// atomic_xor → FA: add=value, boundary=UINT64_MAX
aclshmemi_roce_amo_add<T, true>(dst, nullptr, pe, 0, value, UINT64_MAX, ...);

设计洞察: 所有逻辑原子操作 (and/or/xor) 都通过 CAS 或 FA 的 masked 变体实现，利用 mask 位控制实现条件更新。

6.3 原子操作结果获取

template <typename T>
ACLSHMEM_DEVICE T aclshmemi_roce_get_atomic_fetch_data(uint32_t pe, uint32_t qp_idx) {
    auto amo_addr = sq_context->amo_addr;  // 硬件回写地址
    dcci_cachelines(amo_addr, sizeof(T));
    T data = *(__gm__ T *)amo_addr;
    if constexpr (sizeof(T) == 4 && XSCALE) {
        return aclshmemi_htobe32(data);  // 4B 需要字节序转换
    }
    return data;
}

8. ZB 算子如何调用 SHMEM

8.1 调用层次

moe_dispatch_normal_zero_buffer (kernel entry)
    │
    ▼
MoeDispatchNormalZeroBuffer::Process()
    │
    ├── syncFlag_.IncrementMagic()       ← ZeroBufferSyncFlag (shmem_ptr)
    ├── syncFlag_.BarrierAll()           ← 跨卡 barrier
    ├── InputToDstOutput()               ← 数据搬运
    │     └── shmem_ptr(expandXOutGM, dstRankId)  ← 获取远端对称地址
    │         └── DataCopyPad(dstGT, xOutTensor, ...)  ← 写入远端 HBM
    └── syncFlag_.BarrierAll()           ← 确保所有写入完成

8.2 shmem_ptr — 对称地址转换

// shmem_device_rdma.hpp
ACLSHMEM_DEVICE __gm__ void *aclshmem_roce_ptr(__gm__ void *ptr, int pe) {
    auto device_state = aclshmemi_get_state();
    uint64_t offset = (uint64_t)ptr - (uint64_t)device_state->heap_base;
    uint64_t remote_ptr = (uint64_t)device_state->p2p_device_heap_base[pe] + offset;
    return (__gm__ void *)remote_ptr;
}

原理: 所有 rank 的对称内存在各自地址空间中具有相同偏移量，通过 远端 heap_base + 本地偏移 计算远端可访问地址。

8.3 ZeroBufferSyncFlag — 跨卡同步协议

内存布局 (per rank, 从 gva_gm + 100KB 开始):
┌─────────────────────────────────────────────────────────┐
│ Magic Area (24KB = 48 cores × 512B)                     │
│   slot[coreIdx] — 单调递增, 每 kernel +1               │
├─────────────────────────────────────────────────────────┤
│ Barrier PingPong Buffer 0 (epWorldSize × 32B)           │
├─────────────────────────────────────────────────────────┤
│ Barrier PingPong Buffer 1 (同上)                         │
├─────────────────────────────────────────────────────────┤
│ Flag PingPong Buffer 0 (epWorldSize × slotsPerRank × 32B)│
├─────────────────────────────────────────────────────────┤
│ Flag PingPong Buffer 1 (同上)                            │
└─────────────────────────────────────────────────────────┘

同步协议:
  Writer (rank A, core i):
    1. 完成数据写入 (DMA to remote shmem)
    2. MTE3 fence (flush all prior writes)
    3. SetFlag(destRank=B, eventID=i, phase)
       → 写 flag 到 B 的本地 shmem slot[A][i]

  Reader (rank B, core j):
    1. WaitFlag(srcRank=A, eventID=i, phase)
       → 轮询 B 的本地 shmem 直到 flag 匹配
    2. 返回后, A 的数据保证可见

8.4 Dispatch 同步流程

void Process() {
    // Step 1: IncrementMagic — 获取 magic M+1
    syncFlag_.IncrementMagic();

    // Step 2: BarrierAll — 确保 Notify 结果全局可见
    //         (putOffset, recvData 等已计算完成)
    syncFlag_.BarrierAll();

    // Step 3: 通过 shmem 写 token 数据到远端 rank
    InputToDstOutput();
    //   └── shmem_ptr(expandXOutGM, dstRankId) → 远端地址
    //   └── DataCopyPad(dstGT, xOutTensor, ...) → 写入

    // Step 4: BarrierAll — 确保所有 rank 写入完成
    //         后续 GMM  kernel 可安全消费
    syncFlag_.BarrierAll();
}

8.5 BarrierAll 实现

void BarrierAll() {
    // Step 1: 卡内同步
    SyncAll<true>();

    // Step 2: 分配 rank 到 core (SplitCoreCal)
    // Step 3: 写 barrier flag 到所有远端 rank
    for (r = startRank; r < endRank; r++) {
        if (r == epRankId_) continue;
        remotePtr = shmem_ptr(flagBaseGM_, r);
        DataCopy(remoteSlot, ubFlag, FLAG_ELEM_NUM);
    }

    // Step 4: 轮询本地 slot 等待所有远端 flag
    for (r = startRank; r < endRank; r++) {
        if (r == epRankId_) continue;
        do { DataCopy(ubFlag, localSlot, ...); }
        while (ubFlag.GetValue(0) != barrierFlag);
    }

    // Step 5: 卡内同步
    SyncAll<true>();
}

8.6 数据搬运核心逻辑

void InputToDstOutput() {
    // 1. 读取 putOffset (全局 recv_count 前缀和)
    DataCopyPad(putOffsetTensor, putOffsetGT, ...);

    // 2. 读取 expertIds 和 sendTokenIdx
    DataCopyPad(expertIdsTensor, expertIdsGT[startTokenId], ...);
    DataCopyPad(sendTokenIdxTensor, sendTokenIdxGT[startTokenId], ...);

    // 3. 逐 token 处理
    for (tokenIndex = startTokenId; tokenIndex < endTokenId; ++tokenIndex) {
        dstExpertId = expertIdsTensor(...);
        dstRankId = dstExpertId / moeExpertNumPerRank;
        curExpertIdx = sendTokenIdxTensor(...);
        dstExpertOffset = putOffsetTensor(dstExpertId * epRankSize + epRankId);

        // 计算远端输出地址
        auto ptr = shmem_ptr(expandXOutGM, dstRankId);
        uint64_t expandXByteOffset = hUBAlignSize * (dstExpertOffset + curExpertIdx);
        dstGT.SetGlobalBuffer(ptr + expandXByteOffset);

        // 写入数据
        DataCopyPad(dstGT, xOutTensor, xOutCopyParams);
    }
}

9. 与 master 版 NDA 的差异 — cam_feature 上的 SHMEM 框架演进

9.1 架构分层演进

维度 master (原始 NDA) cam_feature (ZB 算子)
API 层 aclshmemx_roce_put_nbi/get_nbi/atomic_* shmem_ptr() + DataCopyPad
传输模式 RDMA one-sided (NIC 直接读写) 设备端 AIV 直接写 HBM (无需 NIC)
同步机制 aclshmemx_roce_quiet (轮询 CQ) ZeroBufferSyncFlag (flag-based)
内存模型 对称内存 + 远端指针转换 对称内存 + shmem_ptr 地址转换
后端 XSCALE RDMA 硬件 AIV core 直接 store (不走 RDMA)
原子操作 16 种硬件原子 (FA/CAS masked) 不需要 (单方面写入)

9.2 关键差异分析

1. 数据传输路径不同

master 的 NDA 框架使用 RDMA one-sided 操作： - 由 NIC 直接执行 RDMA_WRITE / RDMA_READ - 需要 SQ doorbell / CQ poll 机制 - 数据路径: Local HBM → NIC → (wire) → NIC → Remote HBM

cam_feature 的 ZB 算子使用 AIV core 直接写入： - 由 AI Vector core 执行 DataCopyPad (本质是 store 指令) - 利用对称内存的地址直接可达性 - 数据路径: Local UB → AIV store → Remote HBM (通过内部互联)

2. 同步协议升级

master: aclshmemx_roce_quiet 需要轮询 CQ，等待硬件完成中断。 cam_feature: ZeroBufferSyncFlag 使用 magic + phase 的 flag 协议，支持： - ping-pong buffer 避免 stale flag - per-core / per-rank / per-expert 多级粒度 - 无需硬件中断，纯软件轮询

3. 内存布局精细化

cam_feature 引入了更精细的内存分区：

gva_gm + 0:        Notify 区域 (2MB)
gva_gm + 2MB:      Dispatch 区域 (3MB)
gva_gm + 5MB:      Combine 区域 (3MB)
gva_gm + 100KB:    Flag Area (同步标志区)
  ├── Magic Area (24KB)
  ├── Barrier PingPong (2 × epWorldSize × 32B)
  └── Flag PingPong (2 × epWorldSize × slotsPerRank × 32B)

9.3 演进趋势

  1. 从 RDMA 到 AIV-direct: ZB 算子不直接使用 NDA 的 RDMA API，而是利用对称内存 + AIV 直接写入，避免了 NIC 交互开销。
  2. 从 quiet 到 flag-based sync: 同步机制从硬件 CQ 轮询演进为软件 flag 协议，更灵活且支持复杂 DAG 依赖。
  3. 从粗粒度到细粒度: 同步粒度从 per-QP 演进到 per-core/per-expert，支持更精细的流水线。
  4. SHMEM 作为基础设施: NDA 框架提供的 shmem_ptr 地址转换和对称内存管理成为 ZB 算子的基础，但 RDMA 操作本身被绕过。

11. 关键设计洞察

11.1 编译时多态 vs 运行时多态

NDA 框架大量使用模板特化而非虚函数： - 零运行时开销: 所有后端选择在编译时确定 - 类型安全: 未实现的操作在编译时报错 (static_assert) - 代码生成: 每个后端生成独立的特化代码，无分支预测失败

11.2 内存一致性模型

XSCALE 后端使用 dcci_cachelines (cache line flush) 确保： - WQE 写入对 NIC 可见后再 ring doorbell - CQ polling 时读取最新 CQE - 原子操作结果对 CPU 可见

11.3 超时与容错

// 5 分钟超时 (按 cycle 计数)
constexpr uint64_t ACLSHMEMI_XSC_POLL_CQ_TIMEOUT_DURATION = 5ULL * 60 * 1000000;
constexpr uint64_t ACLSHMEMI_XSC_POLL_CQ_TIMEOUT_CYCLES =
    ACLSHMEMI_XSC_POLL_CQ_TIMEOUT_DURATION * ACLSHMEMI_XSC_CYCLE_TO_TIME_BASE;
// 950M 系列: ×50, 其他: ×1000

11.4 字节序陷阱

XSCALE 硬件使用 big-endian： - 所有 WQE 中的 64-bit 数据需要 htobe64 转换 - 4B fetch/swap 结果为 little-endian，需要额外 htobe32 转换 - 这是设备端代码中容易出错的地方

11.5 SQ 流控

constexpr int XSCALE_POLL_CQ_THRESHOLD = 16;
// 当 SQ 即将满时 (head + 16 == tail), 先 poll_cq 等待空闲
if ((cur_head + XSCALE_POLL_CQ_THRESHOLD) % depth == (depth + cur_tail) % depth) {
    aclshmemi_roce_poll_cq<XSCALE>(pe, qp_idx, cur_head, ...);
}

设计: 提前 16 个 WQE 触发反压，避免 SQ 完全填满导致死锁。

12. 总结

SHMEM/NDA 框架是 UMDK/CAM 中设备端 RDMA 通信的基础设施层，其核心设计特点：

  1. 分层架构: 公共 API → 调度层 → 后端抽象 → 硬件特化，每层职责清晰
  2. 编译时多态: 通过 aclshmemi_backend_traits 实现零开销后端分发
  3. 硬件原生: XSCALE 后端直接操作 WQE/CQE/doorbell，无中间抽象层
  4. Host-Device 分离: Host 负责 QP 建立/MR 注册/信息收集，Device 独立执行 RDMA 操作
  5. ZB 算子演进: cam_feature 上的 ZB 算子利用对称内存绕过 RDMA，使用 AIV 直接写入 + flag 同步，实现了更低延迟的通信

该框架目前仅支持 Ascend950 + XSCALE 后端，IN_DIE 后端尚未实现（编译时 fallback 到 XSCALE）。
