FusedDeepMoE 通信模式 & SHMEM 编译机制深度剖析

代码深读报告 | 聚焦 FusedDeepMoE 的 SHMEM vs 非 SHMEM 通信路径、raw_distributed 通信原语、三条执行路径差异、Workspace 内存布局，以及编译系统的 SHMEM 注入机制。

1. 总体架构概览

FusedDeepMoE 是 CAM 中最复杂的融合算子，将 Dispatch → GMM1 → SwiGLU → GMM2 → Combine 五个阶段融合进单一 kernel。其核心挑战在于：

  • 通信与计算的融合：dispatch/combine 需要跨 EP rank 搬运 token，同时 GMM 需要密集 Cube 计算
  • 三条执行路径：非深度融合 / 深度融合 / 共享专家，各自通信模式不同
  • SHMEM vs HCCL Window：当前默认使用 HCCL Window 通信，SHMEM 路径为未来预留
┌─────────────────────────────────────────────────────────────────────┐
│                     FusedDeepMoE 融合流程                           │
│                                                                     │
│  x ──→ Dispatch ──→ GMM1 ──→ SwiGLU ──→ GMM2 ──→ Combine ──→ out  │
│       (HCCL Window)  (Cube)   (Vector)   (Cube)  (HCCL Window)     │
│                                                                     │
│  shared_x ──→ ShareGMM1 ──→ SwiGLU ──→ ShareGMM2 ──→ share_out    │
│               (Cube)        (Vector)   (Cube)                       │
└─────────────────────────────────────────────────────────────────────┘

Process() 函数揭示的三阶段执行模型（第354-448行）：

template <TemplateMC2TypeClass>
__aicore__ inline void FusedDeepMoe<TemplateMC2TypeFunc>::Process()
{
    // 阶段1: Dispatch（仅非深度融合路径）
    if constexpr ((EXEC_FLAG & EXEC_FLAG_DEEP_FUSE) == 0) {
        MoeDistributeDispatchImpl::CamMoeDistributeDispatch<...> dispatcher;
        dispatcher.Init(...);
        dispatcher.Process();  // ← HCCL Window AlltoAll
    }

    // 阶段2: GMM1 + SwiGLU（Cube 计算，无通信）
    GmmDeqSwigluQuant<...>(...);

    // 阶段3: GMM2 + Combine（融合通信计算）
    MoeDistributeCombineImpl::CamMoeDistributeCombine<...> combiner;
    GmmDeq<...>(..., &combiner);  // ← combiner 在 GMM2 的 AIV 侧执行
}

2.2 通信上下文获取：HCCL Window 模式

关键发现：当前代码中没有任何 SHMEM 分支。 所有通信通过 HcclOpResParam 结构体（HCCL Window 描述符）实现：

// cam_moe_distribute_dispatch.h 第236-237行
winContext_[COMM_EP_IDX] = (__gm__ HcclOpResParam *)AscendC::GetHcclContext<HCCL_GROUP_ID_0>();
winContext_[COMM_TP_IDX] = (__gm__ HcclOpResParam *)AscendC::GetHcclContext<1>();

// cam_moe_distribute_combine.h 第245-246行
auto contextGM0 = AscendC::GetHcclContext<HCCL_GROUP_ID_0>();
epWinContext_ = (__gm__ HcclOpResParam *)contextGM0;

HCCL Window 地址解析（dispatch 第83-91行）：

__aicore__ inline GM_ADDR GetWindAddrByRankId(uint8_t ctxIdx, const int32_t rankId)
{
    uint32_t curRankId = ctxIdx == COMM_EP_IDX ? epRankId_ : tpRankId_;
    if (curRankId == rankId) {
        // 本地 Window：直接访问 localWindowsIn
        return (GM_ADDR)(winContext_[ctxIdx]->localWindowsIn) + winDataSizeOffset_ + rankId * OPT_RANK_OFFSET;
    }
    // 远端 Window：通过 remoteRes 指针访问对端 windowsIn
    return (GM_ADDR)(((HcclRankRelationResV2 *)(winContext_[ctxIdx]->remoteRes[rankId].nextDevicePtr))->windowsIn)
           + winDataSizeOffset_ + rankId * OPT_RANK_OFFSET;
}

通信模式总结： - 数据通道：windowsIn — 存放实际 token 数据（int8/half/bf16） - 状态通道：windowsExp — 存放同步状态标志（flag/sumTarget） - 同步机制：通过 DataCopy 在 Window 上写入 flag，轮询对端 flag 实现握手

3. SHMEM vs 非 SHMEM 代码路径

3.1 当前状态：无 SHMEM 分支

对整个 fused_deep_moe/ 目录进行全量 grep：

grep -rn "aclshmem\|SHMEM\|shmem_" src/cam/comm_operator/ascend_kernels/fused_deep_moe/
# 结果：0 命中

结论：FusedDeepMoE kernel 代码中不存在任何 SHMEM 相关代码路径。 所有通信均通过 HCCL Window（HcclOpResParam）实现。

3.2 编译系统的 SHMEM 预留

虽然 kernel 代码没有 SHMEM 分支，但编译系统已完整预留 SHMEM 注入点：

层级 文件 SHMEM 机制
算子选择 select_ops.py requires_shmem() 过滤
编译脚本 compile_ascend_proj.sh SHMEM_HOME_PATH 检测 + include 注入
CMake 工具 ascendc_impl_build.py SHMEM include 路径硬编码注入

3.3 抽象层设计

当前通信原语通过 HcclOpResParam 抽象，没有引入统一的 CommunicationInterface。dispatch/combine 直接依赖 HCCL 的 Window 结构体：

// 通信上下文是 HCCL 强绑定的
__gm__ HcclOpResParam *winContext_[COMM_NUM];  // dispatch
__gm__ HcclOpResParam *epWinContext_;           // combine
__gm__ HcclOpResParam *tpWinContext_;           // combine (TP domain)

如果要支持 SHMEM，需要： 1. 引入 ICommContext 抽象基类 2. 派生 HcclWindowContext 和 ShmemContext 3. dispatch/combine 通过基类指针访问通信原语

void AlltoAllDispatch()
{
    // 1. 加载 expertIds（决定每个 token 去哪个 expert/rank）
    DataCopyPad(expertIdsTensor_, expertIdsGMTensor_, expertIdsCntParams, copyPadParams);

    // 2. 计算每个 expert 的 token 计数 + 状态更新
    if (enableAivOpt_) {
        CountAndPrepareStatusWithAivOpt(expertIdsCnt);  // 优化路径
    } else {
        for (uint32_t tokenIndex = 0; tokenIndex < expertIdsCnt; ++tokenIndex) {
            expertCountTensor_(tokenIndex) = statusTensor_(expertId * INT32_NUM_PER_BLOCK + 1);
            statusTensor_(expertId * INT32_NUM_PER_BLOCK + 1)++;  // 原子计数
        }
    }

    // 3. 将 token 数据写入目标 rank 的 Window
    SendToMoeExpert();
}

void SendToMoeExpert()
{
    for (uint32_t tokenIndex = startTokenId; tokenIndex < endTokenId; ++tokenIndex) {

}

5.2 路径对比

特性 非深度融合 深度融合 (DEEP_FUSE) 共享专家 (SHARED_EXPERT)
触发条件 moeExpertNumPerRank == 1 moeExpertNumPerRank > 1 share_gmm1_weight != nullptr

核分配 AIC + AIV 1:1 AIC + AIV 1:2 (1C2V) AIC + AIV 1:2

5.3 深度融合路径的通信融合

GMM2 AIV 侧执行流程（grouped_matmul_slice_m_per_token_dequant_multistage_workspace.h 第291-458行）：

template <>
CATLASS_DEVICE void operator()<AscendC::AIV>(Params const &params)
{
    auto *combiner = (CamMoeDistributeCombine *)params.combiner;

    // 1. 先执行 GMM2 的 epilogue（反量化 + SwiGLU 后处理）
    for (uint32_t groupIdx = 0; groupIdx < params.problemCount; ++groupIdx) {
        for (uint32_t loopIdx = startLoopIdx; loopIdx < coreLoops; loopIdx += coreNum) {
            CheckSyncFlag(syncGmAddr + SOFT_SYNC_OFFSET, coreIdx, target);  // 等待 AIC 完成
            blockEpilogue(gmGroupOffsetD, groupIdx, ...);  // epilogue 计算
        }
    }

    // 2. 共享专家 GMM2（如有）
    if constexpr (EXEC_FLAG & EXEC_FLAG_SHARED_EXPERT) {
        // ... 执行共享专家 GMM2 epilogue
    }

    // 3. 融合 Combine 通信
    if constexpr (EXEC_FLAG & EXEC_FLAG_SHARED_EXPERT) {
        if (GetSubBlockIdx() == 1) {
            combiner->ProcessCombine();  // ← 在 GMM2 AIV 上执行 Combine
        }
    } else if constexpr (EXEC_FLAG & EXEC_FLAG_DEEP_FUSE) {
        if (GetSubBlockIdx() == 0) {
            combiner->AllToAllSend();    // ← 发送阶段
        } else {
            combiner->ReducePermute();   // ← 接收+累加阶段
        }
    } else {
        combiner->Process();             // ← 非融合：完整 Combine
    }
}

关键同步机制： - SEND_SYNC_EVENT_ID = 6 — GMM2 epilogue 完成 → Combine 开始 - RECV_SYNC_EVENT_ID = 7 — Combine 接收完成 → 主 kernel 继续 - CheckSyncFlag — AIV 等待 AIC 的 Cube 计算完成（soft sync）

9.2 SHMEM 集成挑战

挑战 说明
同步语义差异 HCCL Window 是双向对称的（localWindowsIn + remoteRes），SHMEM 是单边 RDMA
状态通道设计 当前 windowsExp 用于 flag 同步，SHMEM 需要额外设计 fence/notify 机制
多域通信 dispatch 使用 COMM_EP_IDX + COMM_TP_IDX 两个域，SHMEM 需要对应两个通信组
地址计算 expertPerSizeOnWin_、winDataSizeOffset_ 等偏移量需要重新映射到 SHMEM 内存池
量化融合 dispatch 中量化+Window 拷贝是融合的，SHMEM 路径需要保持同样的融合度

9.3 推荐集成策略

  1. Phase 1：引入 ICommContext 抽象，HCCL 路径保持不变
  2. Phase 2：实现 ShmemContext，在 notify_dispatch 等简单算子上验证
  3. Phase 3：将 SHMEM 路径扩展到 FusedDeepMoE 的 dispatch/combine
  4. Phase 4：性能调优，利用 SHMEM 的单边 RDMA 降低同步开销

11. 结论

  1. FusedDeepMoE 当前完全基于 HCCL Window 通信，kernel 代码中不存在任何 SHMEM 分支。

  2. 通信原语通过 HcclOpResParam 强绑定，dispatch/combine 直接操作 windowsIn（数据）和 windowsExp（状态）。

  3. 三条执行路径的通信差异： - 非深度融合：dispatch 和 combine 独立执行 - 深度融合：combine 融合进 GMM2 AIV 侧，通过 AllToAllSend + ReducePermute 实现 - 共享专家：额外执行共享专家 GMM1+GMM2，combine 通过 ProcessCombine 执行

  4. Workspace 内存布局采用复用策略（ENABLE_REUSE_MEMORY），shareX1/X2 共用、shareX1Scale/X2Scale 共用、swapSpace/shareSwiglu 共用。

  5. 编译系统已完整预留 SHMEM 注入点： - select_ops.py 通过 requires_shmem 过滤 - compile_ascend_proj.sh 通过 SHMEM_HOME_PATH 注入 include - ascendc_impl_build.py 硬编码 SHMEM include 路径（含一个路径 bug）

  6. SHMEM 集成需要引入通信抽象层，将 HcclOpResParam 替换为 ICommContext 接口，并在 dispatch/combine 中通过模板参数化通信后端。

  7. ascendc_impl_build.py 第172行存在路径 bug：SHMEM_HOME_PATH + "shmem/include/device" 缺少 / 分隔符。
