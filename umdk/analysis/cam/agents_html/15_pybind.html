1.3 TORCH_LIBRARY 注册 — 算子 Schema 定义

TORCH_LIBRARY(umdk_cam_op_lib, m) 定义了 6 个算子的完整 schema：

(1) fused_deep_moe — 最复杂的算子

m.def("fused_deep_moe(Tensor x, Tensor expert_ids, Tensor[] gmm1_weight, Tensor[] gmm1_weight_scale, \
    Tensor[] gmm2_weight, Tensor[] gmm2_weight_scale, Tensor expert_scales, \
    Tensor? share_gmm1_weight, Tensor? share_gmm1_weight_scale, \
    Tensor? share_gmm2_weight, Tensor? share_gmm2_weight_scale, \
    Tensor? expert_smooth_scales, Tensor? share_smooth_scales, Tensor? x_active_mask, \
    Tensor[] gmm1_bias, Tensor[] gmm2_bias, \
    Tensor? share_gmm1_bias, Tensor? share_gmm2_bias, \
    str group_ep, int ep_rank_size, int ep_rank_id, int moe_expert_num, \
    int quant_mode, int global_bs) -> Tensor[]");

关键设计点： - 返回 Tensor[]（3 个输出：output, shareOutput, expertTokenNums） - 支持 Shared Expert（share_gmm1/2_weight 可选） - 支持 Smooth Quantization（expert_smooth_scales） - 支持 Active Mask（x_active_mask，用于 MC2 动态 batch） - 支持 Bias（gmm1/2_bias） - quant_mode 参数控制量化模式（0=W8A8, 1=其他）

(2) all2_all_detour — All-to-All 通信 Detour

(3) reduce_scatter_detour — ReduceScatter 通信 Detour

(4) cam_get_comm / cam_free_comm — 通信域管理

m.def("cam_get_comm(int comm_id, int rank, int group_size, str server_ip_port) -> Tensor");
m.def("cam_free_comm(int comm_id) -> Tensor");

(5) a2e — Attention 到 Expert 的数据路由

m.def("a2e(Tensor x, Tensor? expert_ids, Tensor? scales, int batch_size, int hidden_size, \
    int topk, int expert_rank_size, int atten_rank_size, int rank, str group_ep, \

(6) e2a — Expert 到 Attention 的数据回传

m.def("e2a(Tensor expand_x, Tensor atten_batch_size, int batch_size, int hidden_size, \

1.4 统一的 Autograd 注册模式

所有算子遵循完全相同的四层注册架构（以 fused_deep_moe 为例）：

FusedDeepMoeImplAutograd  (入口，调用 apply)
  └─ FusedDeepMoe::apply  (torch::autograd::Function)
      └─ FusedDeepMoe::forward → FusedDeepMoeImpl
          └─ Dispatcher::findSchemaOrThrow → 分发到具体实现

三层 TORCH_LIBRARY_IMPL 注册：

后端 注册函数 用途
PrivateUse1 FusedDeepMoeImplNpu NPU 实际执行
AutogradPrivateUse1 FusedDeepMoeImplAutograd Autograd 支持
Meta FusedDeepMoeImplMeta 元设备（shape 推断）

关键实现细节： - FusedDeepMoeImplNpu 中调用 EXEC_NPU_CMD(aclnnFusedDeepMoe, ...) 执行实际 NPU 命令 - backward 返回全空 Tensor（所有算子反向传播均未实现，仅支持推理） - Meta 实现仅做 shape 推断，不分配实际内存

2. PyTorch NPU Extension Bridge

2.1 pytorch_npu_helper.hpp — 核心执行引擎

这是整个 pybind 层最底层的基础设施，提供了 ATen → ACL 类型转换 和 NPU 命令执行 两大功能。

类型转换体系

at::Tensor → aclTensor*    (ConvertType)
at::Scalar → aclScalar*    (ConvertType)
at::IntArrayRef → aclIntArray*  (ConvertType)
at::TensorList → aclTensorList* (ConvertType)
c10::optional<T> → T* or nullptr  (ConvertType)

ScalarType → ACL DataType 映射表：

constexpr aclDataType kATenScalarTypeToAclDataTypeTable[] = {
    at::ScalarType::Byte → ACL_UINT8,
    at::ScalarType::Char → ACL_INT8,
    at::ScalarType::Short → ACL_INT16,
    at::ScalarType::Int → ACL_INT32,
    at::ScalarType::Long → ACL_INT64,
    at::ScalarType::Half → ACL_FLOAT16,
    at::ScalarType::Float → ACL_FLOAT,
    at::ScalarType::Double → ACL_DOUBLE,
    at::ScalarType::BFloat16 → ACL_BF16,
    // ... Complex 类型
};

Format 推断逻辑： - dim=3 → ACL_FORMAT_NCL - dim=4 → ACL_FORMAT_NCHW - dim=5 → ACL_FORMAT_NC1HWC0 - INT8 + (dim=2 or 3) → ACL_FORMAT_FRACTAL_NZ - 其他 → ACL_FORMAT_ND

EXEC_NPU_CMD 宏 — 核心执行宏

#define EXEC_NPU_CMD(aclnn_api, ...)

这是整个 CAM 算子执行的统一入口，执行流程：

  1. 动态加载 OpAPI 函数：通过 dlsym 从 libopapi.so / libcust_opapi.so 加载
  2. 初始化大页内存：InitHugeMemThreadLocal
  3. 类型转换：ConvertTypes(__VA_ARGS__, workspace_size_addr, executor_addr)
  4. 获取 Workspace 大小：调用 GetWorkspaceSize
  5. 分配 Workspace：通过 at::empty 分配 NPU Tensor
  6. 执行算子：lambda 捕获所有参数，通过 OpCommand 提交到 NPU 流
  7. 释放资源：ReleaseConvertTypes + ReleaseHugeMem + UnInitHugeMemThreadLocal

关键设计点： - 使用 static 局部变量缓存函数地址（首次调用时解析，后续复用） - 线程本地存储 g_hashBuf[8192] 用于算子 hash 计算 - 支持 InitHugeMemThreadLocal / UnInitHugeMemThreadLocal 可选大页内存管理 - 通过 at_npu::native::OpCommand 提交命令，支持自定义 handler

2.2 关于 NPUBridge/NPUStorageImpl

注意： NPUBridge.h/.cpp 和 NPUStorageImpl.h/.cpp 在 cam_feature 分支上不存在。这些文件可能是： - 其他分支（如 main 或 master）的内容 - 旧版 CAM 的 PyTorch 存储扩展实现 - 已被 pytorch_npu_helper.hpp 取代

当前 cam_feature 分支使用 pytorch_npu_helper.hpp 作为唯一的 NPU 交互层，不再需要单独的 Bridge/StorageImpl 层。

4.2 SHMEM 配置机制（从编译系统推断）

虽然 Docker 文件不存在，但从 setup.py 和编译脚本可以完整推断 SHMEM 部署要求：

环境变量依赖

变量 必需 说明
SHMEM_HOME_PATH 否（但 SHMEM 算子需要） Ascend-SHMEM 安装路径
CAM_COMM_PATH 是 cam_comm 库构建路径
ASCEND_HOME_PATH 是 CANN 工具包路径

SHMEM 编译时配置

# setup.py
shmem_home_path = os.environ.get("SHMEM_HOME_PATH")
if shmem_home_path:
    include_dirs.append(os.path.join(shmem_home_path, 'shmem', 'include'))
    library_dirs.append(os.path.join(shmem_home_path, 'shmem', 'lib'))
    libraries.append("shmem")
    compile_args.append("-DSHMEM_ENABLED")

SHMEM 运行时配置

从 buffer.cpp 可知： - 使用 aclshmemx_init_attr 初始化 SHMEM 通信域 - 通过 ip_port（如 tcp://172.16.0.146:8766）建立跨节点连接 - aclshmem_malloc 分配共享内存（默认 3GB） - 分为 ext_info（2MB）和 shmem_workspace（剩余空间）两部分

// buffer.cpp
status = aclshmemx_init_attr(ACLSHMEMX_INIT_WITH_DEFAULT, &attributes);
ext_info_ptr_ = aclshmem_malloc(EXT_INFO_SIZE);  // 2MB
shmem_workspace_ptr_ = aclshmem_malloc(shmem_workspace_size_);  // 剩余

SHMEM 算子自动剔除机制

# select_ops.py
def requires_shmem(registry, op):
    return bool(get_meta(registry, op).get("requires_shmem", False))

当 SHMEM_HOME_PATH 未设置时，所有 requires_shmem=true 的算子自动跳过。

  • 超时机制、连接保活、Channel 多对一
  • UMQ 消息队列（enqueue/dequeue + Post/Poll 两种模型）
  • UMQ Token 安全校验

DLock — 分布式锁

  • 原子锁/公平锁/读写锁的基本和批量操作
  • 分布式对象管理（FAA/CAS 原子操作）
  • 故障恢复（Primary Server 故障后重建）
  • 安全传输加密

UMS — Socket 兼容

  • 标准 Socket API 兼容
  • ums_run 透明替换工具

13 个算子，其中 6 个需要 SHMEM。

选项 功能 cam_feature 特有

-a <ops> 选择算子列表
-q W4A8 量化变体
-d Debug 构建
-t 仅构建 UT
-p 仅构建 whl
-r 仅构建 run 包

7.3 关键编译规则

  • fused_deep_moe 与 fused_deep_moe_w4a8 互斥（共享源文件名）
  • fused_deep_moe_fwk 独立，可与任意变体共存
  • SHMEM 算子在 SHMEM_HOME_PATH 未设置时自动跳过
  • -m 选项仅 cam_feature 有（master 无 cam_comm）

8. 综合洞察与架构研判

8.1 架构演进方向

从通信算子到融合算子的跃迁：

Phase 1 (master): dispatch + combine 分离通信算子
Phase 2 (cam_feature): 
  → FusedDeepMoe: dispatch + GMM1 + SwiGLU + GMM2 + combine 五合一
  → A2E/E2A: MoE-Attention 分离通信
  → SHMEM ZeroBuffer: 零拷贝通信
  → All2All/ReduceScatter Detour: 自定义通信原语

8.2 关键设计决策

  1. 全推理导向：所有算子 backward 返回空 Tensor，表明 CAM 定位为推理加速库，训练依赖 PyTorch 原生 autograd 或外部框架。

  2. 三层后端注册：PrivateUse1（执行）+ AutogradPrivateUse1（梯度）+ Meta（shape 推断）是 PyTorch 自定义算子的最佳实践。

  3. SHMEM 可选编译：通过 SHMEM_HOME_PATH 环境变量控制，不依赖 SHMEM 时自动降级，保证兼容性。

  4. Dispatcher 分发模式：所有算子通过 torch::Dispatcher 分发，支持 lazy execution、symbolic tracing 和 torch.compile。

  5. EXEC_NPU_CMD 统一宏：封装了 OpAPI 加载、workspace 管理、内存释放的完整生命周期，大幅降低算子开发复杂度。

8.3 与竞品对比

特性 CAM (Huawei) NCCL (NVIDIA) MS-AMP (Microsoft)
融合 MoE 算子 ✅ FusedDeepMoe ❌ 分离 ❌ 分离
自定义 All-to-All ✅ Detour ❌ 标准 ❌
零拷贝通信 ✅ SHMEM ZB ✅ GPUDirect ❌
MoE-Attn 分离 ✅ A2E/E2A ❌ ❌
量化支持 ✅ W8A8/W4A8 ❌ ✅ FP8
反向传播 ❌ 仅推理 ✅ ✅

8.4 风险与建议

  1. 测试覆盖严重不足：13 个算子仅 1 个 UT，建议优先补充 FusedDeepMoe 和 SHMEM 测试。

  2. 反向传播缺失：如果未来需要训练支持，需实现完整的 backward 函数。

  3. SHMEM 网络配置：ip_port 硬编码在示例中，生产环境需要动态服务发现。

  4. W4A8 量化：示例文件较大（33KB），包含 CPU 模拟器，说明量化精度验证是难点。

  5. Docker 部署缺失：无 Dockerfile/docker-compose，建议补充容器化部署方案。
