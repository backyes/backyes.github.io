1. 整体架构与融合策略

1.1 算子定义

FusedDeepMoE 是 CAM 中最复杂的融合算子，将 MoE Decode 阶段的完整计算流水线融合进单一 kernel：

标准小算子组合 (5次kernel launch + 多次GM读写):
  Dispatch → GMM1 → Swiglu → GMM2 → Combine

FusedDeepMoE (1次kernel launch, 全在片上完成):
  Dispatch → GMM1 → Swiglu → GMM2 → Combine (全部融合)

1.2 输入/输出签名

输入 (14个): | 索引 | 名称 | 类型 | 说明 | |------|------|------|------| | 0 | x | bf16/fp16 | 输入 token [bs, h] | | 1 | expert_ids | int32 | 专家索引 [bs, topk] | | 2 | gmm1_weight | int8 (TensorList) | 路由专家 GMM1 权重 | | 3 | gmm1_weight_scale | fp32/bf16/fp16 | GMM1 权重 scale | | 4 | gmm2_weight | int8 (TensorList) | 路由专家 GMM2 权重 | | 5 | gmm2_weight_scale | fp32/bf16/fp16 | GMM2 权重 scale | | 6 | expert_scales | fp32 | token级专家权重 [bs, topk] | | 7-8 | share_gmm1_weight/scale | int8/fp32 | 共享专家 GMM1 权重 (可选) | | 9-10 | share_gmm2_weight/scale | int8/fp32 | 共享专家 GMM2 权重 (可选) | | 11 | expert_smooth_scales | fp32 | 平滑 scale (可选) | | 12 | share_smooth_scales | fp32 | 共享专家平滑 scale (可选) | | 13 | x_active_mask | bool | token 激活掩码 (可选) |

输出 (3个): - output: [bs, h] — MoE 路由输出 - share_output: [bs, h] — 共享专家输出 - expert_token_nums: [moeExpertNumPerRank] — 每个专家接收的 token 数

属性 (6个): - group_ep, ep_rank_size, ep_rank_id, moe_expert_num, quant_mode, global_bs

1.3 为什么要融合？

核心动机：消除中间数据的 Global Memory 往返

在标准小算子组合中： 1. Dispatch 写出 expand_x, dynamic_scales 到 GM → GMM1 读回 (读写 2次) 2. GMM1 写出 int32 累加结果到 GM → Swiglu 读回 (读写 2次) 3. Swiglu 写出 int8 + scale 到 GM → GMM2 读回 (读写 2次) 4. GMM2 写出 结果到 GM → Combine 读回 (读写 2次)

总计至少 8 次 GM 访问，对于 Decode 阶段（batch小、延迟敏感），GM 带宽是瓶颈。

FusedDeepMoE 的融合策略： - X1 (GMM1输入)：AIV 在 UB 中完成 Dispatch 量化后，直接通过 workspace 偏移量传递给 AIC 做 GMM1 - Swiglu 输出：GMM1 的 int32 结果在 L0C/UB 中直接做 Swiglu + 动态量化，写出 float 到 workspace - X2 (GMM2输入)：Swiglu 后立即做动态量化为 int8，通过 workspace 传递给 GMM2 - GMM2 输出：直接在 UB 中完成 Combine 的 AlltoAll 发送

关键收益：中间数据 X1、SwigluOut、X2 全部在片上 workspace 中流转，不访问 GM。

2.2 Workspace 布局计算

Workspace 是 FusedDeepMoE 的核心设计，所有中间数据在此共享：

// 各段偏移量计算 (启用 ENABLE_REUSE_MEMORY 时复用内存)
WorkSpaceOffset:
  ├── shareX1TokenOffset / x1TokenOffset      // GMM1 输入 (int8)
  ├── shareX1ScaleOffset / x1ScaleOffset      // GMM1 输入 per-token scale
  ├── swapSpaceOffset / shareSwigluOffset     // GMM1 输出 (int32) / Swiglu 输出 (fp32)
  ├── shareX2TokenOffset / x2TokenOffset      // GMM2 输入 (int8)
  ├── shareX2ScaleOffset / x2ScaleOffset      // GMM2 输入 per-token scale
  ├── y2TokenOffset                           // GMM2 输出 (浅融合使用)
  ├── groupListOffset                         // 专家 token 数前缀和
  ├── expandIdxOffset                         // Dispatch 索引
  ├── epSendCountOffset                       // AlltoAll 发送计数
  └── reservedOffset                          // 保留空间 (256KB)

内存复用策略 (ENABLE_REUSE_MEMORY): - X1 和 X2 共享同一块 workspace 空间（因为 X1 和 X2 生命周期不重叠） - Scale 空间同理复用 - SwapSpace 和 Swiglu 输出复用

2.4 Tiling Key 路由机制

使用华为 ASCENDC_TPL 机制，通过 5 个 bool 标志 + 1 个 ARCH 参数组合出多种执行路径：

// 关键执行标志
EXEC_FLAG_DEEP_FUSE      = 1 << 0   // 是否深度融合 (moeExpertNumPerRank != 1)
EXEC_FLAG_TENSOR_LIST    = 1 << 1   // 权重是否为 TensorList
EXEC_FLAG_X_ACTIVE_MASK  = 1 << 2   // 是否启用 x_active_mask
EXEC_FLAG_SHARED_EXPERT  = 1 << 3   // 是否有共享专家
EXEC_FLAG_SMOOTH_QUANT   = 1 << 4   // 是否启用 smooth quant

Tiling Key 决定： - 是否跳过独立 Dispatch（深度融合时 Dispatch 在 kernel 内完成） - 是否走共享专家路径 - 是否启用 smooth scale 处理

3.1 Kernel 入口

    // 实例化 FusedDeepMoe<...> op; op.Init(...); op.Process();

1C2V 模式：每个 AI Core 组包含 1 个 AIC (AI Core Compute) + 2 个 AIV (AI Core Vector)。AIC 负责 GMM 计算，AIV 负责通信调度。

3.2 FusedDeepMoe::Process() 主流程

void Process() {
    // 1. 非深度融合时，先执行独立 Dispatch
    if constexpr ((EXEC_FLAG & EXEC_FLAG_DEEP_FUSE) == 0) {
        if (g_coreType == AIV) {
            CamMoeDistributeDispatch dispatcher;
            dispatcher.Init(...);
            dispatcher.Process();  // AlltoAll Dispatch
        }
        PipeBarrier<PIPE_ALL>();
        CrossCoreBarrier/Wait();  // AIV→AIC 同步
    }

    // 2. GMM1 + Swiglu + 动态量化 (融合执行)
    GmmDeqSwigluQuant<Gmm1L1TileShape, ...>(...);
    PipeBarrier<PIPE_ALL>();
    CrossCoreBarrier/Wait();  // AIC→AIV 同步

    // 3. 初始化 Combine (AIV 端)
    CamMoeDistributeCombine combiner;
    combiner.Init(...);

    // 4. GMM2 + Combine (融合执行)
    GmmDeq<Gmm2L1TileShape, ...>(..., &combiner);
}

4.2 CamMoeDistributeCombine

文件: raw_distributed/cam_moe_distribute_combine.h

负责将 GMM2 计算结果路由回原始 rank，并做加权求和。

核心流程:

void Process() {
    SyncAll<true>();
    if constexpr (IsNeedReduceScatter) ReduceScatterTrans();
    if constexpr ((EXEC_FLAG & EXEC_FLAG_DEEP_FUSE) == 0) {
        BuffInit();
        SetWaitTpStatusAndDisPatch();  // 非融合模式
    }
    AlltoAllBuffInit();
    SetStatus();
    WaitDispatch();
    LocalWindowCopy();  // 从窗口读取 + 加权求和 + 写出
}

深度融合时的特殊路径:

// 融合模式下，Combine 被拆分为两部分：
void AllToAllSend() {
    // AIV subblock 0: AlltoAll 发送
    // AIV subblock 1: ReducePermute
    if constexpr (EXEC_FLAG & EXEC_FLAG_DEEP_FUSE) {
        CrossCoreSetFlag<0x0, PIPE_MTE3>(SEND_SYNC_EVENT_ID);
        CrossCoreWaitFlag(SEND_SYNC_EVENT_ID);
    }
}

void ReducePermute() {
    // 在 GMM2 的 AIV 端执行
    AlltoAllBuffInit();
    WaitDispatch();
    LocalWindowCopy();
}

加权求和 (LocalWindowCopy):

// 对每个 token，遍历 topK 个专家的结果，加权求和
for (uint32_t tokenIndex = beginIndex; tokenIndex < endIndex; ++tokenIndex) {
    Duplicate(sumFloatBufLocal, 0, axisH_);
    for (uint32_t i = 0; i < axisK_; i++) {
        float scaleVal = expandScalesLocal.GetValue(index);
        // 从对应专家的窗口位置读取
        rowTmpGlobal_.SetGlobalBuffer(wAddr + expertPerSizeOnWin_ * moeExpert + ...);
        Cast(rowTmpFloatLocal, tmpUb, CAST_NONE, processLen);
        Muls(mulBufLocal, rowTmpFloatLocal, scaleVal, processLen);
        Add(sumFloatBufLocal, sumFloatBufLocal, mulBufLocal, processLen);
    }
    Cast(sumBufLocal, sumFloatBufLocal, CAST_RINT, processLen);
    DataCopy(expandOutGlobal_[tokenIndex * axisH_ + tokenOffset], sumBufLocal, processLen);
}

5. gemm 子模块深度分析

5.1 Catlass 集成

FusedDeepMoE 使用 Catlass (CUDA Template Library analogue for Ascend) 作为 GMM 计算框架。Catlass 提供：

  • 类型化的矩阵类型: GemmType<Element, Layout>
  • Block 级矩阵乘法: BlockMmad<DispatchPolicy, L1Tile, L0Tile, AType, BType, CType>
  • Epilogue 融合: BlockEpilogue<DispatchPolicy, CType, ScaleType, PerTokenScaleType, DType, ...>
  • Kernel 级调度: GroupedMatmul<...>

5.2 GroupedMatmul 变体

5.2.1 GroupedMatmulSliceMPerTokenDequantSwigluQuantMultiStageWorkspace

用途: GMM1 — 完成 int8 GMM + PerTokenDequant + Swiglu + 动态量化

文件: gemm/kernel/grouped_matmul_slice_m_per_token_dequant_swiglu_quant_multistage_workspace.h

核心特点: - 1C2V 架构: AIC 做 GMM 计算，AIV 做通信 + Swiglu + 量化 - 三角色分离: - isSendCore (aivIdx % 2 == 1): 负责将量化后的 token 发送给目标 rank - isRecvCore (aivIdx % 2 == 0): 负责接收其他 rank 的 token - isCompCore: 负责 Swiglu + 动态量化计算

AIC 端 (operator()):

// 1. 先处理共享专家 (如果启用)

// 2. 处理路由专家 (按 group 循环)

    WaitGroupTokenNumReady(groupTokenNumStateTensor, expected);
    currentM = groupTokenNumStateTensor.GetValue(GROUP_TOKEN_COUNT);
    // 执行该专家的 GMM1
    blockMmad(gmA[gmGroupOffsetA + gmOffsetA], ...);
}

AIV 端 (operator()):

// 1. 共享专家量化 (shareQuantCoreFunc)

// 4. Swiglu + 动态量化 (CompCoreFunc)

    CompCoreFunc();  // 调用 BlockEpilogue 做 Swiglu
}

// 5. 后处理: Swiglu 输出的动态量化
// 共享专家 Swiglu 输出量化
BlockQuant<ArchTag> blockQuant(resource, quantParams);
// 路由专家 Swiglu 输出量化
BlockQuant<ArchTag> blockQuant(resource, quantParams);

6. epilogue 子模块

6.1 EpilogueAtlasA2PerTokenDequantSwiglu

用途: GMM1 的 Epilogue — int32→float 反量化 + Swiglu 激活

文件: epilogue/block/block_epilogue_per_token_dequant_swiglu.h

计算流程:

// 1. 从 GM 加载 int32 累加结果到 UB
copyGmToUbC(ubC, gmTileC, layoutUbC, layoutGmTileC);

// 2. int32 → float 转换
Cast(ubTmpMxN, ubC, CAST_RINT, TileShape::COUNT);

// 3. 乘以 weight scale (行广播: 每列共享一个 scale)
tileRowBroadcastMul(ubTmpMxN, ubTmpMxN, ubFp32Scale);

// 4. 乘以 per-token scale (行广播: 每行共享一个 scale)

// 5. Swiglu 激活 (左右两半)

// 右半: 保持不变

    Muls(ubDenominatorMxN, ubTmpMxN, -1.0f, TileShape::COUNT);
    Exp(ubDenominatorMxN, ubDenominatorMxN, TileShape::COUNT);
    Adds(ubDenominatorMxN, ubDenominatorMxN, 1.0f, TileShape::COUNT);
    Div(ubD, ubTmpMxN, ubDenominatorMxN, TileShape::COUNT);
} else {
    tileOneBlkColumnBroadcastMul(ubD, ubTmpMxN, ubTmpMx32B);
}

// 6. 写出 float 结果到 workspace
copyUbToGmD(gmTileD, ubD, layoutGmTileD, layoutUbD);

Swiglu 数学公式:

Swiglu(x) = [x_left / (1 + exp(-x_left)), x_right]
          = [x_left * sigmoid(x_left), x_right]

7. Swiglu + Quantization 融合机制

7.1 完整数据流

GMM1 输入: int8 → 累加: int32
     ↓
PerTokenDequant: int32 × weight_scale × per_token_scale → float32
     ↓
Swiglu: [x_left * sigmoid(x_left), x_right] → float32
     ↓
动态量化: float32 → int8 + per_token_scale → workspace
     ↓
GMM2 输入: int8 → 累加: int32
     ↓
PerTokenDequant: int32 × weight_scale × per_token_scale → float32
     ↓
类型转换: float32 → bf16/fp16
     ↓
Combine 发送: 直接到对端窗口

7.2 BlockQuant — 动态量化实现

文件: gemm/kernel/post_swiglu_dynamic_quant.h

// BlockQuant 完成 Swiglu 输出的动态量化
// 输入: float32 Swiglu 输出 [token_count, n]
// 输出: int8 [token_count, n/2] + per_token_scale [token_count]

// 关键步骤:
// 1. 计算 Swiglu 右半部分与左半部分相乘 (完成 Swiglu)
Mul(ubInput, ubInput, ubInputRightHalf, tileCount);

// 2. 取绝对值
Abs(ubAbs, ubInput, tileCount);

// 3. 对每行取 max (分半处理)
for (uint32_t rowIdx = 0; rowIdx < tileRow; ++rowIdx) {
    Max(ubMax[rowIdx * halfTileColumn], ubAbs[rowIdx * tileColumn],
        ubAbs[rowIdx * tileColumn + halfTileColumn], halfTileColumn);
}

// 4. 计算量化 scale = max / 127
Muls(ubDequantScale, ubReduceMax, 1.0f / 127.0f, tileRow);

// 5. 量化: x / scale → int8
Muls(ubInputTmp, ubInput, 127.f, tileCount);
for (uint32_t rowIdx = 0; rowIdx < tileRow; ++rowIdx) {
    Muls(ubQuantF32[rowIdx * tileColumn], ubInputTmp[rowIdx * tileColumn],
          1.f / ubReduceMax.GetValue(rowIdx), tileColumn);
}
Cast(ubQuantS32, ubQuantF32, CAST_RINT, tileCount);
Cast(ubQuantF16, ubQuantS32, CAST_RINT, tileCount);
Cast(ubOutput, ubQuantF16, CAST_RINT, tileCount);

9. 与标准 MoE (DeepSeek-style) 的对应关系

9.1 数学对应

FusedDeepMoe 实现了 DeepSeek-V3 风格的 MoE Decode:

标准 MoE 计算:
  1. Routing: expert_ids = TopK(router(x), k)
  2. Dispatch: 将 token 发送到对应 expert 的 rank
  3. GMM1: y = x @ W1  (int8 量化)
  4. Swiglu: y = Swiglu(y)  (GLU 变体)
  5. GMM2: z = y @ W2  (int8 量化)
  6. Combine: output = Σ(expert_scale_i * z_i)

FusedDeepMoE 对应:
  1. expert_ids 由外部 router 计算后传入
  2. Dispatch → CamMoeDistributeDispatch (融合在 kernel 内)
  3. GMM1 → GroupedMatmulSliceMPerTokenDequantSwigluQuantMultiStageWorkspace
  4. Swiglu → BlockEpilogue 中的 Swiglu 激活
  5. GMM2 → GroupedMatmulSliceMPerTokenDequantMultiStageWorkspace
  6. Combine → CamMoeDistributeCombine (融合在 kernel 内)

9.2 量化策略对应

标准 MoE FusedDeepMoE 说明
BF16 输入 Dynamic Quant → int8 Dispatch 阶段完成
W1 int8 per-channel W1 int8 + per-channel scale GMM1 权重
Swiglu 输出 BF16 Dynamic Quant → int8 Swiglu 后立即量化
W2 int8 per-channel W2 int8 + per-channel scale GMM2 权重
输出 BF16 PerTokenDequant → BF16 GMM2 Epilogue

总结

FusedDeepMoE 是 CAM 库的皇冠级算子，其核心价值在于：

  1. 极致融合: 将 5 个独立算子 (Dispatch + GMM1 + Swiglu + GMM2 + Combine) 融合为 1 个 kernel
  2. 中间数据全片上化: X1、SwigluOut、X2 等中间数据在 workspace 中流转，零 GM 访问
  3. 1C2V 异构并行: AIC 做 GMM 计算，AIV 做通信调度，天然计算通信重叠
  4. 全链路量化: 从输入动态量化到 Swiglu 后动态量化，全程 int8 计算
  5. 精细同步: Soft Sync + CV 双缓冲 + 多级 Flag 同步，最小化同步开销

这是华为昇腾 Ascend 910B 平台上 MoE Decode 阶段的终极优化实现，代表了 NPU 上 MoE 算子融合的最高水平。
