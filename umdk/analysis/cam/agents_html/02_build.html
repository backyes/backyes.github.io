1. 编译系统完整链路分析

CAM 的编译系统采用 三层 shell 脚本 + CMake + 华为 CANN npu_op_* 构建框架 的架构，从顶层入口到内核编译的完整调用链如下：

build/cam/build.sh          ← 顶层入口，解析 module 名 + 转发 -c/-a/-q/-p/-r
   └─ build/cam/comm_operator/build.sh   ← 模块级构建，管理编译阶段开关
        ├─ build/cam/comm_operator/compile_ascend_proj.sh   ← 单 SOC 编译主流程
        │    ├─ select_ops.py          ← 算子选择（registry 解析 + 互斥规则）
        │    ├─ set_conf.py             ← 修改 CMakePresets.json（build_type/vendor）
        │    ├─ msopgen gen             ← CANN 工具：生成 SOC 特定算子工程骨架
        │    ├─ cmake -S . -B build_out --preset=default
        │    └─ cmake --build build_out --target binary/package
        └─ build/cam/comm_operator/build_pybind.sh          ← Python whl 包构建
             └─ src/cam/comm_operator/pybind/setup.py       ← NpuExtension + bdist_wheel
                  └─ pytorch_extension/bdist_wheel_build.py   ← auditwheel repair 封装

1.1 顶层入口 build/cam/build.sh

关键设计： - 通过 getopts ":dhc:" 仅解析 -d（Debug）、-c（SOC 类型）、-h（help），其余选项（-a/-q/-p/-r）原样转发给子模块脚本。 - 导出全局路径变量：ROOT_PATH、SRC_PATH、BUILD_OUT_PATH、SCRIPTS_PATH、TEST_PATH、BUILD_PATH、CAM_THIRD_PARTY_PATH。 - 通过 CPATH 注入 catlass 头文件路径：export CPATH=${CAM_THIRD_PARTY_PATH}:${CAM_THIRD_PARTY_PATH}/catlass/include:${CPATH}。 - prepare_cam_third_party() 负责初始化 catlass 子模块（git submodule update --init --recursive）。 - 模块分发逻辑：all/comm_operator 走 comm_operator 构建；all/nda 且 -c ascend950 走 nda 构建。

关键代码（build.sh:137-156）：

1.3 单 SOC 编译主流程 compile_ascend_proj.sh

这是最核心的编译脚本，完成从算子选择到 .run 包产出的全过程：

核心函数链：

  1. list_soc_versions() — 从 operator_registry.json 读取所有注册的 SOC 版本。
  2. resolve_ops() — 调用 select_ops.py 解析最终算子列表。
  3. copy_ops_include() — include-list 模式：仅拷贝选中算子的 op_host/op_kernel/op_api 到目标工程目录（区别于旧的 full-copy + exclude-list 模式，避免同名算子文件覆盖）。
  4. build_ascend_proj() — 单 SOC 编译主函数。

build_ascend_proj() 关键步骤：

# 1. 调用 msopgen 生成 SOC 特定算子工程
msopgen gen -i ./ascend_kernels/AddCustom.json -c ai_core-${soc_version} -f pytorch -lan cpp -out ${MODULE_BUILD_PATH}/${proj_name}

# 2. 删除模板自带的 AddCustom 示例算子
rm -rf ${MODULE_BUILD_PATH}/${proj_name}/op_host/add_custom*
rm -rf ${MODULE_BUILD_PATH}/${proj_name}/op_kernel/add_custom*

# 3. 拷贝 CMakeLists.txt 和 cmake_files/
cp ./ascend_kernels/CMakeLists.txt ${MODULE_BUILD_PATH}/${proj_name}/
cp ./ascend_kernels/cmake_files/op_host/CMakeLists.txt ${MODULE_BUILD_PATH}/${proj_name}/op_host/
cp ./ascend_kernels/cmake_files/op_kernel/CMakeLists.txt ${MODULE_BUILD_PATH}/${proj_name}/op_kernel/
cp -rf ./ascend_kernels/cmake_files/cmake ${MODULE_BUILD_PATH}/${proj_name}/

# 4. 修改 CMakePresets.json（build_type、ENABLE_SOURCE_PACKAGE、vendor_name）
python3 $SCRIPTS_PATH/comm_operator/set_conf.py ${MODULE_BUILD_PATH}/${proj_name}/CMakePresets.json $build_type True CAM

# 5. 配置 CANN 包路径
export ASCEND_CANN_PACKAGE_PATH="${ASCEND_TOOLKIT_HOME}"  # 默认 /usr/local/Ascend/ascend-toolkit/latest

# 6. 推导 CATLASS_HOME_PATH（从 CPATH 解析）
export CATLASS_HOME_PATH=$(echo "${CPATH}" | tr ':' '\n' | grep "catlass" | sed 's|/include||')

# 7. CMake configure + build
cmake -S . -B build_out --preset=default -DCMAKE_BUILD_TYPE=$build_type
cmake --build build_out --target binary -j${BUILD_JOBS}
cmake --build build_out --target package -j${BUILD_JOBS}

# 8. 产出 .run 包
cp build_out/*.run ${BUILD_OUT_PATH}/comm_operator/run/CAM_${soc_version}_${os_id}_${arch}.run

关键 patch 机制（compile_ascend_proj.sh:177-192）：

由于 npu_op_* 构建系统的 simple_kernel_compile 不读取 custom_compile_options.ini，脚本通过 sed 直接 patch .make 文件来注入 SHMEM/CATLASS 的 include 路径：

KERNEL_COMPILE_OPTS=""
if [ -n "${SHMEM_HOME_PATH}" ]; then
    KERNEL_COMPILE_OPTS="-I${SHMEM_HOME_PATH}/shmem/include -I${SHMEM_HOME_PATH}/shmem/src/device"
fi
if [ -n "${CATLASS_HOME_PATH}" ]; then
    KERNEL_COMPILE_OPTS="${KERNEL_COMPILE_OPTS} -I${CATLASS_HOME_PATH}/include"
fi
find build_out/op_kernel/CMakeFiles -name "build.make" -exec sed -i "s|--compile-options=\"\"|--compile-options=\\\"${KERNEL_COMPILE_OPTS}\\\"|g" {} +

1.4 Python whl 包构建 build_pybind.sh

build_pybind() {
    local build_tmp_dir="${MODULE_BUILD_PATH}/pybind"
    local wheel_out_dir="${MODULE_BUILD_OUT_PATH}/dist"
    rm -rf "${build_tmp_dir}" "${wheel_out_dir}"
    mkdir -p "${wheel_out_dir}" "$(dirname "${build_tmp_dir}")"
    cp -r $MODULE_SRC_PATH/pybind "${build_tmp_dir}"
    export MAX_JOBS=${CAM_BUILD_JOBS:-8}
    cd "${build_tmp_dir}"
    python3 setup.py bdist_wheel --dist-dir="${wheel_out_dir}"
}

注意： cp -r $MODULE_SRC_PATH/pybind "${build_tmp_dir}" 仅在 build_tmp_dir 不存在时才会将 pybind 目录内容直接拷贝到 build_tmp_dir（即 build_tmp_dir/setup.py）。如果 build_tmp_dir 已存在，会嵌套一层 pybind/pybind/。脚本通过 rm -rf "${build_tmp_dir}" 确保每次都是干净构建。

2. 算子选择机制

2.1 算子注册表 operator_registry.json

{
  "soc_versions": {
    "ascend910_93": [
      "dispatch_layout",
      "fused_deep_moe",
      "moe_combine_normal",
      "moe_dispatch_normal",
      "notify_dispatch"
    ]
  },
  "operator_meta": {
    "fused_deep_moe": { "requires_shmem": false, "family": "fused_deep_moe", "variant": "base" },
    "dispatch_layout": { "requires_shmem": false },
    "moe_combine_normal": { "requires_shmem": false },
    "moe_dispatch_normal": { "requires_shmem": false },
    "notify_dispatch": { "requires_shmem": false }
  }
}

设计要点： - 算子身份 = 源码目录名（ascend_kernels/<op_name>/）。 - soc_versions 定义每个 SOC 支持的算子列表（当前仅注册 ascend910_93）。 - operator_meta 存储构建元数据（requires_shmem、family、variant）。 - 注释说明：要支持新 SOC 只需在 soc_versions 添加条目；要添加算子只需在对应 SOC 列表追加目录名。

2.2 算子选择逻辑 select_ops.py

解析流程：

  1. 加载注册表 → load_registry()
  2. 获取 SOC 支持列表 → get_soc_list()
  3. 解析用户 -a 列表 → split_ops()（分号分隔，禁止空条目）
  4. 验证用户选择 → validate_user_ops()（每个 -a 条目必须在 SOC 支持列表中）
  5. resolve() 核心规则：

互斥规则（fused_deep_moe 家族）： - fused_deep_moe（base）和 fused_deep_moe_w4a8 共享相同的源码文件名（fused_deep_moe.*），因此互斥——同一 run 包只能编译其中一个。 - --quant（-q）选择 w4a8 变体，此时 base 被禁止。 - fused_deep_moe_fwk 是独立算子（文件名为 fused_deep_moe_fwk.*），不受互斥约束。 - 全量构建（无 -a）时默认保留 base，丢弃 w4a8（带 NOTE 提示）。

SHMEM 过滤： - 当 SHMEM_HOME_PATH 未设置时，自动丢弃 requires_shmem=true 的算子。

确定性排序： - 最终列表按注册表中的顺序排列，确保构建可复现。

输出格式： - 每行一个算子目录名，由 compile_ascend_proj.sh 通过 tr '\n' ' ' 转为空格分隔列表。

9. 关键设计洞察

9.1 include-list vs exclude-list 演进

compile_ascend_proj.sh 的注释明确说明了从 full-copy + exclude-list 到 include-list 的迁移原因：

Unlike the old full-copy + exclude_list approach, only the operators resolved by select_ops.py are copied here, which fundamentally avoids same-name operators (the fused_deep_moe family) overwriting each other's files.

这是 fused_deep_moe / fused_deep_moe_w4a8 共享文件名的直接后果——全量拷贝会导致文件覆盖。

9.2 build.make patch 的必要性

npu_op_ 构建系统的 simple_kernel_compile 不读取 custom_compile_options.ini，因此 --compile-options="" 默认为空。华为选择通过 sed patch .make 文件来注入 SHMEM/CATLASS 路径，这是一种非侵入式的 workaround*，避免了修改 CANN 构建系统源码。

9.3 算子身份 = 源码目录名

整个算子选择机制的核心设计是算子身份与源码目录名一一对应： - operator_registry.json 中的列表项 = ascend_kernels/ 下的目录名 - select_ops.py 输出 = 目录名列表 - copy_ops_include() 按目录名拷贝

这种设计使得添加新算子只需：创建目录 → 在 registry 中添加条目 → 完成。

9.4 版本匹配的脆弱性

CANN ↔ torch-npu 的版本匹配是硬编码在 Dockerfile 中的： - CANN 8.5 → torch_npu 2.8.0.post2 - CANN 9.0 → torch_npu 2.8.0.post4

升级 CANN 必须同步修改 Dockerfile 中的 wheel URL。setup.py 的 FP8/FP4 探测机制部分缓解了运行时版本差异，但构建时依赖仍需人工维护。

9.5 CAM 不单独打包 RPM

CAM 的 .run 和 .whl 包是独立于 UMDK RPM 体系的。umdk.spec 中的 RPM 打包仅覆盖 URMA/URPC/DLock/UMS 等子模块，CAM 通过独立的安装脚本（run --install-path + pip install）部署。
