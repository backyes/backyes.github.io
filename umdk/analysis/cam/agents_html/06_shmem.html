CAM SHMEM 模式代码深度剖析

1. SHMEM 模式整体架构

1.1 定位与使命

SHMEM 模式是 CAM 的 用户态 RDMA 直通路径。它完全绕过 HCCL 集合通信库，在 AIV（Ascend Intelligent Vector）核心上直接操作 RoCE RDMA 硬件的 WQE/CQE/SQ/CQ 寄存器，实现 kernel 内部的细粒度 P2P 数据传输。

│                                                                       │
│  RdmaTransportManagerV2                                               │
│  ├── OpenDevice()           → phyId 获取 + RDMA IP 解析               │
│  ├── CreateEndpoint()       → HcommEndpointCreate(COMM_PROTOCOL_ROCE) │
│  ├── RegisterMemoryRegion() → HcommMemReg (HBM 注册为 MR)             │

1.2 数据流全景

一次 aclshmemx_roce_put_nbi (RDMA WRITE) 的完整路径：

用户调用 aclshmemx_roce_put_nbi(dst, src, buf, elem_size, pe)
    │
    ├─ 1. aclshmem_ptr(dst, pe)        ← 本地对称地址 → 远端对称地址
    │     计算: remote = p2p_device_heap_base[pe] + (dst - heap_base)
    │
    ├─ 2. aclshmemi_roce_write<B>()    ← 填充 send_wr 结构体
    │     wr.remote_addr = ptr
    │     wr.local_addr  = src
    │     wr.message_len = elem_size * sizeof(T)
    │
    ├─ 3. aclshmemi_roce_post_send<B, OP_RDMA_WRITE>()
    │     │
    │     ├─ 3a. 读取 SQ head/tail (dcci_cachelines 刷新缓存行)
    │     │     若 SQ 即将满 (head+16 == tail)，先 poll_cq 等待空闲
    │     │
    │     ├─ 3b. aclshmemi_roce_fill_wqe<B, OP_RDMA_WRITE>()
    │     │     → 在 HBM 上构造 128B WQE (ctrl_seg + remote_seg + local_seg)
    │     │
    │     ├─ 3c. dcci_cachelines(wqe_addr, 128)  ← 刷回 HBM 确保硬件可见
    │     │
    │     └─ 3d. aclshmemi_roce_ring_sq_doorbell<B>()
    │            → 更新 head_addr (producer index)
    │            → 写 doorbell register: {next_pid, qp_id}
    │
    └─ 4. 硬件自动执行 RDMA WRITE，数据通过 RoCE 网络直达远端 HBM

2. 核心数据结构深度分析

2.1 aclshmemi_rdma_info — 全局 RDMA 信息表

文件: shmemi_device_rdma.h:38-45

struct aclshmemi_rdma_info {
    uint32_t qp_num;   // 每个连接的 QP 数量 (支持多 QP 并行)
    uint64_t sq_ptr;   // SQ 上下文数组地址 [PE_NUM][qp_num]
    uint64_t rq_ptr;   // RQ 上下文数组地址 [PE_NUM][qp_num]
    uint62_t scq_ptr;  // Send CQ 上下文数组地址 [PE_NUM][qp_num]
    uint64_t rcq_ptr;  // Recv CQ 上下文数组地址 [PE_NUM][qp_num]
    uint64_t mem_ptr;  // Memory Region 信息数组地址 [MAX_PE_NUM]
};

设计要点: - 这是 device 端唯一的全局入口，通过 aclshmemi_qp_info_fetch() 从 HBM 固定地址读取 - 数组维度 [PE_NUM][qp_num] 支持多 rank 多 QP 场景 - mem_ptr 存储每个 rank 的 MR (Memory Region) 信息，包含 lkey/rkey 用于 RDMA 访问权限控制

2.2 aclshmemi_rdma_sq_ctx — Send Queue 上下文

文件: shmemi_device_rdma.h:54-66

struct aclshmemi_rdma_sq_ctx {
    uint32_t wqn;       // Work Queue Number (硬件 QP 编号)
    uint64_t buf_addr;  // SQ ring buffer 起始地址 (HBM)
    uint32_t wqe_size;  // 每个 WQE 的大小 (XSCALE=128B)
    uint32_t depth;     // ring buffer 深度 (WQE 数量)
    uint64_t head_addr; // Producer Index 地址 (HBM, 软件写)
    uint64_t tail_addr; // Consumer Index 地址 (HBM, 硬件写)
    aclshmemi_rdma_db_mode_t db_mode;  // HW_DB(0) / SW_DB(1)
    uint64_t db_addr;   // Doorbell register 地址
    uint32_t sl;        // Service Level (RDMA 优先级)
    uint64_t amo_addr;  // 原子操作的本地结果缓冲区地址
    uint32_t amo_lkey;  // 原子操作缓冲区的 local key
};

关键字段解读: - head_addr / tail_addr: 分离的 producer/consumer index，支持硬件异步消费。head_addr 由软件通过 aclshmemi_roce_ring_sq_doorbell 更新，tail_addr 由硬件通过 CQE 完成事件推进。 - db_mode: 区分硬件 doorbell (HW_DB) 和软件 doorbell (SW_DB)，适应不同硬件版本。 - amo_addr / amo_lkey: 原子操作的专用缓冲区。RDMA atomic fetch 操作的结果会写回此地址，软件通过 dcci_cachelines 刷新后读取。

2.3 aclshmemi_rdma_cq_ctx — Completion Queue 上下文

文件: shmemi_device_rdma.h:68-77

struct aclshmemi_rdma_cq_ctx {
    uint32_t cqn;       // Completion Queue Number
    uint64_t buf_addr;  // CQ ring buffer 起始地址
    uint32_t cqe_size;  // 每个 CQE 的大小 (XSCALE=64B, 含 32B padding)
    uint32_t depth;     // ring buffer 深度
    uint64_t head_addr; // Producer Index (硬件写)
    uint64_t tail_addr; // Consumer Index (软件写)
    aclshmemi_rdma_db_mode_t db_mode;
    uint64_t db_addr;   // CQ Doorbell register 地址

与 SQ 的对称性: CQ 的 head_addr 由硬件更新（新 CQE 写入时），tail_addr 由软件更新（aclshmemi_roce_ring_cq_doorbell 在 poll 完成后更新）。

2.5 aclshmemi_rdma_atomic_params — 原子操作参数联合体

文件: shmemi_device_rdma.h:85-119

struct aclshmemi_rdma_atomic_params {
    union {
        // 标准 8B FA/CAS (unmasked)
        struct { uint64_t compare_add; uint64_t swap; };

        // 通用 masked 格式 (XSCALE 使用)
        struct {
            uint64_t swap_add_data;
            uint64_t compare_data;
            uint64_t swap_add_mask;
            uint64_t compare_mask;
        } masked_common;

        // 64-bit masked CAS (双 64-bit 段: swap+compare, mask_swap+mask_compare)
        struct { uint64_t swap_data; uint64_t compare_data;
                 uint64_t swap_mask; uint64_t compare_mask; } masked_cas64;

        // 64-bit masked FA (add_data + field_boundary)
        struct { uint64_t add_data; uint64_t field_boundary; } masked_fa64;

        // 32-bit masked CAS
        struct { uint32_t swap_data; uint32_t compare_data;
                 uint32_t swap_mask; uint32_t compare_mask; } masked_cas32;

        // 32-bit masked FA
        struct { uint32_t add_data; uint32_t field_boundary;
                 uint64_t _reserved; } masked_fa32;
    };
};

设计哲学: 使用 union 实现参数格式的零开销复用。masked_common 是 XSCALE 后端实际使用的格式，其 4 个 64-bit 字段直接映射到 WQE 的 atomic segment。

3.2 RDMA WRITE 完整调用链

// Step 1: 用户调用
aclshmemx_roce_put_nbi<uint32_t>(dst, src, buf, 256, pe);

// Step 2: 地址转换 + 转发
// shmem_device_rdma.hpp:183-197
aclshmem_ptr(dst, pe);  // 本地对称地址 → 远端对称地址
aclshmemi_roce_write<uint32_t, XSCALE>(ptr, src, pe, 0, 1024, ...);

// Step 3: 填充 wr + post_send
// rdma_device_backend_base.h:137-149
wr.remote_addr = ptr;
wr.local_addr = src;
wr.message_len = 1024;
aclshmemi_roce_post_send<XSCALE, OP_RDMA_WRITE>(wr, pe, 0, ...);

// Step 4: XSCALE 后端 post_send
// rdma_device_backend_xscale.hpp:609-667
aclshmemi_xscale_post_send_read_write<OP_CODE>(wr, pe, qp_idx, ...) {
    // 4a. 读取 SQ head/tail
    dcci_cachelines(sq_pi_addr, 8);
    cur_head = *(uint32_t*)sq_pi_addr;

    // 4b. SQ 满检查 (阈值=16)
    if ((cur_head + 16) % depth == (depth + cur_tail) % depth)
        aclshmemi_roce_poll_cq<XSCALE>(...);

    // 4c. 获取 rkey/lkey
    wr.rkey = remote_mem_info->rkey;
    wr.lkey = local_mem_info->lkey;

    // 4d. 填充 WQE
    aclshmemi_roce_fill_wqe<XSCALE, OP_RDMA_WRITE>(wr, sq, wqe_addr, cur_head);

    // 4e. 刷回 + doorbell
    dcci_cachelines(wqe_addr, 128);
    aclshmemi_roce_ring_sq_doorbell<XSCALE>(sq, cur_head+1, ...);
}

3.3 aclshmemx_roce_quiet 同步机制

// shmem_device_rdma.hpp:253-276
template <typename T>
void aclshmemx_roce_quiet(uint32_t pe, __ubuf__ T *buf, uint32_t sync_id) {
    for (qp_idx = 0; qp_idx < qp_num; qp_idx++) {
        sq_context = ...;
        dcci_cachelines(sq_pi_addr, 8);       // 刷新 head 缓存行
        cur_head = *(uint32_t*)sq_pi_addr;     // 读取当前 producer index
        aclshmemi_roce_poll_cq<XSCALE>(pe, qp_idx, cur_head, ...);
        // poll_cq 会阻塞直到 cur_tail == cur_head (所有 WQE 完成)
    }
}

关键设计: quiet 遍历所有 QP，对每个 QP 读取当前 SQ head（即已 post 的 WQE 总数），然后 poll_cq 等待 CQ tail 追上该值。

4. 编译时多态 traits 系统

4.1 设计理念

CAM 使用 C++ 模板特化 实现零运行时开销的后端选择。核心思想：

在编译期通过 ACLSHMEMI_K_RDMA_BACKEND 宏确定后端，所有 aclshmemi_roce_*<B>() 调用在编译期绑定到具体后端实现，运行时无 switch/if 分支。

5. XSCALE 后端 WQE/CQE 格式

5.1 WQE 整体结构 (XSCALE, 128 Bytes)

XSCALE 后端的 WQE 为 128 Bytes (ACLSHMEMI_XSCALE_SND_WQE_SIZE = 1 << 7 = 128)，由以下部分组成：

┌──────────────────────────────────────────────────────┐
│  Control Segment (16 Bytes)                           │
│  ├── msg_opcode    (8 bits)  — 操作码                 │
│  ├── with_imm      (1 bit)   — 是否带 immediate      │
│  ├── csum_en       (2 bits)  — 校验和使能             │
│  ├── ds_data_num   (5 bits)  — Data Segment 数量      │
│  ├── wqe_id        (16/20 bits) — WQE 标识           │
│  ├── msg_len       (32 bits) — 消息长度               │
│  ├── opcode_data   (32 bits) — 操作码扩展数据         │
│  ├── se            (1 bit)   — Solicitation Event     │
│  ├── ce            (1 bit)   — Completion Enable      │
│  ├── in_line       (1 bit)   — Inline 模式            │
│  ├── fence_mode    (2 bits)  — Fence 模式             │
│  └── mask          (2 bits)  — 掩码                   │
├──────────────────────────────────────────────────────┤
│  Remote Data Segment (16 Bytes)                       │
│  ├── seg_len  (31 bits) — 远端数据长度                │
│  ├── rkey     (32 bits) — 远端 Memory Region Key      │
│  └── va       (64 bits) — 远端虚拟地址                │
├──────────────────────────────────────────────────────┤
│  Local Data Segment (16 Bytes)                        │
│  ├── seg_len  (31 bits) — 本地数据长度                │
│  ├── lkey     (32 bits) — 本地 Memory Region Key      │
│  └── va       (64 bits) — 本地虚拟地址                │
├──────────────────────────────────────────────────────┤
│  Atomic Segment (16 Bytes, 仅原子操作)                 │
│  ├── swap_add  (64 bits) — 交换/加数据               │
│  └── compare   (64 bits) — 比较数据                   │
├──────────────────────────────────────────────────────┤
│  Padding (64 Bytes, 补齐到 128B)                      │
└──────────────────────────────────────────────────────┘

5.2 Control Segment 版本差异

XSCALE 支持两个 API 版本，通过 ACLSHMEMI_XSCALE_API_VERSION_VAR 宏选择：

5.4 Owner Bit 机制

XSCALE 使用 owner bit 实现硬件/软件同步，避免昂贵的 atomic 操作：

// rdma_device_backend_xscale.hpp:389-397
bool aclshmemi_roce_xscale_check_cqe_owner(__gm__ aclshmemi_xscdv_cqe64_t *cqe64, 
                                            uint32_t cur_tail, uint32_t depth) {
    bool owner_bit = (cqe64->cqe.owner & ACLSHMEMI_XSC_CQE_OWNER_MASK);
    bool expect_bit = ((cur_tail / depth) & 1);  // 每翻转一轮 depth，owner bit 翻转
    return owner_bit == expect_bit;
}

原理: 硬件每写完一轮 CQ（depth 个 CQE），自动翻转 owner bit。软件通过比较 owner_bit == (cur_tail / depth) & 1 判断 CQE 是否已写入。

6. 原子操作机制

6.1 原子操作 API 全景

CAM 提供 16 种 原子操作 API，覆盖 fetch/set/compare-swap/inc/add/and/or/xor 及其组合：

类别 API 同步性 实现方式
Fetch atomic_fetch(src, pe) 同步 FA(add=0) + quiet + read
Set atomic_set(dst, val, pe) 异步 CAS(swap=val, comp=0, mask_swap=MAX)
CAS atomic_compare_swap(dst, cond, val, pe) 同步 CAS + quiet + read
Swap atomic_swap(dst, val, pe) 同步 CAS(swap=val, comp=0) + quiet + read
Inc atomic_inc(dst, pe) 异步 FA(add=1)
Add atomic_add(dst, val, pe) 异步 FA(add=val)
And atomic_and(dst, val, pe) 异步 CAS(swap=val, mask_swap=~val)
Or atomic_or(dst, val, pe) 异步 CAS(swap=val, mask_swap=val)
Xor atomic_xor(dst, val, pe) 异步 FA(add=val, boundary=MAX)
FetchInc atomic_fetch_inc(dst, pe) 同步 FA(add=1) + quiet + read
FetchAdd atomic_fetch_add(dst, val, pe) 同步 FA(add=val) + quiet + read
FetchAnd atomic_fetch_and(dst, val, pe) 同步 CAS + quiet + read
FetchOr atomic_fetch_or(dst, val, pe) 同步 CAS + quiet + read
FetchXor atomic_fetch_xor(dst, val, pe) 同步 FA + quiet + read

6.2 Masked vs Unmasked 变体

XSCALE 后端对原子操作区分 masked 和 unmasked 两种模式：

// 4B 操作强制使用 masked 模式
static_assert(!(TYPE_BYTES <= BYTES_32 && !IS_MASKED),
    "XSCALE backend doesn't support fetch and add operation with type size of 4B without mask\n");

Masked 模式 允许对 64-bit 字中的特定位字段进行原子操作，通过 swap_add_mask / compare_mask 控制哪些位参与操作。

6.3 原子操作 WQE 构造

以 aclshmemi_rdma_xscdv_fill_wqe_fa 为例：

template <typename T, bool IS_MASKED>
uint32_t aclshmemi_rdma_xscdv_fill_wqe_fa(aclshmemi_rdma_send_wr &wr, ...) {
    constexpr uint32_t DS_DATA_NUM = 3;  // ctrl + data_seg + atomic_seg
    constexpr uint32_t MSG_LEN = (TYPE_BYTES == BYTES_32 && IS_MASKED) ? 8 : 16;
    constexpr uint32_t DATA_LEN = (TYPE_BYTES <= BYTES_32) ? BYTES_32 : BYTES_64;

    // 根据类型选择操作码
    constexpr aclshmemi_xscdv_msg_type_t OPCODE =
        (TYPE_BYTES == BYTES_64 && IS_MASKED) ? ACLSHMEMI_XSCALE_MSG_OPCODE_RDMA_ATOMIC_8B_MSK_FETCH_AND_ADD
        : (TYPE_BYTES == BYTES_64 && !IS_MASKED) ? ACLSHMEMI_XSCALE_MSG_OPCODE_RDMA_ATOMIC_FETCH_AND_ADD
        : ACLSHMEMI_XSCALE_MSG_OPCODE_RDMA_ATOMIC_4B_MSK_FETCH_AND_ADD;

    // 填充 ctrl_seg (inline=1, 因为数据在 WQE 内)
    cur_wqe_addr = aclshmemi_roce_xscale_fill_wqe_ctrl_seg(wqe_addr, DS_DATA_NUM, cur_head, OPCODE, MSG_LEN, 1);
    // 填充 data_seg (remote + local)
    cur_wqe_addr = aclshmemi_roce_xscale_fill_wqe_data_seg(cur_wqe_addr, wr.rkey, wr.remote_addr, wr.lkey, wr.local_addr, DATA_LEN);
    // 填充 atomic_seg
    return aclshmemi_rdma_xscdv_fill_fa_atomic_seg<T, IS_MASKED>(cur_wqe_addr, wr);
}

6.4 字节序转换

ACLSHMEM_DEVICE uint64_t aclshmemi_htobe64(uint64_t host_val) {
    return ((host_val >> 0) & 0xFF) << 56 |
           ((host_val >> 8) & 0xFF) << 48 |
           ((host_val >> 16) & 0xFF) << 40 |
           ((host_val >> 24) & 0xFF) << 32 |
           ((host_val >> 32) & 0xFF) << 24 |
           ((host_val >> 40) & 0xFF) << 16 |
           ((host_val >> 48) & 0xFF) << 8  |
           ((host_val >> 56) & 0xFF) << 0;
}

7.6 原子操作共享内存

// device_rdma_transport_manager_v2.cpp:703-735
bool ReserveRdmaInfoSpace() noexcept {
    // qpInfo 大小: sizeof(AiQpRMAQueueInfo) + rankCount * [2*(sizeof(AiQpRMAWQ) + sizeof(AiQpRMACQ)) + sizeof(RdmaMemRegionInfo)]
    qpInfoSize_ = sizeof(AiQpRMAQueueInfo) + oneQpSize * rankCount_;
    DlAclApi::AclrtMalloc(&ptr, qpInfoSize_, 0);
    qpInfo_ = (AiQpRMAQueueInfo*)ptr;

    // 原子操作共享内存: 128 * 8B * rankCount, 512B 对齐
    uint32_t atomicSize = ATOMIC_MAX_NUM(128) * sizeof(uint64_t) * rankCount;
    atomicSize = ALIGN_UP(atomicSize, MEMORY_ALIGNMENT(512));
    DlAclApi::AclrtMalloc(&atomicSharedMemory_, atomicSize, 0);
}

设计要点: 原子操作共享内存用于存储 fetch-before-write 结果。每个 rank 占用 128 × 8B = 1KB，支持最大 128 个并发原子操作。

7.7 底层协议结构

HCCP 协议 (dl_hccp_def.h): - HccpMrInfo: Memory Region 信息 (addr/size/access/lkey/rkey) - HccpQpExtAttrs: QP 扩展属性 (qpMode/cqAttr/qp_attr/version/mem_align) - ai_data_plane_wq / ai_data_plane_cq: 硬件队列描述 (wqn/cqn/buf_addr/wqebb_size/depth/head/tail/db_reg) - AiQpRMAWQ / AiQpRMACQ: 运行时 SQ/CQ 信息 (含 atomicAddr/atomicLkey)

Hcomm 协议 (dl_hcomm_def.h): - ChannelEntity: 通道实体 (engine/protocol/bufferNum/notifyNum/sqNum/cqNum) - SqContext / CqContext: 队列上下文 (roceSq/roceCq 包含 qpn/cqn/sqVa/cqVa/dbVa 等) - RegedBufferEntity: 注册缓冲区 (addr/size/protectionInfo) - ProtectionInfo: 保护信息 (type=PROTECTION_TYPE_ROCE, lkey/rkey)

9. 与 HCCL Window 模式的本质区别

9.1 架构层面对比

维度 SHMEM 模式 HCCL Window 模式
通信路径 AIV 核心直接操作 RoCE 硬件 WQE/CQE 通过 HCCL Window 机制 (winContext_->localWindowsIn)
运行时开销 单次 kernel 内完成 RDMA 操作，无 HCCL 调度 需经过 HCCL 调度层、Window 资源管理
延迟 微秒级 (直接 doorbell 振铃) 数十微秒级 (HCCL 软件栈开销)
灵活性 支持任意地址、任意长度的 P2P 传输 受限于 Window 预分配区域
原子操作 16 种 RDMA 原子操作 (CAS/FA/AND/OR/XOR) 依赖 HCCL 集合通信原语
计算-通信重叠 同一 kernel 内交替执行 GEMM 和 RDMA 需多 kernel 协作，HBM 读写开销
后端选择 编译期 traits 系统 (IN_DIE/XSCALE) HCCL 内部自动选择
内存注册 显式 MR 注册 (HcommMemReg) HCCL 隐式管理
QP 管理 显式 SQ/CQ 上下文管理 HCCL 抽象

9.2 代码层面对比

SHMEM 模式 (直接 RDMA):

// 算子 kernel 内部直接发起 RDMA
aclshmemx_roce_put_nbi(remote_addr, local_addr, ub_buf, len, remote_pe);
// ↑ 直接构造 WQE → doorbell → 硬件执行

HCCL Window 模式 (当前默认):

// ↑ 需经过 HCCL 调度、Window 资源同步

9.3 性能影响分析

SHMEM 模式的核心优势在于 消除 HCCL 软件栈开销:

  1. 零调度延迟: 直接 doorbell 振铃 (~100ns) vs HCCL 调度 (~10-50μs)
  2. Kernel 内融合: FusedDeepMoE 可在同一 kernel 内交替执行 GEMM 和 RDMA，避免多次 kernel launch 和 HBM 读写
  3. 细粒度 P2P: 支持 token 级别的动态路由，无需预定义通信模式
  4. 硬件原子操作: 利用 RDMA 硬件原子操作实现无锁同步

9.4 启用路径

要将算子从 HCCL Window 模式切换到 SHMEM 模式:

  1. 修改 operator_registry.json: 将目标算子的 requires_shmem 设为 true
  2. 确保 SHMEM 安装: 设置 SHMEM_HOME_PATH 环境变量指向 SHMEM 安装目录
  3. 算子代码改造: 在 kernel 中调用 aclshmemx_roce_* API 替代 HCCL Window 操作
  4. 编译: select_ops.py 会自动包含该算子，编译注入 SHMEM include 路径

10. 关键设计洞察

10.1 用户态 RDMA 的极致优化

CAM 的 SHMEM 模式是 用户态 RDMA 在 AI 芯片上的极致实现:

  • AIV 核心直接驱动: 不同于传统 RDMA 由 CPU 或专用 NIC 驱动，CAM 让 AIV 向量核心直接构造 WQE 和振铃 doorbell
  • Cache-line 精确刷新: 使用 dcci_cachelines() 精确刷新 64B 缓存行，确保 HBM 数据对硬件可见
  • Inline 模式: 原子操作使用 inline 模式 (数据在 WQE 内)，避免额外的 DMA 读取

10.2 编译时多态的工程价值

traits 系统实现了 零成本抽象:

  • 无虚函数表 (vtable) 开销
  • 无运行时 switch/if 分支
  • 未使用的后端代码在编译期完全消除
  • if constexpr 确保编译期分支消除

10.3 版本兼容性的优雅处理

XSCALE 后端的 V1/V2 API 版本通过 std::conditional_t 实现编译期选择:

using aclshmemi_xscdv_wqe_ctrl_seg_t =
    std::conditional_t<ACLSHMEMI_XSCALE_API_VERSION_VAR == 1, 
                       aclshmemi_xscdv_wqe_ctrl_seg_v1,
                       aclshmemi_xscdv_wqe_ctrl_seg_v2>;

这允许同一份代码支持不同硬件世代的 WQE 格式差异。

10.4 同步机制的精心设计

  • SQ head/tail 分离: 软件写 head (producer), 硬件写 tail (consumer via CQE)
  • CQ owner bit: 硬件/软件同步无 atomic 操作，每轮 depth 翻转一次
  • SQ 满检查: 阈值 16 的预检查避免死锁，同时最大化 SQ 利用率
  • 超时保护: 5 分钟超时防止硬件异常导致无限等待

10.5 原子操作的统一抽象

通过 aclshmemi_rdma_atomic_params union + IS_MASKED 模板参数，实现了:

  • 8B/4B 数据类型统一处理
  • masked/unmasked 模式编译期选择
  • 多种原子操作 (FA/CAS/AND/OR/XOR) 的统一 WQE 构造路径

11. 总结

CAM 的 SHMEM 模式是一个 完整的用户态 RDMA 子系统，其核心优势在于:

  1. 硬件直通: AIV 核心直接操作 RoCE 硬件，消除 HCCL 软件栈开销
  2. 编译时多态: traits 系统实现零运行时开销的后端选择
  3. 原子操作丰富: 16 种原子操作变体，支持 8B/4B masked/unmasked 模式
  4. 版本兼容: V1/V2 API 版本编译期选择，支持多硬件世代
  5. 工程完备: 超时保护、错误处理、日志调试机制齐全

当前所有算子默认使用 HCCL Window 模式 (requires_shmem: false)，但 SHMEM 模式基础设施已完整就绪，是 CAM 未来性能突破的关键路径。
