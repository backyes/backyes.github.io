1. 家族概览与设计哲学

FusedDeepMoE 是 CAM (Communication Acceleration for Matrix) 中最复杂的算子家族，将 MoE Decode 阶段的 Dispatch → GMM1 → SwiGLU → GMM2 → Combine 五阶段完整流水线融合为单一 kernel launch。

核心动机：消除中间数据的 Global Memory 往返。 标准小算子组合中，中间数据 X1、SwigluOut、X2 需要至少 8 次 GM 访问；融合后全部在片上 workspace 中流转。

三个变体服务不同的部署场景：

维度 Base (fused_deep_moe) W4A8 (fused_deep_moe_w4a8) FWK (fused_deep_moe_fwk)
定位 通用 EP 融合 MoE INT4/INT8 混合量化 框架原生 + OPX 流水线
Weight 类型 INT8 INT4 (int4b_t) INT8
requires_shmem true false false
创建时间 2025-07-19 2026-06-29 2026-01-20
共享专家 可选 (share_gmm1/2_weight) 可选 + Bias 支持 原生 sharedExpertNum/RankNum
模板参数 W1ScaleType, W2ScaleType 同 Base 仅 ExpandXType
GMM1_L1K/L0K 512 / 128 1024 / 256 512 / 128
Bias 输入 无 gmm1/2_bias + share_bias 无
通信机制 SHMEM + HCCL Window HCCL Window (MC2) HCCL Window + OPX LibOps

    icache_preload(8);
    REGISTER_TILING_DEFAULT(FusedDeepMoeTilingData);

关键设计: 64 个 tiling key 通过宏展开为编译期 if constexpr 链，每个 key 对应不同的 EXEC_FLAG 组合。这允许编译器为每种配置生成特化代码。

3.3 FusedDeepMoe::Init — 参数初始化

Init 方法（fused_deep_moe.h L352-412）完成：

  1. GM 地址绑定: 所有输入/输出/权重的 Global Memory 指针
  2. 维度推导: - maxTokenNum_ = maxBs * epRankSize * min(topK, moeExpertNumPerRank) — 单 rank 需处理的最大 token 数 - gmm2InputDim_ = gmm1OutputDim_ / 2 — SwiGLU 将隐藏维度减半 - shareGmm2InputDim_ = shareGmm1OutputDim_ / 2
  3. SHMEM workspace 指针: shmemWorkspaceGM_ 从 tiling 数据的 shmemWorkspacePtr 获取
  4. statusDataSpace 偏移计算: 用于 cross-rank 状态同步的多缓冲区布局

3.4 FusedDeepMoe::Process — 五阶段融合主流程

Process() 是整个算子的核心（fused_deep_moe.h L414-558），分两大阶段：

阶段 A: Problem Shape 与 Workspace 布局

GemmCoord gmm1ProblemShape{maxTokenNum_, gmm1OutputDim_, tokenHiddenSize_};  // [M, N, K]
GemmCoord gmm2ProblemShape{maxTokenNum_, gmm2OutputDim_, gmm2InputDim_};

Workspace 内存布局（全部在 GM 上，但中间数据通过 SHMEM 在片上流转）：

shmemWorkspaceGM_ (Base 独有，SHMEM 双缓冲):
├── gmShareX1 / gmX1          — GMM1 输入 token (int8)
├── gmShareX2 / gmX2          — GMM2 输入 token (int8)
├── gmTokenFlag               — token 到达标志
├── gmAllExpertTokenNums      — 全局专家 token 计数
└── gmCombineSend             — Combine 发送缓冲

workspaceGM_ (普通 GM workspace):
├── gmCVSwap                  — Cube 计算 workspace (blockDim * L1M*L1N * STAGES * int32)
├── gmShareSwigluOut/gmSwigluOut — SwiGLU 输出 (float)
├── gmGroupList               — 专家分组列表
├── gmExpandIdx               — 展开索引
├── gmEpSendCount             — EP 发送计数
├── gmResvered                — 保留空间 (256KB)
└── gmAllEpRecvCount          — 全局 EP 接收计数

阶段 B: 非深度融合路径 (Shallow Dispatch)

当 EXEC_FLAG & EXEC_FLAG_DEEP_FUSE == 0 时，执行独立的 Dispatch 阶段：

if constexpr ((EXEC_FLAG & EXEC_FLAG_DEEP_FUSE) == 0) {
    if constexpr (g_coreType == AscendC::AIV) {
        // AIV 上执行 Dispatch：量化 + AlltoAll 发送
        CamMoeDistributeDispatch<...> dispatcher;
        dispatcher.Init(gmX_, gmexpertIds_, ...);
        dispatcher.Process();
        icache_preload(8);  // 预取 GMM 指令到 icache
    }
    // AIV 完成 → CrossCoreFlag 通知 AIC
    PipeBarrier<PIPE_ALL>();
    Arch::CrossCoreFlag gmm1AivFinished{0};
    if constexpr (g_coreType == AscendC::AIV) {
        Arch::CrossCoreSetFlag<0x2, PIPE_MTE3>(gmm1AivFinished);
    } else {
        Arch::CrossCoreWaitFlag(gmm1AivFinished);
    }
}

阶段 C: GMM1 + SwiGLU + Quant 融合

GmmDeqSwigluQuant<TemplateMC2TypeFunc, Gmm1L1TileShape, Gmm1L0TileShape, ...>(
    gmm1ProblemShape, groupCount_, gmGroupList, gmX1, layoutX1, ...);

这是第一个 GMM kernel，在 AIC+AIV 上执行，内部完成： - GMM1: INT8 × INT8 → INT32 (GroupedMatmul) - PerTokenDequant: INT32 × weightScale × perTokenScale → float - SwiGLU: x * sigmoid(x) * x_right (half-glu) - BlockQuant: float → INT8 + dequantScale (为 GMM2 准备输入)

阶段 D: GMM2 + Combine 融合

// 先初始化 Combiner（AIV 上）
CamMoeDistributeCombine<...> combiner;
combiner.Init(gmGmm2DepOut, gmexpertIds_, gmExpandIdx, ...);

// 再执行 GMM2 kernel（AIC+AIV），Combiner 作为回调注入
GmmDeq<TemplateMC2TypeFunc, Gmm2L1TileShape, ...>(
    gmm2ProblemShape, ..., &combiner);

GMM2 kernel 的 Epilogue 在完成 L0C→GM 写入后，直接调用 combiner.AllToAllSend() 将结果通过 HCCL Window 发送回源 rank。

4. W4A8 变体 (fused_deep_moe_w4a8) — INT4 量化融合

源文件位置: src/cam/comm_operator/ascend_kernels/fused_deep_moe_w4a8/

4.1 设计定位

W4A8 是面向 W4A8 (Weight-INT4, Activation-INT8) 量化 MoE 模型的专用变体。与 Base 的核心差异：

  1. 权重类型从 INT8 降为 INT4 (AscendC::int4b_t)，权重存储量减半
  2. 新增 Bias 输入 (gmm1_bias, gmm2_bias, share_gmm1_bias, share_gmm2_bias)
  3. 更大的 L1/L0 tile (L1K=1024, L0K=256) — 补偿 INT4 计算密度的降低
  4. 独立的 host tiling.cpp — 更复杂的参数校验和 workspace 计算
  5. 无 SHMEM 依赖 (requires_shmem: false)

4.2 输入签名差异

W4A8 的 kernel 入口比 Base 多 4 个 Bias 输入：

extern "C" __global__ __aicore__ void fused_deep_moe(

    GM_ADDR share_gmm1_bias, GM_ADDR share_gmm2_bias, // ← 新增
    // ... 输出 + workspace + tiling ...
)

对应的 FusedDeepMoe::Init 也扩展了 Bias 地址绑定：

4.4 block_mmad_preload_async_with_callback_w4a8.hpp — 核心优化

这是 W4A8 最独特的文件，实现了 INT4 的 2:1 压缩存储与计算补偿。

4.4.1 INT4 打包比

constexpr int CONSTANT_INT4_PACKING_RATIO = 2;  // 2 个 INT4 打包进 1 个 INT8

// L1/L0 tile 大小计算（注意除以打包比）
static constexpr uint32_t L1A_TILE_SIZE =
    L1TileShape::M * L1TileShape::K * sizeof(ElementA) / CONSTANT_INT4_PACKING_RATIO;
static constexpr uint32_t L1B_TILE_SIZE =
    L1TileShape::N * L1TileShape::K * sizeof(ElementB) / CONSTANT_INT4_PACKING_RATIO;

含义: INT4 权重在内存中以 INT8 存储（2 个 INT4 打包），实际 L1 存储需求减半。这使得 W4A8 可以使用更大的 L1 tile (L1K=1024)。

4.4.3 计算流程

W4A8 的 BlockMmad::operator() 流程与 Base 一致： 1. GM→L1 预取: K 轴分 tile，L1 双缓冲 2. L1→L0 搬运: K 轴进一步切分到 L0 tile 3. TileMmad: 调用硬件 tileMmad 指令完成 INT4×INT4→INT32 乘累加 4. Callback: K 轴最后一轮时触发 callbackBeforeFixpipe / callbackAfterFixpipe（用于触发 Epilogue 的 SwiGLU+Quant）

4.4.4 更大的 Tile Shape

5. FWK 变体 (fused_deep_moe_fwk) — 框架原生范式

源文件位置: src/cam/comm_operator/ascend_kernels/fused_deep_moe_fwk/

5.1 设计定位

FWK 是面向框架原生集成的变体，核心特征：

  1. OPX 框架集成: 通过 lib_ops.h 提供 OPX 兼容的 Op 类
  2. 原生共享专家: sharedExpertNum / sharedExpertRankNum 作为一等公民
  3. 无 SHMEM 依赖: 纯 HCCL Window 通信
  4. 简化模板: 去掉 Scale 类型参数
  5. Permuted Weight: 输入权重名为 gmm1_permuted_weight（预置换布局）
  6. HCCL Context 直连: GetHcclContext<AscendC::HCCL_GROUP_ID_0>()

5.2 输入签名差异

// FWK kernel 入口 — 无 share_gmm1/2_weight，无 x_active_mask

    GM_ADDR x, GM_ADDR expert_ids,
    GM_ADDR gmm1_permuted_weight, GM_ADDR gmm1_permuted_weight_scale,  // ← permuted
    GM_ADDR gmm2_weight, GM_ADDR gmm2_weight_scale,
    GM_ADDR expert_smooth_scales, GM_ADDR expert_scales,               // ← 无 share_smooth
    GM_ADDR output, GM_ADDR expertTokenNums,                           // ← 无 share_output
    GM_ADDR workspace, GM_ADDR tiling)

关键差异: - 无 share_output 输出（共享专家输出直接写回源 rank） - 无 x_active_mask 输入 - 权重名为 permuted_weight（暗示框架侧已做权重置换）

6. 三条执行路径的 cam_feature 演进

6.1 路径 1: 非深度融合 (Shallow Dispatch)

AIV: Dispatch (量化 + AlltoAll 发送) ──────────────────────▶
AIC:  等待 ◀── CrossCoreFlag ── 等待结束 ──▶ GMM1+Swiglu+Quant ──▶ GMM2+Combine

触发条件: EXEC_FLAG & EXEC_FLAG_DEEP_FUSE == 0（即 moeExpertNumPerRank == 1）

特点: - Dispatch 在 AIV 上独立执行 - AIV→AIC 通过 CrossCoreFlag 同步 - 适用于单专家 per rank 场景（通信量小）

6.2 路径 2: 深度融合 (Deep Fuse)

AIV: [Dispatch 量化+发送] ─┐
                           ├─▶ 在 GMM Kernel 内部流水线化
AIC: [GMM1 计算] ──────────┘     [SwiGLU+Quant] ──▶ [GMM2+Combine]

触发条件: EXEC_FLAG & EXEC_FLAG_DEEP_FUSE != 0（即 moeExpertNumPerRank > 1）

特点: - Dispatch 嵌入 GMM kernel 的 GroupedMatmulSliceMPerTokenDequantSwigluQuantMultiStageWorkspace - AIV 在 GMM1 计算前完成量化+发送，AIC 同时计算 - 通过 workspace 流水线（WORKSPACE_STAGES=4）实现计算-通信重叠 - 适用于多专家 per rank 场景（通信量大，需要隐藏延迟）

6.3 路径 3: 共享专家 (Shared Expert)

路由专家 Rank:
  Dispatch → GMM1_route → SwiGLU → GMM2_route → Combine

共享专家 Rank (epRankId < sharedExpertRankNum):
  GMM1_share (全部 token) → SwiGLU → GMM2_share → 直接写回

FWK 的实现:

bool isShareExpert = (epRankId_ < sharedExpertRankNum_);
if (isShareExpert) {
    m_ = maxBs_ * epRankSize_ / sharedExpertRankNum_;  // 处理全部 token
} else {

Base 的实现: 通过 EXEC_FLAG_SHARED_EXPERT 标志，在 workspace 中为共享专家分配独立区域（gmShareX1, gmShareX2 等），与路由专家并行计算。

6.4 三条路径的协同

在 cam_feature 上，三条路径可以组合： - 非深度融合 + 无共享专家: tilingKey bit0=0, bit3=0 - 深度融合 + 无共享专家: tilingKey bit0=1, bit3=0 - 深度融合 + 共享专家: tilingKey bit0=1, bit3=1

  → GM (int8) + dequantScale (float)

8.4 GroupedMatmul Kernel

文件: grouped_matmul_slice_m_per_token_dequant_swiglu_quant_multistage_workspace.h

这是 GMM1 的 kernel 级别实现，核心特征：

  1. Grouped Matmul: 多个专家权重组成一个 group，每个 token 选择对应专家的权重做 GMM
  2. Slice M: M 维按 token 切片，每个 AIV 处理一部分 token
  3. PerTokenDequant: 每个 token 有独立的 dequant scale
  4. SwigluQuant: SwiGLU 激活后立即量化
  5. MultiStageWorkspace: 4 级 workspace 流水线

8.4.1 深度融合变体

GMM1 kernel 内部嵌入了 BlockQuant 类，用于 SwiGLU 输出后的量化：

template <class ArchTag>

8.5 GMM2 的 ResidentA 策略

GMM2 使用 MmadAtlasA2PreloadAsyncWithCallbackResidentA 策略：

using Gmm2DispatchPolicy =
    Gemm::MmadAtlasA2PreloadAsyncWithCallbackResidentA<
        CUSTOM_PRELOAD_STAGES, GMM2_L1A_STAGES, GMM2_L1B_STAGES,
        GMM2_L0A_STAGES, GMM2_L0B_STAGES, CUSTOM_L0C_STAGES,
        CUSTOM_ENABLE_UNIT_FLAG, CUSTOM_ENABLE_SHUFFLE_K>;

ResidentA 含义: 激活矩阵 A (X2) 在 L1 中常驻（4 级缓冲），因为 GMM2 的输入 X2 来自 SwiGLU 的输出（已在 workspace 中），不需要频繁从 GM 读取。权重 B 仍然需要预取。

9. SHMEM 集成机制

9.1 Base 变体的 SHMEM 使用

Base 变体是唯一 requires_shmem: true 的变体，通过 SHMEM 实现跨 rank 的元数据同步。

9.1.1 SHMEM Workspace 分配

    shmemWorkspaceGM_ = (GM_ADDR)(tilingData->disGmmDeqSwigluQuantGmmDeqComInfo.shmemWorkspacePtr);

shmemWorkspacePtr 来自 tiling 数据，指向 SHMEM 分配的窗口空间。

9.1.2 SHMEM Workspace 布局

shmemWorkspaceGM_:
├── gmShareX1 / gmX1          — GMM1 输入 token (int8)
├── gmShareX2 / gmX2          — GMM2 输入 token (int8)
├── gmShareX1Scale / gmX1Scale — GMM1 输入 scale
├── gmShareX2Scale / gmX2Scale — GMM2 输入 scale
├── gmTokenFlag               — token 到达标志
├── gmAllExpertTokenNums      — 全局专家 token 计数
└── gmCombineSend             — Combine 发送缓冲

9.1.3 SHMEM 指针转换

// combine 中的 SHMEM 地址计算
__aicore__ GM_ADDR GetShmemAddrByRankId(GM_ADDR baseAddr, int64_t rankId) {
    if (epRankId_ == rankId) return baseAddr;
    return (GM_ADDR)aclshmem_ptr(baseAddr, rankId);  // SHMEM 指针转换
}

aclshmem_ptr 是 CAM 提供的 SHMEM 指针转换函数，将本地 SHMEM 地址转换为远端 rank 可访问的指针。

9.1.4 双缓冲机制

// dataState 在 {0, 1} 之间交替
winDataSizeOffset_ = dataState * epRankSize * expertPerSizeOnWin * moeExpertNumPerRank;

// 窗口地址 = base + winDataSizeOffset_ + rankId * OPT_RANK_OFFSET
GM_ADDR GetWindowDataAddrByRankId(uint32_t rankId) {
    return baseAddr + winDataSizeOffset_ + rankId * WIN_OPT_RANK_OFFSET;
}

双缓冲作用: 避免上一次迭代的通信数据被当前迭代覆盖。dataState 在每次迭代后翻转。

9.1.5 SHMEM 清理

void ShmemCleanUp() {
    if (coreIdx_ != 0) return;  // 仅 core 0 执行
    // 清零本地 notify 数据
    AscendC::LocalTensor<int32_t> cleanTempLt = cleanUpBuf_.Get<int32_t>();
    AscendC::Duplicate(cleanTempLt, (int32_t)0, moeExpertNum_);
    // DataCopyPad 到远端窗口的 notify 区域
    DataCopyPad(localNotifyDataTensor, cleanTempLt, ...);
}

9.2 W4A8/FWK 的非 SHMEM 通信

W4A8 和 FWK 使用纯 HCCL Window 通信（通过 MC2 框架），不需要 SHMEM：

  • W4A8: 依赖 Mc2InitTiling / Mc2CcTiling 进行 AlltoAll 通信配置
  • FWK: 通过 winContext->localWindowsIn/Exp 直接访问窗口
