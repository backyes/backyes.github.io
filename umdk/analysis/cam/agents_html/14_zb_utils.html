1.2 sync_collectives.h — Detour 原语与 Offset 重载

文件路径： src/cam/comm_operator/ascend_kernels/utils/op_kernel/sync_collectives.h

这是 cam_feature 中变化最大的共享工具文件。SyncCollectives 类是 Ascend C kernel 间同步的核心原语库，管理两套 flag 区域（Memory A 卡内同步 / Memory B 卡间同步）。

1.2.1 新增 InitDetour — 共享内存偏移初始化

__aicore__ inline void InitDetour(int rank, int rankSize, GM_ADDR *shareAddrs,
                                  int64_t shareMemOffset, TBuf<QuePosition::VECCALC> &tBuf)
{
    // ...
    localSyncAddr = (__gm__ int64_t*)(shareAddrs[rank] + shareMemOffset);
    basicSyncAddr = (__gm__ int64_t*)(shareAddrs[rank] + shareMemOffset) + GetBlockIdx() * FLAG_UNIT_INT_NUM;
    blockOuterSyncAddr = (__gm__ int64_t*)(shareAddrs[rank] + shareMemOffset) + segmentCount + GetBlockIdx() * FLAG_UNIT_INT_NUM;
    // ...
}

设计意图： 在 ZB 系列中，共享内存窗口被划分为多个逻辑区域（combine 区域、notify-dispatch 区域等）。shareMemOffset 允许同一算子内的不同 kernel 使用同一块共享内存的不同偏移段，避免 flag 区域冲突。这是 Detour 算子的核心需求——detour 算子需要在同一共享内存窗口内复用同步原语。

1.2.2 新增 WaitSyncFlag 6 参数重载

__aicore__ inline void WaitSyncFlag(int32_t magic, int32_t value, int32_t eventID,
                                     int32_t rank, int64_t flagNum, int64_t offset)
{
    int64_t v = MergeMagicWithValue(magic, value);
    WaitOneRankPartFlag((__gm__ int64_t*)(shareAddrs[rank] + offset) + eventID * FLAG_UNIT_INT_NUM, flagNum, v);
}

相比 5 参数版本，新增 offset 参数，支持在共享内存中指定偏移后等待 flag。

1.2.3 新增 SetOuterFlagDetour / WaitOuterFlagDetour

__aicore__ inline void SetOuterFlagDetour(int32_t magic, int32_t eventID,
    int64_t setRank, int64_t setBlock, int64_t shareMemOffset)
{
    __gm__ int64_t* flagAddr = GetOuterFlagAddrDetour(setRank, setBlock, shareMemOffset);
    int64_t value = MergeMagicWithValue(magic, eventID);
    SetFlag(flagAddr, value);
}

__aicore__ inline void WaitOuterFlagDetour(int32_t magic, int32_t eventID,
    int64_t waitRank, int64_t waitBlock, int64_t shareMemOffset)
{
    int64_t value = MergeMagicWithValue(magic, eventID);
    __gm__ int64_t* flagAddr = GetOuterFlagAddrDetour(waitRank, waitBlock, shareMemOffset);
    WaitOneRankPartFlag(flagAddr, 1, value);
}

1.2.4 新增 GetOuterFlagAddrDetour

__aicore__ inline __gm__ int64_t* GetOuterFlagAddrDetour(int64_t flagRank, int64_t flagBlock, int64_t shareMemOffset)
{
    return (__gm__ int64_t*)(shareAddrs[flagRank] + shareMemOffset) + segmentCount + flagBlock * FLAG_UNIT_INT_NUM;
}

总结： 所有 Detour 变体都在原始地址计算基础上增加了 shareMemOffset，使得同步原语可以在共享内存窗口的任意偏移处工作。这是 ZB 系列算子（moe_combine_*_zb、moe_dispatch_*_zb、moe_notify_dispatch_zb）的核心基础设施。

1.2.5 同步原语内存布局

共享内存窗口 (peerMems[rank])
├── [0, segmentCount * FLAG_UNIT_INT_NUM * 8)     ← Memory A: 卡内同步 flag
│   ├── block 0: [flag0, flag1, flag2, flag3]       (FLAG_UNIT_INT_NUM = 4)
│   ├── block 1: [flag0, flag1, flag2, flag3]
│   └── ...
├── [segmentCount * 4 * 8, ...)                    ← Memory B: 卡间同步 flag
│   ├── block 0: [flag0, flag1, flag2, flag3]
│   └── ...
└── [IPC_CHUNK_FLAG, ...)                          ← Chunk Flag 区域 (IPC 用)
  • segmentCount = blockNum * FLAG_UNIT_INT_NUM
  • FLAG_UNIT_INT_NUM = 4（每个同步单元 4 个 int64_t = 32 Bytes）
  • MAGIC_OFFSET = 32（magic 占高 32 bits，value 占低 32 bits）

1.5 mc2_tiling_utils.h — HCCL_BUFFSIZE 解析

class Mc2TilingUtils {
public:
    static uint64_t GetMaxWindowSize() {
        uint16_t defaultWindowSize = 200;  // 默认 200MB
        if (getenv("HCCL_BUFFSIZE") != nullptr) {
            try {
                std::string envStr(getenv("HCCL_BUFFSIZE"));

1.7 data_copy.h — GM↔GM Ping-Pong 拷贝

文件路径： src/cam/comm_operator/ascend_kernels/utils/op_kernel/data_copy.h

核心函数 CpGM2GMPingPong：

template <typename K, typename U = K>
FORCE_INLINE_AICORE void CpGM2GMPingPong(int64_t dataSizeRemain,
    const GlobalTensor<U> &sendDataInputGt,
    const GlobalTensor<K> &recvDataOutputGT, int op)

设计要点： 1. Ping-Pong 双缓冲：UB 分为两个区域（UB_HEAD_OFFSET 和 UB_MID_OFFSET），MTE2（GM→UB）和 MTE3（UB→GM）流水线化 2. 类型转换支持：当 K != U 时（如 half→float），UB 按 sizeof(U):sizeof(K) 比例分割 3. 原子操作：通过 SetAtomic<K>(op) 支持 ADD/MUL/MAX/MIN 归约 4. 事件同步：使用 MTE3_MTE2、MTE2_MTE3、MTE2_V、V_MTE3 等硬事件

# Patch kernel compile .make files to pass custom include paths to OPC tool.
KERNEL_COMPILE_OPTS=""
if [ -n "${SHMEM_HOME_PATH}" ]; then
    KERNEL_COMPILE_OPTS="-I${SHMEM_HOME_PATH}/shmem/include -I${SHMEM_HOME_PATH}/shmem/src/device"

if [ -n "${CATLASS_HOME_PATH}" ]; then
    KERNEL_COMPILE_OPTS="${KERNEL_COMPILE_OPTS}-I${CATLASS_HOME_PATH}/include"
fi
if [ -n "${KERNEL_COMPILE_OPTS}" ]; then
    find build_out/op_kernel/CMakeFiles -name "build.make" \
        -exec sed -i "s|--compile-options=\"\"|--compile-options=\\\"${KERNEL_COMPILE_OPTS}\\\"|g" {} +

设计意图： npu_op_* 构建系统的 simple_kernel_compile 不读取 custom_compile_options.ini，导致 --compile-options 为空。该 patch 通过 sed 直接修改 .make 文件，将 SHMEM 和 Catlass 的 include 路径注入到编译选项中。

3.2 FlexAlloc — 双索引弹性分配器

文件路径： src/cam/utilities/hfa/src/flex_alloc.h / flex_alloc.cpp

3.2.1 数据结构

class FlexAlloc {
private:
    struct Range { length_t size; };

    // 主索引：按起始偏移排序
    using StartMap = std::map<offset_t, Range>;
    // 辅助索引：按大小排序，值为 StartMap 的迭代器
    using SizeMap = std::multimap<length_t, typename StartMap::iterator>;

    StartMap m_free_ranges_by_offset;   // 按偏移索引的空闲区间
    SizeMap  m_free_ranges_by_size;     // 按大小索引的空闲区间
    size_t m_mem_size { 0 };
    size_t m_total_alloc_bytes { 0 };
    size_t m_total_free_bytes { 0 };
    size_t m_total_alloc_objs { 0 };
};

设计亮点： 双索引结构——StartMap 支持 O(log n) 的邻居合并，SizeMap 支持 O(log n) 的最佳适配查找。SizeMap 存储的是 StartMap 的迭代器，避免数据冗余。

3.2.4 对齐策略

static size_t const OBJ_SIZE_BITS = 6;           // 2^6 = 64 字节对齐
static size_t const MIN_OBJ_SIZE = (1UL << OBJ_SIZE_BITS);  // 最小 64 字节

template <typename T>
T round_up_bits(T num, uint32_t nbits) {
    T shift = ((static_cast<T>(1)) << nbits) - 1;
    return (num + shift) & ~(shift);
}

3.3.3 碎片整理算法（defrag）

输入: 空闲块列表 (AclFreeChunks)
  │
  ├─ 1. 遍历每个空闲块，提取完整物理页
  │   ├─ 对齐的首页 → 加入 free_pages
  │   ├─ 对齐的尾页 → 加入 free_pages
  │   └─ 中间完整页 → 加入 free_pages
  │
  ├─ 2. 重新映射完整页到虚拟内存末尾
  │   for each page in free_pages:
  │     if 完整页: aclrtMapMem 到末尾
  │     else: 作为不完整块保留
  │
  └─ 3. 输出: 一个连续大块 + 若干不完整边块

关键 API： - aclrtMapMem：将物理页映射到指定虚拟地址 - aclrtUnmapMem：解除映射 - aclrtFreePhysical：释放物理内存 - aclrtReleaseMemAddress：释放虚拟地址空间

3.4.2 分配流程

alloc(size)
  │
  ├─ 1. mp_fa->alloc(size, offset, alloc_size)
  │   ├─ 成功 → 返回对象地址
  │   └─ 失败且为 FRAGMENTATION → 进入 defrag 流程
  │
  ├─ 2. handle_defrag(size, offset, alloc_size)
  │   ├─ try_defrag()
  │   │   ├─ 获取所有空闲区间
  │   │   ├─ 调用 mp_avm->defrag() 重新映射物理页
  │   │   └─ 将新区间插回 FlexAlloc
  │   └─ 再次尝试 mp_fa->alloc()
  │
  └─ 3. 记录到 m_alloc_map，返回地址

3.4.3 释放流程

bool HbmFastAlloc::free(void *ptr)
{
    std::lock_guard<std::mutex> lock(m_alloc_map_lock);
    auto iter = m_alloc_map.find(static_cast<uint8_t*>(ptr));
    if (iter == m_alloc_map.end()) return false;

    // 通过 FlexAlloc 释放
    mp_fa->free(iter->second.first, iter->second.second);
    m_alloc_map.erase(iter);
    return true;
}

3.6 HFA 与 Zero-Buffer 的关系

HFA 虽然位于 src/cam/utilities/hfa/ 公共目录，但其设计目标是为 ZB 系列算子提供共享内存窗口的动态分配能力：

  1. ZB 算子的内存需求： ZB 系列（moe_combine_*_zb、moe_dispatch_*_zb、moe_notify_dispatch_zb）需要在 HBM 上预分配大块共享内存窗口，用于卡间 flag 同步和数据交换
  2. HFA 的角色： 提供高效的 HBM 分配/释放/碎片整理，使 ZB 算子能动态管理共享内存窗口
  3. 碎片整理： ZB 算子的内存分配模式（频繁 alloc/free 不同大小块）容易产生碎片，HFA 的 defrag 通过 ACL API 重新映射物理页解决

性能指标（README 描述）： - ~99.7% HBM 利用率（所有对象大小） - ~0.5 µs 分配/释放延迟（超低且稳定）

6.6 架构演进总结

cam_feature 分支代表了 CAM 库从标准 HCCL 通信向SHMEM 共享内存加速的架构转型：

  1. ZB 系列算子： 7 个新的 ZB 算子（全部 requires_shmem=true）使用自定义共享内存窗口替代部分 HCCL 通信，降低延迟
  2. Detour 同步原语： SyncCollectives 新增 Detour 变体，支持在共享内存窗口任意偏移处工作
  3. HFA 内存管理： 全新的 HBM 快速分配器，提供 O(log n) 分配/释放 + 碎片整理能力
  4. pybind 扩展： 新增 6 个通信管理 API + 1 个 ZB Fused MOE Buffer 类
  5. 构建系统： 修复 SHMEM 路径 bug，新增 Catlass 架构宏，支持更灵活的算子选择
