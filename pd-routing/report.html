1. vLLM — NixlConnector (Pull mode) + NixlPushConnector + Lease Renewal

核心思想：Router/Proxy 先将请求只发 1 个 token 做 prefill leg（P 端，do_remote_decode=True），拿到 kv_transfer_params 后再发 decode leg（D 端，do_remote_prefill=True）。KV 在 P→D 之间通过 NIXL 传输，由 NixlConnectorScheduler 协调 scheduler/worker 两侧抽象传输。

1.1 Pull 模式生命周期 设计文档

① Request arrives → Router 选 P：Router/Proxy 挑一个可用 prefill instance（策略由上层决定，vLLM 自身仅暴露 do_remote_decode/do_remote_prefill 两个标志位）。

② P 执行 prefill：跑 full prefill，得到 KV cache blocks，提交 NIXL READ-ready 状态。

③ P 返回 kv_transfer_params：通过 HTTP response 将 P 的 NIXL 侧信道 (host/port/tp_size/engine_id/block 元信息) 给 Proxy。

④ Proxy 选 D 并签发 decode leg：D 端 worker pre-allocates 等量 KV blocks，对 P 发起 RDMA READ。

⑤ D 读取完成后通知 P：P 释放 blocks（lease 到期或 transfer 完成即释放）。

出处：NIXL KV Cache Lease Renewal 设计文档 docs/design/nixl_kv_cache_lease.md

1.2 Push 模式（NixlPushConnector，更晚的内部实现） 设计文档

动机：默认 pull 模式下 D 必须主动拉，两侧握手顺序和 watchdog 逻辑复杂；push 让 P 直接 WRITE 到 D 预注册内存，"发送即完成"。

新增 nixl-push-writer 线程：每个 TP rank 起一个 background writer 线程，仅负责 nixl_wrapper.get_new_notifs / make_prepped_xfer / transfer(WRITE)。

Writer-local matching tables 实现两侧顺序无关的 request_id 匹配：_pending_d_registrations（D 侧发来的 PUSH_REG）与 _push_finished_blocks（P 侧 scheduler stage 的 blocks）。时间对齐用 get_base_request_id() 剥离 cmpl-uuid-index 后的 8-hex 随机 suffix。关键：这是 Proxy 给两腿的同一 X-Request-Id 被 input_processor 随机化后的解同步手段

Wire format：D→P 的 PUSH_REG 通过 NIXL notification 发 PUSH_REG:<msgpack>，字段：request_id, decode_engine_id, decode_host, decode_port, decode_tp_size, local_block_ids (per-group), remote_engine_id。

Transfer 流程：

D allocate → stash → worker build_connector_meta → meta.push_registrations
→ writer enqueue PUSH_REG → NIXL send_notif → P writer 收到 → _handle_push_reg_notif
P prefill finish → meta.push_finished_blocks → writer 匹配 → NIXL WRITE → D
→ D 收到 completion notif → update_connector_output(finished_recving) → stop watchdog

Wake model：三个事件源 + 1ms self-poll：① start_load_kv 握手产生新材料；② get_finished 始终唤醒（让 writer drain HB / completion / late PUSH_REG）；③ handshake 完成回调。

1.3 Lease / Heartbeat 机制 设计文档

设计动机：单一大超时 480s 遇 D crash 保活浪费显存；过短则易误杀排队的 D。解法：lease renewal via heartbeat。

Lease 默认参数：initial lease 30s；heartbeat interval ≈5s（lease_duration//6）；每次续租延长 ≈20s（2/3 lease）；P 校验用 max(old_expiry, now + lease_extension)（never 缩短）。

关键设计：D 端 on_new_request() 从 request 进入 scheduler waiting queue 就开始发 HB，不等被 schedule 执行。所以 prefill 完成时 P blocks 立即进入 "leased" 状态。

HB 基于 request：不是 per-instance，而是 per-request。这样 P 无需知道 "哪个 D 持有我的 blocks"，路由解耦。

Transport：复用 NIXL send_notif/get_new_notifs，后端 IB/RoCE/TCP 自动 fallback。HB 消息格式 HB:<payload>，每个 HB 消息可批量续租同一 P 的所有 in-flight requests。

Proactive handshake：D 发 HB 时若尚未与 P 握手（请求还在排队），就触发 background thread 主动完成握手，加速后续 KV READ。

1.4 vLLM 的 PD 路由抽象层 设计文档

vLLM 自身不实现 Router，而是提供 Connector / LookupBuffer / Pipe 三层抽象：

  • Connector：kv consumer 从 kv producer 拉取 KV caches 的入口。
  • LookupBuffer：SQL 语义的 insert（非阻塞）+ drop_select（阻塞）KV cache 缓冲区。
  • Pipe：单向 FIFO tensor 传输，send_tensor / recv_tensor。

每个进程一个 connector：Scheduler connector（调度 KV transfer ops）+ Worker connectors（执行 KV transfer ops）。

Connector 实现：NixlConnector、LMCacheConnectorV1、MooncakeConnector、MoRIIOConnector、MultiConnector、OffloadingConnector、FlexKVConnectorV1。

出处：docs/features/disagg_prefill.md

2. NVIDIA Dynamo — KV-Aware Router + Planner + KVBM

定位：Dynamo 是编排层，不替代 SGLang/TRT-LLM/vLLM，而是把多 GPU 集群变成协调推理系统。Router 是 Dynamo 的内置组件，不是外部 proxy。

2.1 路由拓扑 GitHub 官方文档

两种入口：

  • Dynamo-native Frontend routing：client → Frontend → Router → workers，无需外部 gateway。
  • Gateway API routing with GAIE：client → Gateway → EPP (Endpoint Picker Plugin) → Frontend sidecar (router-mode direct) → workers。

Router 决策基于 worker load + KV cache overlap（KV-aware routing），消除冗余 prefill 计算。

3. SGLang — PD Disaggregation (Mooncake/NIXL + sgl_router)

核心思想：SGLang 的 PD 分离是 push 模式 — P 端 sender 主动把 KV 推到 D 端 receiver 预分配的地址。Router 是独立的 sgl_router 进程，--pd-disaggregation 模式把请求先送 P 再送 D。

3.1 整体流程 用户文档

启动命令：

# P 端
python -m sglang.launch_server --model-path ... --disaggregation-mode prefill --port 30000
# D 端
python -m sglang.launch_server --model-path ... --disaggregation-mode decode --port 30001
# Router
python -m sglang_router.launch_router --pd-disaggregation \
  --prefill http://127.0.0.1:30000 --decode http://127.0.0.1:30001 --host 0.0.0.0 --port 8000

Router 把请求先送 P，P 完成 prefill 后把 transfer_info 给 Router，Router 再送 D。

3.2 P 端生命周期（prefill.py） 源码

四个阶段：

1. Bootstrap Queue
   a. 为每个 request 初始化 sender
   b. 用 queue 存储 bootstrap 未完成的 requests
   c. poll senders 检查 bootstrap 状态
   d. bootstrap 完成后移到 Waiting Queue

2. Waiting Queue
   a. PrefillAdder pop requests
   b. 跑 forward
   c. 移到 Inflight Queue

3. Inflight Queue
   a. 非阻塞 poll sender
   b. transfer 完成后 return request

Sender 通过 bootstrap_room 与 D 端 receiver 握手，bootstrap_addr 是 P 端的 HTTP bootstrap server 地址。

3.3 D 端生命周期（decode.py） 源码

四个阶段：

1. PreallocQueue
   a. 为每个 request 初始化 receiver
   b. request 先 handshake，有可用 KV 时 pre-allocate
   c. 移到 TransferQueue

2. TransferQueue
   a. poll receiver 检查 transfer 状态
   b. transfer 完成后移到 waiting queue

3. WaitingQueue
   a. 用 queue 中的 requests 构造 PrebuiltExtendBatch
   b. 跳过 prefill forward，只填充 metadata

4. RunningBatch
   a. 把 PrebuiltExtendBatch 合并进 running batch 跑 decoding

关键：D 端 跳过 prefill forward，直接用 P 端传来的 KV 构造 PrebuiltExtendBatch，省掉 prefill 计算。

3.4 Bootstrap Server 与 KV transfer 协调 源码

CommonKVBootstrapServer：只在 P 端启动，HTTP 服务，D 端 receiver 通过它发现 P 端 sender。

CommonKVManager：每个 TP rank 一个，bind ZMQ PULL socket，管理 request_status / socket_cache / failure_records。

CommonKVSender.init(num_kv_indices, aux_index)：设置本地 request 的 index metadata。

CommonKVSender.send(kv_indices, state_indices)：把指定 indices 的 KV cache 推到 D 端。

CommonKVSender.poll()：检查 transfer 状态，返回 KVPoll (Failed/Bootstrapping/WaitingForInput/Transferring/Success)。

DP attention 下的 prefill dp_rank 注册：_register_prefill_dp_rank() 把 P 端的 dp_rank 注册到 bootstrap server，D 端据此选择正确的 P rank。follow_bootstrap_room 模式用 bootstrap_room % dp_size 决定。

3.5 NIXL 后端（nixl/conn.py） 源码

2717 行，核心：

  • _KVXferMemSegment：描述 VRAM/DRAM 内存段。
  • _normalize_kv_mem_kinds / _pack_kv_mem_kinds：支持 VRAM + DRAM 混合 KV 内存。
  • NIXL 错误处理：nixlRemoteDisconnectError / nixlBackendError / nixlCancelledError 统一捕获。
  • Staging buffer 支持：异构 TP 时 P 端 gather KV head slices → contiguous buffer → bulk RDMA → D 端 scatter，2-5x 吞吐提升。

3.7 HiCache — KV Cache 恢复优化 源码

3.8 SGLang Router 的 PD 模式 用户文档

sgl_router --pd-disaggregation --prefill <urls> --decode <urls>

Router 把请求先送 P，P 完成 prefill 后把 transfer_info 给 Router，Router 再送 D。

环境变量：

  • SGLANG_DISAGGREGATION_THREAD_POOL_SIZE：每 TP rank 的 KV transfer 线程数（默认 0.75*cpu_count/8，限制 4-12）。
  • SGLANG_DISAGGREGATION_QUEUE_SIZE：并行 transfer 队列数（默认 4），多 D 请求共享带宽。
  • SGLANG_DISAGGREGATION_BOOTSTRAP_TIMEOUT：bootstrap 超时（默认 300s）。
  • SGLANG_DISAGGREGATION_HEARTBEAT_INTERVAL / MAX_FAILURE：D→P 心跳（默认 5s / 2 次失败标记 offline）。

4.3 P2P KV Cache Sharing 设计文档

问题：多节点部署时，每个节点独立缓存，相同 prefix 在不同节点需重复 prefill。

解法：P2P 把 per-node caches 变成逻辑上的一个 cache。

三组件：

  • Coordinator：一个小型 HTTP 服务（每个 deployment 一个），跟踪 LMCache server 的存活。
  • LMCache server：每个 server 的 P2P controller 定期从 coordinator 获取 peer list，为每个 live peer 开连接。
  • Transfer channel：RDMA 层，每个 server 启动时注册 L1 buffer，peers 可直接 read。

流程：cache miss → 向拥有 prefix 的 peer 请求 lock-and-locate → 拿到远端地址 → RDMA read 到本地 L1 → 服务请求。

特点：one-sided RDMA read，拥有数据的 node 不被中断；peer 自动发现/断开。

7. 深度洞察

1. Push vs Pull 的本质是 "谁拥有调度权"

Pull 模式下 D 主动拉，D 端 scheduler 决定何时传输，与本地 batch 调度天然对齐；但 D 端需要知道 P 的地址和 block 布局，耦合度高。Push 模式下 P 主动推，P 端完成即可发送，D 端只需等待 poll；但 D 端需要预分配内存，且需要处理 "P 推完了但 D 还没 prealloc" 的竞态。vLLM 的 NixlPushConnector 用 writer-local matching tables 解决这个竞态。

2. Lease 机制是 PD 分离的 "垃圾回收"

vLLM 的 lease + heartbeat 设计精妙之处在于：lease 是 per-request 不是 per-instance。P 无需知道哪个 D 持有它的 blocks，D 也无需知道 P 的 block 布局。这实现了 P/D 的完全解耦，Router 可以自由选择 D 而不影响 P 的 block 管理。

3. Bootstrap Server 是 SGLang PD 的 "控制平面"

SGLang 的 bootstrap server 是 P 端的一个 HTTP 服务，D 端 receiver 通过它发现 P 端 sender。这个设计把 "P 在哪里" 的发现逻辑从 Router 解耦出来，Router 只需要知道 P 的 bootstrap server 地址即可。bootstrap_room 是 request 级别的会话标识。

4. HiCache — KV Cache 的 "增量传输"

SGLang 的 HiCache 在 D 端 radix cache 中匹配已有 prefix，只传输 "缺失的 suffix"。这大幅减少 KV 传输量，尤其在多轮对话场景。DecodePrefixMatch + HiCacheRestoreGatedKVReceiver 的组合是工程上的精巧设计。

5. LMCache MP — "KV Cache 即服务"

LMCache 把 KV cache 从 vLLM 进程中完全剥离，变成独立 daemon。这带来：① 进程隔离（cache 问题不 crash 推理引擎）；② 跨 pod 共享 L1 缓存；③ 多级存储 (L1 CPU DRAM + L2 NVMe/remote)；④ P2P 跨节点 RDMA 共享。这是 "KV Cache as a Service" 的范式。
